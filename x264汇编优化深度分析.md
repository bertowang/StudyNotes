# x264 汇编优化深度分析

> 手写汇编 vs C + 编译器：为什么在视频编码领域，手工 SIMD 汇编仍然更快？

---

## 一、汇编是否"一定"比 C 快？

**不一定，但有前提条件：**

| 场景 | 结果 |
|:---|:---|
| ✅ x264 类场景（视频编码：DCT/SAD/SATD/运动估计/像素滤波） | 手工汇编（MMX/SSE2/AVX2/NEON）几乎总是更快（5%~数倍） |
| ❌ 普通业务逻辑、不规则分支、指针别名严重的情况 | 手写汇编可能不如编译器自动向量化、更难维护、易被新 CPU 微架构反超 |

> x264 只把热点函数（profile 确认 >90% 时间）用汇编重写，其余仍是 C。

---

## 二、为什么即使 GCC/Clang 很强，汇编仍然更快？

### 2.1 编译器不能随意假设"你允许 SIMD"

C 版本的 SAD 通常是标量循环：

```c
for (i=0;i<N;i++) sum += a[i]*b[i];
```

即使开启 `-O3 -march=native`，编译器必须证明：
- 无越界
- 无指针别名（`restrict` 不够强）
- 循环次数可向量化
- 浮点/整数语义完全一致

很多编码核函数（跨行、半像素插值、饱和运算）打破这些假设 → 编译器保守地不向量化或生成次优指令。

手工汇编直接使用：`vpmaddwd`、`vpabsd`、`vpshufb` 等专用指令，多通道并行（unroll + interleave）。

### 2.2 编译器无法完美调度流水线与寄存器

x264 汇编是**针对特定微架构手工调度**：
- 指令重排以避免 RAW/WAW 停顿
- 精确控制寄存器压力（避免 spill）
- 加载/计算重叠（prefetch + ALU 并行）
- 利用 CPU 隐性行为（延迟槽等价、port 分配）

现代编译器尝试做调度，但通常受限于抽象 IR，不会为某个 CPU tuning 到寄存器级。

### 2.3 编码算法有大量"非标准"整数操作

x264 常用操作：

| 操作 | C 表达 | 汇编 |
|:---|:---|:---|
| 饱和减法 | `if(a<b) diff=b-a; else diff=a-b;` | `psubusb` |
| 横向绝对值求和 | 循环累加 | `psadbw` |
| 水平加/打包加 | 多条运算+临时变量 | `phadd`/`hadd` |
| 半像素插值（6-tap FIR） | 浮点/整数展开 | SIMD multiply-add |

C 表达会被拆成多条运算、引入临时变量 → 编译器很难还原成单条 SIMD 指令。

### 2.4 调用约定 & ABI 开销控制

| 方面 | 编译器生成 | x264 汇编 |
|:---|:---|:---|
| 寄存器保留 | 遵守 ABI 保存部分寄存器 | 内部全寄存器分配 |
| 栈帧 | 可能有 prologue/epilogue | 不做不必要栈帧 |
| 参数传递 | 可能经栈（尤其旧调用约定） | 常与 C inline stub 配合 |

### 2.5 多年"人类经验"注入

x264 的 asm 来自 FFmpeg/VLC/x264 社区十几年打磨：
- 实际 benchmark 在真实 CPU 上微调
- 同一函数有多套实现：`sad_8x8_sse2.asm`、`sad_8x8_avx2.asm`、`sad_16x16_neon.S`
- 这是经验 + profiling + 微架构知识，远超通用优化 pass

## 三、为什么不用 Intrinsics 代替汇编？

| 方式 | 优点 | 缺点 |
|:---|:---|:---|
| Intrinsics (`_mm_*`) | 可移植、可读 | 编译器仍乱排指令、spill 寄存器 |
| 手写 ASM (`.asm`/`.S`) | 精确控制指令、调度、寄存器 | 难写、平台相关 |

- 对极限性能 + 固定平台：nasm/yasm 汇编仍胜出
- 新项目（如部分 AV1 enc）有时选 intrinsics 换可维护性

## 四、SAD 8×8 实例——C vs 编译器 vs 汇编

### 4.1 什么是 SAD（x264 核心热点）

SAD（Sum of Absolute Differences）用于运动估计，每秒被调用数百万次：

$$SAD = \sum |A[i] - B[i]|$$

特点：纯整数、大量内存访问、可完全向量化、**没有任何分支逻辑**。

### 4.2 C 语言版本（朴素写法）

```c
int sad_8x8_c(const uint8_t *a, const uint8_t *b)
{
    int sum = 0;
    for (int i = 0; i < 64; i++)
        sum += abs(a[i] - b[i]);
    return sum;
}
```

GCC/Clang `-O3 -march=native` 可能尝试 auto-vectorization，但现实中经常 fallback 到 scalar，或生成 sub-optimal SIMD。

### 4.3 x264 的 SSE2 汇编版本（简化）

```nasm
; SAD 8x8 (SSE2)
movdqu   xmm0, [rdi]      ; load A
movdqu   xmm1, [rsi]      ; load B
psubusb  xmm0, xmm1       ; unsigned saturating sub
psubusb  xmm1, xmm0
por      xmm0, xmm1       ; abs diff
psadbw   xmm0, xmm2       ; horizontal sum (64→16)
```

关键点：
- ✅ 一条 `psubusb` 完成 16 个字节的 abs diff
- ✅ 一条 `psadbw` 完成横向求和
- ✅ 没有循环、没有分支
- ✅ 寄存器、流水线完全可控

### 4.4 直观对比（8×8 块）

| 实现 | 指令数 | 处理方式 |
|:---|:---|:---|
| C（标量） | ~200 | 1 byte / 指令 |
| GCC auto-SIMD | ~40–60 | 保守、多次 unpack |
| x264 SSE2 | <10 | 16 bytes / ins |
| x264 AVX2 | ~5 | 32 bytes / ins |

> 性能差距：**2～4 倍很常见**

### 4.5 为什么编译器做不到这一步？

1. **C 语言无法直接表达「饱和减法」** — C 必须写成 `if(a<b) diff=b-a; else diff=a-b;`，编译器不敢替换成 `psubusb`
2. **编译器不敢用 `psadbw`** — 只在 SSE2+，有隐式横向累加行为，编译器很少主动选择
3. **内存访问模式无法完美推断** — 汇编精确 prefetch + 对齐访问；C 代码抽象访问，编译器只能猜测
4. **寄存器分配 vs 手工调度**

| 方面 | 编译器 | 手写 ASM |
|:---|:---|:---|
| 寄存器 | 溢出风险 | 零 spill |
| 指令顺序 | 保守 | 精确流水 |
| 微架构感知 | 弱 | 强 |

## 五、PSADBW 指令深度解析

### 5.1 指令行为（SSE2+）

```nasm
PSADBW mm/xmm/ymm, mm/xmm/ymm
```

对两个 64/128/256-bit 向量：
1. 按字节做 $|A[i] - B[i]|$（无符号饱和 abs diff）
2. 将高/低 64-bit lane 内 8 个字节的绝对差横向相加
3. 结果存入对应 64-bit lane 的低 16-bit，其余位清零

```
输入:  A = [a0..a7 | a8..a15], B = [b0..b7 | b8..b15]
输出:  [Σ|a0-b0..a7-b7|, 0,0,0 | Σ|a8-b8..a15-b15|, 0,0,0]
```

> 一条指令 = 16 次 sub + 16 次 abs + 横向 reduce

### 5.2 为什么编译器不会自动用它？

1. **C 语言没有"横向字节绝对差求和"语义** — `psadbw` 是无符号字节饱和 abs + 横向归约，GCC/LLVM tree-vectorizer 根本没有匹配规则
2. **归约宽度不匹配** — C 循环返回 int(32-bit)，`psadbw` 产出 16-bit word，编译器必须证明不会溢出
3. **多步骤 pattern matching 太脆弱** — 需要编译器识别 5~6 层 IR pattern：load→sub→abs→hsum→accumulate

## 六、VPDPBUSD 指令深度解析

### 6.1 指令行为（VNNI/AVX512VNNI）

```nasm
VPDPBUSD zmm/k, zmm, zmm   ; 或 ymm/xmm
```

操作（每 32-bit lane = 4 字节组）：

$$\text{acc}[i] += (u8\_a[4j..4j+3] \times s8\_b[4j..4j+3])\ \text{横向求和} \to \text{int32}$$

即 INT8 卷积/矩阵乘法的 fused 指令：4 次 mul + 3 次 add → 一条 FMA-like 指令。

### 6.2 为什么编译器几乎不会自动生成它？

1. **C 标准无 "BYTE×BYTE DOT + WIDEN + ACC INTO INT32" 原语** — 编译器看到的是独立 extends + muls + adds，不是 fused 4-byte dot product
2. **符号/无符号混合不匹配** — `vpdpbusd` 强制规定 A=unsigned byte, B=signed byte；C 里两同类型不匹配 VNNI 语义
3. **归约窗口太小 + 依赖链敏感** — 只在恰好 4 字节一组、恰好 int32 累加时有意义
4. **它是新扩展，不是通用 ISA** — VNNI (2017+) 主要用于 DL 推理，编译器不在通用优化 pass 中匹配

## 七、PSADBW vs VPDPBUSD 对比总结

| 维度 | PSADBW | VPDPBUSD |
|:---|:---|:---|
| C 是否有等价语义 | ❌（无符号字节横向 abs 归约） | ❌（u8×s8 4-way dot→i32） |
| 编译器 pattern match | 极难（多步横向 op） | 极难（符号混合 + fused） |
| 出现场景 | 视频编码 SAD/SATD | INT8 CNN / Transformer |
| 编译器主动生成 | 几乎从不 | 几乎从不 |
| 实际使用 | ✅ x264/FFmpeg/dav1d 手写 NASM | 新 AV1 enc 用 intrinsics |

---

## 八、一句话总结

> x264 的汇编并不是"魔法"，它比 C + 好编译器快，是因为：
> - 编译器受语言和 ABI 约束不能做某些事
> - 手工汇编直接映射最优 SIMD + 精确流水线/寄存器调度
> - 编码算法极度规则、可向量化，却超出编译器自动推导能力
>
> **不是汇编天生快，而是在这个特定问题域，人类 + SIMD + 微架构知识 > 通用优化器。**
>
> PSADBW/VPDPBUSD 不是"编译器生成得不好的指令"——而是 **C 抽象机根本没有对应的语义原语**，编译器既无法证明等价，也无法做跨多层 IR 的模式匹配。手写汇编的意义就在于：你告诉 CPU"就用这条指令"，跳过整个语义证明过程。
