# 第一部分：屏幕采集基础原理

> 在动手写代码之前，必须先理解操作系统如何"绘制屏幕"。  
> 无论桌面还是移动平台，屏幕采集本质上都是**从系统合成器那里"截胡"一份画面**——  
> Windows 上是 DWM，macOS 上是 WindowServer，Android 上是 SurfaceFlinger，iOS 上是 RenderServer。

---

## 一、整体采集架构概览

```
┌─────────────────────────────────────────────────────────────────┐
│                        应用层（用户空间）                          │
│  ┌──────────┐   ┌──────────────┐   ┌──────────────────────┐     │
│  │ 权限申请  │──▶│ 采集会话建立  │──▶│ 帧数据接收/处理       │     │
│  │ (系统弹窗)│   │ (系统 API)    │   │ (Bitmap/GL纹理/编码器)│     │
│  └──────────┘   └──────────────┘   └──────────────────────┘     │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                   系统合成层（SurfaceFlinger / RenderServer）      │
│                                                                   │
│  ┌────────────┐   ┌────────────┐   ┌────────────┐   ┌────────┐ │
│  │ App Layer1 │──▶│ App Layer2 │──▶│ 状态栏图层 │──▶│ 合成器  │ │
│  │ (SurfaceA) │   │ (SurfaceB) │   │ (StatusBar)│   │(GPU合成)│ │
│  └────────────┘   └────────────┘   └────────────┘   └────────┘ │
│                                                          │       │
│                                                          ▼       │
│                                              ┌──────────────┐   │
│                                              │ 最终帧缓冲    │   │
│                                              │ (Framebuffer) │   │
│                                              └──────────────┘   │
└─────────────────────────────────────────────────────────────────┘
                              │
                    ┌─────────┴─────────┐
                    ▼                   ▼
          ┌──────────────────┐  ┌──────────────────┐
          │  物理显示器       │  │  虚拟显示/采集通道 │
          │  (DisplayList)   │  │  (Capture Target) │
          └──────────────────┘  └──────────────────┘
```

**核心思想**：屏幕采集不是"重新绘制画面"，而是**让系统合成器把已经合成好的画面复制/共享一份给我们**。

## 二、Android 屏幕合成体系

### 2.1 SurfaceFlinger —— Android 的画面合成守护进程

```
┌─────────────────────────────────────────────────────────────┐
│                    SurfaceFlinger 工作流程                    │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  App1  ──BufferQueue──▶ ┐                                   │
│  App2  ──BufferQueue──▶ ├─▶ [Layer合成] ──▶ HWComposer      │
│  App3  ──BufferQueue──▶ ┘        │                          │
│                                   ▼                          │
│                            ┌──────────────┐                 │
│                            │ 硬件合成 (HWC) │──▶ 物理显示器  │
│                            │ 或 GPU 合成    │                │
│                            └──────────────┘                 │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**关键概念**：

| 概念 | 说明 |
|------|------|
| **Surface** | 应用绘制的目标，本质是一块 GraphicBuffer 的生产端 |
| **BufferQueue** | 生产者-消费者队列，App 是生产者，SurfaceFlinger 是消费者 |
| **Layer** | SurfaceFlinger 内部对应一个 Surface 的合成单元 |
| **HWComposer (HWC)** | 硬件合成器，能直接用显示控制器叠加多个 Layer |
| **VirtualDisplay** | 虚拟显示器，SurfaceFlinger 会把合成结果输出到指定 Surface |

### 2.2 权限模型

Android 从 5.0 (Lollipop, API 21) 开始通过 `MediaProjection` 提供**用户可控的屏幕采集**：

```
┌────────────────────────────────────────────────────┐
│  用户 App                                           │
│    ↓ requestScreenCapture()                        │
│  MediaProjectionManager                             │
│    ↓ createScreenCaptureIntent()                   │
│  系统弹窗（"允许 XX 录制屏幕吗？"）                    │
│    ↓ 用户点击"允许"                                  │
│  onActivityResult() → MediaProjection 对象           │
│    ↓ createVirtualDisplay(surface)                 │
│  SurfaceFlinger 开始向指定 Surface 输出合成结果       │
└────────────────────────────────────────────────────┘
```

> ⚠️ **Android 14 (API 34) 之后**：必须启动前台服务 `foregroundServiceType="mediaProjection"` 才能采集。

## 三、iOS 屏幕合成体系

### 3.1 Core Animation / RenderServer

iOS 的合成体系与 Android 类似，但更加封闭：

```
┌─────────────────────────────────────────────────────────────┐
│                     iOS 合成流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  App (UIKit)                                                 │
│    ↓ CALayer 树 (通过 IPC 提交)                              │
│  backboardd / RenderServer (系统进程)                        │
│    ↓ 合成所有 App 的 layer 树                                │
│  GPU 合成 (Metal)                                            │
│    ↓                                                         │
│  IOSurface (跨进程共享的 GPU 内存对象)                        │
│    ↓                                                         │
│  Display Controller → 屏幕                                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**关键概念**：

| 概念 | 说明 |
|------|------|
| **CALayer** | Core Animation 图层，UIView 的底层实现 |
| **RenderServer** | 系统合成进程（backboardd 的一部分） |
| **IOSurface** | 内核对象，可在多个进程/GPU/CPU 之间零拷贝共享 |
| **CVPixelBuffer** | 对 IOSurface 的封装，供 Video/Camera 框架使用 |
| **ReplayKit** | iOS 9+ 官方屏幕采集/录制/直播框架 |

### 3.2 权限模型

iOS 的屏幕采集有**两种模式**：

```
┌──────────────────────────────────────────────────────────────┐
│  模式一：App 内录屏（RPScreenRecorder）                        │
│  • 只能录制自己 App 的画面                                     │
│  • 系统权限弹窗一次即可                                        │
│                                                               │
│  模式二：全系统录屏（Broadcast Upload Extension）              │
│  • 可录制整个系统画面                                          │
│  • 需要用户从控制中心手动开启                                   │
│  • 采集运行在独立进程（Extension），内存限制 50MB              │
└──────────────────────────────────────────────────────────────┘
```

## 四、Windows 屏幕合成体系

### 4.1 DWM (Desktop Window Manager) —— Windows 的画面合成守护进程

自 Windows Vista 起，桌面合成从"每个 App 直接画到显存"转为**统一由 DWM 合成**。它是 Windows 桌面 GPU 加速渲染的核心：

```
┌─────────────────────────────────────────────────────────────┐
│                     DWM 工作流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  App1 (D3D/GDI) ──DXGI Swapchain──▶ ┐                       │
│  App2 (D3D/GDI) ──DXGI Swapchain──▶ ├─▶ [DWM 合成]           │
│  App3 (D3D/GDI) ──DXGI Swapchain──▶ ┘        │              │
│                                               ▼              │
│                                    ┌────────────────────┐   │
│                                    │ D3D11/12 合成引擎   │   │
│                                    │ + MPO 硬件叠加平面  │──▶ 显示器
│                                    └────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**关键概念**：

| 概念 | 说明 |
|------|------|
| **DWM (dwm.exe)** | 桌面窗口管理器，系统级合成进程，运行在会话 1 |
| **DXGI Swapchain** | App 与 DWM 之间的帧缓冲交换链，本质是 GPU 共享纹理 |
| **Composition Surface** | DWM 内部对每个可见窗口维护的合成表面 |
| **MPO (Multi-Plane Overlay)** | 硬件叠加平面，视频/全屏 App 可直连显示控制器（省电、低延迟） |
| **Windows.Graphics.Capture** | Windows 10 1903+ 官方采集 API，由 DWM 直接提供帧 |
| **DXGI Desktop Duplication** | Windows 8+ 底层桌面复制 API，主要面向远程桌面/录屏软件 |

**Windows 特有能力**：
- **可采集被遮挡/最小化的窗口**：DWM 为每个窗口保留独立的合成缓冲区（"Buffered Presentation"）
- **可跨 App 采集任意窗口**：不受沙箱限制，仅需用户授权
- **多显示器独立采集**：每台显示器一路 Swapchain，可分别抓取

### 4.2 权限模型

Windows 的屏幕采集权限模型是**四大平台中最宽松**的：

```
┌────────────────────────────────────────────────────┐
│  用户 App                                           │
│    ↓ GraphicsCaptureItem 枚举可采窗口/屏幕          │
│  GraphicsCapturePicker (WinRT)                      │
│    ↓ 系统 UI 让用户选择要采集的内容                  │
│  用户选择 → 返回 GraphicsCaptureItem                │
│    ↓ Direct3D11CaptureFramePool.Create()           │
│    ↓ session.StartCapture()                        │
│  DWM 开始向 FramePool 输出 ID3D11Texture2D          │
└────────────────────────────────────────────────────┘
```

**权限特点**：

| 场景 | 权限要求 |
|------|---------|
| **UWP / MSIX 打包应用** | 通过 `GraphicsCapturePicker` 弹窗选择，无需额外声明 |
| **Win32 桌面应用** | 直接调用 API，无系统弹窗（用户对进程有信任） |
| **DXGI Desktop Duplication** | 无系统弹窗，任何进程都可用（远程桌面软件常用） |
| **UAC 提升** | 采集 UAC 弹窗/管理员窗口需要提升到 High IL |
| **DRM 保护内容** | 自动黑帧（受保护视频、部分银行 App） |

> ⚠️ **Windows 11 24H2+** 引入"应用隔离"设置，用户可禁用某些采集能力；企业策略也可通过 GPO 关闭 Desktop Duplication。

## 五、macOS 屏幕合成体系

### 5.1 WindowServer + Skylight —— macOS 的画面合成守护进程

macOS 的合成体系与 iOS 高度同源（都基于 Core Animation），但多了**桌面窗口管理**这一层：

```
┌─────────────────────────────────────────────────────────────┐
│                   macOS 合成流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  App (AppKit / SwiftUI)                                      │
│    ↓ CALayer 树 (通过 Mach IPC 提交)                         │
│  WindowServer (系统进程, Skylight.framework)                 │
│    ↓ 合成所有窗口 + 菜单栏 + Dock                            │
│  GPU 合成 (Metal)                                            │
│    ↓                                                         │
│  IOSurface (跨进程共享的 GPU 内存对象)                        │
│    ↓                                                         │
│  Display Controller → 屏幕                                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**关键概念**：

| 概念 | 说明 |
|------|------|
| **WindowServer** | 系统级合成进程，负责桌面窗口管理 + 事件分发 |
| **Skylight.framework** | WindowServer 内部使用的私有合成引擎（对应 iOS RenderServer） |
| **CGWindow** | 每个窗口在 WindowServer 中对应一个 CGWindow 记录 |
| **IOSurface** | 与 iOS 完全一致的跨进程共享 GPU 缓冲对象 |
| **ScreenCaptureKit** | macOS 12.3+ 官方现代采集 API，直接从 WindowServer 拿帧 |
| **CGDisplayStream** | 传统屏幕采集 API（macOS 10.8+），已被 SCK 逐步替代 |

**macOS 特有能力**：
- **窗口保留缓冲**：同 Windows DWM，可采集被遮挡/后台窗口
- **多空间（Spaces）支持**：可跨虚拟桌面采集
- **系统级录屏（Cmd+Shift+5）**：底层同样使用 ScreenCaptureKit 通道

### 5.2 权限模型

macOS 从 **10.15 (Catalina)** 起对屏幕采集实施严格授权：

```
┌────────────────────────────────────────────────────┐
│  用户 App                                           │
│    ↓ 首次调用 SCShareableContent.current           │
│  系统弹窗（"XX 想要录制此电脑的屏幕"）                │
│    ↓ 用户前往 系统设置 → 隐私与安全性 → 屏幕录制      │
│    ↓ 手动勾选允许，可能需要重启 App                  │
│  再次启动 → SCShareableContent 返回真实内容          │
│    ↓ SCContentFilter + SCStream.startCapture()     │
│  WindowServer 通过 IOSurface 输出到 App             │
└────────────────────────────────────────────────────┘
```

**权限特点**：

| 场景 | 权限要求 |
|------|---------|
| **首次采集** | 必须在 "系统设置 → 屏幕录制" 中手动授权 |
| **授权后** | 可枚举并采集**任意窗口 / 任意屏幕**，无需二次弹窗 |
| **App 沙箱应用** | 需在 Entitlements 声明；Mac App Store 版本审核较严 |
| **命令行工具** | 首次运行会由父终端（Terminal / iTerm）代为申请权限 |
| **DRM 保护内容** | 自动黑帧（Netflix、Apple TV+ 等） |

> ⚠️ **macOS 15 (Sequoia)+** 加强了提醒：授权后仍会每周弹一次通知提醒用户"某 App 正在使用屏幕录制"，长时间直播类应用需在 UX 上告知用户。

## 六、采集的两大关键问题

无论 Android 还是 iOS，屏幕采集都要解决**两个核心问题**：

### 6.1 数据从哪来？

```
路径 A: 系统合成后回读 (Readback)          路径 B: 系统合成时直连 (Direct)
┌─────────────────────┐                    ┌─────────────────────┐
│ GPU 帧缓冲           │                    │ SurfaceFlinger      │
│    ↓ (拷贝下来)      │                    │    ↓ (直接输出到)    │
│ CPU 内存 (Bitmap)    │                    │ 我们提供的 Surface   │
└─────────────────────┘                    └─────────────────────┘
慢，每帧数百 MB/s 拷贝                       快，零拷贝，GPU→GPU
```

### 6.2 数据去哪用？

采集到的画面，最终会流向 **6 大典型用途**。不同的用途，对**数据形态、延迟、吞吐、色彩空间、是否可有损**的要求差异极大。这决定了链路末端应该以什么样的形式承载数据——是 CPU 内存、GL/Metal 纹理、还是 IOSurface/AHardwareBuffer？

#### 6.2.1 六大典型用途总览

```
                    ┌─── 用途 A: 本地显示预览 ───────→ GPU 纹理 → SurfaceView/CAMetalLayer
                    │
                    ├─── 用途 B: 编码后网络推流 ─────→ MediaCodec / VideoToolbox → RTMP/WebRTC
                    │
   采集到的一帧 ────┼─── 用途 C: 本地录制到文件 ────→ MediaMuxer / AVAssetWriter → .mp4
                    │
                    ├─── 用途 D: 图像识别 / AI 推理 ─→ ML Kit / Vision / TFLite / CoreML
                    │
                    ├─── 用途 E: 云端上传原图 ───────→ HTTP/OSS（PNG/JPEG）
                    │
                    └─── 用途 F: 二次合成 / 特效 ────→ GL/Metal Shader → 回写编码器/预览
```

#### 6.2.2 六大用途的关键属性对比

| 用途 | 典型场景 | 数据流向 | 延迟要求 | 吞吐要求 | 是否可有损 | 首选数据形态 |
|------|---------|---------|---------|---------|-----------|-------------|
| **A. 本地显示预览** | 屏幕镜像、投屏预览小窗 | 采集 → 屏幕 | **极低**（< 16ms） | 中（30~60fps） | 否 | GPU 纹理（零拷贝） |
| **B. 编码推流** | 直播、云游戏、远程协助 | 采集 → 编码器 → 网络 | 低（< 100ms） | **极高**（1080p@60） | 是（H.264/265 有损） | 编码器 Input Surface / CVPixelBuffer |
| **C. 本地录制** | 录屏保存 mp4 | 采集 → 编码器 → 文件 | 宽松（秒级也可） | 高 | 是 | 编码器 Input Surface |
| **D. AI 推理 / 识别** | OCR、物体检测、内容审核 | 采集 → CPU/NPU | 中（< 200ms） | 低（1~5fps 足够） | 否（需精确像素） | CPU Bitmap / NPU Tensor |
| **E. 云端上传原图** | 客服截屏、Bug 上报 | 采集 → PNG/JPEG → HTTP | 宽松 | 极低（单帧） | 通常无损 | CPU Bitmap（RGBA/JPEG） |
| **F. 二次合成 / 特效** | 加水印、美颜、马赛克敏感区 | 采集 → GPU Shader → 编码/预览 | 低 | 高 | 视下游 | GPU 纹理（可读写） |

#### 6.2.3 用途 × 数据形态 匹配矩阵

不同用途对数据形态的要求差别很大，选错形态就会引入不必要的拷贝和转换：

| 数据形态 → | **CPU Bitmap** (RGBA_8888) | **GPU 纹理** (Texture2D / OES) | **YUV Buffer** (NV12/I420) | **平台零拷贝句柄** (IOSurface / AHardwareBuffer) |
|-----------|:---:|:---:|:---:|:---:|
| A. 本地预览 | 🟡 需上传 GPU | ✅ **最佳** | 🟡 需 shader 转 RGB | ✅ 最佳（零拷贝） |
| B. 编码推流 | 🔴 CPU→编码器拷贝 | 🟡 需 EGL/CVPixelBuffer 桥接 | ✅ 最佳（编码器原生吃 YUV） | ✅ **最佳** |
| C. 本地录制 | 🔴 同上 | 🟡 同上 | ✅ 最佳 | ✅ **最佳** |
| D. AI 推理 | ✅ **最佳** | 🟡 需 glReadPixels 回读 | 🟡 需转 RGB | 🟡 需 map 到 CPU |
| E. 云端上传原图 | ✅ **最佳** | 🔴 需回读 CPU | 🔴 需转 RGB 再编码 | 🟡 需 map 到 CPU |
| F. 二次合成 | 🔴 CPU 处理慢 | ✅ **最佳** | 🔴 shader 复杂 | ✅ 最佳（GPU 直接采样） |

> 图例：✅ 最佳匹配 ｜ 🟡 可用但有转换开销 ｜ 🔴 严重不匹配，会成为性能瓶颈

#### 6.2.4 各用途的色彩空间与格式要求

| 用途 | 常用颜色空间 | 常用像素格式 | Alpha 通道 | 特殊要求 |
|------|-------------|-------------|-----------|---------|
| A. 本地预览 | sRGB / Display P3 | RGBA_8888 / BGRA | 保留 | 需与显示屏色域匹配 |
| B. 编码推流 | **BT.709 / BT.601** | **NV12 / I420** (YUV) | **丢弃** | 编码器要求 YUV，且色彩转换矩阵需一致 |
| C. 本地录制 | BT.709 | NV12 | 丢弃 | 同上 |
| D. AI 推理 | 线性 RGB / sRGB | RGB_888 / RGBA_8888 | 视模型而定 | 部分模型要求归一化到 [0,1] 或 [-1,1] |
| E. 云端上传原图 | sRGB | RGBA_8888 / JPEG YUV | 保留 | 无损或高质量压缩 |
| F. 二次合成 | 与预览一致 | RGBA_8888 | **必须保留** | 供混合运算 |

> 💡 **关键洞察**：**预览走 RGB，编码走 YUV** —— 这是最容易忽视的分歧。如果链路末端接的是编码器，中间就应尽早转到 YUV（或直接让编码器吃 Surface，由驱动内部完成 RGB→YUV），避免多余的 CPU 转色空间。

#### 6.2.5 性能敏感度对比（1080p@30fps 为基准）

| 用途 | CPU 占用 | GPU 占用 | 内存带宽 | 电量消耗 | 首选优化方向 |
|------|:-------:|:-------:|:-------:|:-------:|-------------|
| A. 本地预览 | 极低 | 中 | 中 | 中 | 保持 GPU 直采链路 |
| B. 编码推流 | 中（网络+编码控制） | 中（RGB→YUV） | 高（≈120MB/s 原始） | **高** | 用硬编 Input Surface，全链路零拷贝 |
| C. 本地录制 | 低 | 中 | 高 | 中 | 同上 |
| D. AI 推理 | **高**（如无 NPU） | 低~中 | 中 | 中~高 | 降频（如 5fps）、下采样、用 NPU |
| E. 云端上传原图 | 中（PNG/JPEG 压缩） | 无 | 低 | 低 | 单帧触发，避免持续采集 |
| F. 二次合成 | 低 | **高** | 高 | 中~高 | 合并 Shader Pass，避免中间纹理往返 |

#### 6.2.6 常见误配陷阱

以下是实践中反复出现的"用途 vs 数据形态"错配问题，每一个都对应显著性能损耗：

| 错配模式 | 表现 | 根因 | 正确做法 |
|---------|------|------|---------|
| 用 CPU Bitmap 做预览 | 掉帧、GPU 空闲、CPU 100% | 走了 glReadPixels 或 PixelCopy | 让采集直接产出到 SurfaceTexture，GPU 直接采样 |
| 用 GPU 纹理做 AI 推理 | 推理前每帧 glReadPixels 卡顿 | GPU→CPU 同步阻塞 | 用 PBO 异步回读，或让模型直接吃 GPU 纹理（Delegate） |
| 用 RGBA 送编码器 | 编码器内部隐式转 YUV，功耗高 | 未用编码器原生格式 | 让编码器直接接 Surface，或提前转 NV12 |
| YUV 数据做特效合成 | Shader 代码复杂、色差 | 未在 GPU 侧统一到 RGB | 采样时用 `samplerExternalOES` 或 YUV→RGB LUT |
| 每帧都 encode PNG 上传 | CPU 长时间高占用、掉帧 | PNG 压缩慢、体积大 | 事件触发单帧，或用 JPEG，或降到 720p |

---

**每种"来源"和"用途"的组合都对应一种最佳实践**——这也是第二部分 10 个演进示例（5 级移动端 + 5 级桌面端）要逐一展开的内容：从"CPU Bitmap 预览"（最简单也最低效）一路演进到"编码器 Input Surface 全链路零拷贝"（最高效但最复杂）。

## 七、屏幕采集 vs 窗口采集 —— 概念与平台差异

> ⚠️ "屏幕采集"和"窗口采集"这两个术语在**桌面平台**和**移动平台**含义完全不同。  
> 本章先讲清桌面语境下的原始定义，再对照移动端的语义偏移，  
> 帮助从桌面转向移动开发的读者建立正确的心智模型，避免选错 API、选错权限模型。

### 7.1 桌面平台（Windows / macOS）的屏幕 vs 窗口采集

在桌面操作系统上，"屏幕采集"和"窗口采集"是**两个真正并列且都可用的选项**，用户可以自由选择：

```
┌──────────────────────────────────────────────────────────────────────┐
│                    屏幕采集 vs 窗口采集（桌面）                         │
├──────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  ┌───────────────────────┐         ┌───────────────────────┐        │
│  │   屏幕采集             │         │   窗口采集             │        │
│  │  (Screen / Display)    │         │  (Window Capture)      │        │
│  ├───────────────────────┤         ├───────────────────────┤        │
│  │ · 采集整块显示器画面    │         │ · 采集任意一个可见窗口 │        │
│  │ · 可选主屏 / 副屏       │         │ · 甚至可采后台/被遮挡  │        │
│  │ · 含任务栏、通知、其他  │         │   的窗口（DWM 保留合成）│       │
│  │   App 的窗口            │         │ · 可跨 App 抓另一个    │        │
│  │ · 分辨率 = 显示器分辨率  │         │   App 的窗口（授权即可）│       │
│  └───────────────────────┘         └───────────────────────┘        │
│                                                                       │
│  ✅ 桌面平台**允许跨 App 窗口采集**：这是与移动端最本质的差异。         │
│                                                                       │
└──────────────────────────────────────────────────────────────────────┘
```

**Windows 平台的主要 API 路径**：

| 采集类型 | 现代 API (Windows 10 1903+) | 传统 / 兼容 API |
|----------|----------------------------|----------------|
| **屏幕采集** | `Windows.Graphics.Capture` (WinRT) + `GraphicsCaptureItem.CreateFromDisplayId` | GDI `BitBlt` from screen DC / DXGI `Desktop Duplication API` |
| **窗口采集** | `Windows.Graphics.Capture` + `GraphicsCaptureItem.CreateFromWindowId` | GDI `PrintWindow` / `BitBlt` from window DC |
| **零拷贝路径** | 直接得到 `IDirect3D11Texture2D`（DXGI 表面）→ D3D11 / D3D12 直接采样 | 无（都要 GPU→CPU 回读） |
| **典型应用** | OBS Studio、Teams 屏幕共享、GeForce Experience | 老版本录屏工具、远程桌面 |

**macOS 平台的主要 API 路径**：

| 采集类型 | 现代 API (macOS 12.3+) | 传统 API |
|----------|-----------------------|---------|
| **屏幕采集** | `ScreenCaptureKit` — `SCContentFilter(display: ...)` | `CGDisplayStream` / `AVCaptureScreenInput` |
| **窗口采集** | `ScreenCaptureKit` — `SCContentFilter(desktopIndependentWindow: ...)` | `CGWindowListCreateImage` |
| **零拷贝路径** | `CMSampleBuffer` 内含 `IOSurface`，可直接桥 `MTLTexture` | 无 |
| **典型应用** | macOS Sonoma 原生屏幕共享、OBS 15+、FaceTime 屏幕分享 | 老版本 QuickTime、旧 OBS |

**桌面平台的关键特性**：

1. **合成器保留了每个窗口的独立缓冲**（Windows DWM / macOS WindowServer），  
 因此**即使窗口被遮挡或最小化到后台，仍能取到最新画面**（有例外，如某些游戏全屏独占模式）。
2. **权限模型相对宽松**：Windows 10 之前几乎无阻碍；macOS 从 10.15 起需要"屏幕录制"权限（系统设置 → 隐私），  
 但**授权后即可枚举并采集任意窗口/屏幕**。
3. **DRM 保护内容会被自动黑屏**（Netflix / Apple TV+ 播放器等），这是操作系统硬约束。
4. **多显示器友好**：可以只采一台屏，或分屏各采一路。

> 💡 桌面平台的"窗口采集"是**真正的"抓某一个 App 的画面"**——这在移动端完全不存在。

### 7.2 移动平台（Android / iOS）的语义偏移

到了移动端，两个概念的含义发生根本变化：

```
┌──────────────────────────────────────────────────────────────────────┐
│                    屏幕采集 vs 窗口采集（移动端）                       │
├──────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  ┌───────────────────────┐         ┌───────────────────────┐        │
│  │   屏幕采集             │         │   窗口采集             │        │
│  │  (Screen Capture)      │         │  (Window Capture)      │        │
│  ├───────────────────────┤         ├───────────────────────┤        │
│  │ · 采集整个设备画面      │         │ · 只采集自己 App 内容   │        │
│  │ · 包含所有 App +        │         │ · 或某个 View / Layer  │        │
│  │   状态栏 + 通知         │         │ · 移动端不能采别的 App │        │
│  │ · 用户需授权（弹窗）    │         │ · 无需系统级权限        │        │
│  │ · 走系统合成器输出       │         │ · 走应用内绘制层        │        │
│  │ · 有前台服务 / 独立      │         │ · 主 App 进程直接完成   │        │
│  │   Extension 约束        │         │                        │        │
│  └───────────────────────┘         └───────────────────────┘        │
│                                                                       │
│  ⚠️ 移动端**没有跨 App 的"窗口采集"**：iOS/Android 都不允许一个 App     │
│     采集另一个 App 的画面（沙箱隔离）。这一点与桌面完全不同。            │
│                                                                       │
└──────────────────────────────────────────────────────────────────────┘
```

**为什么移动端不允许跨 App 窗口采集？**

- **沙箱隔离**：每个 App 都在独立进程 + 独立 SELinux/Seatbelt 沙箱内，绘制缓冲区被内核隔离
- **隐私合规**：iOS App Store 与 Google Play 明确禁止未授权采集他 App 内容
- **合成器优化**：SurfaceFlinger / RenderServer 为省电做了大量激进优化，非当前前台的 App 缓冲区经常被回收，即使内核允许也无画面可取

### 7.3 移动端两种场景的技术路径对照

| 维度 | 屏幕采集（Screen） | 窗口采集（Window / In-App） |
|------|-------------------|---------------------------|
| **典型场景** | 直播推流、系统录屏、远程协助 | 教程截图、App 内录屏、水印分享 |
| **Android 主 API** | `MediaProjection` + `VirtualDisplay` | `PixelCopy` / `View.draw(Canvas)` / `SurfaceView` 自采 |
| **iOS 主 API** | `RPBroadcastSampleHandler`（Extension） | `RPScreenRecorder`（App 内）/ `UIGraphicsImageRenderer` |
| **权限** | 系统弹窗必需 | 无系统权限（或仅 Photos） |
| **数据源** | SurfaceFlinger / RenderServer 合成结果 | 本 App 的 CALayer / SurfaceView |
| **是否能录他 App** | ✅ 可以（用户授权） | ❌ 绝对不能 |
| **进程模型** | Android 前台服务 / iOS 独立 Extension | 主 App 进程内完成 |
| **常见坑** | 权限被拒 / 前台服务被杀 / 50MB OOM | GPU→CPU 回读慢 / 只能录自己 |
| **本文档对应示例** | 02、03、04、05 | 01（PixelCopy / UIGraphicsRenderer） |

### 7.5 混合场景：App 内录屏但要"看起来像屏幕录制"

有时业务需要"录制 App 内主要 View + 悬浮水印/摄像头小窗"这类合成，属于**窗口采集 + 应用内合成**：

- **Android**：应用侧创建离屏 `SurfaceTexture` + OpenGL ES / Vulkan 合成 → 送 MediaCodec InputSurface
- **iOS**：`UIViewController.view.layer` 到 `CGContext` 或直接用 `Metal` 离屏渲染 → 送 VTCompressionSession

这种场景**不需要 MediaProjection / Broadcast Extension**，但需要"渲染 API"（下一节）参与合成。

### 7.6 桌面 vs 移动 —— 一张表看清所有差异

| 维度 | 桌面（Windows / macOS） | 移动（Android / iOS） |
|------|------------------------|---------------------|
| **屏幕采集** | ✅ 全屏 + 多显示器可选 | ✅ 整台设备（单屏） |
| **窗口采集（他人 App）** | ✅ 授权后可采任意窗口 | ❌ 完全禁止（沙箱） |
| **窗口采集（自己 App）** | ✅ 通过 HWND / NSWindow 抓自己 | ✅ 抓自己的 View / Activity |
| **后台/遮挡窗口采集** | ✅ DWM/WindowServer 保留缓冲 | ⚠️ 一般只能采前台可见 |
| **权限模型** | Win：宽松；Mac：一次授权后放开 | 每次采集都要弹窗授权（屏幕采集） |
| **进程隔离要求** | 无（主进程即可） | iOS 强制 Extension；Android 强制前台服务 |
| **零拷贝载体** | D3D11 Texture / IOSurface | AHardwareBuffer / IOSurface |
| **DRM 保护** | 自动黑屏 | 自动黑屏 |
| **典型应用** | OBS、Teams、远程桌面 | 直播 App、系统录屏、App 内教程 |
| **本文档深度** | 本章系统对照 + 关键 API 索引 | ★ 后续所有章节深入 5 级演进 |

> 💡 **核心结论**：桌面上"窗口采集"是**真正的跨 App 抓画面**；  
> 移动端上"窗口采集"退化成**"抓自己的 View"**——名字相同，语义完全不同。  
> 
> 📌 **文档定位**：后续 5 级演进（示例 01~05）以移动端为主线深入讲解；  
> **桌面平台的核心概念（合成器、零拷贝、编码器直连）与移动端高度同构**——  
> 例如 Windows 的 `Windows.Graphics.Capture → D3D11 Texture → Media Foundation` 对应 Android 的  
> `MediaProjection → SurfaceTexture → MediaCodec`，macOS 的 `ScreenCaptureKit → IOSurface → VideoToolbox`  
> 与 iOS 几乎完全一致。**读者可将移动端的优化思路直接迁移到桌面端**。

## 八、渲染 API 的选择 —— OpenGL ES / Vulkan / Metal 在采集链路中的角色

> 屏幕采集本身**不"渲染"** —— 画面是系统合成器已经画好的。  
> 但**采集到之后**（预览 / 滤镜 / 二次合成 / 编码前处理），必须挂到某个 GPU API 上。  
> 这一节澄清：**在什么时候需要考虑渲染 API？如何选？**

### 8.1 渲染 API 在采集链路中的定位

```
┌───────────────────────────────────────────────────────────────────────┐
│              采集链路的三个阶段：谁需要渲染 API？                        │
├───────────────────────────────────────────────────────────────────────┤
│                                                                        │
│  阶段 A: 系统合成（不需要渲染 API）                                     │
│  ─────────────────────────────                                        │
│  SurfaceFlinger / RenderServer 内部已用硬件合成完成，                    │
│  开发者拿不到、也不需要 GPU API。                                       │
│                                                                        │
│           ↓ 输出到采集通道                                              │
│                                                                        │
│  阶段 B: 采集通道（GPU API 影响"零拷贝能力"）                           │
│  ─────────────────────────────────────                              │
│  · 想让采集帧留在 GPU 显存里 → 必须选一个 GPU API 承载它               │
│  · Android: OpenGL ES (SurfaceTexture) 或 Vulkan (AHardwareBuffer)   │
│  · iOS:     Metal (CVMetalTextureCache)                              │
│                                                                        │
│           ↓ 需要滤镜/水印/缩放？                                        │
│                                                                        │
│  阶段 C: 二次处理（渲染 API 全面参与）                                  │
│  ────────────────────────────────                                    │
│  · 美颜、水印、马赛克、缩放、色彩空间转换等                              │
│  · 必须写 Shader，此时渲染 API 的选择决定开发效率与性能                  │
│                                                                        │
└───────────────────────────────────────────────────────────────────────┘
```

### 8.2 三大 GPU API 在采集场景的对比

| 维度 | OpenGL ES | Vulkan | Metal |
|------|-----------|--------|-------|
| **Android 支持** | ✅ 全平台（API 8+） | ✅ Android 7+ | ❌ 不支持 |
| **iOS 支持** | ⚠️ 已弃用（iOS 12+） | ❌ 不支持 | ✅ 官方唯一推荐 |
| **屏幕采集集成** | `GL_TEXTURE_EXTERNAL_OES` 天然支持 | 需 `AHardwareBuffer` + 扩展 | `CVMetalTextureCache` 天然支持 |
| **零拷贝** | ✅ SurfaceTexture 路径 | ✅ 但代码量大 3~5× | ✅ IOSurface 天然共享 |
| **多线程录屏 + 编码** | 单 Context 受限 | 多队列，可分线程 | 天生多线程 |
| **上手成本** | 低（生态成熟） | 高（初始化数百行） | 中 |
| **典型采集场景选型** | 90% Android 直播 App | 高帧率 / 4K / 云游戏 | 100% iOS |

### 8.3 场景决策矩阵

```
┌──────────────────────────────────────────────────────────────────────┐
│           "我做什么样的采集应用" → "选什么渲染 API"                     │
├──────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  场景 1: 只截图，不做实时预览/编码                                      │
│    → 不需要渲染 API（本文示例 01 即可）                                 │
│                                                                       │
│  场景 2: 直播推流，无滤镜                                              │
│    Android → OpenGL ES（SurfaceTexture）或直接编码器直连（示例 05）    │
│    iOS     → Metal（如需处理）或直接 VTCompression（示例 05）          │
│    Windows → D3D11（Windows.Graphics.Capture → MediaFoundation）      │
│    macOS   → Metal（ScreenCaptureKit → VideoToolbox）                 │
│                                                                       │
│  场景 3: 直播 + 美颜/水印                                              │
│    Android → OpenGL ES（生态成熟，滤镜库多）                           │
│    iOS     → Metal（唯一选择）                                         │
│    Windows → D3D11 / D3D12（HLSL Compute Shader 做后处理）            │
│    macOS   → Metal（同 iOS，代码可复用）                               │
│                                                                       │
│  场景 4: 云游戏 / 4K@60fps 高帧率采集                                  │
│    Android → Vulkan（多队列压榨硬件）                                  │
│    iOS     → Metal（Apple GPU TBDR 天然低带宽）                        │
│    Windows → D3D12 / Vulkan（NVENC / AMF 硬件编码器直连）             │
│    macOS   → Metal（VTVideoEncoderSpecification 硬编）                │
│                                                                       │
│  场景 5: 跨端一套代码（Flutter / RN / Electron 类）                     │
│    通用层 → OpenGL ES / ANGLE（Windows 用 ANGLE 后端最容易 shim）      │
│    Apple 层 → Metal（iOS/macOS 12+ 建议逐步迁移）                      │
│    Windows → 单独一份 D3D11 后端                                       │
│                                                                       │
└──────────────────────────────────────────────────────────────────────┘
```

> 💡 **移动 vs 桌面的 GPU API 对应关系**：  
> - Android OpenGL ES ≈ Windows D3D11（生态成熟、上手快）  
> - Android Vulkan ≈ Windows D3D12 / Vulkan（榨性能、多队列）  
> - iOS Metal ≈ macOS Metal（代码几乎完全可复用）  
> 这也是很多跨端引擎（Unity / Unreal / FFmpeg）的技术选型逻辑。

### 9.4 平台对应关系速查

**同级示例解决同一类瓶颈，可直接横向对照**：

> 📌 **编号约定**：本节使用 **S1~S5（Stage 1~5）** 表示“**演进阶段**”，与第 10~11 章中表示“**数据流路径分层**”的 L1/L2/L3（详见 [10.1.3](#1013-数据流路径的三级分层l1l2l3-术语定义)）**属于两套不同的记号体系**：
> - **S1~S5**：以“**API 新旧与能力升级**”为维度的 5 个演进阶段，一对一对应 A1→A5 / B1→B5
> - **L1/L2/L3**：以“**数据到底走不走 CPU**”为维度的 3 条物理路径，多个 S 阶段可能属于同一条 L 路径

| 等级 | 移动示例 | 桌面示例 | 共同瓶颈突破点 | 归属路径 |
|------|---------|---------|--------------|:--------:|
| **S1 基础** | A1 Bitmap Readback | B1 BitBlt / CGWindowListCreateImage | 引入平台“最简”截图 API | L1 |
| **S2 流式** | A2 VirtualDisplay + ImageReader | B2 DXGI Desktop Duplication | 从“拉截图”变成“推流” | L1→L2 |
| **S3 GPU 纹理** | A3 SurfaceTexture (OES) | B3 Windows.Graphics.Capture (D3D11) | 数据留在 GPU，不再回读 CPU | L2 |
| **S4 零拷贝** | A4 ReplayKit + IOSurface | B4 ScreenCaptureKit + IOSurface | 内核共享缓冲，跨进程/API 零拷贝 | L2 |
| **S5 编码直连** | A5 MediaCodec / VTCompression | B5 Media Foundation / VTCompression | 采集缓冲 = 编码器输入 | L3 |

> 💡 **关键洞察**：Apple 生态（iOS / macOS）在 S4/S5 上**代码几乎完全可复用**（都基于 IOSurface + VideoToolbox）；Windows 与 Android 则是“思想同构、API 不同”。

### 9.5 A 组：移动端 5 级演进

> 示例 01~05 覆盖 Android 与 iOS 两大移动平台，  
> 从 CPU 回读一路演进到全链路零拷贝的硬编直连。

### 9.5.1 移动示例 01: 基础位图截图（Bitmap Readback）

#### 核心思想

**一句话总结**：让 GPU 合成好的画面"下载"回 CPU 内存，得到一个 Bitmap（Android）或 UIImage（iOS）。

#### Android 实现（PixelCopy API）

```java
// API 24+ 推荐方式
PixelCopy.request(
    surfaceView,       // 源 SurfaceView
    bitmap,            // 目标 Bitmap（提前分配好）
    (int copyResult) -> {
        if (copyResult == PixelCopy.SUCCESS) {
            // bitmap 中已经填充好像素
            processBitmap(bitmap);
        }
    },
    new Handler(Looper.getMainLooper())
);
```

#### iOS 实现（UIGraphicsImageRenderer）

```swift
// 只能截自己 App 的画面
let renderer = UIGraphicsImageRenderer(size: view.bounds.size)
let image = renderer.image { ctx in
    view.drawHierarchy(in: view.bounds, afterScreenUpdates: true)
}
```

#### 底层发生了什么

```
┌─────────────────────────────────────────────────────────────┐
│                    Bitmap Readback 路径                       │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  GPU 显存 (RGBA)                                             │
│       │                                                      │
│       │ glReadPixels / vkCmdCopyImageToBuffer                │
│       │ (同步阻塞，等待 GPU 完成所有绘制)                     │
│       ▼                                                      │
│  Pinned 内存 (驱动分配)                                      │
│       │                                                      │
│       │ memcpy                                              │
│       ▼                                                      │
│  Java 堆 (Bitmap.mBuffer) / Swift 对象                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### 性能瓶颈

⚠️ **三重开销**：

1. **GPU 管线停顿**：`glReadPixels` 会强制 GPU 完成所有前置命令，破坏流水线
2. **PCIe/总线传输**：1080p RGBA = 8.3MB/帧，30fps 需要 250MB/s
3. **CPU 内存分配**：每次都新建 Bitmap 对象（Java GC 压力）

#### 局限性

- 只能截**自己 App 的画面**，无法录整个系统
- 帧率上限约 10~15fps
- 大量 GC 抖动

**这就是为什么专业录屏软件不用这种方案。**

### 9.5.2 移动示例 02: VirtualDisplay + ImageReader (Android)

#### 核心思想

**一句话总结**：让 SurfaceFlinger 直接把整个系统画面输出到一个虚拟屏幕，再从虚拟屏幕的 Surface 上读取像素。

#### 与 01 的关键区别

```
01 的做法（拉模式）:                    02 的做法（推模式）:
┌─────────────────────┐                ┌─────────────────────┐
│ App 主动调用 API     │                │ 建立虚拟显示通道后   │
│ → 触发一次 GPU 回读   │                │ 系统持续推送每一帧   │
│ 需要什么时候截，就截  │                │ 帧率与屏幕刷新同步   │
│ (低频，一次性)       │                │ (高频，流式)         │
└─────────────────────┘                └─────────────────────┘
```

#### 关键代码

##### 1. 申请权限

```java
MediaProjectionManager pm = (MediaProjectionManager) 
    getSystemService(Context.MEDIA_PROJECTION_SERVICE);
Intent intent = pm.createScreenCaptureIntent();
startActivityForResult(intent, REQUEST_CODE);

// onActivityResult 拿到 MediaProjection
mediaProjection = pm.getMediaProjection(resultCode, data);
```

##### 2. 创建 ImageReader

```java
ImageReader imageReader = ImageReader.newInstance(
    width, height,
    PixelFormat.RGBA_8888,
    2  // maxImages: 缓冲队列深度
);

imageReader.setOnImageAvailableListener(reader -> {
    Image image = reader.acquireLatestImage();  // ★ 取最新的，丢弃旧的
    if (image != null) {
        Image.Plane plane = image.getPlanes()[0];
        ByteBuffer buffer = plane.getBuffer();
        int pixelStride = plane.getPixelStride();
        int rowStride = plane.getRowStride();
        // 处理像素...
        image.close();  // ★ 必须 close，否则会阻塞后续帧
    }
}, backgroundHandler);
```

##### 3. 创建 VirtualDisplay

```java
VirtualDisplay display = mediaProjection.createVirtualDisplay(
    "ScreenCapture",
    width, height, densityDpi,
    DisplayManager.VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR,
    imageReader.getSurface(),  // ★ 把 ImageReader 的 Surface 接给虚拟显示
    null, null
);
```

#### 底层数据流

```
┌────────────────────────────────────────────────────────────────┐
│                  VirtualDisplay 数据流                           │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  SurfaceFlinger                                                 │
│       │                                                         │
│       │ 每次 VSync 合成完毕                                     │
│       ▼                                                         │
│  ImageReader 的 Surface (BufferQueue 生产端)                    │
│       │                                                         │
│       │ dequeueBuffer / queueBuffer                            │
│       ▼                                                         │
│  BufferQueue                                                    │
│       │                                                         │
│       │ acquireBuffer                                          │
│       ▼                                                         │
│  ImageReader (消费端) ──▶ onImageAvailable 回调                 │
│                            │                                    │
│                            ▼                                    │
│                       Image.Plane.getBuffer()                   │
│                       (DirectByteBuffer，映射自 GraphicBuffer)   │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

#### 关键优化点：`acquireLatestImage`

```java
// ❌ 错误用法：acquireNextImage，如果 CPU 处理慢会累积延迟
Image img = reader.acquireNextImage();

// ✅ 正确用法：acquireLatestImage，跳过所有旧帧，只处理最新的
Image img = reader.acquireLatestImage();
```

**为什么？**

```
生产速度 > 消费速度时：
  队列深度 = 2 (maxImages)
  帧号:   [1][2][3][4][5][6][7]...
                        ↑
              CPU 只能处理到这里
  
  acquireNextImage: 拿到旧的帧2，画面延迟严重
  acquireLatestImage: 直接拿最新的帧7，前面的自动丢弃
```

#### rowStride 坑点

```java
// ImageReader 返回的 buffer 可能有行对齐 padding
int rowStride = plane.getRowStride();  // 可能 > width * 4
int pixelStride = plane.getPixelStride();  // 通常 = 4

// 逐行拷贝，跳过 padding
for (int y = 0; y < height; y++) {
    buffer.position(y * rowStride);
    buffer.get(dst, y * width * 4, width * 4);
}
```

（这与 OpenGL 中 `GL_UNPACK_ROW_LENGTH` 的问题完全一样。）

#### 性能提升原理

1. **推模式减少空转**：只在有新帧时才回调，不再是主动轮询
2. **系统级 GraphicBuffer**：底层是 GPU 可访问的共享内存（gralloc），比 Java Bitmap 高效
3. **DirectByteBuffer**：数据不经过 Java 堆，避免 GC

#### 局限性

- 数据仍然要**回到 CPU**才能处理，无法送给 GPU 直接渲染
- 每帧仍有 GPU→系统内存的拷贝（约 8.3MB@1080p）
- 如果目标是编码或 GL 处理，这一步拷贝完全是浪费

### 9.5.3 移动示例 03: SurfaceTexture + GPU 直采 (Android)

#### 核心思想

**一句话总结**：让屏幕内容直接输出到一个 **OpenGL 纹理**，全程不下载到 CPU。

#### 与 02 的根本区别

```
02 的数据路径:                          03 的数据路径:
┌──────────┐                           ┌──────────┐
│ 系统合成  │                           │ 系统合成  │
└────┬─────┘                           └────┬─────┘
     │ (GPU→CPU 拷贝)                        │ (GPU 内直接共享)
     ▼                                       ▼
┌──────────┐                           ┌──────────┐
│ CPU 内存  │                           │ GL 纹理   │
│ ImageReader                           │ Surface  │
└────┬─────┘                           │ Texture  │
     │                                 └────┬─────┘
     ▼                                       │
[需要再上传回GPU]                       [已经在GPU]
```

#### SurfaceTexture 是什么？

`SurfaceTexture` 是 Android 的**"GPU 纹理版本的 ImageReader"**：
- 它有一个 Surface（作为 SurfaceFlinger 的输出目标）
- 它内部关联一个 **OpenGL 外部纹理**（`GL_TEXTURE_EXTERNAL_OES`）
- 当新帧到达时，通过 `updateTexImage()` 把系统 GraphicBuffer 绑定到那个 GL 纹理上

#### 关键代码

##### 1. 创建 GL 环境并生成外部纹理

```java
int[] tex = new int[1];
GLES20.glGenTextures(1, tex, 0);
int textureId = tex[0];

// ★ 注意目标是 GL_TEXTURE_EXTERNAL_OES，不是 GL_TEXTURE_2D
GLES20.glBindTexture(GLES11Ext.GL_TEXTURE_EXTERNAL_OES, textureId);
GLES20.glTexParameterf(GLES11Ext.GL_TEXTURE_EXTERNAL_OES,
    GLES20.GL_TEXTURE_MIN_FILTER, GLES20.GL_LINEAR);
GLES20.glTexParameterf(GLES11Ext.GL_TEXTURE_EXTERNAL_OES,
    GLES20.GL_TEXTURE_MAG_FILTER, GLES20.GL_LINEAR);
```

##### 2. 建立 SurfaceTexture → VirtualDisplay 通路

```java
SurfaceTexture surfaceTexture = new SurfaceTexture(textureId);
surfaceTexture.setDefaultBufferSize(width, height);

Surface surface = new Surface(surfaceTexture);

VirtualDisplay display = mediaProjection.createVirtualDisplay(
    "ScreenCapture", width, height, dpi,
    DisplayManager.VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR,
    surface, null, null
);

surfaceTexture.setOnFrameAvailableListener(st -> {
    // 通知 GL 线程有新帧
    glHandler.post(() -> renderFrame());
});
```

##### 3. GL 线程中拉取并渲染

```java
void renderFrame() {
    surfaceTexture.updateTexImage();  // ★ 关键：把最新帧绑定到 GL 纹理
    
    float[] mtx = new float[16];
    surfaceTexture.getTransformMatrix(mtx);  // 处理方向/翻转矩阵
    
    // 现在可以用普通的 shader 采样这个纹理了
    GLES20.glUseProgram(program);
    GLES20.glActiveTexture(GLES20.GL_TEXTURE0);
    GLES20.glBindTexture(GLES11Ext.GL_TEXTURE_EXTERNAL_OES, textureId);
    GLES20.glUniformMatrix4fv(uMtxLoc, 1, false, mtx, 0);
    GLES20.glDrawArrays(GLES20.GL_TRIANGLE_STRIP, 0, 4);
}
```

##### 4. 特殊的 Fragment Shader

```glsl
#extension GL_OES_EGL_image_external : require    // ★ 必须的扩展

precision mediump float;
varying vec2 vTexCoord;
uniform samplerExternalOES sTexture;              // ★ 特殊采样器类型

void main() {
    gl_FragColor = texture2D(sTexture, vTexCoord);
}
```

#### `GL_TEXTURE_EXTERNAL_OES` 底层原理

```
┌─────────────────────────────────────────────────────────────┐
│              外部纹理 vs 普通纹理                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  GL_TEXTURE_2D (普通纹理):                                   │
│  ┌──────────────┐   glTexImage2D    ┌──────────────┐        │
│  │ CPU 内存像素  │ ──────────────▶  │ GPU 纹理内存  │        │
│  └──────────────┘                   └──────────────┘        │
│                                                              │
│  GL_TEXTURE_EXTERNAL_OES (外部纹理):                         │
│  ┌──────────────┐                   ┌──────────────┐        │
│  │ GraphicBuffer│ ─── 直接引用 ───▶ │ 纹理句柄     │        │
│  │ (系统共享内存)│    (零拷贝!)      │ 指向同一块内存│        │
│  └──────────────┘                   └──────────────┘        │
│         │                                                    │
│         │ 由 SurfaceFlinger 直接写入                          │
│         │ (GPU 硬件路径)                                     │
│                                                              │
│  • 可能是 YUV 格式（相机常见）                                │
│  • 采样时驱动自动做 YUV→RGB 转换                              │
│  • 可能需要 transform 矩阵校正方向                            │
└─────────────────────────────────────────────────────────────┘
```

#### 性能提升原理

| 优化维度 | 效果 |
|---------|------|
| 零 CPU 拷贝 | 数据全程在 GPU 内存，8.3MB/帧的拷贝完全消除 |
| GPU 直接可用 | 拿到就是纹理，可以直接做滤镜、缩放、编码 |
| 内存低占用 | 无需 Bitmap/ByteBuffer，也就没有 GC 压力 |
| 延迟极低 | SurfaceFlinger → SurfaceTexture 约 1 帧 |

#### 应用场景

- 实时预览 + 滤镜（美颜、水印）
- 屏幕录制 + 实时编码
- 云游戏客户端画面回传

### 9.5.4 移动示例 04: ReplayKit + IOSurface 零拷贝 (iOS)

#### 核心思想

**一句话总结**：iOS 的 CMSampleBuffer 内部就是 IOSurface，天生就是跨进程零拷贝的 GPU 内存对象。

#### iOS 全系统录屏的架构

```
┌──────────────────────────────────────────────────────────────┐
│                  Broadcast Upload Extension 架构                │
├──────────────────────────────────────────────────────────────┤
│                                                                │
│  主 App 进程                                                   │
│    ↕ App Group / XPC                                          │
│  Broadcast Extension 进程 (独立进程!)                          │
│    ↕                                                          │
│  ReplayKit 系统服务                                            │
│    ↕                                                          │
│  RenderServer (合成整个系统画面)                                │
│                                                                │
└──────────────────────────────────────────────────────────────┘
```

**关键约束**：Extension 内存上限 **50MB**，稍不注意就 OOM 崩溃。所以 iOS 上"零拷贝"不是优化，而是**生存必需**。

#### 关键代码

```swift
import ReplayKit

class SampleHandler: RPBroadcastSampleHandler {
    
    override func processSampleBuffer(_ sampleBuffer: CMSampleBuffer,
                                       with sampleBufferType: RPSampleBufferType) {
        guard sampleBufferType == .video else { return }
        
        // ★ CMSampleBuffer 内部是 CVPixelBuffer，进一步就是 IOSurface
        guard let pixelBuffer = CMSampleBufferGetImageBuffer(sampleBuffer) else {
            return
        }
        
        // ★ 拿到 IOSurface 引用，可以跨进程/跨 GPU 上下文共享
        let ioSurface = CVPixelBufferGetIOSurface(pixelBuffer)?.takeUnretainedValue()
        
        // 方案 A: 直接送编码器（VideoToolbox）
        encodeAndPush(sampleBuffer)
        
        // 方案 B: 送 Metal 渲染管线
        renderWithMetal(pixelBuffer)
    }
}
```

#### IOSurface 的独特能力

```
┌─────────────────────────────────────────────────────────────┐
│                     IOSurface 的多重身份                       │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│              ┌─────────────────┐                             │
│              │   IOSurface     │                             │
│              │ (内核对象)       │                             │
│              └────────┬────────┘                             │
│                       │                                      │
│      ┌────────────────┼────────────────┐                    │
│      ▼                ▼                ▼                    │
│  CVPixelBuffer    MTLTexture      CIImage                   │
│  (视频/相机)       (Metal 渲染)    (Core Image 滤镜)          │
│                                                              │
│      ▼                ▼                ▼                    │
│  编码/解码         GPU 绘制         图像处理                   │
│                                                              │
│  所有这些操作共享同一块物理内存，无任何拷贝！                     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### CVPixelBuffer → MTLTexture 零拷贝

```swift
import Metal
import CoreVideo

var textureCache: CVMetalTextureCache?
CVMetalTextureCacheCreate(nil, nil, mtlDevice, nil, &textureCache)

func makeMetalTexture(from pixelBuffer: CVPixelBuffer) -> MTLTexture? {
    let width = CVPixelBufferGetWidth(pixelBuffer)
    let height = CVPixelBufferGetHeight(pixelBuffer)
    
    var cvTextureOut: CVMetalTexture?
    CVMetalTextureCacheCreateTextureFromImage(
        nil, textureCache!, pixelBuffer, nil,
        .bgra8Unorm, width, height, 0,
        &cvTextureOut
    )
    
    // ★ 返回的 MTLTexture 和原 pixelBuffer 共享同一块内存
    return cvTextureOut.flatMap { CVMetalTextureGetTexture($0) }
}
```

#### 与 Android SurfaceTexture 的对比

| 方面 | Android (SurfaceTexture) | iOS (IOSurface) |
|------|-------------------------|-----------------| 
| 底层对象 | GraphicBuffer (gralloc) | IOSurface (内核对象) |
| 跨进程共享 | 支持（Binder + fd 传递） | 天然支持（Mach port） |
| GPU 后端 | OpenGL ES / Vulkan | Metal / OpenGL ES（已废弃） |
| 采样目标 | `GL_TEXTURE_EXTERNAL_OES` | `MTLTexture` (bgra 直接可用) |
| YUV 处理 | 驱动自动转换 | 需要手动创建 Y/CbCr 双纹理 |

#### 性能特征

- **零拷贝真正意义**：数据物理上只有一份，多个消费者共享
- **延迟极低**：Extension 收到帧时，画面刚合成完不到 16ms
- **50MB 限制的救赎**：如果每帧都复制成 UIImage，2~3 帧就 OOM 了

### 9.5.5 移动示例 05: 硬件编码器直连（MediaCodec / VideoToolbox）

#### 核心思想

**一句话总结**：不要把画面"拉出来"再"送进去"编码，直接让**采集的 Surface = 编码器的输入 Surface**，形成一条 GPU 内部的闭环管道。

#### Android: MediaCodec Input Surface

```java
// 1. 配置编码器
MediaFormat format = MediaFormat.createVideoFormat("video/avc", width, height);
format.setInteger(MediaFormat.KEY_COLOR_FORMAT,
    MediaCodecInfo.CodecCapabilities.COLOR_FormatSurface);  // ★ 关键
format.setInteger(MediaFormat.KEY_BIT_RATE, 6_000_000);
format.setInteger(MediaFormat.KEY_FRAME_RATE, 30);
format.setInteger(MediaFormat.KEY_I_FRAME_INTERVAL, 2);

MediaCodec encoder = MediaCodec.createEncoderByType("video/avc");
encoder.configure(format, null, null, MediaCodec.CONFIGURE_FLAG_ENCODE);

// 2. ★ 从编码器要一个输入 Surface
Surface encoderInputSurface = encoder.createInputSurface();

// 3. 让 VirtualDisplay 直接把画面往编码器的 Surface 里塞！
mediaProjection.createVirtualDisplay(
    "ScreenRecord", width, height, dpi,
    DisplayManager.VIRTUAL_DISPLAY_FLAG_AUTO_MIRROR,
    encoderInputSurface,   // ★ 系统合成结果 → 编码器输入
    null, null
);

// 4. 启动编码器，从输出端拿 H.264 NAL Unit
encoder.start();
while (recording) {
    int idx = encoder.dequeueOutputBuffer(bufferInfo, 10000);
    if (idx >= 0) {
        ByteBuffer nal = encoder.getOutputBuffer(idx);
        writeToFile(nal);  // 或 RTMP 推流
        encoder.releaseOutputBuffer(idx, false);
    }
}
```

#### 全链路零拷贝的意义

### 9.6.1 桌面示例 D1: BitBlt / CGWindowListCreateImage 基础截图

#### 核心思想

**一句话总结**：让 GPU 合成好的桌面画面"下载"回 CPU 内存，得到一份 DIB 位图（Windows）或 CGImage（macOS）。**对应移动端示例 01**。

#### Windows 实现（GDI BitBlt）

```cpp
// 传统 GDI 方式：从整个屏幕 DC 复制到内存 DC
HDC hScreen = GetDC(NULL);                       // 整个桌面的 DC
HDC hDC     = CreateCompatibleDC(hScreen);
int cx = GetSystemMetrics(SM_CXSCREEN);
int cy = GetSystemMetrics(SM_CYSCREEN);

HBITMAP hBmp = CreateCompatibleBitmap(hScreen, cx, cy);
SelectObject(hDC, hBmp);

// ★ 阻塞式像素拷贝：GPU→系统内存→GDI 位图
BitBlt(hDC, 0, 0, cx, cy, hScreen, 0, 0, SRCCOPY);

// 后续可 GetDIBits 拿到像素数组
DeleteDC(hDC);
ReleaseDC(NULL, hScreen);
```

#### macOS 实现（Core Graphics）

```swift
import CoreGraphics

// macOS 14 前的经典 API（已在新版本标记 deprecated，但仍可用）
if let cgImage = CGWindowListCreateImage(
    .infinite,
    .optionOnScreenOnly,      // 或 .optionIncludingWindow 采单个窗口
    kCGNullWindowID,
    [.bestResolution, .boundsIgnoreFraming]
) {
    let bitmap = NSBitmapImageRep(cgImage: cgImage)
    let pngData = bitmap.representation(using: .png, properties: [:])
}
```

#### 底层发生了什么

```
┌─────────────────────────────────────────────────────────────┐
│               D1 桌面基础截图路径                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  DWM / WindowServer 合成好的帧缓冲 (GPU 显存)                 │
│       │                                                      │
│       │ Windows: BitBlt 触发 GDI 从 GPU 回读                  │
│       │ macOS:   CG 从 WindowServer IPC 拿一份 CGImage        │
│       │ (同步阻塞，等待合成器完成当前帧)                       │
│       ▼                                                      │
│  系统内存 (BGRA 像素数组)                                     │
│       │                                                      │
│       │ 用户代码再拷贝一次                                    │
│       ▼                                                      │
│  应用堆里的 Bitmap / CGImage                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### 性能特征

- **1080p 全屏截图约 30~60ms**：几乎无法用于实时采集
- **每帧拷贝 ~8MB**：内存带宽压力大
- **完全丢失 GPU 加速优势**：即使后续要 GPU 编码，还得再上传一次

#### 适用场景

- 手动截图工具（QQ 截图、Snipping Tool）
- 单帧调试快照
- 老代码维护，不追求实时性

#### 局限性

- 无法达到 30fps 采集（BitBlt 单次就要 30ms+）
- CGWindowListCreateImage 每次都触发一次跨进程 IPC 到 WindowServer
- macOS 14+ 已提示迁移到 ScreenCaptureKit（对应 D4）

### 9.6.2 桌面示例 D2: DXGI Desktop Duplication (Windows)

#### 核心思想

**一句话总结**：让 DWM 主动把"下一帧桌面"推送到一个 D3D11 共享纹理里，App 用 GPU API 直接访问。**对应移动端示例 02**（推模式流式采集）。

#### 与 D1 的关键区别

```
D1 的做法（拉模式）:                    D2 的做法（推模式）:
┌─────────────────────┐                ┌─────────────────────┐
│ App 主动调用 BitBlt  │                │ 建立 Duplication 后 │
│ → 触发一次 GPU 回读   │                │ AcquireNextFrame    │
│ 每次都阻塞           │                │ DWM 有新帧就返回     │
│ (低频，同步)         │                │ (高频，流式)         │
└─────────────────────┘                └─────────────────────┘
```

#### 关键代码（C++ / D3D11）

```cpp
// 1. 创建 D3D11 设备
ID3D11Device*        device;
ID3D11DeviceContext* ctx;
D3D11CreateDevice(nullptr, D3D_DRIVER_TYPE_HARDWARE, nullptr, 0,
                  nullptr, 0, D3D11_SDK_VERSION, &device, nullptr, &ctx);

// 2. 拿到 DXGI Output（一台显示器 = 一个 Output）
IDXGIDevice*      dxgiDevice;   device->QueryInterface(&dxgiDevice);
IDXGIAdapter*     adapter;      dxgiDevice->GetAdapter(&adapter);
IDXGIOutput*      output;       adapter->EnumOutputs(0, &output);
IDXGIOutput1*     output1;      output->QueryInterface(&output1);

// 3. ★ 建立 Desktop Duplication
IDXGIOutputDuplication* dup;
output1->DuplicateOutput(device, &dup);

// 4. 采集循环
while (running) {
    DXGI_OUTDUPL_FRAME_INFO info;
    IDXGIResource* res;
    HRESULT hr = dup->AcquireNextFrame(16 /*ms 超时*/, &info, &res);
    if (hr == DXGI_ERROR_WAIT_TIMEOUT) continue;    // 桌面没变化
    
    // 拿到桌面的 D3D11 纹理
    ID3D11Texture2D* desktopTex;
    res->QueryInterface(&desktopTex);
    
    // 方案 A: 拷贝到我们的纹理，交给编码器/渲染
    ctx->CopyResource(myTexture, desktopTex);
    
    // 方案 B: 直接 CopySubresourceRegion 到分享内存
    
    dup->ReleaseFrame();
}
```

#### 底层数据流

```
┌────────────────────────────────────────────────────────────────┐
│                DXGI Desktop Duplication 数据流                   │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  DWM (dwm.exe)                                                  │
│       │                                                         │
│       │ 每次合成完毕后触发 Present                              │
│       ▼                                                         │
│  DXGI Duplication 内部帧队列 (D3D11 共享纹理)                    │
│       │                                                         │
│       │ AcquireNextFrame(timeout)                              │
│       ▼                                                         │
│  App 收到 IDXGIResource                                         │
│       │                                                         │
│       ├── QueryInterface → ID3D11Texture2D                     │
│       │        │                                                │
│       │        │ 可直接作为 Shader Resource View 采样          │
│       │        ▼                                                │
│       │   自家 D3D11 渲染管线（不回 CPU）                        │
│       │                                                         │
│       └── 也可 CopyResource 到用户纹理再处理                     │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

#### 关键优化点：DirtyRects 与 MoveRects

```cpp
// info.TotalMetadataBufferSize > 0 时才有增量信息
UINT bufSize = info.TotalMetadataBufferSize;
std::vector<BYTE> metadata(bufSize);
UINT dirtySize = 0;
dup->GetFrameDirtyRects(bufSize, (RECT*)metadata.data(), &dirtySize);

// ★ 只拷贝变化区域，大幅降低带宽
for (UINT i = 0; i < dirtySize / sizeof(RECT); i++) {
    RECT& r = ((RECT*)metadata.data())[i];
    D3D11_BOX box = { r.left, r.top, 0, r.right, r.bottom, 1 };
    ctx->CopySubresourceRegion(myTexture, 0,
        r.left, r.top, 0, desktopTex, 0, &box);
}
```

**为什么重要？** 对于办公场景（大部分区域静止），DirtyRects 优化可以让实际拷贝量下降 90%。

#### 性能提升原理

1. **推模式减少空转**：只在有新帧时才醒来，静止桌面 0 拷贝
2. **数据全程在 GPU**：D3D11 共享纹理，无 CPU 回读
3. **DirtyRects 增量更新**：办公场景下带宽下降 10 倍以上

#### 局限性

- **不能采集单个窗口**，只能采整台显示器
- **UAC/安全桌面切换时会失效**，需要重建 Duplication
- **DRM 内容自动黑帧**（Netflix 全屏等）
- **macOS 无对应 API**：桌面同级方案要看 D4（ScreenCaptureKit）

### 9.6.3 桌面示例 D3: Windows.Graphics.Capture + D3D11 Texture

#### 核心思想

**一句话总结**：Windows 10 1903+ 引入的现代采集 API，DWM 把每帧直接送到 App 的 D3D11 设备上，**支持单窗口 / 单显示器 / 单应用**采集。**对应移动端示例 03**（GPU 纹理直采）。

#### 关键代码（C++/WinRT）

```cpp
#include <winrt/Windows.Graphics.Capture.h>
using namespace winrt::Windows::Graphics::Capture;

// 1. 让用户选择要采集的目标（屏幕或窗口）
GraphicsCapturePicker picker;
GraphicsCaptureItem   item = co_await picker.PickSingleItemAsync();
if (!item) return;

// 2. 从 D3D11 设备创建 WinRT IDirect3DDevice
auto d3dDevice = CreateDirect3DDevice(dxgiDevice);

// 3. ★ 创建 FramePool：D3D11 纹理的共享池
Direct3D11CaptureFramePool pool = Direct3D11CaptureFramePool::Create(
    d3dDevice,
    DirectXPixelFormat::B8G8R8A8UIntNormalized,
    2,                        // 池深度
    item.Size()
);

// 4. 每帧回调：拿到即用的 D3D11 纹理
pool.FrameArrived([](auto&& sender, auto&& args) {
    auto frame = sender.TryGetNextFrame();
    if (!frame) return;
    
    // ★ 关键：直接拿到 ID3D11Texture2D，与 App D3D 设备兼容
    ID3D11Texture2D* tex = GetD3D11Texture(frame.Surface());
    
    // 直接送渲染管线 / 编码器（后续 D5 讲解）
    RenderOrEncode(tex);
    // frame 会自动 close 归还池
});

// 5. 启动采集
GraphicsCaptureSession session = pool.CreateCaptureSession(item);
session.StartCapture();
```

### 9.6.4 桌面示例 D4: ScreenCaptureKit + IOSurface (macOS)

#### 核心思想

**一句话总结**：macOS 12.3+ 引入的现代采集 API，WindowServer 通过 IOSurface 把帧送到 App，**与 iOS 使用同一套 IOSurface / Metal / VideoToolbox 技术栈**。**对应移动端示例 04**。

#### 关键代码（Swift）

```swift
import ScreenCaptureKit

class Capturer: NSObject, SCStreamOutput {
    var stream: SCStream?
    
    func start() async throws {
        // 1. 枚举可采集内容（屏幕、窗口、App）
        let content = try await SCShareableContent.current
        guard let display = content.displays.first else { return }
        
        // 2. 配置过滤器（可指定要采/要排除的窗口）
        let filter = SCContentFilter(display: display, excludingWindows: [])
        
        // 3. 配置流参数
        let config = SCStreamConfiguration()
        config.width  = display.width  * 2   // Retina
        config.height = display.height * 2
        config.minimumFrameInterval = CMTime(value: 1, timescale: 60)  // 60fps
        config.pixelFormat = kCVPixelFormatType_32BGRA
        config.queueDepth = 3
        
        // 4. 启动流
        stream = SCStream(filter: filter, configuration: config, delegate: nil)
        try stream?.addStreamOutput(self, type: .screen,
                                    sampleHandlerQueue: .global(qos: .userInteractive))
        try await stream?.startCapture()
    }
    
    // ★ 每帧回调：CMSampleBuffer 内部就是 IOSurface
    func stream(_ stream: SCStream,
                didOutputSampleBuffer sampleBuffer: CMSampleBuffer,
                of type: SCStreamOutputType) {
        guard type == .screen,
              let pixelBuffer = CMSampleBufferGetImageBuffer(sampleBuffer) else { return }
        
        // 路径 A：直接送 VideoToolbox 编码（对应 D5）
        encodeWithVT(sampleBuffer)
        
        // 路径 B：CVPixelBuffer → MTLTexture 零拷贝
        renderWithMetal(pixelBuffer)
    }
}
```

#### CVPixelBuffer → MTLTexture 零拷贝（与 iOS 完全一致）

```swift
import Metal
import CoreVideo

var textureCache: CVMetalTextureCache?
CVMetalTextureCacheCreate(nil, nil, mtlDevice, nil, &textureCache)

func makeMetalTexture(from pixelBuffer: CVPixelBuffer) -> MTLTexture? {
    let width  = CVPixelBufferGetWidth(pixelBuffer)
    let height = CVPixelBufferGetHeight(pixelBuffer)
    var cvTex: CVMetalTexture?
    CVMetalTextureCacheCreateTextureFromImage(
        nil, textureCache!, pixelBuffer, nil,
        .bgra8Unorm, width, height, 0, &cvTex
    )
    // ★ 返回的 MTLTexture 与 pixelBuffer 共享同一块物理内存
    return cvTex.flatMap { CVMetalTextureGetTexture($0) }
}
```

#### 与 Windows WGC 的对照

| 方面 | Windows (WGC) | macOS (ScreenCaptureKit) |
|------|--------------|------------------------|
| 授权模型 | 首次弹窗，Win32 无弹窗 | 系统设置 → 屏幕录制手动授权 |
| 数据载体 | ID3D11Texture2D | IOSurface / CVPixelBuffer |
| 通知机制 | FramePool.FrameArrived | SCStreamOutput 回调 |
| 目标粒度 | 屏幕 / 窗口 / App | 屏幕 / 窗口 / App，还可排除窗口 |
| 音频采集 | 需要单独 WASAPI Loopback | ✅ 内置系统音频采集 |
| 平台整合 | Win10 1903+ | macOS 12.3+ |

#### 独特能力：系统音频同步采集

ScreenCaptureKit 的一大亮点是 **`type == .audio`**：可以在同一流中同步采集系统音频，无需额外 API，这是 Windows WGC 目前尚未提供的能力（Windows 需要 WASAPI Loopback 单独接入）。

#### 性能特征

- **零拷贝的物理意义**：pixelBuffer / MTLTexture / CIImage / VT 编码器输入是同一块内存
- **延迟极低**：WindowServer 合成完毕直接送 App，通常 <16ms
- **CPU 占用极低**：Retina 5K 30fps 采集 CPU 通常 <3%

### 9.6.5 桌面示例 D5: 硬件编码器直连（Media Foundation / VideoToolbox）

#### 核心思想

**一句话总结**：不要把画面"拉出来"再"送进去"编码，直接让**采集纹理 = 编码器的输入缓冲**，形成一条 GPU 内部的闭环管道。**对应移动端示例 05**。

#### Windows: Media Foundation + D3D11 输入

```cpp
#include <mfapi.h>
#include <mftransform.h>
#include <codecapi.h>

// 1. 建立 MF 设备管理器，绑定 D3D11 设备（与 WGC 同一个）
UINT resetToken;
IMFDXGIDeviceManager* devMgr;
MFCreateDXGIDeviceManager(&resetToken, &devMgr);
devMgr->ResetDevice(d3dDevice, resetToken);

// 2. 创建 H.264 编码器 MFT
IMFTransform* encoder;
CoCreateInstance(CLSID_MSH264EncoderMFT, nullptr, CLSCTX_INPROC_SERVER,
                 IID_PPV_ARGS(&encoder));

// 3. ★ 关键：让编码器使用同一个 D3D11 设备
encoder->ProcessMessage(MFT_MESSAGE_SET_D3D_MANAGER, (ULONG_PTR)devMgr);

// 4. 配置输出格式
IMFMediaType* outType;
MFCreateMediaType(&outType);
outType->SetGUID(MF_MT_MAJOR_TYPE, MFMediaType_Video);
outType->SetGUID(MF_MT_SUBTYPE, MFVideoFormat_H264);
outType->SetUINT32(MF_MT_AVG_BITRATE, 6'000'000);

MFSetAttributeRatio(outType, MF_MT_FRAME_RATE, 30, 1);

// 5. 输入格式：NV12（GPU 转换更快）

MFCreateMediaType(&inType);
inType->SetGUID(MF_MT_SUBTYPE, MFVideoFormat_NV12);
// ... 其他属性
encoder->SetInputType(0, inType, 0);

// 6. 采集循环（在 WGC FrameArrived 中调用）
void OnFrame(ID3D11Texture2D* wgcTex) {
    // ★ 把 D3D11 纹理包装成 IMFSample 送入编码器
    IMFMediaBuffer* buf;
    MFCreateDXGISurfaceBuffer(__uuidof(ID3D11Texture2D), wgcTex, 0, FALSE, &buf);
    IMFSample* sample;
    MFCreateSample(&sample);
    sample->AddBuffer(buf);
    sample->SetSampleTime(GetTimeStamp100ns());
    
    encoder->ProcessInput(0, sample, 0);
    
    // 从输出端拿到 H.264 NAL Unit
    MFT_OUTPUT_DATA_BUFFER out{};
    DWORD status;
    if (encoder->ProcessOutput(0, 1, &out, &status) == S_OK) {
        WriteNALUnit(out.pSample);   // 或推流
    }
}
```

#### 10.1.3 数据流路径的三级分层（L1/L2/L3 术语定义）

> 🎯 **本节是全文 `L1 / L2 / L3` 记号的唯一权威定义**。第 10~11 章后续所有涉及 L1/L2/L3 的表述，均以此处为准。

**为什么要分三级？** 从“数据是否需要经过 CPU”这一核心问题出发，无论移动还是桌面，所有采集方案本质上都只有**三种物理路径**：

| 路径等级 | 关键特征 | GPU→CPU 拷贝次数 | 移动端代表 | 桌面端代表 |
|:-------:|---------|:---------------:|-----------|-----------|
| **L1 · CPU 拷贝路径** | 数据必经 CPU 内存，本质上是“截屏 + 拷回” | ≥ 1 次全帧 | A1 Bitmap Readback、A2 ImageReader | B1 BitBlt / CGWindowListCreateImage |
| **L2 · GPU 零拷贝路径** | 数据全程在 GPU / 内核共享内存，仅传递句柄 | 0 | A3 SurfaceTexture、A4 iOS IOSurface | B2 DXGI Duplication、B3 WGC、B4 ScreenCaptureKit |
| **L3 · 编码器直连路径** | 采集缓冲直接送入硬件编码器，全在 GPU/ASIC 内部完成 | 0（仅编码后比特流流出） | A5 编码器 Input Surface | B5 MF / VT 硬编直连 |

**三级之间的关系**：

```
   L1 (CPU 拷贝)
      ↓  把“回读”换成“共享内存句柄”
   L2 (GPU 零拷贝)
      ↓  把“纹理手动送入编码器”换成“采集直接写入编码器输入 Surface”
   L3 (编码直连)
```

> 💡 **核心结论**：从 L1 到 L3 每上一级，1080p@30fps 场景下的 **CPU 占用大约下降一个量级**，功耗和发热同步下降。

> ⚠️ **不要混淆 L1/L2/L3 与 9.4 的 S1~S5**：S 表示“API 新旧阶段”（具体到方案序号），L 表示“物理数据路径”（数据是否绕开 CPU）。一个 L2 内可以包含 S3、S4 两个阶段。

### 10.2 API 层次对比（四大平台）

同一等级的方案，在 Android / iOS / Windows / macOS 上使用的是**完全不同的 API 组合**，但**架构角色一一对应**。理解这张映射表，可以帮你在跨平台工程中共享设计：

| 层次 | Android | iOS | Windows | macOS |
|------|---------|-----|---------|-------|
| **权限入口** | `MediaProjectionManager.createScreenCaptureIntent` | `RPScreenRecorder` / Broadcast Extension | UAC / `GraphicsCaptureAccess` | TCC (`Screen Recording` 权限) |
| **采集会话** | `VirtualDisplay` | `RPBroadcastSampleHandler` | `IDXGIOutputDuplication` / `GraphicsCaptureSession` | `SCStream` / `CGDisplayStream` |
| **画面来源** | `SurfaceFlinger` 合成结果 | `RenderServer` 合成结果 | DWM 交换链 | `WindowServer` 合成结果 |
| **内存表示** | `Surface` / `GraphicBuffer` / `AHardwareBuffer` | `CVPixelBuffer` / `IOSurface` | `ID3D11Texture2D` / `IDXGIResource` | `CVPixelBuffer` / `IOSurface` |
| **GPU 集成** | `SurfaceTexture` (GLES) / Vulkan AHB 扩展 | `CVMetalTextureCache` | D3D11 SRV / D3D11-on-12 | `CVMetalTextureCache` |
| **编码集成** | `MediaCodec.createInputSurface` | `VTCompressionSession` + `CVPixelBuffer` | Media Foundation `IMFTransform` / NVENC | `VTCompressionSession` |
| **典型延迟** | 40~80 ms | 40~80 ms | 20~50 ms | 20~50 ms |

> 🎯 **移动 vs 桌面**：桌面平台由于没有 Extension 内存限制、且合成器天然缓存窗口原始 buffer，因此**从 L1 直接跨到 L2 的门槛更低**。移动端在到 L3 之前，必须先解决权限、Extension 生命周期、跨进程通信等一系列非技术问题。

### 10.4 跨平台方案对应关系

在做跨端工程（如同时支持 Android+iOS+Win+macOS 的直播 SDK）时，可以按**同一台阶**匹配对应实现：

> 编号沿用 [10.1.3](#1013-数据流路径的三级分层l1l2l3-术语定义) 的 L1/L2/L3 定义。

| 数据流路径 | 移动端 | 桌面端 | 共同特征 |
|:---------:|-------|-------|---------|
| **L1 · CPU 拷贝路径** | A1 Bitmap Readback、A2 ImageReader | B1 BitBlt / CGWindowListCreateImage | CPU 全帧拷贝，最兼容 |
| **L2 · GPU 零拷贝路径** | A3 SurfaceTexture、A4 iOS IOSurface | B2 DXGI Duplication、B3 WGC、B4 SCK | 数据一直在 GPU / 内核共享内存 |
| **L3 · 编码器直连路径** | A5 Encoder Input Surface | B5 MF Transform / VideoToolbox | 采集缓冲 = 编码器输入，全 GPU |

**跨端设计建议**：

1. **抽象数据形态**：定义一个统一的 `PlatformFrame` 接口，四个平台各自返回自己的原生零拷贝句柄（AHardwareBuffer / IOSurface / ID3D11Texture2D / IOSurface）。
2. **统一编码器抽象**：`IEncoder::submitFrame(PlatformFrame&)`，在每个平台内部走各自的 L3 直连路径。
3. **回退策略**：在权限不足或旧系统上，统一回退到 L1，确保功能可用性。

## 十一、性能分析与瓶颈

> 本章将第十章的 10 个方案（移动端 A1~A5 + 桌面端 B1~B5）在**瓶颈类型、数据量、功耗、突破范式**四个维度做统一分析。目的是让读者不仅知道"哪个方案快"，更能理解"为什么快、还能怎么再快、以及每种方案卡在哪里"。

> 📌 **术语约定回顾**（详见 [10.1.3 数据流路径的三级分层](#1013-数据流路径的三级分层l1l2l3-术语定义)）：
> - **L1 · CPU 拷贝路径**：GPU→CPU 至少 1 次全帧拷贝（如 A1/A2/B1）
> - **L2 · GPU 零拷贝路径**：数据一直在 GPU / 内核共享内存（如 A3/A4/B2/B3/B4）
> - **L3 · 编码器直连路径**：采集缓冲直接送入硬件编码器（如 A5/B5）
>
> 本章 11.1 / 11.2 / 11.4 中提到的 "L1 卡在…、L2 卡在…、L3 达到…" 均基于此定义。

### 11.1 每种方案的核心瓶颈

**A 组 · 移动端**：

| 方案 | 瓶颈原因 | 优化方向 |
|------|---------|---------|
| A1 Bitmap Readback | `glReadPixels` 破坏 GPU 流水线，同步阻塞主线程 | 换 VirtualDisplay 推模式 |
| A2 ImageReader | GPU→CPU 拷贝浪费带宽，`acquireLatestImage()` 有锁 | 保持在 GPU：用 SurfaceTexture |
| A3 SurfaceTexture | 编码前仍需一次 GL→编码器的纹理拷贝 | 直接把采集 Surface 交给编码器 |
| A4 iOS Broadcast + IOSurface | Extension 50MB 内存墙 + 跨进程传输 | 编码在 Extension 内部完成，仅传编码后 NAL |
| A5 编码器 Input Surface 直连 | 已达硬件极限，几乎无软件瓶颈 | 仅剩码率控制、GOP 结构等业务级调优 |

**B 组 · 桌面端**：

| 方案 | 瓶颈原因 | 优化方向 |
|------|---------|---------|
| B1 BitBlt / CGWindowListCreateImage | CPU 全帧拷贝 + GDI/Quartz 合成同步 | 换 DXGI Duplication / ScreenCaptureKit |
| B2 DXGI Desktop Duplication | 需自行处理脏矩形、光标叠加、旋转 | 保留 GPU 纹理直送编码器；用 dirty rects 降带宽 |
| B3 Windows.Graphics.Capture | `FramePool` 队列长度 + Compositor 帧率上限 | 调 `MinUpdateInterval`，紧跟 vsync |
| B4 ScreenCaptureKit + IOSurface | `SCStreamOutput` 回调在私有队列，主线程回投有开销 | 直接在回调线程走 VideoToolbox |
| B5 硬件编码器直连（MF / VideoToolbox） | 已达硬件极限 | 仅剩码率与 GOP 调优 |

**一句话总结**：**L1 卡在 CPU 拷贝，L2 卡在跨模块传递，L3 已到硬件天花板**。

### 11.2 理论数据量计算（1080p@30fps 基准）

以 1920 × 1080 × 4 字节 = **8.3 MB / 帧** 为单位，10 个方案的每秒数据流量：

| 方案 | 每帧 CPU 拷贝量 | CPU 内存带宽 | 备注 |
|:----:|:---------------:|:-----------:|------|
| A1 Bitmap Readback | 8.3 MB | **~300 MB/s**（含 Java 堆分配） | 主要瓶颈 |
| A2 ImageReader | 8.3 MB | ~250 MB/s | 少了 Java 堆分配 |
| A3 SurfaceTexture | **0 MB** | ~0 MB/s | 仅传纹理句柄 |
| A4 iOS IOSurface | **0 MB** | ~0 MB/s | 跨进程内核对象共享 |
| A5 Encoder Input Surface | **0 MB** | 750 KB/s（仅编码后 6Mbps） | 全链路 GPU |
| B1 BitBlt / CGWindowList | 8.3 MB | ~200 MB/s | 单帧拷贝 |
| B2 DXGI Duplication | **0 MB** | ~0 MB/s | GPU 纹理句柄 |
| B3 Windows.Graphics.Capture | **0 MB** | ~0 MB/s | 同上 |
| B4 ScreenCaptureKit | **0 MB** | ~0 MB/s | IOSurface 共享 |
| B5 MF / VT 直连 | **0 MB** | 750 KB/s | 全链路 GPU |

**关键洞察**：

- L2 及以上方案，**CPU 内存带宽几乎为 0**（仅指针/句柄传递）
- L3 方案，CPU 只搬运编码后的 6Mbps ≈ 750 KB/s 数据
- L1 方案每秒消耗 200~300 MB 内存带宽，占典型手机总带宽（20~30 GB/s）的 1%~1.5%，看似不多，但**引起 GPU 停顿的间接损失远大于此**

### 11.3 电量、温度与散热

**移动端**：电池容量受限，功耗是必须考虑的**首要指标**。  
**桌面端**：笔记本电池模式和长时间直播场景同样重要，尤其是 macOS 无风扇机型。

| 方案 | 主要功耗来源 | 移动端相对功耗 | 桌面端相对功耗 |
|:----:|-------------|:-------------:|:-------------:|
| A1 Bitmap | CPU 忙 + GPU 停顿 + 频繁 GC | ★★★★★ | — |
| A2 ImageReader | 内存拷贝 + CPU 处理 | ★★★★ | — |
| A3 SurfaceTexture | GPU 渲染 | ★★ | — |
| A4 IOSurface（+ Extension） | GPU 渲染 + 跨进程调度 | ★★ | — |
| A5 Encoder Input Surface | 硬件编码器（Fixed Function） | ★ | — |
| B1 BitBlt / CGWindowList | CPU 拷贝 + GDI 合成 | — | ★★★★ |
| B2 DXGI Duplication | GPU 拷贝 | — | ★★ |
| B3 WGC | GPU 拷贝 + WinRT 调度 | — | ★★ |
| B4 ScreenCaptureKit | GPU + WindowServer | — | ★★ |
| B5 MF / VT 直连 | 硬件编码器 | — | ★ |

> 💡 **硬件编码器**（H.264/H.265 Fixed Function 单元）在移动端和桌面端功耗都仅为 CPU 编码的 1/10~1/20，是**长时间录屏、直播、云游戏的唯一可行方案**。

**温度触发的降频行为**：

| 平台 | 触发信号 | 系统行为 | 应用应对 |
|------|---------|---------|---------|
| Android | `PowerManager.getCurrentThermalStatus()` | CPU/GPU 降频，可能强杀前台服务 | 监听 → 主动降码率/降分辨率 |
| iOS | `NSProcessInfo.thermalState` | 降 CPU 频率，Extension 优先被回收 | 监听 → 降到 720p 或 15fps |
| Windows | `IPowerSettingSubscription` / EC 传感器 | CPU 降频，风扇转速上升 | 侦测 AC/DC → DC 时降码率 |
| macOS | `NSProcessInfo.thermalState`（同 iOS） | GPU/编码器降频，风扇转速上升 | 与 iOS 同策略 |

### 11.4 瓶颈突破的通用范式

无论平台如何，突破每一级瓶颈的思路是**相同的四步**：

```
┌──────────────────────────────────────────────────────────────────┐
│  L1 → L2：把数据搬到 GPU/共享内存                                    │
│    - Android:  glReadPixels  → SurfaceTexture / AHardwareBuffer   │
│    - iOS:      CGImage      → CVPixelBuffer / IOSurface           │
│    - Windows:  BitBlt       → DXGI Duplication / WGC              │
│    - macOS:    CGWindowList → ScreenCaptureKit                    │
│                                                                    │
│  L2 → L3：让编码器直接吃采集输出，绕开中间纹理                          │
│    - Android:  Encoder.createInputSurface()                        │
│    - iOS:      VTCompressionSession + CVPixelBuffer                │
│    - Windows:  Media Foundation IMFTransform + D3D11              │
│    - macOS:    VideoToolbox + IOSurface                            │
│                                                                    │
│  L3 → 极限：调优码率控制器、GOP 结构、脏矩形、色彩空间转换                │
└──────────────────────────────────────────────────────────────────┘
```

**当你的方案卡在某一级时**，可按下面的"体检清单"逐项检查：

| 检查项 | 通过标准 | 常用工具 |
|-------|---------|---------|
| CPU 占用 | 单核 < 30% | Perfetto / Instruments / Xperf / Instruments |
| GPU 占用 | < 50% | Systrace / Xcode GPU / GPUView / Metal HUD |
| 内存带宽 | 帧路径上无 GPU→CPU 全帧拷贝 | Perfetto 内存视图 / Instruments Allocations |
| 编码功耗 | 硬编而非 x264 软编 | 检查是否有 `createInputSurface` / `IMFTransform` |
| 端到端延迟 | 采集到编码器 < 50ms | 打时间戳日志或 QuickTime 录屏对比 |

### A.1 常见坑点清单（四大平台）

#### A.1.1 移动端常见坑

| 问题 | 平台 | 原因 | 解决 |
|------|:----:|------|------|
| 采集画面上下颠倒 | Android | GL 坐标系 Y 轴与显示相反 | 用 `getTransformMatrix()` 或翻转 UV |
| 采集画面绿屏 | Android | YUV 平面顺序错误 | 检查 `linesize` 和 stride |
| VirtualDisplay 黑屏 | Android | Surface 未 detach 就重建 | 先 release VirtualDisplay 再释放 |
| 编码器输出无 SPS/PPS | Android | 未处理 `BUFFER_FLAG_CODEC_CONFIG` | 首帧特判，把 SPS/PPS 拼到 IDR 前 |
| 前台服务被杀 | Android 14+ | 未声明 `mediaProjection` 类型 | manifest 中加 `foregroundServiceType` |
| iOS Extension 崩溃 | iOS | 内存超过 50MB | 必须零拷贝，禁止 CGImage/UIImage |
| Extension 编码结果传主 App 慢 | iOS | 每帧走 App Group 磁盘序列化 | 用 XPC 或 Darwin Notification + 共享内存 |
| ThermalState 转 Serious 后掉帧 | iOS | 系统降频 | 主动降分辨率/帧率，监听 `NSProcessInfoThermalStateDidChange` |

#### A.1.2 桌面端常见坑

| 问题 | 平台 | 原因 | 解决 |
|------|:----:|------|------|
| BitBlt 抓不到全屏游戏 | Windows | 独占全屏绕过 DWM 合成 | 改用 DXGI Duplication 或 WGC |
| 抓 UWP / Chrome 出现黑屏 | Windows | 硬件覆盖层 / DRM 保护 | 用 WGC（自动处理）或提示用户禁用硬件加速 |
| 多显示器只抓到一屏 | Windows | Duplication 需按 `IDXGIOutput` 分别创建 | 每个 Output 建立独立会话 |
| WGC 首次调用弹出黄框提示 | Windows 11 | 系统 UI 通知用户当前正被采集 | 属正常安全提示，无法屏蔽；文档中告知用户 |
| ScreenCaptureKit 首帧延迟大 | macOS | 系统构建 SCStream 的初始化耗时 | 应用启动即预建 stream，或允许 200ms 冷启动 |
| SCK 采集自身窗口导致回环 | macOS | 未过滤自己 | 用 `SCContentFilter(display:excludingWindows:)` 排除自身 |
| Retina 屏采到 2× 像素 | macOS | 逻辑像素 vs 物理像素 | 通过 `SCStreamConfiguration.width/height` 显式设定物理像素 |
| DPI 缩放下抓的位图变形 | Windows | 未设置进程 DPI 感知 | manifest 中声明 `PerMonitorV2` DPI Awareness |
| 长时间录制文件损坏 | 桌面双端 | 未在退出时 flush 编码器 / 关闭 muxer | `Finalize()` / `VTCompressionSessionCompleteFrames()` |

6. 深入 **A3 / A4 / B3 / B4**（SurfaceTexture / IOSurface / WGC / SCK），理解"零拷贝"在四大平台的具体形式
7. 最终实现 **A5 / B5**（编码器直连），构建生产级的采集→编码闭环
8. 结合具体业务（直播 / 云游戏 / 远程协助 / 录屏软件）选择合适的组合，并参考第 10 章的 4 平台生产链路
