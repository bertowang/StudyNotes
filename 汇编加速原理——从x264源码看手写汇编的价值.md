## 一、编译器已经做了什么？

现代编译器在 `-O2` / `-O3` 优化级别下，会自动执行大量优化。先了解这些，才能理解"为什么还不够"。

### 1.1 编译器自动优化清单

| 优化类别 | 具体技术 | 效果 |
|----------|----------|------|
| **标量优化** | 常量折叠、死代码消除、公共子表达式消除 | 减少无用计算 |
| **循环优化** | 循环展开、循环向量化、循环交换、循环分裂 | 提升循环效率 |
| **内存优化** | 寄存器分配、指令调度、预取提示 | 减少访存延迟 |
| **自动向量化** | 自动识别可 SIMD 化的循环，生成 SIMD 指令 | 数据并行加速 |
| **过程间优化** | 内联展开、链接时优化（LTO）、Profile-Guided Optimization (PGO) | 全局最优 |

### 1.2 编译器自动向量化示例

```c
// 编译器看到这段代码，会自动向量化
void add_arrays(int *a, int *b, int *c, int n) {
    for (int i = 0; i < n; i++) {
        c[i] = a[i] + b[i];
    }
}
```

用 `gcc -O3 -march=native` 编译后，编译器会自动生成类似这样的 SIMD 代码：

```asm
; x86-64 自动向量化结果（简化）
.loop:
    vmovdqu  ymm0, [rdi + rcx*4]    ; 一次加载8个int (256位)
    vpaddd   ymm0, ymm0, [rsi + rcx*4]  ; 8个int同时相加
    vmovdqu  [rdx + rcx*4], ymm0    ; 一次存储8个int
    add      rcx, 8
    cmp      rcx, rax
    jl       .loop
```

#### 为什么这段 SIMD 汇编和上面的 C 代码是等价的？——逐行对照

先回忆 C 代码在做什么：

```
循环体：c[i] = a[i] + b[i]
含义：从数组 a 取一个数，从数组 b 取一个数，相加后存入数组 c
每次循环处理 1 个元素，循环 n 次
```

再看 SIMD 汇编在做什么——**完全相同的事情，只是一次处理 8 个**：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  C 代码逻辑              →    SIMD 汇编指令                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  读取 a[i]               →    vmovdqu ymm0, [rdi + rcx*4]                   │
│  （从数组a取数据）              （一次取8个int，放入256位寄存器ymm0）           │
│                                                                             │
│  a[i] + b[i]            →    vpaddd ymm0, ymm0, [rsi + rcx*4]              │
│  （加法运算）                   （ymm0中的8个int，和数组b的8个int同时相加）     │
│                                                                             │
│  c[i] = 结果             →    vmovdqu [rdx + rcx*4], ymm0                   │
│  （存入数组c）                  （把8个结果一次性写回数组c）                    │
│                                                                             │
│  i++                     →    add rcx, 8                                    │
│  （下标+1）                    （下标+8，因为一次处理了8个）                    │
│                                                                             │
│  i < n ?                 →    cmp rcx, rax / jl .loop                       │
│  （循环条件判断）               （还没处理完？跳回去继续）                      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**关键对应关系解释**：

| 概念 | C 代码 | SIMD 汇编 | 为什么能对上？ |
|------|--------|-----------|---------------|
| 数组 a 的地址 | 指针 `a` | 寄存器 `rdi` | 函数参数按约定存在 rdi 中 |
| 数组 b 的地址 | 指针 `b` | 寄存器 `rsi` | 第二个参数存在 rsi 中 |
| 数组 c 的地址 | 指针 `c` | 寄存器 `rdx` | 第三个参数存在 rdx 中 |
| 当前下标 i | 变量 `i` | 寄存器 `rcx` | 编译器选了 rcx 当计数器 |
| 元素偏移 | `a[i]` = `*(a + i)` | `[rdi + rcx*4]` | 每个 int 占 4 字节，所以 ×4 |
| 一次处理量 | 1 个 | 8 个 | ymm 寄存器是 256 位 = 8×32位int |
| 循环步长 | i++ (每次+1) | rcx += 8 (每次+8) | 因为一轮处理了 8 个元素 |

**用图来理解"一次处理8个"**：

```
C 代码的执行方式（标量，一次1个）：
  第1轮: a[0] + b[0] → c[0]
  第2轮: a[1] + b[1] → c[1]
  第3轮: a[2] + b[2] → c[2]
  ...
  第n轮: a[n-1] + b[n-1] → c[n-1]
  → 共循环 n 次

SIMD 的执行方式（向量，一次8个）：
  第1轮: a[0..7] + b[0..7] → c[0..7]     ← 8个加法同时完成！
  第2轮: a[8..15] + b[8..15] → c[8..15]
  第3轮: a[16..23] + b[16..23] → c[16..23]
  ...
  → 共循环 n/8 次（循环次数减少为1/8！）

ymm0 寄存器内部（256位 = 8个int）：
  ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
  │ a[0] │ a[1] │ a[2] │ a[3] │ a[4] │ a[5] │ a[6] │ a[7] │  ← vmovdqu 加载
  └──┬───┴──┬───┴──┬───┴──┬───┴──┬───┴──┬───┴──┬───┴──┬───┘
     +      +      +      +      +      +      +      +       ← vpaddd 同时加
  ┌──┴───┬──┴───┬──┴───┬──┴───┬──┴───┬──┴───┬──┴───┬──┴───┐
  │ b[0] │ b[1] │ b[2] │ b[3] │ b[4] │ b[5] │ b[6] │ b[7] │
  └──┬───┴──┬───┴──┬───┴──┬───┴──┬───┴──┬───┴──┬───┴──┬───┘
     ‖      ‖      ‖      ‖      ‖      ‖      ‖      ‖
  ┌──┴───┬──┴───┬──┴───┬──┴───┬──┴───┬──┴───┬──┴───┬──┴───┐
  │ c[0] │ c[1] │ c[2] │ c[3] │ c[4] │ c[5] │ c[6] │ c[7] │  ← vmovdqu 存储
  └──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
```

> 🔑 **一句话总结**：SIMD 没有改变"取数→相加→存数"的逻辑，只是把"一次处理1个"变成了"一次处理8个"。就像超市收银：标量是一件一件扫码，SIMD 是把8件商品放在传送带上一次性扫过去。

> 💡 **小白理解**：编译器已经能把简单的循环变成"一次处理多个数据"的并行代码。那为什么还需要手写？

## 二、编译器做不到什么？——手写汇编的六大理由

### 理由一：编译器的向量化是"保守"的

```
编译器的思维模式：
  "我必须保证正确性。如果我不能100%证明这个优化是安全的，我就不做。"

人类的思维模式：
  "我知道这些数据的对齐方式、范围、不会溢出——编译器不知道这些。"
```

**具体例子**：

```c
// 编译器看到这段代码，可能不敢向量化
void process(uint8_t *src, uint8_t *dst, int stride, int n) {
    for (int i = 0; i < n; i++) {
        dst[i * stride] = (src[i * stride] + src[(i+1) * stride]) >> 1;
    }
}
// 原因：
// 1. src 和 dst 可能重叠（aliasing）→ 编译器不敢并行
// 2. stride 是变量 → 编译器不确定内存访问模式
// 3. 跨步访问 → 不是连续内存，向量化困难
```

**手写汇编的人知道**：
- `src` 和 `dst` 绝对不重叠（视频编码中帧缓冲区是分开的）
- `stride` 总是 16 的倍数（编码器保证对齐）
- 可以用特殊的 gather/scatter 指令处理跨步访问

### 理由二：编译器不了解算法的"全局语义"

```
视频编码中的 SAD（Sum of Absolute Differences）计算：

C 语言版本：
  for (int y = 0; y < 16; y++)
      for (int x = 0; x < 16; x++)
          sad += abs(src[y*stride1+x] - ref[y*stride2+x]);

编译器看到的：一个嵌套循环，里面有减法和绝对值
编译器不知道的：
  - 这是一个 16×16 块的 SAD，是视频编码中调用最频繁的函数
  - CPU 有专门的 SAD 指令（x86: PSADBW, ARM: UABD+UADALP）
  - 可以用水平累加指令一步完成求和
  - 可以同时计算多个候选块的 SAD（多参考帧）
```

### 理由三：特殊指令无法从 C 代码自动推导

现代 CPU 有很多"特殊用途"指令，编译器很难从普通 C 代码中推导出应该使用它们：

| 特殊指令 | 功能 | C 代码中的等价操作 | 编译器能自动用？ |
|----------|------|-------------------|-----------------|
| `PSADBW` (x86) | 8字节绝对差之和 | 循环+abs+累加 | 极少数情况 |
| `PMADDUBSW` (x86) | 无符号×有符号+水平加 | 乘法+加法组合 | ❌ 几乎不会 |
| `PSHUFB` (x86) | 按字节查表/重排 | 复杂的索引操作 | ❌ |
| `UABD` (ARM) | 无符号绝对差 | abs(a-b) | 有时可以 |
| `TBL/TBX` (ARM) | 字节查表 | 数组索引 | ❌ |
| `UDOT/SDOT` (ARMv8.2) | 点积累加 | 乘加循环 | 较新编译器可以 |

### 理由四：寄存器分配和指令调度的"人类直觉"

```
编译器的寄存器分配是"通用算法"（如图着色法），
但人类可以根据具体算法的数据流，做出更优的安排：

例如：流水线化（Software Pipelining）

编译器生成的代码：
  加载数据A → 处理A → 存储A → 加载数据B → 处理B → 存储B
  [等待内存]   [计算]   [等待]   [等待内存]   [计算]   [等待]

手写汇编的流水线：
  加载A → 加载B → 处理A → 加载C → 处理B → 存储A → 处理C → 存储B → ...
  [内存]   [内存]   [计算]   [内存]   [计算]   [内存]   [计算]   [内存]

→ 内存延迟被计算操作"隐藏"了，CPU 几乎没有空闲周期！
```

### 理由五：编译器不做"跨函数 SIMD 优化"

```
视频编码中，经常需要连续调用多个函数：
  1. 运动估计（SAD计算）
  2. 残差计算
  3. DCT变换
  4. 量化

编译器：每个函数独立编译，数据在寄存器和内存之间反复搬运
手写汇编：可以让数据一直留在 SIMD 寄存器中，跨步骤复用

  编译器：寄存器 → 内存 → 寄存器 → 内存 → 寄存器（大量 load/store）
  手写：  寄存器 → 寄存器 → 寄存器 → 寄存器（数据不落地！）
```

### 理由六：精确控制对齐和内存访问模式

```
x264 中的实际情况：
- 像素数据是 uint8_t（1字节）
- 一个宏块是 16×16 = 256 字节
- 参考帧可能不对齐（运动向量是半像素精度）

手写汇编可以：
1. 对齐的数据用对齐加载（更快）：vmovdqa
2. 不对齐的数据用非对齐加载：vmovdqu
3. 混合使用，编译器通常只能统一用非对齐加载（更慢但安全）
```

### 总结：编译器 vs 手写汇编

```
┌─────────────────────────────────────────────────────────────┐
│                    性能优化的层次                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  算法优化（选择更好的算法）     ← 人类做，收益最大            │
│       │                                                     │
│       ▼                                                     │
│  数据结构优化（缓存友好等）     ← 人类做                     │
│       │                                                     │
│       ▼                                                     │
│  编译器优化（-O3, LTO, PGO）   ← 编译器做，覆盖80%场景      │
│       │                                                     │
│       ▼                                                     │
│  Intrinsics（C中嵌入SIMD）     ← 人类指导，编译器辅助       │
│       │                                                     │
│       ▼                                                     │
│  手写汇编                       ← 人类完全控制，榨干最后20%  │
│                                                             │
└─────────────────────────────────────────────────────────────┘

x264 的选择：对热点函数（占运行时间80%+的SAD/DCT/MC等），
            用手写汇编榨干最后的性能。
```

## 三、SIMD 指令集速览

### 3.1 什么是 SIMD？

```
SIMD = Single Instruction, Multiple Data（单指令多数据）

普通标量指令：
  ADD R1, R2, R3    → 1次加法，处理1个数据

SIMD 指令：
  VADD.8 Q0, Q1, Q2  → 1次加法，同时处理16个字节！

类比：
  标量 = 一个人搬砖，一次搬1块
  SIMD = 一个人开叉车，一次搬16块
```

### 3.2 主流 SIMD 指令集对比

| 指令集 | 平台 | 寄存器宽度 | 一次处理 | 代表 CPU |
|--------|------|-----------|----------|----------|
| **SSE2** | x86 | 128位 | 16字节 / 4个float | Pentium 4+ |
| **AVX2** | x86 | 256位 | 32字节 / 8个float | Haswell+ (2013) |
| **AVX-512** | x86 | 512位 | 64字节 / 16个float | Skylake-X+ |
| **AMX** | x86 | 矩阵(16×64B) | 1024字节 | Sapphire Rapids+ (2023) |
| **NEON** | ARM | 128位 | 16字节 / 4个float | ARMv7+ / 所有手机 |
| **SVE/SVE2** | ARM | 128~2048位(可变) | 可变 | ARMv9 / 新服务器 |

### 3.3 x264 中的 SIMD 使用概况

```
x264 源码结构（汇编相关）：
x264/
├── common/
│   ├── x86/          ← x86 汇编（SSE2/AVX2/AVX-512）
│   │   ├── sad-a.asm         ← SAD 计算
│   │   ├── pixel-a.asm       ← 像素操作
│   │   ├── dct-a.asm         ← DCT 变换
│   │   ├── mc-a.asm          ← 运动补偿
│   │   └── quant-a.asm       ← 量化
│   │
│   ├── aarch64/      ← ARM64 汇编（NEON）
│   │   ├── sad-a.S           ← SAD 计算
│   │   ├── pixel-a.S         ← 像素操作
│   │   ├── dct-a.S           ← DCT 变换
│   │   └── mc-a.S            ← 运动补偿
│   │
│   ├── pixel.c       ← C 语言参考实现（用于对比/回退）
│   ├── dct.c
│   └── mc.c
│
└── encoder/
    └── ...           ← 编码器主逻辑（纯C）
```

> x264 的策略：**运行时检测 CPU 能力，动态选择最优实现**。
> 同一个函数有 C / SSE2 / AVX2 / NEON 多个版本，启动时自动选最快的。

## 四、x264 源码实战分析

### 4.1 案例一：SAD（Sum of Absolute Differences）

SAD 是视频编码中**调用频率最高**的函数——运动估计时，每个宏块要和数十个候选位置比较，每次比较都要算 SAD。

#### C 语言参考实现

```c
// x264/common/pixel.c（简化版）
static int pixel_sad_16x16(uint8_t *pix1, intptr_t stride1,
                           uint8_t *pix2, intptr_t stride2) {
    int sum = 0;
    for (int y = 0; y < 16; y++) {
        for (int x = 0; x < 16; x++) {
            sum += abs(pix1[x] - pix2[x]);
        }
        pix1 += stride1;
        pix2 += stride2;
    }
    return sum;
}
```

**性能分析**：
- 16×16 = 256 次减法 + 256 次绝对值 + 256 次累加
- 每次只处理 1 个像素（标量）
- 循环开销：256 次循环判断和跳转

#### x86 AVX2 手写汇编实现

```asm
; x264/common/x86/sad-a.asm（简化注释版）
; 函数：pixel_sad_16x16_avx2
; 一次处理 16 字节 = 一整行！

pixel_sad_16x16_avx2:
    pxor     xmm0, xmm0          ; 累加器清零
    mov      ecx, 16             ; 循环16行

.loop:
    ; 加载一整行（16字节）的 pix1 和 pix2
    vmovdqu  xmm1, [rdi]         ; xmm1 = pix1 的一行（16像素）
    vmovdqu  xmm2, [rsi]         ; xmm2 = pix2 的一行（16像素）
    
    ; PSADBW：一条指令完成 16 个字节的 |a-b| 求和！
    ; 这条指令做了：abs(a[0]-b[0]) + abs(a[1]-b[1]) + ... + abs(a[15]-b[15])
    psadbw   xmm1, xmm2          ; xmm1 = [高8字节SAD, 低8字节SAD]
    
    ; 累加到总和
    paddq    xmm0, xmm1
    
    ; 移动到下一行
    add      rdi, r8             ; rdi += stride1
    add      rsi, r9             ; rsi += stride2
    dec      ecx
    jnz      .loop

    ; 水平求和（高64位 + 低64位）
    movhlps  xmm1, xmm0
    paddq    xmm0, xmm1
    movd     eax, xmm0           ; 返回结果
    ret
```

**为什么快？**

```
C 版本 vs 汇编版本对比：

C 版本（编译器优化后）：
  每行：16次 load + 16次 sub + 16次 abs + 16次 add = 64 条指令
  16行：64 × 16 = 1024 条指令（最少）
  + 循环开销约 100 条

汇编版本：
  每行：2次 load + 1次 PSADBW + 1次 PADDQ = 4 条指令
  16行：4 × 16 = 64 条指令
  + 循环开销约 48 条

加速比：约 1024 / 64 ≈ 16 倍！

关键：PSADBW 这一条指令 = C 代码中的 48 条指令
     （16次减法 + 16次绝对值 + 15次加法）
```

### 4.2 案例二：SATD（Hadamard 变换 + 绝对值求和）

SATD 比 SAD 更精确（考虑了频域信息），但计算量更大。x264 在模式决策中大量使用 SATD。

#### C 语言参考实现

```c
// 4×4 Hadamard 变换 + 绝对值求和
static int satd_4x4(uint8_t *pix1, intptr_t stride1,
                    uint8_t *pix2, intptr_t stride2) {
    int16_t tmp[4][4], d[4][4];
    
    // 1. 计算残差
    for (int i = 0; i < 4; i++)
        for (int j = 0; j < 4; j++)
            tmp[i][j] = pix1[i*stride1+j] - pix2[i*stride2+j];
    
    // 2. 水平 Hadamard 变换
    for (int i = 0; i < 4; i++) {
        int a0 = tmp[i][0] + tmp[i][1];
        int a1 = tmp[i][2] + tmp[i][3];
        int a2 = tmp[i][0] - tmp[i][1];
        int a3 = tmp[i][2] - tmp[i][3];
        d[i][0] = a0 + a1;
        d[i][1] = a2 + a3;
        d[i][2] = a0 - a1;
        d[i][3] = a2 - a3;
    }
    
    // 3. 垂直 Hadamard 变换
    int sum = 0;
    for (int j = 0; j < 4; j++) {
        int a0 = d[0][j] + d[1][j];
        int a1 = d[2][j] + d[3][j];
        int a2 = d[0][j] - d[1][j];
        int a3 = d[2][j] - d[3][j];
        sum += abs(a0 + a1) + abs(a2 + a3) + abs(a0 - a1) + abs(a2 - a3);
    }
    return sum >> 1;
}
```

#### 汇编优化思路

```
手写汇编的优化策略：

1. 把 4×4 矩阵整体放入 SIMD 寄存器
   → 4行数据 = 4个 64位值 → 刚好放入一个 128位寄存器的4个16位通道

2. Hadamard 变换用"蝶形运算"实现
   → 加法和减法可以用 SIMD 的 ADD/SUB 并行完成
   → 4个元素的变换 = 2条 SIMD 指令（而不是8条标量指令）

3. 转置操作用 SIMD shuffle 指令
   → 水平变换后需要转置再做垂直变换
   → 标量需要逐个搬运，SIMD 一条 shuffle 搞定

4. 绝对值求和用 PABSW + PHADDW
   → 一条指令完成4个绝对值
   → 一条指令完成水平累加

结果：C 版本约 120 条指令 → 汇编约 20 条指令，加速 6 倍
```

### 4.3 案例三：运动补偿插值（MC, Motion Compensation）

视频编码支持"半像素"和"四分之一像素"精度的运动向量，需要对参考帧进行插值。

#### C 语言参考实现

```c
// 半像素水平插值（6-tap FIR 滤波器）
// H.264 标准规定的滤波器系数：[1, -5, 20, 20, -5, 1] / 32
static void mc_hpel_h(uint8_t *dst, uint8_t *src, intptr_t stride, int width, int height) {
    for (int y = 0; y < height; y++) {
        for (int x = 0; x < width; x++) {
            int val = src[x-2] - 5*src[x-1] + 20*src[x] + 20*src[x+1] 
                    - 5*src[x+2] + src[x+3];
            dst[x] = clip_uint8((val + 16) >> 5);  // 四舍五入并裁剪到 [0,255]
        }
        src += stride;
        dst += stride;
    }
}
```

**为什么编译器优化不好？**

```
编译器面临的困难：
1. 每个输出像素依赖6个输入像素 → 数据依赖复杂
2. clip_uint8 涉及条件判断 → 分支预测开销
3. 乘以常数 5 和 20 → 编译器可能用 shift+add，但不一定最优
4. 相邻像素的输入有重叠（src[x+1] 是下一个像素的 src[x]）
   → 编译器不一定能利用这个数据复用
```

#### ARM NEON 手写汇编的优化策略

```
手写汇编的做法：

1. 一次加载 16+5 = 21 个字节（一行的所有需要数据）
2. 用 VEXT 指令生成6个偏移版本：
   v0 = src[x-2 .. x+13]   (16字节)
   v1 = src[x-1 .. x+14]   (VEXT #1)
   v2 = src[x   .. x+15]   (VEXT #2)
   v3 = src[x+1 .. x+16]   (VEXT #3)
   v4 = src[x+2 .. x+17]   (VEXT #4)
   v5 = src[x+3 .. x+18]   (VEXT #5)

3. 用 VMULL/VMLAL 做乘累加：
   result = v0*1 + v1*(-5) + v2*20 + v3*20 + v4*(-5) + v5*1
   → 16个像素同时计算！

4. 用 VQRSHRUN 一步完成：右移5位 + 四舍五入 + 饱和裁剪到 [0,255]
   → 替代了 C 代码中的 (val+16)>>5 和 clip_uint8

结果：16个像素的插值 = 约10条 NEON 指令
     C 版本：16个像素 = 约 16×10 = 160 条标量指令
     加速比：约 16 倍
```

### 4.4 案例四：DCT 变换（离散余弦变换）

H.264 使用 4×4 整数 DCT 变换，是编码流水线中的核心步骤。

#### C 语言参考实现

```c
// 4×4 DCT 变换（H.264 整数近似）
static void dct4x4(int16_t d[16]) {
    int16_t tmp[16];
    
    // 水平变换
    for (int i = 0; i < 4; i++) {
        int s03 = d[i*4+0] + d[i*4+3];
        int d03 = d[i*4+0] - d[i*4+3];
        int s12 = d[i*4+1] + d[i*4+2];
        int d12 = d[i*4+1] - d[i*4+2];
        tmp[i*4+0] = s03 + s12;
        tmp[i*4+1] = 2*d03 + d12;
        tmp[i*4+2] = s03 - s12;
        tmp[i*4+3] = d03 - 2*d12;
    }
    
    // 垂直变换（同样的蝶形运算，方向不同）
    for (int j = 0; j < 4; j++) {
        int s03 = tmp[0*4+j] + tmp[3*4+j];
        int d03 = tmp[0*4+j] - tmp[3*4+j];
        int s12 = tmp[1*4+j] + tmp[2*4+j];
        int d12 = tmp[1*4+j] - tmp[2*4+j];
        d[0*4+j] = s03 + s12;
        d[1*4+j] = 2*d03 + d12;
        d[2*4+j] = s03 - s12;
        d[3*4+j] = d03 - 2*d12;
    }
}
```

#### 汇编优化的核心技巧

```
SIMD 优化 DCT 的关键洞察：

1. 4×4 矩阵刚好放入 4 个 64位 SIMD 通道
   → 每行4个 int16 = 64位 → 4行 = 256位 = 一个 YMM 寄存器

2. 蝶形运算天然适合 SIMD：
   s03 = row[0] + row[3]  →  VADD 一条指令，4个元素同时算
   d03 = row[0] - row[3]  →  VSUB 一条指令，4个元素同时算

3. 转置用 VTRN/VZIP 指令（ARM）或 PUNPCKL/SHUFPS（x86）
   → 4×4 转置只需 4 条指令（标量需要 12 次交换）

4. "×2" 用左移1位实现：VSHL #1

C 版本：约 80 条标量指令
汇编版本：约 16 条 SIMD 指令
加速比：约 5 倍
```

### 4.5 案例五：量化（Quantization）与反量化

```c
// H.264 量化（简化版）
static void quant_4x4(int16_t dct[16], int mf, int bias) {
    for (int i = 0; i < 16; i++) {
        if (dct[i] > 0)
            dct[i] = (dct[i] * mf + bias) >> 16;
        else
            dct[i] = -(((-dct[i]) * mf + bias) >> 16);
    }
}
```

**编译器的困难**：`if-else` 分支导致向量化困难。

**汇编的解法**：

```
SIMD 无分支量化：

1. 取绝对值：PABSW → 去掉符号
2. 乘以量化因子：PMULHUW → 无符号高位乘法（等效于 ×mf >> 16）
3. 加偏置后右移：PADDW + PSRAW
4. 恢复符号：PSIGNW → 根据原始符号恢复正负

4条指令完成16个系数的量化，无任何分支！
C 版本有16个 if-else → 可能导致分支预测失败 → 流水线停顿
```

## 五、ARM NEON 实例精讲

### 5.1 NEON 基础知识

```
ARM NEON 寄存器：
  - 32 个 128位寄存器：V0 ~ V31（AArch64）
  - 每个寄存器可以看作：
    16 × 8位 (uint8)   → 处理像素
    8 × 16位 (int16)   → 处理 DCT 系数
    4 × 32位 (int32)   → 处理累加器
    2 × 64位 (int64)   → 处理大数值

常用 NEON 指令（视频编码相关）：
  LD1  {v0.16b}, [x0]     → 加载16字节
  UABD v0.16b, v1.16b, v2.16b  → 无符号绝对差（|a-b|）
  UADALP v0.8h, v1.16b    → 绝对差累加（成对相加）
  EXT  v0.16b, v1.16b, v2.16b, #n  → 字节提取（滑动窗口）
  SQRSHRUN v0.8b, v1.8h, #5  → 饱和右移+窄化（插值用）
```

### 5.2 实例：16×16 SAD 的 NEON 实现

```asm
// x264/common/aarch64/sad-a.S（简化注释版）
// 函数原型：int sad_16x16_neon(uint8_t *pix1, intptr_t stride1,
//                              uint8_t *pix2, intptr_t stride2)

function sad_16x16_neon
    // 初始化累加器
    movi    v0.4s, #0           // v0 = [0, 0, 0, 0]（32位×4 累加器）
    
    mov     w4, #16             // 循环计数器 = 16行
.loop_sad16:
    // 加载 pix1 和 pix2 的一行（各16字节）
    ld1     {v1.16b}, [x0], x1  // v1 = pix1[0..15], x0 += stride1
    ld1     {v2.16b}, [x2], x3  // v2 = pix2[0..15], x2 += stride2
    
    // 计算绝对差：|pix1[i] - pix2[i]|，16个像素同时算
    uabd    v3.16b, v1.16b, v2.16b   // v3[i] = |v1[i] - v2[i]|
    
    // 累加到32位累加器（防止溢出）
    // uadalp: 成对相加并累加 (8位→16位)
    uadalp  v0.8h, v3.16b      // v0 的 8 个 16位通道分别累加
    
    subs    w4, w4, #1
    b.ne    .loop_sad16
    
    // 水平求和：8个16位 → 1个32位结果
    uaddlv  h0, v0.8h          // 所有通道求和
    fmov    w0, s0             // 结果放入返回寄存器
    ret
endfunc
```

**逐行解析**：

```
指令数对比：
  C 版本（标量）：每行约 64 条指令 × 16 行 = 1024+ 条
  NEON 版本：每行 3 条核心指令 × 16 行 + 收尾 = 约 52 条

关键指令解析：
  UABD（Unsigned Absolute Difference）：
    → 一条指令完成 16 个字节的 |a-b| 计算
    → 等价于 C 代码中的 16 次减法 + 16 次 abs()

  UADALP（Unsigned Add and Accumulate Long Pairwise）：
    → 把 16 个 8位结果成对相加，累加到 8 个 16位通道
    → 防止溢出（16个 255 的和 = 4080，超过 8 位范围）
    
  UADDLV（Unsigned Add Long across Vector）：
    → 把所有通道的值加在一起，得到最终标量结果
```

### 5.3 实例：半像素插值的 NEON 实现

```asm
// 半像素水平插值 NEON 实现（简化版）
// 6-tap 滤波器：[1, -5, 20, 20, -5, 1] / 32

function mc_hpel_filter_h_neon
    // 预加载滤波器系数到寄存器
    movi    v20.8h, #20         // 系数 20
    movi    v21.8h, #5          // 系数 5（负号在运算中处理）
    
.loop_row:
    // 加载一行 + 额外5字节（滤波器需要左右各2~3个像素）
    ld1     {v0.16b, v1.16b}, [x1]  // 加载 32 字节
    
    // 用 EXT 生成6个偏移版本
    ext     v2.16b, v0.16b, v1.16b, #0   // src[x-2]
    ext     v3.16b, v0.16b, v1.16b, #1   // src[x-1]
    ext     v4.16b, v0.16b, v1.16b, #2   // src[x]
    ext     v5.16b, v0.16b, v1.16b, #3   // src[x+1]
    ext     v6.16b, v0.16b, v1.16b, #4   // src[x+2]
    ext     v7.16b, v0.16b, v1.16b, #5   // src[x+3]
    
    // 扩展到 16 位进行计算（避免溢出）
    uxtl    v10.8h, v2.8b       // src[x-2] → 16位
    uxtl    v11.8h, v3.8b       // src[x-1] → 16位
    uxtl    v12.8h, v4.8b       // src[x]   → 16位
    uxtl    v13.8h, v5.8b       // src[x+1] → 16位
    uxtl    v14.8h, v6.8b       // src[x+2] → 16位
    uxtl    v15.8h, v7.8b       // src[x+3] → 16位
    
    // 计算：1×src[x-2] - 5×src[x-1] + 20×src[x] + 20×src[x+1] - 5×src[x+2] + 1×src[x+3]
    add     v16.8h, v10.8h, v15.8h       // src[x-2] + src[x+3]  (系数1)
    add     v17.8h, v12.8h, v13.8h       // src[x] + src[x+1]
    add     v18.8h, v11.8h, v14.8h       // src[x-1] + src[x+2]
    
    mla     v16.8h, v17.8h, v20.8h       // += 20 × (src[x] + src[x+1])
    mls     v16.8h, v18.8h, v21.8h       // -= 5 × (src[x-1] + src[x+2])
    
    // 右移5位 + 饱和裁剪到 [0, 255]
    sqrshrun v0.8b, v16.8h, #5   // 一条指令完成：右移+四舍五入+饱和+窄化
    
    // 存储结果
    st1     {v0.8b}, [x0], x2
    
    // ... 处理高8字节（类似）...
    
    subs    w5, w5, #1
    b.ne    .loop_row
    ret
endfunc
```

**关键优化点**：

| 优化点 | C 代码做法 | NEON 做法 | 收益 |
|--------|-----------|-----------|------|
| 生成偏移数据 | 每个像素单独索引 | EXT 指令滑动窗口 | 避免重复加载 |
| 乘以常数 | 通用乘法 | MLA/MLS（乘累加/乘累减） | 融合运算 |
| 裁剪到 [0,255] | if (val<0) val=0; if (val>255) val=255; | SQRSHRUN（饱和窄化） | 无分支 |
| 四舍五入 | (val + 16) >> 5 | SQRSHRUN 自带四舍五入 | 省一条加法 |

### 5.4 实例：4×4 Hadamard 变换 NEON 实现

```asm
// 4×4 SATD 的 NEON 核心（Hadamard 变换部分）
// 输入：v0-v3 各含4个 int16 值（4×4 矩阵的4行）

// ---- 水平 Hadamard 蝶形运算 ----
    // 第一级蝶形
    add     v4.4h, v0.4h, v3.4h     // a = row0 + row3
    sub     v5.4h, v0.4h, v3.4h     // b = row0 - row3
    add     v6.4h, v1.4h, v2.4h     // c = row1 + row2
    sub     v7.4h, v1.4h, v2.4h     // d = row1 - row2
    
    // 第二级蝶形
    add     v0.4h, v4.4h, v6.4h     // out0 = a + c
    sub     v2.4h, v4.4h, v6.4h     // out2 = a - c
    add     v1.4h, v5.4h, v7.4h     // out1 = b + d
    sub     v3.4h, v5.4h, v7.4h     // out3 = b - d

// ---- 转置 4×4 矩阵 ----
    trn1    v4.4h, v0.4h, v1.4h     // 交错取元素
    trn2    v5.4h, v0.4h, v1.4h
    trn1    v6.4h, v2.4h, v3.4h
    trn2    v7.4h, v2.4h, v3.4h
    trn1    v0.2s, v4.2s, v6.2s     // 完成转置
    trn2    v2.2s, v4.2s, v6.2s
    trn1    v1.2s, v5.2s, v7.2s
    trn2    v3.2s, v5.2s, v7.2s

// ---- 垂直 Hadamard 蝶形运算（同上）----
    add     v4.4h, v0.4h, v3.4h
    sub     v5.4h, v0.4h, v3.4h
    add     v6.4h, v1.4h, v2.4h
    sub     v7.4h, v1.4h, v2.4h
    add     v0.4h, v4.4h, v6.4h
    sub     v2.4h, v4.4h, v6.4h
    add     v1.4h, v5.4h, v7.4h
    sub     v3.4h, v5.4h, v7.4h

// ---- 绝对值求和 ----
    abs     v0.4h, v0.4h            // 取绝对值
    abs     v1.4h, v1.4h
    abs     v2.4h, v2.4h
    abs     v3.4h, v3.4h
    add     v0.4h, v0.4h, v1.4h    // 累加
    add     v0.4h, v0.4h, v2.4h
    add     v0.4h, v0.4h, v3.4h
    addv    h0, v0.4h               // 水平求和得到最终 SATD
```

**指令数对比**：

```
C 版本：约 120 条标量指令（含循环开销）

而且 NEON 版本没有任何内存访问（数据全在寄存器中），

```

## 六、x86 AVX/AVX2 实例精讲

### 6.1 AVX/AVX2 基础知识

```
x86 SIMD 寄存器演进：
  SSE:    XMM0~XMM15  (128位)  → 16字节 / 4个float
  AVX:    YMM0~YMM15  (256位)  → 32字节 / 8个float
  AVX-512: ZMM0~ZMM31 (512位)  → 64字节 / 16个float

x264 中常用的 AVX2 指令：
  VPSADBW   → 32字节的绝对差求和（SAD 核心）
  VPMADDUBSW → 无符号×有符号 + 水平加（插值核心）
  VPSHUFB   → 字节级重排（万能瑞士军刀）
  VPADDD    → 32位整数加法（累加器）
  VPERM2I128 → 128位通道间交换（AVX2 特有）
  VINSERTI128 → 插入128位到256位寄存器
```

### 6.2 实例：SAD 16×16 的 AVX2 实现

```asm
; x264/common/x86/sad-a.asm（AVX2 版本，简化注释）
; 利用 AVX2 的 256 位宽度，一次处理 2 行！

INIT_YMM avx2
cglobal pixel_sad_16x16, 4, 4, 4
    ; 参数：rdi=pix1, rsi=stride1, rdx=pix2, rcx=stride2
    
    vpxor    ymm0, ymm0, ymm0      ; 累加器清零
    
    ; 循环展开：每次处理2行（利用256位 = 2×128位）
    %rep 8                          ; 重复8次 = 16行
        ; 加载 pix1 的两行（各16字节），拼成一个 256位
        vmovdqu  xmm1, [rdi]        ; 低128位 = 第1行
        vinserti128 ymm1, ymm1, [rdi + rsi], 1  ; 高128位 = 第2行
        
        ; 加载 pix2 的两行
        vmovdqu  xmm2, [rdx]
        vinserti128 ymm2, ymm2, [rdx + rcx], 1
        
        ; VPSADBW：256位绝对差求和
        ; 一条指令处理 32 字节 = 2行×16像素！
        vpsadbw  ymm1, ymm1, ymm2
        
        ; 累加
        vpaddd   ymm0, ymm0, ymm1
        
        ; 移动到下2行
        lea      rdi, [rdi + rsi*2]
        lea      rdx, [rdx + rcx*2]
    %endrep
    
    ; 水平求和（4个64位通道 → 1个结果）
    vextracti128 xmm1, ymm0, 1     ; 取高128位
    vpaddd   xmm0, xmm0, xmm1      ; 高+低
    pshufd   xmm1, xmm0, 0x4E      ; 交换高低64位
    vpaddd   xmm0, xmm0, xmm1
    movd     eax, xmm0              ; 返回结果
    
    vzeroupper                       ; 清除 AVX 状态（避免性能惩罚）
    ret
```

**性能分析**：

```
AVX2 vs SSE2 vs C：

C 版本（标量）：
  256 次 load + 256 次 sub + 256 次 abs + 256 次 add
  ≈ 1024 条指令，约 300 个时钟周期

SSE2 版本（128位）：
  每行：2 load + 1 PSADBW + 1 PADDD = 4 条
  16行：64 条指令，约 50 个时钟周期

AVX2 版本（256位）：
  每2行：2 load + 1 VPSADBW + 1 VPADDD = 4 条
  16行（8次循环）：32 条指令，约 25 个时钟周期

加速比链：C → SSE2 ≈ 6倍，SSE2 → AVX2 ≈ 2倍，C → AVX2 ≈ 12倍
```

### 6.3 实例：运动补偿插值的 AVX2 实现

```asm
; 半像素水平插值 AVX2 版本
; 利用 VPMADDUBSW 实现高效的 FIR 滤波

cglobal mc_hpel_filter_h, 5, 7, 8
    ; 预加载滤波器系数
    ; H.264 6-tap: [1, -5, 20, 20, -5, 1]
    ; 技巧：把相邻系数打包成字节对，用 VPMADDUBSW 一步完成乘加
    vbroadcasti128 ymm7, [filter_coeff]  ; 系数 = [1,-5, 20,20, -5,1, ...]
    
.loop:
    ; 加载源数据（需要额外5字节用于滤波器窗口）
    vmovdqu  ymm0, [rsi - 2]     ; 加载 32 字节（从 src[x-2] 开始）
    
    ; 用 VPSHUFB 生成交错的字节对：
    ; 把 [a,b,c,d,e,f,...] 变成 [(a,b),(b,c),(c,d),(d,e),(e,f),...]
    ; 这样 VPMADDUBSW 就能一步完成 a×coef0 + b×coef1
    vpshufb  ymm1, ymm0, ymm_shuf_01   ; 字节对 (src[x-2], src[x-1])
    vpshufb  ymm2, ymm0, ymm_shuf_23   ; 字节对 (src[x], src[x+1])
    vpshufb  ymm3, ymm0, ymm_shuf_45   ; 字节对 (src[x+2], src[x+3])
    
    ; VPMADDUBSW：无符号字节 × 有符号字节 + 水平相加 → 16位结果
    ; 一条指令完成：src[x-2]*1 + src[x-1]*(-5) 对所有像素
    vpmaddubsw ymm1, ymm1, [coeff_1_n5]   ; 1×src[x-2] + (-5)×src[x-1]
    vpmaddubsw ymm2, ymm2, [coeff_20_20]  ; 20×src[x] + 20×src[x+1]
    vpmaddubsw ymm3, ymm3, [coeff_n5_1]   ; (-5)×src[x+2] + 1×src[x+3]
    
    ; 三部分相加
    vpaddw   ymm1, ymm1, ymm2
    vpaddw   ymm1, ymm1, ymm3
    
    ; 右移5位 + 饱和裁剪到 [0,255]
    vpaddw   ymm1, ymm1, [round_16]     ; +16（四舍五入）
    vpsraw   ymm1, ymm1, 5              ; >>5
    vpackuswb ymm1, ymm1, ymm1          ; 饱和窄化到 uint8
    
    ; 存储 16 字节结果
    vmovdqu  [rdi], xmm1
    
    add      rsi, rax            ; src += stride
    add      rdi, rax            ; dst += stride
    dec      ecx
    jnz      .loop
    
    vzeroupper
    ret
```

**VPMADDUBSW 的威力**：

```
VPMADDUBSW 指令做了什么？（这是 x264 汇编中最常用的"神器"）

输入A（无符号字节）：[a0, a1, a2, a3, a4, a5, ...]  (uint8)
输入B（有符号字节）：[b0, b1, b2, b3, b4, b5, ...]  (int8)

输出（有符号16位）：
  out[0] = a0×b0 + a1×b1    (16位)
  out[1] = a2×b2 + a3×b3    (16位)
  out[2] = a4×b4 + a5×b5    (16位)
  ...

一条指令完成了：
  - 32 次乘法（uint8 × int8）
  - 16 次加法（相邻乘积相加）
  = 48 次运算！

在 FIR 滤波中的应用：
  把像素和滤波器系数打包成字节对
  → 一条 VPMADDUBSW 就完成了 16 个像素的部分滤波计算
  → C 代码需要 16×6 = 96 次乘法 + 16×5 = 80 次加法
```

### 6.4 实例：量化的 AVX2 实现

```asm
; x264/common/x86/quant-a.asm（简化版）
; 无分支量化：利用 SIMD 的符号操作避免 if-else

cglobal quant_4x4, 3, 3, 5
    vmovdqu  ymm0, [rdi]          ; 加载 16 个 DCT 系数（int16）
    
    ; 1. 保存符号
    vpabsw   ymm1, ymm0           ; ymm1 = |dct[i]|（绝对值）
    
    ; 2. 乘以量化因子（无符号高位乘法）
    vpmulhuw ymm1, ymm1, [rsi]    ; ymm1 = |dct[i]| × mf >> 16
    
    ; 3. 加偏置
    vpaddw   ymm1, ymm1, [rdx]    ; ymm1 += bias
    
    ; 4. 右移（最终量化步骤）
    vpsrlw   ymm1, ymm1, 1        ; ymm1 >>= 1
    
    ; 5. 恢复符号（关键！无分支！）
    vpsignw  ymm1, ymm1, ymm0     ; if (dct[i] < 0) result = -result
                                    ; if (dct[i] == 0) result = 0
                                    ; if (dct[i] > 0) result = result
    
    ; 6. 存储结果
    vmovdqu  [rdi], ymm1
    
    ; 7. 检查是否全为零（用于跳过后续编码）
    vptest   ymm1, ymm1           ; 设置 ZF 标志
    setnz    al                    ; 返回：是否有非零系数
    
    vzeroupper
    ret
```

**无分支的秘密——VPSIGNW**：

```
C 代码的分支版本：
  if (dct[i] > 0)
      result = (abs_val * mf + bias) >> 16;
  else if (dct[i] < 0)
      result = -((abs_val * mf + bias) >> 16);
  else
      result = 0;

→ 16 个系数 = 16 次分支判断
→ 分支预测失败率约 30%（DCT系数正负随机）
→ 每次失败惩罚 15~20 个时钟周期

VPSIGNW 的做法：
  VPSIGNW dest, src, sign_source
  → 根据 sign_source 的每个元素的符号，决定 src 的符号
  → 正数不变，负数取反，零变零
  → 一条指令替代 16 个 if-else，零分支！
```

## 七、AMX 简介与未来趋势

### 7.1 什么是 AMX？

```
AMX = Advanced Matrix Extensions（高级矩阵扩展）
Intel 从 Sapphire Rapids（第4代至强，2023年）开始引入

AMX vs 传统 SIMD 的区别：

传统 SIMD（SSE/AVX）：
  操作对象：向量（一维数组）
  寄存器：XMM/YMM/ZMM（128/256/512位）
  一次操作：最多 64 字节

AMX：
  操作对象：矩阵（二维 Tile）
  寄存器：TMM0~TMM7（每个最大 16行×64字节 = 1024字节！）
  一次操作：整个矩阵乘法！

类比：
  SIMD = 一次搬一排砖
  AMX = 一次搬一面墙
```

### 7.2 AMX 的 Tile 寄存器

```
AMX Tile 配置：
  - 8 个 Tile 寄存器：TMM0 ~ TMM7
  - 每个 Tile 最大：16 行 × 64 字节 = 1024 字节
  - 支持的数据类型：
    INT8:  16×64 矩阵（1024个元素）
    BF16:  16×32 矩阵（512个元素）
    FP16:  16×32 矩阵（512个元素）

核心指令：
  LDTILECFG  → 配置 Tile 的行列数
  TILELOADD  → 从内存加载矩阵到 Tile
  TDPBSSD    → INT8 矩阵乘法（有符号×有符号→INT32累加）
  TDPBF16PS  → BF16 矩阵乘法（→FP32累加）
  TILESTORED → 将 Tile 存回内存
  TILERELEASE → 释放 Tile 资源
```

### 7.3 AMX 在视频编码中的潜力

```
AMX 目前主要用于 AI 推理（矩阵乘法），但在视频编码中也有潜力：

1. 批量 SAD 计算：
   - 运动估计需要计算一个块与多个候选位置的 SAD
   - 可以把多个候选块排成矩阵，用 AMX 一次算完
   - 传统：逐个计算 → AMX：批量矩阵运算

2. 大块 DCT/DST 变换：
   - H.266/VVC 支持最大 64×64 的变换块
   - 64×64 的 DCT = 矩阵乘法 → 天然适合 AMX

3. 帧内预测的矩阵运算：
   - 某些帧内预测模式涉及矩阵运算
   - AMX 可以一步完成

注意：截至目前（2024），x264 尚未使用 AMX，
     但下一代编码器（如 SVT-AV1）正在探索 AMX 优化。
```

### 7.4 AMX 代码示例（概念性）

```c
// AMX 矩阵乘法示例（使用 Intel Intrinsics）
#include <immintrin.h>

// 配置 Tile
struct __tilecfg {
    uint8_t palette_id;
    uint8_t start_row;
    uint8_t reserved[14];
    uint16_t colsb[16];  // 每个 tile 的列数（字节）
    uint8_t rows[16];    // 每个 tile 的行数
};

void amx_matmul_example() {
    struct __tilecfg cfg = {0};
    cfg.palette_id = 1;
    
    // 配置 TMM0: 16×64 (A矩阵，INT8)
    cfg.rows[0] = 16;
    cfg.colsb[0] = 64;
    
    // 配置 TMM1: 16×64 (B矩阵，INT8)
    cfg.rows[1] = 16;
    cfg.colsb[1] = 64;
    
    // 配置 TMM2: 16×16 (C矩阵，INT32结果)
    cfg.rows[2] = 16;
    cfg.colsb[2] = 64;  // 16个INT32 = 64字节
    
    _tile_loadconfig(&cfg);
    
    // 加载矩阵
    _tile_loadd(0, A, stride_a);   // TMM0 = A
    _tile_loadd(1, B, stride_b);   // TMM1 = B
    _tile_zero(2);                  // TMM2 = 0
    
    // 矩阵乘法：TMM2 += TMM0 × TMM1
    // 一条指令完成 16×64 × 64×16 = 16×16 的矩阵乘法！
    _tile_dpbssd(2, 0, 1);         // INT8 矩阵乘累加
    
    // 存储结果
    _tile_stored(2, C, stride_c);
    
    // 释放
    _tile_release();
}

// 对比：
// 传统 AVX2 做 16×16 矩阵乘法：需要约 100+ 条指令
// AMX 做 16×16 矩阵乘法：3条指令（load + dpbssd + store）
// 而且 AMX 的吞吐量远高于 AVX2
```

### 7.5 SIMD 技术演进时间线

```
x86 平台：
  1997: MMX        (64位)   → 多媒体起步
  1999: SSE        (128位)  → 浮点 SIMD
  2001: SSE2       (128位)  → 整数 SIMD 标配
  2011: AVX        (256位)  → 浮点翻倍
  2013: AVX2       (256位)  → 整数也翻倍 ← x264 主力
  2017: AVX-512    (512位)  → 再翻倍（但功耗高）
  2023: AMX        (矩阵)   → 矩阵运算时代

ARM 平台：
  2005: NEON       (128位)  → ARM 的 SIMD 标配 ← x264 ARM 主力
  2016: SVE        (可变宽) → 服务器级 SIMD
  2021: SVE2       (可变宽) → 通用计算增强
  2020: DOTPROD    (128位)  → 点积加速（AI推理）
  2022: SME        (矩阵)   → ARM 的 AMX 等价物

趋势：从"向量"走向"矩阵"，从"固定宽度"走向"可变宽度"
```

## 八、何时该写汇编？决策指南

### 8.1 决策流程图

```
你的代码需要优化吗？
│
├── 先 Profile！找到热点函数
│   └── 热点函数占总时间 < 5%？ → 不值得优化，优化其他地方
│
├── 热点函数占总时间 > 20%？
│   │
│   ├── 先试编译器优化：-O3 -march=native -flto -fprofile-generate
│   │   └── 够快了？ → 停止，不需要手写汇编
│   │
│   ├── 试 Intrinsics（C 中嵌入 SIMD）
│   │   └── 够快了？ → 停止，Intrinsics 比纯汇编好维护
│   │
│   └── Intrinsics 还不够快？
│       │
│       ├── 是否涉及特殊指令（PSADBW、PMADDUBSW 等）？ → 手写汇编
│       ├── 是否需要精确控制寄存器分配？ → 手写汇编
│       ├── 是否需要软件流水线？ → 手写汇编
│       └── 是否需要跨平台？ → 考虑用汇编宏（如 x264 的方式）
│
└── 结论：只有在"最后 20% 性能"真的重要时，才值得手写汇编
```

### 8.2 Intrinsics vs 纯汇编

| 对比项 | Intrinsics | 纯汇编 |
|--------|-----------|--------|
| 可读性 | ⭐⭐⭐ 较好（C 语法） | ⭐ 差（汇编语法） |
| 可移植性 | ⭐⭐ 同指令集内可移植 | ⭐ 完全不可移植 |
| 性能控制 | ⭐⭐ 编译器仍会"帮忙"调度 | ⭐⭐⭐ 完全控制 |
| 寄存器控制 | ⭐⭐ 编译器分配 | ⭐⭐⭐ 手动分配 |
| 维护成本 | ⭐⭐ 中等 | ⭐ 高 |
| 适用场景 | 大多数 SIMD 优化 | 极致性能热点 |

**Intrinsics 示例**（同样的 SAD 计算）：

```c
// 用 Intrinsics 写的 SAD（可读性好，但性能可能略逊于纯汇编）
#include <immintrin.h>

int sad_16x16_avx2(uint8_t *pix1, int stride1, uint8_t *pix2, int stride2) {
    __m256i sum = _mm256_setzero_si256();
    
    for (int i = 0; i < 16; i += 2) {
        // 加载两行
        __m128i row1a = _mm_loadu_si128((__m128i*)(pix1));
        __m128i row1b = _mm_loadu_si128((__m128i*)(pix1 + stride1));
        __m256i rows1 = _mm256_inserti128_si256(_mm256_castsi128_si256(row1a), row1b, 1);
        
        __m128i row2a = _mm_loadu_si128((__m128i*)(pix2));
        __m128i row2b = _mm_loadu_si128((__m128i*)(pix2 + stride2));
        __m256i rows2 = _mm256_inserti128_si256(_mm256_castsi128_si256(row2a), row2b, 1);
        
        // PSADBW
        sum = _mm256_add_epi64(sum, _mm256_sad_epu8(rows1, rows2));
        
        pix1 += stride1 * 2;
        pix2 += stride2 * 2;
    }
    
    // 水平求和
    __m128i lo = _mm256_castsi256_si128(sum);
    __m128i hi = _mm256_extracti128_si256(sum, 1);
    __m128i total = _mm_add_epi64(lo, hi);
    total = _mm_add_epi64(total, _mm_srli_si128(total, 8));
    return _mm_cvtsi128_si32(total);
}
```

### 8.3 x264 的工程实践：运行时分发

```c
// x264 的做法：编译时包含所有版本，运行时选最快的

// pixel.c 中的函数指针表
typedef struct {
    int (*sad[8])(uint8_t*, intptr_t, uint8_t*, intptr_t);
    // sad[0] = sad_4x4, sad[1] = sad_4x8, ..., sad[7] = sad_16x16
} x264_pixel_function_t;

// 初始化时根据 CPU 能力选择实现
void x264_pixel_init(int cpu, x264_pixel_function_t *pf) {
    // 默认：C 语言实现
    pf->sad[PIXEL_16x16] = pixel_sad_16x16;
    
    // 如果支持 SSE2
    if (cpu & X264_CPU_SSE2) {
        pf->sad[PIXEL_16x16] = x264_pixel_sad_16x16_sse2;
    }
    
    // 如果支持 AVX2（更快）
    if (cpu & X264_CPU_AVX2) {
        pf->sad[PIXEL_16x16] = x264_pixel_sad_16x16_avx2;
    }
}

// 使用时：直接调用函数指针，运行时自动用最快版本
int cost = h->pixf.sad[PIXEL_16x16](src, stride_src, ref, stride_ref);
```

```
这种设计的好处：
1. 一份代码支持所有 CPU（从老旧的 SSE2 到最新的 AVX-512）
2. 用户不需要关心自己的 CPU 支持什么指令集
3. 新增优化版本只需添加新函数 + 注册到分发表
4. C 版本始终作为"保底"和"正确性参考"
```
