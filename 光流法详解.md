## 0. 光流法要解决什么问题？

先看一个最朴素的问题：

> 你有两张连续拍摄的照片，中间只隔了 `1/30` 秒。你能否算出**照片里的每一个像素**从第一张到第二张"移动"了多少？

这个"每个像素的位移"就叫**光流（Optical Flow）**。它是无数计算机视觉任务的基石：

| 应用场景 | 光流的作用 |
|---------|-----------|
| **视频压缩**（H.264/H.265）| 只存储相邻帧的差异，用光流预测下一帧 |
| **动作识别**（打架检测、跌倒检测）| 从光流场提取运动特征 |
| **视觉里程计 / SLAM** | 估计相机自身的运动 |
| **视频稳定** | 找出抖动方向并反向补偿 |
| **光学防抖 / 慢动作插帧** | 中间帧 = 前后帧沿光流方向插值 |
| **自动驾驶** | 检测运动的行人、车辆 |

**核心问题**：光流本质是一个**二元函数**——对图像中的每个像素 `(x, y)`，输出一个二维向量 `(u, v)` 表示它的运动。

```
              光流场
   图像 I₁, I₂  ────────▶  每个像素 (x, y) 的位移 (u, v)
```

## 1. 直观理解：什么是"光流"

想象你在看一段视频：

```
第 t 帧：                 第 t+1 帧：
┌───────────┐            ┌───────────┐
│           │            │           │
│  🟠       │            │      🟠   │      橙球从左移到右
│           │            │           │
└───────────┘            └───────────┘
```

**橙球所在像素的光流** ≈ `(+3, 0)`（表示向右移动 3 个像素、上下不动）。

而**背景那些白色像素的光流** ≈ `(0, 0)`（因为白墙没变化）。

于是整幅图像会得到一张**光流场**：

```
每个像素 (x,y) 都有一个箭头 (u,v)，画出来像一场"箭雨"：

                    · · · · · · · · ·
                    · · · · · · · · ·
                    · · →→→ · · · · ·        ← 橙球所在位置的光流箭头
                    · · →→→ · · · · ·
                    · · · · · · · · ·
```

> 🔑 **一个必须澄清的区别**：
> - **真实运动场（Motion Field）** = 3D 世界中物体运动在图像平面的投影；
> - **光流（Optical Flow）** = 由**图像亮度变化**推断出的表观运动。
>
> 它们**通常一致**，但**不总是相等**。经典反例：一个纯白色的旋转球，光流几乎为 0（因为像素值不变），但真实运动场不为 0。光流本质是"看图像亮度的变化"。

## 2. 核心假设与基本方程

### 2.1 亮度恒定假设

光流法所有推导的起点，是一个非常朴素的假设：

> **亮度恒定假设（Brightness Constancy）**：同一个物体点在相邻两帧中的**亮度保持不变**。

用数学表达：设图像亮度函数为 `I(x, y, t)`，如果一个物体点在时刻 `t` 位于 `(x, y)`，在时刻 `t + Δt` 移动到了 `(x + Δx, y + Δy)`，那么：

```
I(x, y, t) = I(x + Δx, y + Δy, t + Δt)
```

**注意这个假设的隐含条件**：

1. **光照恒定**：光源不能突然变亮变暗；
2. **没有遮挡**：那个点在下一帧还得看得见；
3. **物体表面漫反射**：不同角度看颜色差不多（不能是镜面反射）；
4. **两帧间隔很短**：位移 `Δx, Δy` 是"小量"，可以做泰勒展开。

真实世界这些假设经常不成立，所以现代光流算法会用各种手段（金字塔、鲁棒代价函数、深度学习）弥补。

---

### 2.2 光流约束方程 (OFCE) 的推导

从亮度恒定出发，我们来推导**光流约束方程（Optical Flow Constraint Equation）**。

**Step 1**：把右边做一阶泰勒展开（因为 Δx, Δy, Δt 都很小）：

```
I(x + Δx, y + Δy, t + Δt)
    ≈ I(x, y, t) + ∂I/∂x · Δx + ∂I/∂y · Δy + ∂I/∂t · Δt + O(高阶项)
```

**Step 2**：代入亮度恒定 `I(x,y,t) = I(x+Δx, y+Δy, t+Δt)`，两边消去 `I(x,y,t)`：

```
0 = ∂I/∂x · Δx + ∂I/∂y · Δy + ∂I/∂t · Δt
```

**Step 3**：两边除以 `Δt`：

```
0 = ∂I/∂x · (Δx/Δt) + ∂I/∂y · (Δy/Δt) + ∂I/∂t
```

**Step 4**：定义速度分量 `u = Δx/Δt`（每秒移动多少像素）、`v = Δy/Δt`；同时缩写偏导数：

```
I_x = ∂I/∂x     (x 方向的空间梯度，"图像水平变化率")
I_y = ∂I/∂y    (y 方向的空间梯度)
I_t = ∂I/∂t    (时间梯度，"两帧的差")
```

得到**光流约束方程（OFCE）**：

```
┌────────────────────────────────────┐
│   I_x · u  +  I_y · v  +  I_t = 0   │
└────────────────────────────────────┘
```

或者写成向量形式：

```
∇I · (u, v)ᵀ + I_t = 0
```

其中 `∇I = (I_x, I_y)` 是图像的**空间梯度**。

---

### 2.3 孔径问题（Aperture Problem）

看看 OFCE：

```
I_x · u  +  I_y · v  +  I_t = 0
```

**这是一个方程，两个未知数（u 和 v）——欠定！**

几何上看：这个方程在 `(u, v)` 平面上是一条直线，直线上任何一点都是"合法解"。也就是说，只用**单个像素**的信息，我们只能确定运动**沿梯度方向的分量**，无法确定**沿等亮度线方向的分量**。

这就是著名的**孔径问题**：

```
     ┌─────────┐
     │         │
     │   ═════ │     ← 一根黑色横条
     │         │
     └─────────┘

   透过一个小圆孔看这根横条，你只能看出它"上下"运动了没有，
   看不出它"左右"运动了没有——因为左右移动时，圆孔里看到的
   还是一根一模一样的黑条。
```

**破解孔径问题的两条思路**：

| 思路 | 代表算法 | 补充的约束 |
|------|---------|----------|
| **局部法**：假设一小块邻域内运动一致 | Lucas-Kanade | 邻域内所有像素共享同一个 (u, v) |
| **全局法**：假设整幅图的运动"平滑" | Horn-Schunck | 整幅图的运动场变化不能太剧烈 |

## 3. 经典算法一：Lucas-Kanade（局部法）

### 3.1 核心思想：邻域内运动一致

Lucas-Kanade（下称 **LK**）的关键假设：

> 在一个小窗口（比如 3×3 或 5×5）内的所有像素，**共享同一个光流 `(u, v)`**。

这样一来，一个 3×3 窗口就有 9 个像素，每个像素都提供一条 OFCE 方程：

```
I_x(p₁)·u + I_y(p₁)·v = -I_t(p₁)
I_x(p₂)·u + I_y(p₂)·v = -I_t(p₂)
     ⋮                    ⋮
I_x(p₉)·u + I_y(p₉)·v = -I_t(p₉)
```

**9 个方程、2 个未知数**——超定！可以用**最小二乘**求解。

### 3.2 最小二乘求解

写成矩阵形式：

```
      A               x        =     b
┌ I_x(p₁)  I_y(p₁) ┐   ┌ u ┐    ┌ -I_t(p₁) ┐
│ I_x(p₂)  I_y(p₂) │ · │   │  = │ -I_t(p₂) │
│    ⋮        ⋮   │   └ v ┘    │    ⋮      │
└ I_x(p₉)  I_y(p₉) ┘            └ -I_t(p₉) ┘
   9×2                2×1         9×1
```

法方程解：

```
    ┌──────────────────────────────────────┐
    │   (u, v)ᵀ = (AᵀA)⁻¹ AᵀB              │
    └──────────────────────────────────────┘
```

其中：

```
              ┌  Σ I_x²      Σ I_x I_y  ┐               ┌ -Σ I_x I_t ┐
   AᵀA  =    │                       │      AᵀB  =   │            │
              └  Σ I_x I_y   Σ I_y²    ┘               └ -Σ I_y I_t ┘
```

（求和是在窗口内所有像素上进行）

**这个 2×2 矩阵 `M = AᵀA` 就是大名鼎鼎的"结构张量（Structure Tensor）"`——和 Harris 角点检测的核心矩阵完全相同。

---

### 3.3 什么样的点算得准？——好特征点

矩阵 `M = AᵀA` 是否可逆，决定了 LK 能否解出光流。看它的两个特征值 `λ₁ ≥ λ₂`：

| 情况 | 特征值 | 图像局部结构 | 光流可解性 |
|------|--------|-------------|-----------|
| **平坦区域**（一片白墙）| `λ₁ ≈ λ₂ ≈ 0` | 没有梯度 | ❌ M 不可逆，无解 |
| **边缘**（一根线）| `λ₁ 大，λ₂ ≈ 0` | 只有一个方向有梯度 | ❌ 孔径问题，只能解出一个分量 |
| **角点 / 纹理丰富**| `λ₁ ≈ λ₂ 都大` | 两个方向都有梯度 | ✅ 可以稳定求解 |

这也是为什么 LK 光流经常配合**Shi-Tomasi 角点检测**（`cv2.goodFeaturesToTrack`）：先找出图中最好追踪的点，只对这些点算光流——这就是**稀疏光流**。

## 4. 经典算法二：Horn-Schunck（全局法）

Horn-Schunck（HS）走另一条路：**整幅图共同求解，加上"运动场平滑"作为约束**。

它把光流场当作一个**能量最小化问题**：

```
E(u, v) = ∫∫  (I_xu + I_y v + I_t)²    +  α² · (|∇u|² + |∇v|²)   dx dy
                └─── 数据项 ───┘         └────── 平滑项 ──────┘
```

- **数据项**：让每个像素满足 OFCE；
- **平滑项**：让相邻像素的光流不要差太多（`α` 是权衡系数，控制"平滑"的强度）；
- 用**变分法**（欧拉-拉格朗日方程）+ 迭代求解。

**LK vs HS 对比**：

| 项 | Lucas-Kanade | Horn-Schunck |
|---|--------------|--------------|
| 类型 | 局部 | 全局 |
| 输出 | 稀疏（几百个点）或稠密 | 稠密（每个像素都有值） |
| 平滑性 | 无 | 强制光滑 |
| 计算量 | 小 | 大（需要迭代） |
| 抗噪 | 中 | 强 |
| 是否假设小位移 | 是 | 是 |

现代深度学习方法（FlowNet、RAFT 等）已经在稠密光流的精度上大幅超越 HS，但 LK 因为**又快又稳**，至今仍是稀疏光流的首选。

## 5. 稠密光流 vs 稀疏光流

| | 稀疏光流 (Sparse) | 稠密光流 (Dense) |
|---|-------------------|------------------|
| **计算对象** | 图像上少数好特征点（几百个）| 图像每一个像素 |
| **代表算法** | Lucas-Kanade | Horn-Schunck / Farneback / RAFT |
| **计算量** | 小（实时可用）| 大（GPU 加速常见）|
| **典型应用** | 目标追踪、SLAM 特征跟踪 | 视频压缩、慢动作插帧、动作识别 |

一个直观对比：

```
稀疏：只在角点处画箭头            稠密：每个像素都有一个箭头
     · · · · · · · ·                 →→→→→→→→→→→→→→→→
     · · →→ · · · · ·                →→→→→→→→→→→→→→→→
     · · · · · · · ·                 →→→→→→→→→→→→→→→→
     · →→ · · · →→ ·                 →→→→→→→→→→→→→→→→
```

## 6. 完整实例：手算一个 3×3 区域的光流

现在用一个**可以手算验证**的最小例子把上面的 LK 全流程跑一遍。

### 6.1 输入两帧图像

设我们有两张 3×3 的灰度图，其中"物体"在两帧之间**向右移动了 1 像素**（真实光流应为 `(u=1, v=0)`）：

**第 1 帧 `I₁`（时刻 t）：**

```
        x=0   x=1   x=2
      ┌─────┬─────┬─────┐
y=0   │  0  │ 100 │ 200 │
      ├─────┼─────┼─────┤
y=1   │  0  │ 100 │ 200 │
      ├─────┼─────┼─────┤
y=2   │  0  │ 100 │ 200 │
      └─────┴─────┴─────┘
```

**第 2 帧 `I₂`（时刻 t+1）：**（整个图像整体向右平移了 1 像素）

```
        x=0   x=1   x=2
      ┌─────┬─────┬─────┐
y=0   │  0  │  0  │ 100 │
      ├─────┼─────┼─────┤
y=1   │  0  │  0  │ 100 │
      ├─────┼─────┼─────┤
y=2   │  0  │  0  │ 100 │
      └─────┴─────┴─────┘
```

**观察**：这是一张"垂直边缘"图，梯度只沿 x 方向存在，y 方向为 0。这是一个**典型的孔径问题场景**——但因为是完整平移，我们仍然可以求出 u（v 由于孔径问题会有不确定性，但我们会看到最小二乘会给出一个合理答案）。

---

### 6.2 计算梯度 I_x, I_y, I_t

用**中心差分 / 前向差分**（这里用最简单的前向差分，只在中间像素 `(x=1, y=1)` 及其邻域上算）。为了简化，我们对 3×3 窗口的每个像素都用**在第 1 帧上的前向差分**计算 I_x 和 I_y，用 `I₂ - I₁` 计算 I_t。

**I_x（x 方向梯度，用第 1 帧的前向差分）**：

```
I_x(x, y) = I₁(x+1, y) - I₁(x, y)
```

对 x=0, x=1 有效；x=2 越界，为简化我们复制 x=1 的值：

```
        x=0   x=1   x=2
      ┌─────┬─────┬─────┐
y=0   │ 100 │ 100 │ 100 │
y=1   │ 100 │ 100 │ 100 │
y=2   │ 100 │ 100 │ 100 │
      └─────┴─────┴─────┘
```

**I_y（y 方向梯度）**：所有像素在 y 方向都相同 → 全 0。

```
        x=0   x=1   x=2
      ┌─────┬─────┬─────┐
y=0   │  0  │  0  │  0  │
y=1   │  0  │  0  │  0  │
y=2   │  0  │  0  │  0  │
      └─────┴─────┴─────┘
```

**I_t（时间梯度 = I₂ - I₁）**：

```
                          I₂           -           I₁
        x=0    x=1    x=2                x=0    x=1    x=2
y=0   [  0  ,   0  , 100 ]           y=0   [  0  , 100 , 200 ]
y=1   [  0  ,   0  , 100 ]     -     y=1   [  0  , 100 , 200 ]
y=2   [  0  ,   0  , 100 ]           y=2   [  0  , 100 , 200 ]

= I_t：
        x=0    x=1    x=2
y=0   [  0  , -100 , -100 ]
y=1   [  0  , -100 , -100 ]
y=2   [  0  , -100 , -100 ]
```

---

### 6.3 组装 Lucas-Kanade 方程组

窗口 = 整个 3×3 图（9 个像素），需要计算这 4 个量：

```
Σ I_x²      = ?
Σ I_y²     = ?
Σ I_x · I_y = ?
Σ I_x · I_t = ?
Σ I_y · I_t = ?
```

**Σ I_x²**：所有 9 个 I_x 都是 100 → `9 × 100² = 90000`

**Σ I_y²**：所有 I_y 都是 0 → `0`

**Σ I_x · I_y**：任何数乘 0 = 0 → `0`

**Σ I_x · I_t**：逐个计算

```
                   I_x      I_t     I_x·I_t
(0,0):            100    ×    0  =    0
(1,0):            100    × -100  = -10000
(2,0):            100    × -100  = -10000
(0,1):            100    ×    0  =    0
(1,1):            100    × -100  = -10000
(2,1):            100    × -100  = -10000
(0,2):            100    ×    0  =    0
(1,2):            100    × -100  = -10000
(2,2):            100    × -100  = -10000

Σ I_x · I_t = -60000
```

**Σ I_y · I_t**：I_y 全是 0 → `0`

---

### 6.4 解出光流 (u, v)

代入 LK 法方程：

```
    ┌ Σ I_x²      Σ I_x I_y ┐   ┌ u ┐     ┌ -Σ I_x I_t ┐
    │                     │ · │   │  =  │            │
    └ Σ I_x I_y   Σ I_y²   ┘   └ v ┘     └ -Σ I_y I_t ┘

    ┌ 90000    0 ┐   ┌ u ┐     ┌ 60000 ┐
    │            │ · │   │  =  │       │
    └     0    0 ┘   └ v ┘     └     0 ┘
```

**注意**：这个 M 矩阵是**奇异**的（第二行全 0）——正是**孔径问题**！它意味着 v 无法确定，可以是任何值。

**从第一行**：`90000 · u + 0 · v = 60000`，解得：

```
    u = 60000 / 90000 = 2/3 ≈ 0.667
```

**为什么不是 1（真实位移）？**——因为我们只用了 9 个像素、且用了粗糙的前向差分，属于近似解。如果换成中心差分、加大窗口、加金字塔，会越来越接近 1。**这正说明 LK 在实际工程里为什么需要金字塔——大位移（比如这里的 1 像素其实相对于 3×3 窗口已经算大）会破坏泰勒展开的近似**。

**v 的处理**：在真实实现里通常用**伪逆**（`np.linalg.lstsq` 或加一个正则项 `M + εI`）取得数值稳定的解 `v = 0`（因为它对应的第二行是 0=0，最小范数解就是 0）。

---

### 6.5 验证结果

- **理论值**：`(u, v) = (1, 0)`
- **手算结果**：`(u, v) ≈ (0.67, 0)`（v 靠伪逆得 0）

方向正确，v 分量正确，u 分量因为大位移下的一阶近似而偏小——这是**教科书级的展示**：

1. 光流方向**总是可以准确判断**（→ 向右）；
2. 光流大小**在小位移下准确**，大位移会低估；
3. **孔径问题真实存在**——纯垂直边缘的图，v 分量本来就无法确定。

**修正方法**（工程中标配）：

- **迭代 LK**：算出 (u, v) 后把 `I₂` 反向 warp，再次算残差光流，累加；
- **金字塔 LK**：把图像下采样到 1/2、1/4，先在小图上算大位移，再逐层细化。

---

### 6.6 拿到 (u, v) 之后能做什么？——常见应用场景

前面辛辛苦苦把 `(u, v)` 算出来了，那么**这个二维矢量到底能拿去干什么**？下面按"从简单到复杂"的顺序，把工程上最常见的用法归纳成 8 类，并说明每一类是怎么"消费" `(u, v)` 的。

> 记号约定：设某像素 `(x, y)` 在两帧之间的光流为 `(u, v)`，帧率为 `fps`（帧/秒），下一帧对应位置即 `(x+u, y+v)`。

---

#### ① 目标追踪（Object Tracking）

**做什么**：在第 1 帧圈出一个目标（比如人脸、车辆），要求在后续每一帧持续跟踪它的位置。

**怎么用 (u, v)**：

```
新位置 = 旧位置 + (u, v)
    (x_new, y_new) = (x_old + u, y_old + v)
```

- **稀疏 LK 追踪**：只对目标框内的角点算光流，用**多个点位移的中位数/RANSAC**估计整个目标框的运动（抗遮挡、抗噪）。
- **代表实现**：`cv2.calcOpticalFlowPyrLK` + Shi-Tomasi 角点。
- **典型场景**：SLAM 特征点跟踪、无人机跟拍、AR 贴图跟随。

---

#### ② 相机自身运动估计（视觉里程计 / SLAM）

**做什么**：相机装在机器人/车上，要通过图像反推"我这一帧相对上一帧移动了多少、转了多少"。

**怎么用 (u, v)**：

- 收集图像上多个点的光流 `{(u_i, v_i)}`；
- 结合**相机内参 K**（👉 参见相机标定文档），把光流反投影为 3D 运动约束；
- 用**8 点法 / 5 点法 / PnP** 求解相机的旋转 `R` 和平移 `t`。

**核心思想**：光流 = 相机运动在图像上的"投影痕迹"，反过来解方程就能恢复相机运动。

**典型场景**：ORB-SLAM、VINS-Mono、扫地机器人定位、AR 眼镜的 6DoF 追踪。

---

#### ③ 视频稳像（Video Stabilization）

**做什么**：手持拍摄的视频抖动厉害，要"把抖动去掉"。

**怎么用 (u, v)**：

1. 对每一帧的一堆特征点算光流；
2. 用 RANSAC 拟合出**全局仿射/单应变换 H**（描述整帧相对上一帧的抖动）；
3. 对 H 序列做**低通滤波**（把高频抖动滤掉，保留低频的平缓运动）；
4. 用滤波后的 H 反向 warp 每一帧 → 稳定视频。

**典型场景**：手机相机的"运动模式"、GoPro 后期防抖、YouTube Stabilizer。

---

#### ④ 慢动作插帧（Frame Interpolation）

**做什么**：30fps 的视频想变成 60fps / 120fps，需要"凭空造出中间帧"。

**怎么用 (u, v)**：

```
中间帧像素 I_{t+0.5}(x, y)
    ≈ 0.5 · I_t(x - 0.5u, y - 0.5v)  +  0.5 · I_{t+1}(x + 0.5u, y + 0.5v)
```

即：**沿光流方向做双向 warp + 混合**。

- 对每个像素都要有光流 → **需要稠密光流**（Farneback / RAFT）；
- 遮挡区域要额外处理（一侧看得见、另一侧看不见）。

**典型场景**：手机的"960fps 慢动作"、DAIN / RIFE 等 AI 插帧、电视的"运动补偿" MEMC。

---

#### ⑤ 视频压缩中的运动补偿（Motion Compensation）

**做什么**：H.264 / H.265 / AV1 编码器不想为每一帧存储完整像素，而是"存差异"。

**怎么用 (u, v)**：

- 编码器把当前帧切成 16×16 的**宏块**；
- 对每个宏块找到它在参考帧里的**运动矢量 MV**（其实就是块级光流）；
- 只存储 `MV + 残差`（预测误差），码率立刻下降 10~100 倍。

> 严格说编码器用的是**块匹配（Block Matching）**而非光流，但两者数学本质相同——都在找"这块像素在上一帧的哪里"。光流思想是运动补偿的理论根基。

**典型场景**：所有主流视频编码标准。

---

#### ⑥ 动作识别 / 行为分析（Action Recognition）

**做什么**：判断视频里的人在"打架、跌倒、挥手、跑步"等等。

**怎么用 (u, v)**：

- 把每一帧的光流场 `(u, v)` 当作一张**"运动图像"**；
- 常见做法：把 `u` 做 R 通道、`v` 做 G 通道、幅值 `√(u²+v²)` 做 B 通道，得到伪彩色光流图；
- 送进 CNN（Two-Stream Network、I3D、SlowFast）与 RGB 帧联合分类。

**典型场景**：安防监控中的跌倒检测、打架检测、抖音的动作分类、体育视频分析。

---

#### ⑦ 运动分割 / 前景提取（Motion Segmentation）

**做什么**：从固定摄像头视频中把"动的东西"抠出来（背景静止，前景运动）。

**怎么用 (u, v)**：

```
if √(u² + v²) > 阈值T:
    像素 (x, y) 属于前景
else:
    背景
```

- 更鲁棒的做法：先估计**全局背景光流**（常常≈0），把偏离全局的像素归为前景；
- 相比帧差法，光流能区分**运动方向**——两个反向运动的物体不会合并。

**典型场景**：交通监控中的车辆计数、运动目标追踪的初始化、体育回放中的球员分割。

---

#### ⑧ 深度估计与 3D 重建（Structure from Motion）

**做什么**：从单目相机的运动视频反推场景的 3D 结构。

**怎么用 (u, v)**：

- 相机做纯平移时，**光流大小 ∝ 1 / 深度**——远处像素动得慢、近处动得快（"运动视差"）；
- 结合相机运动（②）+ 光流大小 → 反推每个像素的深度 Z；
- 数学关系（简化版，仅平移 `t_z=0`, `t_x≠0`）：

```
    Z = f · t_x / u
```

**典型场景**：单目 SLAM 的稠密建图、无人机避障、AR 场景理解、自动驾驶的伪 3D 感知。

---

#### 应用总览：一张表看懂

| # | 应用 | 光流密度 | 典型算法 | 关键消费方式 |
|:--|:--|:--|:--|:--|
| ① | 目标追踪 | 稀疏 | LK + 角点 | `新位置 = 旧位置 + (u,v)` |
| ② | 视觉里程计 / SLAM | 稀疏 | LK + 8 点法/PnP | 反解相机 R, t |
| ③ | 视频稳像 | 稀疏 | LK + RANSAC + 低通滤波 | 拟合全局 H，反向 warp |
| ④ | 插帧慢动作 | 稠密 | Farneback / RAFT | 沿 (u,v) 双向 warp + 混合 |
| ⑤ | 视频编码 | 块级 | 块匹配 | 存 MV + 残差 |
| ⑥ | 动作识别 | 稠密 | Farneback / TV-L1 | 光流场作为 CNN 输入 |
| ⑦ | 运动分割 | 稠密 | 任何稠密光流 | 幅值阈值 / 与全局差异 |
| ⑧ | 深度 / 3D 重建 | 稠密 | LK + SfM | `Z ∝ 1 / |flow|` |

---

#### 一句话记忆

> **`(u, v)` 就是"每个像素下一步去哪"的路标**——顺着它走可以追踪、拟合它可以稳像、沿着它 warp 可以插帧、反解它可以恢复相机运动和 3D 结构。所有光流应用，本质上都是这个二维矢量的不同"消费方式"。

## 7. 工程实现要点

### 7.1 金字塔多尺度光流

**动机**：LK 假设"小位移"（因为用了泰勒一阶展开）。位移 20 像素这种情况直接算会失败。

**做法**：

```
     原图 512×512    ─── 1/2 ────▶  256×256  ─── 1/2 ────▶  128×128  ...
        ▲                                                       │
        │                                                       │
        │                                                       ▼
   累加得到最终光流  ◀─── 上采样 ×2 + 精修 ◀─── 上采样 ×2 + 精修 ◀── 在小图上先算大概
```

- 顶层小图上，一个大位移（如 20 像素）变成小位移（约 5 像素），LK 就成立；
- 逐层放大 + 用当前光流 warp 图像 + 算残差光流，最终得到高精度光流。

这就是 `cv2.calcOpticalFlowPyrLK` 里的 "Pyr"。

### 7.2 OpenCV 代码示例

**稀疏光流（LK + 金字塔）**：

```python
import cv2, numpy as np

cap = cv2.VideoCapture("video.mp4")
ret, prev = cap.read()
prev_gray = cv2.cvtColor(prev, cv2.COLOR_BGR2GRAY)

# ① 找好追踪的点（Shi-Tomasi 角点）
p0 = cv2.goodFeaturesToTrack(prev_gray, maxCorners=200,
                             qualityLevel=0.3, minDistance=7)

while True:
    ret, frame = cap.read()
    if not ret: break
    gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)

    # ② 计算这些点在新一帧的位置（金字塔 LK）
    p1, status, err = cv2.calcOpticalFlowPyrLK(
        prev_gray, gray, p0, None,
        winSize=(15, 15), maxLevel=2)

    # ③ 只保留追踪成功的点
    good_new = p1[status == 1]
    good_old = p0[status == 1]

    # 画箭头
    for (x1, y1), (x0, y0) in zip(good_new, good_old):
        cv2.arrowedLine(frame, (int(x0), int(y0)), (int(x1), int(y1)),
                        (0, 255, 0), 2)

    cv2.imshow("flow", frame)
    if cv2.waitKey(1) == 27: break

    prev_gray = gray
    p0 = good_new.reshape(-1, 1, 2)
```

**稠密光流（Farneback）**：

```python
flow = cv2.calcOpticalFlowFarneback(
    prev_gray, gray, None,
    pyr_scale=0.5, levels=3, winsize=15,
    iterations=3, poly_n=5, poly_sigma=1.2, flags=0)

# flow 形状 (H, W, 2)：flow[y, x] = (u, v)
```

## 8. 常见疑问 FAQ

### Q1：光流的单位是什么？

通常是**"像素每帧"**（`pixel/frame`）。要换成"米每秒"，需要知道相机内参、场景深度和帧率——这是视觉里程计要做的事。

### Q2：光流一定和真实运动一致吗？

不一定。经典反例：

- **纯白色旋转球**：像素亮度不变 → 光流为 0，但真实运动不为 0；
- **理发店旋转灯柱**：图案沿轴向"流动"，光流指向轴向，但柱子本身只是绕轴旋转；
- **光照突变**：整幅图突然变亮 → 光流全体乱套，但物体没动。

### Q3：LK 为什么必须假设小位移？

因为推导用的是**一阶泰勒展开**，只对无穷小位移严格成立。位移越大、二阶及以上项就越不能忽略。工程上用**金字塔 + 迭代**弥补。

### Q4：亮度恒定假设不成立时怎么办？

常见解法：

- **归一化**：用图像块的零均值归一化（NCC）代替原始亮度；
- **鲁棒代价**：用 Charbonnier / Huber 等函数代替平方误差；
- **梯度恒定假设**：假设梯度不变，而非亮度不变；
- **深度学习**：让网络自己去学（RAFT, FlowNet 等）。

### Q5：光流和"帧差法"有什么区别？

| | 帧差法 | 光流法 |
|--|--------|--------|
| 输出 | 一张二值图（哪里有变化）| 每个像素一个 2D 矢量 |
| 能否知道方向 | ❌ | ✅ |
| 能否知道速度 | ❌ | ✅ |
| 计算量 | 极小 | 中～大 |

### Q6：为什么 LK 常和角点检测配合？

因为 LK 需要**结构张量 M 可逆**，即两个特征值都够大——这正好是**角点**的定义。所以 Shi-Tomasi 角点 + LK 是天生一对。

### Q7：现在还需要学传统光流吗，深度学习不是更好吗？

需要。理由：

1. **实时场景**（无人机、AR/VR、嵌入式）没算力跑深度网络，LK 依然是首选；
2. **深度光流网络的训练目标、损失设计**都建立在传统理论上，不懂 OFCE 就看不懂 RAFT 论文；
3. **SLAM/VIO 系统**里 LK 追踪特征点仍是主流；
4. **快速原型**：`cv2.calcOpticalFlowPyrLK` 一行搞定。

---

## 9. 一句话总结

> **光流法 = 一个假设（亮度恒定）+ 一条方程（OFCE：`I_xu + I_y v + I_t = 0`）+ 一个补丁（额外约束破解孔径问题）**。
>
> - 如果补丁选"邻域内运动一致" → **Lucas-Kanade**（稀疏、快）；
> - 如果补丁选"运动场平滑" → **Horn-Schunck**（稠密、慢）；
> - 如果补丁交给神经网络 → **FlowNet / RAFT**（现代深度光流）。
>
> 无论算法多复杂，核心都是在解那条 `I_xu + I_y v + I_t = 0`——记住它，你就掌握了光流法的灵魂。
