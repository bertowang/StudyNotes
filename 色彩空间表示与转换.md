## 零、写给小白：30 分钟看懂所有色彩空间

> **本章目的**：在深入细节前，先用最朴素的语言把"色彩空间到底是什么、为什么有这么多种、转换公式到底在干嘛"讲清楚。看完本章，再看后面的推导就会非常顺畅。

### 0.1 一句话理解什么是"色彩空间"

```
最朴素的比喻：色彩空间 = 描述颜色的"坐标系"

  就像描述地球上的一个位置，可以有多种坐标系：
  ├── 经纬度坐标（GPS）
  ├── 北京街道地址（朝阳区xx路xx号）
  └── 直角坐标系（x, y, z 米）
  
  同一个位置，三种描述方式都对，只是用途不同：
  ├── GPS 适合机器导航
  ├── 街道地址适合人类沟通
  └── 直角坐标适合数学计算

  ───────────────────────────────────────────────

  颜色也是同样道理：同一个颜色（比如这朵红玫瑰），
  可以用不同的"坐标系"描述：
  
  ├── RGB：红=220, 绿=20,  蓝=60   （显示器友好）
  ├── YUV：亮=80, U=-30, V=120     （视频压缩友好）
  ├── HSV：色相=350°, 饱和=91%, 明度=86%（人类直觉友好）
  └── XYZ：X=0.41, Y=0.21, Z=0.06  （物理测量友好）

  这些都是同一朵红玫瑰，只是用了不同的"坐标系"。
  色彩空间转换 = 同一个颜色在不同坐标系之间的换算。
```

### 0.2 为什么会有那么多色彩空间？

每一种色彩空间的诞生都有一个**非常朴素的现实需求**。理解了需求，公式就容易理解了。

```
┌─────────────┬──────────────────────┬───────────────────────────────────┐
│ 色彩空间    │ 诞生的"质朴原始目的" │ 一句话解释                        │
├─────────────┼──────────────────────┼───────────────────────────────────┤
│ CIE XYZ     │ 给颜色一个"国际标准  │ 1931年科学家说："要科学研究颜色， │
│ (1931)      │ 度量衡"              │ 必须先有统一的、客观的颜色坐标"   │
├─────────────┼──────────────────────┼───────────────────────────────────┤
│ RGB         │ 让显示器能"画"出颜色 │ 显示器只有红绿蓝三种小灯泡，      │
│             │                      │ 必须告诉它"每个灯泡多亮"          │
├─────────────┼──────────────────────┼───────────────────────────────────┤
│ sRGB        │ 让所有显示器画出的   │ 1996年互联网普及，"我画的红色，   │
│ (1996)      │ 颜色"长得一样"       │ 你看到的也得是同一个红色"         │
├─────────────┼──────────────────────┼───────────────────────────────────┤
│ YUV/YCbCr   │ 省带宽 + 兼容黑白电视│ 黑白电视只看亮度，彩色电视多      │
│             │                      │ 加两个"色差"信号；亮度比颜色重要，│
│             │                      │ 可以让颜色少存一点（4:2:0）       │
├─────────────┼──────────────────────┼───────────────────────────────────┤
│ HSV/HSL     │ 让人能"直觉地"调颜色 │ 设计师不想算 RGB=(220,20,60)，    │
│             │                      │ 只想说"我要一个鲜艳一点的红"      │
├─────────────┼──────────────────────┼───────────────────────────────────┤
│ CIE Lab     │ 让"颜色之间的距离"   │ RGB 中差 10 个值，肉眼差异时大时小│
│ (1976)      │ 符合人眼感知         │ Lab 中差 10 个值，肉眼感觉差不多  │
├─────────────┼──────────────────────┼───────────────────────────────────┤
│ BT.2020     │ 表示更鲜艳的颜色     │ 4K/HDR 时代，sRGB 的色域不够用了  │
│ (2012)      │ （宽色域）           │ 比如纯纯的红、纯纯的绿，sRGB 画不出│
├─────────────┼──────────────────────┼───────────────────────────────────┤
│ PQ / HLG    │ 表示更亮、更暗的画面 │ SDR 最亮 100nit，HDR 要到 10000nit│
│ (2016)      │ （HDR）              │ 需要新的"亮度编码曲线"            │
├─────────────┼──────────────────────┼───────────────────────────────────┤
│ Oklab       │ Lab 的现代改良版     │ Lab 用了几十年发现还是不够准，    │
│ (2020)      │                      │ 重新设计一个对设计工具更友好的    │
└─────────────┴──────────────────────┴───────────────────────────────────┘

  关键认知：
  ├── 没有"最好的色彩空间"，只有"最适合某个任务的色彩空间"
  ├── 显示用 RGB、压缩用 YUV、调色用 HSV、对比用 Lab
  └── 转换是为了"换个角度看同一个颜色"，不会改变颜色本身
```

### 0.3 所有转换公式的统一原理

虽然后面的章节会出现各种矩阵和公式，看起来很复杂，但**所有转换其实只有三种类型**：

```
─────────────────────────────────────────────────────────────────────────
类型 ①：线性变换（矩阵乘法）—— 最常见
─────────────────────────────────────────────────────────────────────────

  形式：    ┌ Y' ┐   ┌ a₁₁ a₁₂ a₁₃ ┐ ┌ X ┐
            │ U' │ = │ a₂₁ a₂₂ a₂₃ │ │ Y │
            └ V' ┘   └ a₃₁ a₃₂ a₃₃ ┘ └ Z ┘

  质朴理解："新坐标 = 老坐标的加权混合"
  ├── 新颜色的每个分量，都是老颜色三个分量的"加权平均"
  └── 权重就是矩阵里的数字（怎么来的？后面章节有推导）

  应用：
  ├── RGB ↔ XYZ：矩阵乘法（不同 RGB 空间矩阵不同）
  ├── RGB ↔ YUV：矩阵乘法（BT.601/709/2020 矩阵不同）
  └── 不同 RGB 之间：矩阵乘法（如 sRGB ↔ Adobe RGB）

  典型例子（BT.709 RGB → YUV）：
       Y =  0.2126·R + 0.7152·G + 0.0722·B   ← 亮度（绿色权重最大）
       U = -0.1146·R - 0.3854·G + 0.5000·B   ← 蓝-亮度
       V =  0.5000·R - 0.4542·G - 0.0458·B   ← 红-亮度

       为什么 Y 里绿色权重最大？
       └── 因为人眼对绿色最敏感（视锥细胞 M 型最多）

─────────────────────────────────────────────────────────────────────────
类型 ②：非线性变换（Gamma 校正、对数变换等）
─────────────────────────────────────────────────────────────────────────

  形式：    Y_显示 = (Y_存储)^γ        （Gamma 曲线）
  或者：    Y_存储 = (Y_线性)^(1/γ)     （反 Gamma）

  质朴理解："人眼对暗部敏感、对亮部不敏感"
  ├── 8bit 只能存 256 个亮度值
  ├── 如果均匀分配，暗部不够细、亮部太奢侈
  └── 用 Gamma 曲线"压一压亮部、放大暗部"，就能省比特

  应用：
  ├── sRGB 的 Gamma ≈ 2.2（暗部精细，亮部粗略）
  ├── HDR 的 PQ 曲线（专门为 0~10000nit 设计的非线性曲线）
  └── Lab 中的立方根：L* = 116·(Y/Yn)^(1/3) - 16

─────────────────────────────────────────────────────────────────────────
类型 ③：几何变换（直角坐标 ↔ 极坐标）
─────────────────────────────────────────────────────────────────────────

  形式：    （直角）→（角度 + 半径）
            R, G, B  →  色相 H（角度）+ 饱和度 S（半径）+ 明度 V

  质朴理解："把立方体掰成圆柱体"
  ├── RGB 是立方体（三个轴：R/G/B 各 0~255）
  ├── HSV 是圆柱体（H 是角度，S 是半径，V 是高度）
  └── 转换 = 找到 RGB 立方体上某个点对应的 HSV 柱坐标

  应用：
  ├── RGB ↔ HSV
  ├── RGB ↔ HSL
  └── Lab → LCh（明度+色度+色相角）

─────────────────────────────────────────────────────────────────────────

  总结：所有色彩空间转换，本质就是这三种变换的组合：
       
       ┌──────────────────────────────────────────────────────┐
       │  矩阵乘法（换坐标系）+ Gamma（线性 ↔ 非线性）        │
       │  + 极坐标（直角 ↔ 圆柱）= 任意色彩空间转换           │
       └──────────────────────────────────────────────────────┘
```

### 0.4 三类转换的"质朴原始目的"速查表

下表把后面章节会出现的所有转换公式，用"为什么要做这个转换"的角度梳理一遍：

```
┌────────────────────┬─────────────────────────────────────────────────┐
│ 转换               │ 质朴原始目的（为什么要做这个转换？）            │
├────────────────────┼─────────────────────────────────────────────────┤
│ RGB → XYZ          │ "我有显示器的 RGB 值，想知道这个颜色客观上是    │
│                    │ 什么颜色"（接入色彩科学的世界）                 │
├────────────────────┼─────────────────────────────────────────────────┤
│ XYZ → RGB          │ "我知道某个客观颜色，想让显示器画出来"          │
├────────────────────┼─────────────────────────────────────────────────┤
│ sRGB → Adobe RGB   │ "我的图片是 sRGB 拍的，要送到 Adobe RGB 印刷厂" │
│ (RGB 空间互转)     │ → 先 sRGB→XYZ，再 XYZ→Adobe RGB                 │
├────────────────────┼─────────────────────────────────────────────────┤
│ RGB → YUV          │ "我要压缩这张图/视频，亮度可以保留全部精度，    │
│                    │ 颜色可以只存 1/4，节省带宽"                     │
├────────────────────┼─────────────────────────────────────────────────┤
│ YUV → RGB          │ "解码视频，让显示器能画出来"                    │
├────────────────────┼─────────────────────────────────────────────────┤
│ RGB → HSV          │ "用户想拖动滑块调'更鲜艳一点'，不想算 RGB"     │
├────────────────────┼─────────────────────────────────────────────────┤
│ HSV → RGB          │ "用户调好了 HSV，要传给显示器"                  │
├────────────────────┼─────────────────────────────────────────────────┤
│ XYZ → Lab          │ "我要比较两个颜色的差异（ΔE），需要在感知均匀  │
│                    │ 的空间里算欧氏距离"                             │
├────────────────────┼─────────────────────────────────────────────────┤
│ Lab → XYZ          │ "颜色编辑完了，要回到显示链路"                  │
├────────────────────┼─────────────────────────────────────────────────┤
│ 线性 RGB ↔ sRGB    │ "存储/传输用 sRGB（省比特），运算用线性 RGB     │
│ (Gamma 校正)       │ （混合/缩放才物理正确）"                        │
├────────────────────┼─────────────────────────────────────────────────┤
│ SDR → HDR (PQ)     │ "我要把亮度从 100nit 扩展到 10000nit，需要新的  │
│                    │ 编码曲线把这么大的动态范围塞进 10/12bit"        │
├────────────────────┼─────────────────────────────────────────────────┤
│ HDR → SDR (Tone    │ "HDR 视频要在普通显示器上播放，得把高亮度       │
│ Mapping)           │ '压'到 100nit 范围内，还得好看"                 │
└────────────────────┴─────────────────────────────────────────────────┘
```

#### ⭐ 程序员的灵魂拷问：这些公式到底是实验测的，还是数学推的？

> 这是初学者最容易困惑的点。简单回答：**只有 1 组实验数据是真正"测"出来的，其他全是基于这组数据 + 工程约定 推导出来的**。下面用程序员的视角来拆解。

```

  │   唯一的"实验真值"（唯一的"根"）：                       │
  │   ┌─────────────────────────────────────┐               │
  │   │  CIE 1931 颜色匹配函数               │               │
  │   │  x̄(λ)、ȳ(λ)、z̄(λ)                │               │
  │   │  （Wright 1928 + Guild 1931 测的）  │               │
  │   └─────────────────────────────────────┘               │
  │                    │                                    │
  │                    │ 一切都是从这里"算"出来的            │
  │                    ▼                                    │
  │   ┌─────────────────────────────────────────────┐       │
  │   │ 所有色彩空间的"原色坐标"                     │       │
  │   │ 所有色彩空间之间的"转换矩阵"                 │       │
  │   │ 所有 Gamma 曲线                              │       │
  │   │ 所有色域定义                                 │       │
  │   └─────────────────────────────────────────────┘       │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

##### 问题 1：所有色彩空间的"坐标系"都是实验测出来的吗？

**答：不是。只有 CIE XYZ 是基于实验数据建立的，其他色彩空间的坐标系都是"工程师定的"。**

```
程序员视角的类比：
  └── CIE XYZ ≈ "操作系统内核"（实验+理论建立，所有人都依赖它）
      其他色彩空间 ≈ "上层应用"（工程师拍脑袋根据需求定的）

具体例子：

┌──────────────┬────────────────────────────────────────────────────┐
│ 色彩空间     │ 坐标系是怎么"建"的？                               │
├──────────────┼────────────────────────────────────────────────────┤
│ CIE XYZ      │ ⭐ 实验数据 + Grassmann 线性代数理论 + 4 个工程约束 │
│ (1931)       │ → 这是唯一一个"有实验根基"的坐标系                 │
├──────────────┼────────────────────────────────────────────────────┤
│ sRGB         │ ❌ 不是实验测的！                                   │
│              │ 工程师 1996 年开会决定："就用这三个原色坐标吧！"   │
│              │   红 = (0.640, 0.330)                              │
│              │   绿 = (0.300, 0.600)                              │
│              │   蓝 = (0.150, 0.060)                              │
│              │ → 这三个坐标是"挑"的，不是"测"的                  │
│              │   挑的依据：当时主流 CRT 显像管荧光粉能发出的颜色 │
├──────────────┼────────────────────────────────────────────────────┤
│ Adobe RGB    │ ❌ 不是实验测的！                                   │
│ (1999)       │ Adobe 公司选了一组比 sRGB 大的原色坐标             │
│              │ 目的：覆盖 CMYK 印刷色域                           │
├──────────────┼────────────────────────────────────────────────────┤
│ DCI-P3       │ ❌ 不是实验测的！                                   │
│ (2007)       │ 数字电影协会选的，覆盖电影常用色彩                 │
├──────────────┼────────────────────────────────────────────────────┤
│ BT.2020      │ ❌ 不是实验测的！                                   │
│ (2012)       │ ITU 选的，原色干脆放在马蹄形边界上（最大色域）     │
├──────────────┼────────────────────────────────────────────────────┤
│ YCbCr        │ ❌ 不是实验测的！                                   │
│              │ 工程师定义："Y = R/G/B 的加权和，权重看 BT 标准"  │
│              │ 权重根据"亮度感知"原理推导                         │
├──────────────┼────────────────────────────────────────────────────┤
│ HSV / HSL    │ ❌ 不是实验测的！                                   │
│ (1978)       │ 程序员 Alvy Ray Smith 定义的几何变换               │
│              │ 把 RGB 立方体"几何上"重新参数化为圆柱体             │
├──────────────┼────────────────────────────────────────────────────┤
│ CIE Lab      │ ⚠️ 半实验半数学                                    │
│ (1976)       │ 立方根公式的"曲线形状"参考了感知实验数据           │
│              │ 但具体参数（如 116、16）是工程师拟合 + 取整         │
└──────────────┴────────────────────────────────────────────────────┘

→ 结论：除了 CIE XYZ，其他色彩空间的"原色坐标"都是
   工程师/委员会"选定"的工程参数，不是测量值。

→ 类比代码：
   ```c
   // CIE XYZ：实验测出来的常量（永远不会变）
   const float CMF_X[401];  // x̄(λ)，CIE 1931 测得
   const float CMF_Y[401];  // ȳ(λ)，CIE 1931 测得
   const float CMF_Z[401];  // z̄(λ)，CIE 1931 测得
   
   // sRGB：工程师拍板定下的常量（可能哪天换标准就变）
   #define SRGB_R_xy {0.640, 0.330}   // 不是测的，是定的
   #define SRGB_G_xy {0.300, 0.600}   // 不是测的，是定的
   #define SRGB_B_xy {0.150, 0.060}   // 不是测的，是定的
   #define SRGB_WHITE_xy {0.3127, 0.3290}  // D65 白点，CIE 选定
   ```
```

##### 问题 2：所有转换矩阵都是实验测出来的吗？

**答：完全不是。所有转换矩阵都是"用线性代数算出来的"。**

```
关键洞察：转换矩阵 = 输入色彩空间的原色坐标 + 输出色彩空间的原色坐标
        → 一旦双方原色坐标定下，矩阵就唯一确定（套公式即可）！

也就是说：
  ├── 工程师只需要"定原色坐标"（4 个 xy 坐标即可）
  └── 转换矩阵自动从公式算出（不需要任何额外实验）

矩阵推导的"标准算法"（任何 RGB → XYZ 矩阵都用这个）：

  输入：
    红原色坐标 (xr, yr)
    绿原色坐标 (xg, yg)
    蓝原色坐标 (xb, yb)
    白点坐标   (xw, yw)
  
  输出：3×3 转换矩阵 M（让 M·[1,1,1]ᵀ = 白点）
  
  算法步骤（3 行线性代数）：
  ┌────────────────────────────────────────────────┐
  │ 1. 把每个原色 xy 还原成 XYZ 列向量             │
  │    （已知 xy 和 Y=1，反算 XYZ）                │
  │ 2. 解线性方程：3 个原色向量 × 缩放系数 = 白点   │
  │ 3. 把每列乘以缩放系数，得到最终矩阵 M           │
  └────────────────────────────────────────────────┘

  → 整个过程零实验！全是数学。

举例：常见 RGB → XYZ 矩阵的"出生证"

┌──────────────┬────────────────────────────────────────────────┐
│ 矩阵         │ 是怎么"算"出来的？                             │
├──────────────┼────────────────────────────────────────────────┤
│ sRGB → XYZ   │ 用 sRGB 的 4 个 xy 坐标 + 上面的标准算法       │
│              │ 跑一遍线性代数 → 得到：                        │
│              │  [0.4124  0.3576  0.1805]                      │
│              │  [0.2126  0.7152  0.0722]                      │
│              │  [0.0193  0.1192  0.9505]                      │
│              │ → 这个矩阵不是测的，是 5 行 Python 代码算的！  │
├──────────────┼────────────────────────────────────────────────┤
│ BT.709 RGB   │ 同样的算法，换一组 xy 坐标，再跑一遍           │
│ → XYZ        │ → 因为 sRGB 和 BT.709 用同样的原色，           │
│              │ 矩阵和 sRGB → XYZ 完全相同                     │
├──────────────┼────────────────────────────────────────────────┤
│ BT.2020 RGB  │ 换 BT.2020 的 xy 坐标，跑同样算法              │
│ → XYZ        │ → 得到完全不同的矩阵                           │
├──────────────┼────────────────────────────────────────────────┤
│ RGB → YCbCr  │ 不是矩阵直接测的，是从"人眼亮度感知"出发推导： │
│              │ Y = 0.2126·R + 0.7152·G + 0.0722·B            │
│              │ ↑      ↑         ↑         ↑                  │
│              │ 这三个权重 = ȳ(λ) 在三个原色波长的积分         │
│              │                                                │
│              │ → 权重确实"间接来自实验"（通过 ȳ(λ)）          │
│              │ → 但不是"测 YUV 矩阵"，是从 ȳ(λ) 算出来的！    │
├──────────────┼────────────────────────────────────────────────┤
│ XYZ → Lab    │ 公式形式（立方根、116、16）是工程师设计的       │
│              │ 设计目标：拟合 1942 年 Munsell 颜色感知实验数据 │
│              │ → 公式是"凑"的，参数是"拟合"的，不是直接测的  │
└──────────────┴────────────────────────────────────────────────┘

→ 类比代码：所有转换矩阵都不是常量表，而是函数算出来的：

  ```python
  def compute_rgb_to_xyz_matrix(r_xy, g_xy, b_xy, w_xy):
      """
      输入：4 个 xy 坐标（工程师选定的，不是实验测的）
      输出：3x3 转换矩阵（纯线性代数算出来的）
      """
      # 第 1 步：xy → XYZ 列向量
      r_XYZ = xy_to_XYZ(r_xy)
      g_XYZ = xy_to_XYZ(g_xy)
      b_XYZ = xy_to_XYZ(b_xy)
      w_XYZ = xy_to_XYZ(w_xy)
      
      # 第 2 步：解 [r_XYZ | g_XYZ | b_XYZ] · S = w_XYZ
      S = np.linalg.solve([r_XYZ, g_XYZ, b_XYZ].T, w_XYZ)
      
      # 第 3 步：缩放
      M = np.array([r_XYZ * S[0], g_XYZ * S[1], b_XYZ * S[2]]).T
      return M
  
  # sRGB 矩阵 = 这个函数的一次调用结果，不是实验测的！
  M_sRGB = compute_rgb_to_xyz_matrix(
      r_xy=(0.640, 0.330),     # sRGB 选定的原色
      g_xy=(0.300, 0.600),
      b_xy=(0.150, 0.060),
      w_xy=(0.3127, 0.3290),   # D65 白点
  )
  ```
```

##### 问题 3：选定原色坐标后，转换出来的颜色和实际观察到的能一致吗？

> 这是最容易让人困惑、也是色彩工程最核心的问题。简短回答：**只要每个环节都"诚实"，理论上能一致；但现实中有 4 个层级的不确定性，需要通过"色彩管理系统（CMS）"逐层校准来逼近一致。**

```
先看一个最容易被忽略的事实：

  sRGB 标准定义的原色坐标 (0.640, 0.330) 等，
  本质上是一个"承诺"或"契约"：

  ┌──────────────────────────────────────────────────────────┐
  │  "我（这台显示器）保证：当我收到 RGB=(255,0,0) 时，       │
  │   屏幕实际发出的光，其 CIE xy 色度坐标就是 (0.640, 0.330)" │
  └──────────────────────────────────────────────────────────┘

  → 这是一份"硬件应当遵守的合约"
  → 转换矩阵 = 在"假设硬件遵守合约"的前提下推导出来的
  
  所以问题就变成了：硬件真的遵守这个合约吗？
```

**色彩一致性的"4 层不确定性"** ⭐

```
理论上的完美链路（如果每一环都精确无误）：

  XYZ 真值 ──[公式]──→ sRGB 数值 ──[显示器]──→ 屏幕实际发光
   (客观)              (合约)                  (希望与XYZ一致)

  但现实中每一环都有"误差源"：

  ┌────────────────────────────────────────────────────────────┐
  │ 第 1 层：标准定义层（理论可保证一致 ✅）                    │
  │  └── 数学上：选定原色 xy + 白点 xy + Gamma 曲线后，          │
  │      RGB↔XYZ 是数学严格的双射，不存在误差                   │
  │      → 这一层是"完美"的                                     │
  ├────────────────────────────────────────────────────────────┤
  │ 第 2 层：显示设备层（最大误差源 ⚠️）                        │
  │  └── 真实显示器永远做不到 100% 符合 sRGB 合约               │
  │      ① 红色荧光粉/LED 发出的光，xy 不是精确的 (0.640,0.330) │
  │      ② 白点不精确等于 D65                                   │
  │      ③ Gamma 不精确等于 2.2                                 │
  │      ④ 不同批次、不同温度、使用年限 → 性能漂移              │
  │      → 这是"硬件违约"，会导致显示偏色                       │
  ├────────────────────────────────────────────────────────────┤
  │ 第 3 层：观察环境层（人眼适应 ⚠️）                          │
  │  └── 即使屏幕完美发光，看到的颜色仍受环境影响：              │
  │      ① 环境光色温（暖光/冷光下看屏幕，感知不同）             │
  │      ② 眼睛白平衡适应（盯久了会"自动校正"）                  │
  │      ③ 屏幕周围背景色（同色异常对比效应）                   │
  ├────────────────────────────────────────────────────────────┤
  │ 第 4 层：观察者个体差异（无法消除 ⚠️）                      │
  │  └── CIE 标准观察者是"统计平均"，每个人眼略有不同：           │
  │      ① 锥细胞密度、感光蛋白基因型差异                       │
  │      ② 8% 男性有色弱（红绿混淆）                             │
  │      ③ 年龄（晶状体黄化）→ 老人看蓝色会偏暗                  │
  │      → 这层无解，但也"够用"（差异通常 < 5%）                │
  └────────────────────────────────────────────────────────────┘

  程序员理解：
  └── sRGB 标准 ≈ 一份 API 文档
      显示器 ≈ 第三方厂商实现这份 API
      "颜色不一致" ≈ 厂商实现有 bug 或精度不足
```

**工程上如何"逼近一致"？—— 色彩管理系统（CMS）的 3 道防线** ⭐⭐

```
真实工业界，从来不假设"硬件守约"。而是用 3 道工程防线来纠正偏差：

═══════════════ 防线 1：硬件出厂校准 ═══════════════════════
  
  显示器/相机/打印机出厂前，工厂用专业仪器（分光光度计）测量：
  
       ┌─────────────┐                  ┌──────────────┐
       │  显示器     │ ──发光──→        │ 分光光度计   │
       │  显示纯红色 │                  │ 测真实 xy    │
       └─────────────┘                  └──────────────┘
                                               │
                                               ▼
                            实测 (0.638, 0.328)
                            合约 (0.640, 0.330)
                            → 偏差 = 0.002
                            → 写入硬件 LUT 校正表
  
  → 这一步直接物理校正，让硬件尽量"守约"
  → 高端显示器（艺卓、LG UltraFine）能做到 ΔE < 2

═══════════════ 防线 2：ICC 色彩配置文件 ═══════════════════
  
  即使校正后还有残差，操作系统会用 ICC Profile 软件层补偿：

  ┌────────────────────────────────────────────────────────┐
  │ ICC Profile 文件（.icc）= "这台设备的颜色身份证"        │
  │                                                        │
  │ 内容：                                                 │
  │ ├── 这台显示器的"实测 xy 坐标"（不是 sRGB 标称值）     │
  │ ├── 这台显示器的"实测 Gamma 曲线"                      │
  │ ├── 这台显示器的"实测白点"                             │
  │ └── 一张 RGB→XYZ 的查找表（LUT）                       │
  └────────────────────────────────────────────────────────┘

  操作系统的颜色转换流程（macOS/Windows ColorSync）：
  
       图片                               显示器
     (sRGB) ──┐                          ┌── (实际显示)
              │                          │
              ▼                          ▼
       ① 用 sRGB.icc        ② 用 [本机].icc
          sRGB → XYZ           XYZ → 这台显示器的 RGB
       (按标准合约)         (按这台机器的真实特性)
       
  → 不管硬件多"违约"，软件都能反向补偿
  → 这就是为什么图片在不同电脑上颜色不一样：
    没装/没启用 ICC 配置文件 → 走"假设守约"的捷径 → 偏色

═══════════════ 防线 3：用户级专业校色 ═══════════════════
  
  对色彩极敏感的场景（修图、印刷、影视调色）：
  
       ┌───────────────────────────────────┐
       │ 用户买专业校色仪（Spyder/i1Display）│
       │       │                            │
       │       │ 贴在屏幕上                  │
       │       ▼                            │
       │  ┌─────────┐                       │
       │  │ 显示器   │ ←播放校色软件         │
       │  │ 显示色块 │  让屏幕轮流显示       │
       │  │         │  数十种颜色           │
       │  └─────────┘                       │
       │       │                            │
       │       ▼ 实测每种色块的真实 xy        │
       │  生成本机专属 ICC                  │
       │       │                            │
       │       ▼                            │
       │  替换系统默认 ICC                  │
       └───────────────────────────────────┘
  
  → 校色后，不同显示器看同一张图能做到色差 ΔE < 1
    （ΔE < 2 = 肉眼几乎无法分辨差异）
```

**ΔE 色差 —— 量化"一致性"的尺子**

```
工业界用 ΔE 来量化"两个颜色看起来差多少"（在 Lab 空间算欧氏距离）：

  ΔE 数值      |  人眼感知
  ───────────────────────────────────
  ΔE < 1       | 完全察觉不到差异 ✅
  ΔE = 1 ~ 2   | 仔细对比能发现 ✅
  ΔE = 2 ~ 3.5 | 普通人能发现 ⚠️
  ΔE = 3.5 ~ 5 | 明显能看出 ❌
  ΔE > 5       | 严重偏色 ❌

  典型设备的色差水平：
  ├── 普通办公显示器（无校准）  → ΔE ≈ 5~10  （够用就行）
  ├── 中端显示器（出厂校准）    → ΔE ≈ 2~4   （日常 OK）
  ├── 专业显示器（艺卓 CG 系列）→ ΔE < 2     （设计/印刷）
  └── 用户校色仪校准后          → ΔE < 1     （影视调色级）
```

**回到原始问题：会不会不一致？怎么保证？**

```
会不会不一致？  → 一定会，只是程度问题。

为什么会？
  ├── 标准本身没问题（数学严密）
  ├── 但硬件无法 100% 实现标准（物理限制 + 制造公差）
  └── 不同设备误差不同 → 同一图片在不同屏幕颜色略有差异

怎么保证（工程实践）？
  ├── 第 1 道：硬件出厂校准（让硬件尽量守约）
  ├── 第 2 道：ICC 配置文件（软件层反向补偿残差）
  ├── 第 3 道：专业校色仪（用户级精细校准）
  └── 用 ΔE 数值监控一致性程度

程序员视角的类比：
  ┌──────────────────────────────────────────────────────┐
  │  sRGB 标准   ≈ HTTP/JSON 标准（理论严密）             │
  │  显示器硬件  ≈ 各家服务器实现（可能有 bug）           │
  │  ICC Profile ≈ 客户端的兼容补丁                      │
  │  ΔE          ≈ 自动化测试的断言阈值                  │
  │  专业校色仪  ≈ 持续集成的标准化测试设备              │
  └──────────────────────────────────────────────────────┘

  → 颜色一致性是一个"工程问题"，不是"数学问题"
  → 数学公式只保证"if 硬件守约 then 颜色一致"
  → 真正的一致性 = 数学正确 + 硬件校准 + 软件补偿 三位一体
```

## 一、从人眼说起——色彩的本质

### 1.1 光是什么？

```
可见光是电磁波，波长范围约 380nm ~ 780nm：

  紫外线  紫   蓝   青   绿   黄   橙   红  红外线
  ←──────────────────────────────────────────→
  <380nm  380  450  490  520  570  590  620  >780nm

  颜色 ≠ 物理量
  颜色 = 人眼对特定波长电磁波的主观感知

  关键认知：
  ├── 同一种"颜色"可以由不同的光谱组合产生（同色异谱）
  ├── 不同人对颜色的感知略有差异
  └── 颜色是三维的（人眼有三种视锥细胞）
```

### 1.2 人眼的三种视锥细胞

```
人眼视网膜上有三种视锥细胞（Cone Cells）：

  L 型（Long，长波）：
  ├── 峰值敏感波长：约 564nm（黄绿色）
  └── 通常被称为"红色感受器"（不准确但常用）

  M 型（Medium，中波）：
  ├── 峰值敏感波长：约 534nm（绿色）
  └── 通常被称为"绿色感受器"

  S 型（Short，短波）：
  ├── 峰值敏感波长：约 420nm（蓝紫色）
  └── 通常被称为"蓝色感受器"

  三色视觉理论（Young-Helmholtz，1802/1850年）：
  └── 任何颜色都可以用 L、M、S 三个响应值唯一描述
      这是所有色彩空间"三维性"的生理基础！

  视锥细胞响应曲线（示意）：

        ↑
    1.0 │    S        M    L
        │   ╭╮      ╭╮   ╭╮

        │ ╭╯  ╰╮  ╭╯  ╰╯  ╰╮
    0.0 └─────────────────────→ 波长(nm)
        380  450  520  590  700

  📷 真实曲线（强烈推荐查看）：

### 1.3 颜色的三个属性

```

  └── 颜色的"种类"：红、橙、黄、绿、蓝、紫...

## 三、CIE XYZ——色彩科学的基石（1931年）

> **本章核心问题**：CIE XYZ 这个坐标系到底是怎么建出来的？是纯实验测出来的吗？还是数学推出来的？
>
> **答案（一句话）**：先做实验测出"人眼对每个波长的响应数据"，再用线性代数把这堆数据**变换成一个工程上好用的坐标系**。
>
> **程序员视角**：CIE XYZ ≈ 一个**人眼颜色感知的"标准 API"**——输入光谱，输出 3 个数（X, Y, Z）。这个 API 的实现 = 一组实验数据 + 一次线性变换。下面会一步步拆解这个"标准 API"是怎么被设计出来的。

### 3.0 程序员速读：3 分钟看懂整章

```
如果你只有 3 分钟，记住这张图就够了：

  ┌──────────────────────────────────────────────────────────┐
  │  Step 1：实验（1928-1931）                               │

  │  让人盯着屏幕调旋钮，测出每个波长 λ 需要多少 R/G/B 才能匹配│
  │  → 得到 3 条曲线：r̄(λ), ḡ(λ), b̄(λ)                       │
  │  → 问题：r̄ 在某些波长是负数 ❌                            │
  ├──────────────────────────────────────────────────────────┤
  │  Step 2：理论（基于 1853 年 Grassmann 定律）             │
  │  ────────────────────────                                │
  │  数学已证明：颜色构成一个 3 维线性空间                    │
  │  → 推论：可以用 3×3 矩阵换基（就像编程里的坐标变换）       │
  ├──────────────────────────────────────────────────────────┤
  │  Step 3：工程取舍（CIE 1931 委员会决议）                 │
  │  ────────────────────────                                │

  │  ④ Z 在长波 ≈ 0（简化计算）                              │
  ├──────────────────────────────────────────────────────────┤
  │  Step 4：求解（线性规划）                                │
  │  ────────────────────────                                │
  │  解出唯一的 M，应用后得到 CIE XYZ 标准                    │
  │  → 沿用 90+ 年，所有色彩空间（sRGB/Lab/YUV）都建在它之上  │
  └──────────────────────────────────────────────────────────┘

  类比理解：
  └── 这个过程像不像设计一个 RESTful API？
      ① 收集真实业务数据（实验数据）
      ② 选择合适的设计模式（线性代数）
      ③ 加上工程约束（非负、独立、对称、简化）
      ④ 实现并发布标准（CIE 1931）
```

  ├── 用 10 名观察者，在 NPL 国家物理实验室

  ├── 用另一批观察者，独立做了同样的实验
  └── 与 Wright 的数据相互印证 → 提高了可信度

  1931年  CIE（国际照明委员会）— 统一标准制定者 ⭐⭐⭐
  ├── 整合 Wright + Guild 的实验数据 → "标准观察者"
  ├── 但发现：r̄(λ) 中有负值，工程上不便
  └── 用线性代数变换：把 (R,G,B) 空间 → (X,Y,Z) 空间

      （之后 90 年再没被推翻，是色彩科学的基石）

  → 此后所有色彩空间（sRGB、AdobeRGB、Lab、YUV 等）
    都建立在 CIE XYZ 之上！

  ───────────────────────────────────────────────────────
  
  关键认知：CIE XYZ 是 100 多年理论 + 30 多年实验的结晶。
  
  ├── 理论支柱（Grassmann 1853）：颜色是 3 维线性空间
  ├── 实验数据（Wright 1928 + Guild 1931）：测出真实人眼响应
  └── 工程整合（CIE 1931）：用矩阵变换让数据非负、亮度独立

### 3.2 颜色匹配实验：实验是怎么做的？

CIE XYZ 的"地基"是**颜色匹配实验**（Color Matching Experiment）。这是一种**心理物理实验**——既不是纯物理测量，也不是纯主观问卷，而是"让人眼充当测量仪器"。

```
程序员类比：
  这个实验本质是在"测一个未知函数"：
    f: 波长 λ  →  (R, G, B) 系数
  方法：穷举每个 λ，让人调旋钮直到颜色匹配，记录 (R,G,B)。
  最后得到三张"查找表"，就是 r̄(λ), ḡ(λ), b̄(λ) 函数。
```

```
3.2.1 实验装置（Wright 1928 / Guild 1931）

  ┌─────────────────────────────────────────────────────┐
  │           黑暗的房间，2° 视场分割屏                  │
  │                                                     │
  │   左半视场（参考光）         右半视场（测试光）       │
  │   ┌──────────────┐           ┌──────────────┐       │
  │   │              │           │              │       │
  │   │  R·R₀ +      │           │  单色光     │       │
  │   │  G·G₀ +      │   分割线  │  C(λ)       │       │
  │   │  B·B₀        │     │     │  波长 λ     │       │
  │   │              │     │     │             │       │
  │   └──────────────┘     │     └──────────────┘       │
  │                                                     │
  │   观察者通过 2° 小孔（模拟视网膜中央凹）观察          │
  └─────────────────────────────────────────────────────┘

  ─────────────────────────────────────────────────────
  【A】右半视场的"单色测试光"如何构建？
  ─────────────────────────────────────────────────────

  核心装置：单色仪（Monochromator）—— 从白光中"切"出任意波长

  光路流程：

    白炽灯（钨丝灯，发出 380~780nm 连续光谱）
        │
        ▼
    ┌───────────────────────────────────────────────┐
    │              单色仪内部                        │
    │                                               │
    │  入射狭缝 → 准直镜 → 棱镜/光栅 → 聚焦镜 → 出射狭缝  │
    │                                               │
    │  工作原理：                                    │
    │  1. 棱镜/光栅将白光按波长"展开"成彩虹带        │
    │  2. 旋转棱镜/光栅角度 → 不同波长对准出射狭缝   │
    │  3. 出射狭缝只放过极窄一段 → 输出近似单色光     │
    └───────────────────────────────────────────────┘
        │
        ▼
    波长 λ 的单色光（带宽约 2~5nm）→ 投射到右半视场

  程序员类比：
    白炽灯 = 全频段白噪声信号源
    棱镜   = FFT（把"时域"展开到"频域"）
    狭缝   = 带通滤波器（只取一个频率分量）
    旋钮   = 调节中心频率

  为什么不用滤光片？
    ├── 滤光片带宽太宽（10~30nm），不够"单色"
    ├── 每个波长需要一片，380~780nm 需要几十片
    └── 单色仪一台仪器、一个旋钮就能连续扫描全波段

  波长精度如何保证？—— 用已知谱线校准：
    1. 将水银灯放入单色仪入射口
    2. 水银灯有物理常数级别的精确谱线：
       546.074nm（绿）、435.835nm（蓝）、404.656nm（紫）...
    3. 找到这些谱线对应的角度刻度 → 建立"角度→波长"校准曲线
    4. 换回白炽灯后，根据角度读数即可精确知道输出波长
    （类比：用"标准砝码"校准天平）

  实际使用的仪器：
    ├── Wright (1928)：Hilger 恒偏向棱镜单色仪，400~700nm，每 10nm 一个点
    └── Guild (1931)：高精度棱镜单色仪，380~780nm，每 5nm 一个点

  ─────────────────────────────────────────────────────
  【B】左半视场的"三原色参考光"如何选择？
  ─────────────────────────────────────────────────────

  三原色（CIE 1931 标准）：
  ├── R₀ = 700.0 nm（红）—— 单色仪从白炽灯中切出
  ├── G₀ = 546.1 nm（绿）—— 水银灯绿色谱线，直接使用
  └── B₀ = 435.8 nm（蓝）—— 水银灯蓝色谱线，直接使用

  选择依据（必须同时满足两个条件）：

  条件①：波长精确可复现（不同实验室能得到一模一样的光）
     ├── 546.1 / 435.8：水银灯谱线，物理常数级别的精确
     │   （水银原子能级跃迁决定，全世界任何水银灯都一样）
     └── 700.0：单色仪 + 白炽灯，精度取决于仪器（稍差但够用）

  条件②：三个波长尽量"分散"在可见光谱上（红/绿/蓝三区各一个）
     ├── 这样三原色线性无关，能覆盖尽可能大的色域
     ├── 700nm 选在红端极限 → 只激发 L 锥，最纯的红
     ├── 546nm 在绿区中心 → 水银灯最强谱线，亮度充足
     └── 436nm 在蓝区 → 水银灯次强谱线，亮度充足

  为什么红色不也用水银灯？
     ├── 水银灯有红色谱线（615nm、623nm），但强度太弱
     └── 且波长不够"红"（会同时激发 M 锥，不够纯）
     → 所以宁可用单色仪切出 700nm 整数波长

  小结：
     546.1 / 435.8 nm → "水银灯自带的"（物理上方便且精确）
     700.0 nm         → "人为挑的整数"（需单色仪切出，为了最纯的红）

3.2.2 实验过程（伪代码视角）

  for λ in range(380, 781, 5):              # 每 5nm 取一个点
      右屏 = 单色光(λ)                       # 固定的单色测试光
      for observer in observers:             # 多人重复
          while 左右屏看起来不一致:           # 人眼充当 == 比较器
              调节 R, G, B 旋钮
          记录(λ, observer, R, G, B)
  
  r̄(λ), ḡ(λ), b̄(λ) = 对所有 observer 取均值

  → 这就是"标准观察者"（Standard Observer）的由来：
    一个统计平均出来的"标准化人眼"

3.2.3 得到的实验数据：颜色匹配函数

  把每个波长 λ 对应的 R、G、B 值画成曲线，得到三条函数：

       响应
        ↑
    3.0 │       ḡ(λ)
        │      ╱╲
    2.0 │     ╱  ╲      
        │  ╲ ╱    ╲   r̄(λ)
    1.0 │   ╳      ╲ ╱╲
        │  ╱ ╲      ╳   ╲
    0.0 ─╱───╲────╱─╲────╲──→ λ (nm)
       380   450  550  650  700
        ↑      ↑          ↑
        b̄(λ)   ↑          ↑
              （负值区！）  
        ─────╲╱───────────
            r̄(λ) 在 440~545nm 区间为负

  📷 真实的 CIE 1931 RGB 颜色匹配函数（含负值，强烈推荐查看）：
  ├── Wikipedia：https://en.wikipedia.org/wiki/CIE_1931_color_space#Color_matching_functions
  └── Wikimedia 高清图：https://commons.wikimedia.org/wiki/File:CIE1931_RGBCMF.svg
     （清晰可见 r̄(λ) 在 440~545nm 区间深入负值区）

  曲线含义（以 r̄(λ) 为例）：

  ┌─ 实验场景回顾 ──────────────────────────────────────────┐
  │  左屏：R·(700nm红) + G·(546nm绿) + B·(436nm蓝)          │
  │  右屏：1 个单位的 580nm 黄光（待匹配的目标）              │
  │                                                          │
  │  观察者调旋钮，直到左右看起来一样。                        │
  │  此时记录下 R=0.5, G=?, B=?                              │
  └──────────────────────────────────────────────────────────┘

  所以 r̄(580) = 0.5 的完整含义是：

    "580nm 黄光"这个颜色，在人眼看来，
    等价于一个混合光配方：0.5份红(700nm) + G份绿(546nm) + B份蓝(436nm)

    r̄(580) = 0.5 只是这个"配方"中红色原料的用量
    完整配方还需要 ḡ(580) 和 b̄(580) 才能完整描述这个颜色

  ┌─ 常见误解 vs 正确理解 ──────────────────────────────────┐
  │                                                          │
  │  ❌ 错误：580nm 黄光"里面包含"0.5 份红光                 │
  │     （单色光是纯的，里面不"包含"任何其他波长的光）        │
  │                                                          │
  │  ✅ 正确：580nm 黄光"在人眼看来等效于"一个混合光配方，    │
  │     这个配方中红色原料的用量是 0.5                        │
  │                                                          │
  │  关键词：不是"含有"，是"匹配时需要"                      │
  └──────────────────────────────────────────────────────────┘

  程序员类比：

    r̄(580) = 0.5 不是在说"580 这个文件里有 0.5 个红色字节"
    而是在说"要用三原色'合成'出和 580 一样的视觉效果，红色通道设为 0.5"

    就像 CSS 里：
      color: rgb(128, ?, ?)   ← 128/255 ≈ 0.5
    这不是说目标颜色"含有"红色像素，
    而是说"用 RGB 模型表达这个颜色时，R 分量是 0.5"

  一句话总结：
  └── r̄(λ) 不是"λ 波长光中红色的含量"，
      而是"用三原色混合来模拟 λ 波长光时，红色原料的配比"。
      它描述的是人眼的等效感知，不是光的物理组成。
```

#### ⭐ 关键问题：为什么 r̄(λ) 会出现负值？

```
程序员视角：负值 = "代数等式的移项"！

  回想初中代数：
    A + B = C        ← 移项 →    A = C - B
                                     ↑
                                  出现"负的 B"

  颜色匹配同理：
    当测试光是 500nm 青色时：
    
    左屏（R+G+B 混合） == 右屏（青色）       ← 调不出来！
    
    实验员只好把 R 灯"挪到右边"：
    左屏（G+B）        == 右屏（青色 + R）   ← 这下能匹配了！
    
    数学上记为（移项）：
       青色 = G + B - R
                     ↑
                "负的红"出现了

  所以 r̄(500nm) = -0.072 的真实含义是：
  └── 不是说红光是负的（物理上不存在负光）
  └── 而是说："要匹配青色，需要从测试光那边减去 0.072 个红光"
  └── = "把红光加到了测试光这一侧"

  这就跟编程里的 unsigned int 减法溢出类似：
  └── 数学上完美自洽
  └── 但工程上每次都要写 if 判断符号 → 烦！
  
  → CIE 决定：用数学方法构造一组"假想原色"，
    让所有匹配函数都 ≥ 0，从根本上消除负值
```

#### 为什么用 700/546/435 三个真实原色无法覆盖所有可见色？

```
原因：人眼三种锥细胞的响应曲线"互相重叠"！

       响应
        ↑
    1.0 │     L 锥(红)     M 锥(绿)
        │    ╱╲            ╱╲
        │   ╱  ╲          ╱  ╲
        │  ╱    ╲╲      ╱╱    ╲
    0.5 │ ╱      ╳      ╳      ╲
        │╱      ╱ ╲    ╱ ╲      ╲
        │      ╱   ╲╲╱╱   ╲      ╲
    0.0 ─────────────────────────→ λ
            S 锥(蓝)

  📷 真实的 L/M/S 锥归一化响应曲线（看一眼就懂"重叠"二字）：
  ├── Wikipedia 中文：https://zh.wikipedia.org/wiki/%E8%A6%96%E9%8C%90%E7%B4%B0%E8%83%9E
  ├── Wikipedia 英文：https://en.wikipedia.org/wiki/Cone_cell
  └── Wikimedia 高清图：https://commons.wikimedia.org/wiki/File:Cones_SMJ2_E.svg

  关键：M 锥和 L 锥曲线重叠很多
  └── 没有任何一个波长能"只激发 M 锥而不动 L 锥"
  └── 所以"纯绿色感觉"无法用任何真实光产生
  └── → 必须用"减法"才能合成 = 匹配函数出现负值
```

### 3.3 从实验数据到 XYZ 坐标系：纯实验？还是实验+理论？

> **核心答案**：CIE XYZ 的建立 = **实验数据**（提供真值）+ **线性代数理论**（消除负值、解耦亮度）。
>
> **程序员一句话**：把 (r̄,ḡ,b̄) 这组"原始 API 数据"通过一次**矩阵变换**重新打包成 (x̄,ȳ,z̄) 这组"工程友好版 API"。

```
3.3.1 总体路线图

  ┌────────────────────┐    Grassmann 颜色定律
  │ 实验测出 r̄、ḡ、b̄ │ ────────────────────────►
  │  (有负值，不便用) │   颜色 = 3 维线性空间
  └────────────────────┘    （理论支柱）
              │
              │ 关键洞察：
              │ 既然颜色空间是线性的，
              │ 那就可以用 3×3 矩阵换基！
              ▼
  ┌────────────────────┐
  │ 设计变换矩阵 M     │ ← 加 4 个工程约束（见下文）
  └────────────────────┘
              │
              ▼
  ┌────────────────────┐
  │  x̄、ȳ、z̄ 函数    │ ← CIE XYZ 颜色匹配函数
  │  全部 ≥ 0         │
  │  Y = 亮度          │
  └────────────────────┘
```

#### 3.3.2 理论基础：Grassmann 颜色定律 = 颜色界的"线性代数"

```
Grassmann 1853 年证明（这是数学定理，不是经验！）：

  人对颜色的感知，构成一个 3 维线性向量空间：
  
  ① 加法封闭：颜色 A + 颜色 B = 还是一个颜色
  ② 数乘封闭：k × 颜色 A = 还是一个颜色（亮度变化）
  ③ 满足线性性：(A + B) 匹配 (C + D)，则 A 匹配 C 时 B 匹配 D

程序员视角的重要类比 ⭐：

  "颜色是 3 维线性空间"  ≈  "颜色就像一个 3D 向量"
  
  这意味着我们可以套用所有线性代数的工具，比如：
  
  ┌──────────────────────────────────────────────────────────┐
  │ 编程里：3D 坐标可以从"世界坐标系"变换到"相机坐标系"      │
  │   world_coord = M @ camera_coord    # 矩阵乘法换基       │
  │                                                          │
  │ 颜色里：颜色可以从"RGB 基"变换到"XYZ 基"                  │
  │   xyz_color = M @ rgb_color         # 完全相同的套路！    │
  └──────────────────────────────────────────────────────────┘

推论（线性代数标准操作）：
└── 任何 3 个"线性无关"的颜色都可以作为基（原色）
    不同基之间，可以用 3×3 矩阵互相变换！

这就是 CIE XYZ 能"凭空构造"的理论依据：
├── (R, G, B) 是一组基（实验测得，700/546/435nm 三原色）
├── (X, Y, Z) 是另一组基（数学构造，假想原色）
└── 两组基之间用矩阵 M 联系：

     ┌ X ┐   ┌ ? ? ? ┐ ┌ R ┐
     │ Y │ = │ ? ? ? │ │ G │
     └ Z ┘   └ ? ? ? ┘ └ B ┘
            ↑
        关键问题：M 的 9 个参数怎么定？
```

#### 3.3.3 CIE 给变换矩阵 M 加的 4 个工程约束

```
程序员视角：
  矩阵 M 有 9 个未知参数 → 需要 9 个方程来确定
  CIE 1931 加了 4 组工程约束（每组提供 2~3 个方程），刚好把 M "钉死"。
  这就像在解一个"超定方程组"：

      9 个未知数（M 的元素）
    + 4 组工程约束（提供 9+ 个方程）
    ───────────────────────────────
    = 用线性规划解出唯一的 M

下面看这 4 组约束分别是什么、为什么这么选：
```

```
约束 ①：所有颜色匹配函数 ≥ 0（消除负值）⭐ 最重要
  └── 几何含义：选三个"假想原色"放在色度图马蹄形之外
      使得真实可见色全部能用正系数表达
      
       y
        │     ╔════ X ★（虚拟原色，色域外）
        │   ╔═╝
        │  ╔(马蹄形：可见色域)
        │ ╔═╝     ╔═══ Z ★
        │╔════════╝
        ★────────────────── x
       Y（虚拟原色）
       
       三个 ★ 围成的三角形完全包住马蹄形 → 可见色全部为正

  📷 真实的 CIE 1931 色度图（带马蹄形 + sRGB/AdobeRGB 等三角形对照）：
  ├── Wikipedia 中文：https://zh.wikipedia.org/wiki/CIE_1931%E8%89%B2%E5%BD%A9%E7%A9%BA%E9%97%B4
  ├── Wikipedia 英文：https://en.wikipedia.org/wiki/CIE_1931_color_space
  └── Wikimedia 彩色版：https://commons.wikimedia.org/wiki/File:CIE1931xy_blank.svg
     （彩色填充版能直接看到马蹄形彩色区域 + 各色域三角形 + D65 白点）
  
  程序员视角：
  └── 相当于"扩大值域到非负"，从此函数没有负值，
      代码里就不需要处理负数特例（unsigned 比 signed 好用）

约束 ②：Y = 人眼亮度响应（解耦亮度）⭐ 次重要
  └── 让 ȳ(λ) 等于已知的"光视效率函数"V(λ)
      （V(λ) 是 1924 年 CIE 已经标定好的人眼亮度感知曲线）
      → 这样 Y 分量直接代表亮度，X、Z 只代表"色度"

  程序员视角：
  └── 相当于"关注点分离"（Separation of Concerns）：
      ├── Y 通道：只管亮度
      └── X、Z 通道：只管色调
      → 后来视频压缩 YUV、HDR 都吃这个红利

约束 ③：等能白光 E（X=Y=Z）落在白点
  └── 要求 ∫x̄(λ)dλ = ∫ȳ(λ)dλ = ∫z̄(λ)dλ
      → 让坐标系"对称"，方便归一化

  程序员视角：
  └── 相当于"标准化常量"，让 (1,1,1) 自动是白色，
      就像 RGB(255,255,255) 是白色一样直观

约束 ④：Z 分量在长波（红光区）尽量为 0
  └── 让 z̄(λ) 在 600nm 以上几乎为 0
      → Z 主要表征"短波（蓝紫）信息"

  程序员视角：
  └── 相当于"稀疏化"：让矩阵在某些区域出现 0，
      可以省一些乘法运算（在 1930 年代手工计算时很重要！）
```

#### 3.3.4 推导全过程：从 r̄/ḡ/b̄ 数据集 → 一步步算出 XYZ 矩阵 ⭐

> 这一节回答一个关键问题：
> **拿到了 r̄(λ), ḡ(λ), b̄(λ) 这三张"查找表"之后，CIE 是怎么算出变换矩阵的？**
>
> 程序员一句话：把 4 组约束翻译成 9 个线性方程，解一个 9 元一次方程组。

##### Step 0：把"实验数据集"看成什么？

```
我们手头有什么？

  ├── r̄(λ)：一张表，输入波长 λ，输出红原色配比（含负值）
  ├── ḡ(λ)：一张表，输入波长 λ，输出绿原色配比
  └── b̄(λ)：一张表，输入波长 λ，输出蓝原色配比

  例如（采样间隔 5nm，从 380nm 到 780nm，共 81 行）：
  
       λ      r̄(λ)     ḡ(λ)    b̄(λ)
     ─────  ───────  ───────  ───────
     380    0.0003  -0.0001   0.0117
     ...
     500   -0.0717   0.0854   0.4789    ← 注意 r̄ 是负的
     580    0.5000   0.4143   0.0008    ← 上一节举的例子
     ...
     780    0.0000   0.0000   0.0000

程序员视角：
  这就是一个 81 × 3 的 numpy 数组：cmf_rgb.shape == (81, 3)
  我们要把它变换成 cmf_xyz.shape == (81, 3)，且全部 ≥ 0
```

##### Step 1：线性代数告诉我们"换基只需一个 3×3 矩阵"

```
由 Grassmann 定律（颜色空间是 3 维线性空间），换基的运算只是一次矩阵乘法：

  ┌ x̄(λ) ┐         ┌ m11  m12  m13 ┐ ┌ r̄(λ) ┐
  │ ȳ(λ) │   =     │ m21  m22  m23 │ │ ḡ(λ) │      ← 对每一个 λ 都成立
  └ z̄(λ) ┘         └ m31  m32  m33 ┘ └ b̄(λ) ┘
       ↑                    ↑                ↑
   想求的"新表"         9 个未知数        已知的"实验表"

写成代码就是：

  cmf_xyz = cmf_rgb @ M.T     # (81,3) = (81,3) @ (3,3)

→ 整个问题就归结为：求出这 9 个 m_ij。
→ 9 个未知数 → 需要 9 个独立方程 → 4 组约束刚好提供 9 个方程。
```

##### Step 2：把 4 组约束翻译成方程（这是核心！）

```
约束 ②（Y = 亮度）→ 提供 3 个方程 ★ 最简单

  要求：ȳ(λ) ≡ V(λ)  对所有 λ 都成立
                    （V(λ) 是 1924 年已经标定好的"明视觉光视效率函数"）
  
  即：m21·r̄(λ) + m22·ḡ(λ) + m23·b̄(λ) = V(λ)   ── (eq A)
  
  代入 3 个不同波长的实验值（比如 λ = 480, 555, 650nm），
  得到 3 个独立方程，正好确定 m21, m22, m23。
  
  程序员理解：
  └── 这是一个超定最小二乘问题：用 81 个 λ 的数据拟合 3 个未知数
      实际上 CIE 用了"积分等价"+"曲线匹配"的方法精确求解。

─────────────────────────────────────────────────────────────

约束 ③（等能白光 X=Y=Z）→ 提供 2 个方程

  等能白光 E：所有波长能量相等（一束"扁平"的白光）
  
  要求：∫ x̄(λ) dλ = ∫ ȳ(λ) dλ = ∫ z̄(λ) dλ
  
  把 x̄(λ) = m11·r̄ + m12·ḡ + m13·b̄ 代入积分：
  
    m11·∫r̄ + m12·∫ḡ + m13·∫b̄ = m21·∫r̄ + m22·∫ḡ + m23·∫b̄    ── (eq B)
    m31·∫r̄ + m32·∫ḡ + m33·∫b̄ = m21·∫r̄ + m22·∫ḡ + m23·∫b̄    ── (eq C)
  
  其中 ∫r̄, ∫ḡ, ∫b̄ 都是已知数（直接对实验数据求和即可）。
  → 关于 (m11..m13) 一个方程，关于 (m31..m33) 一个方程。

  程序员理解：
    sum_r = cmf_rgb[:, 0].sum()    # 已知
    sum_g = cmf_rgb[:, 1].sum()    # 已知
    sum_b = cmf_rgb[:, 2].sum()    # 已知
    # 上面两式是关于 M 元素的线性约束

─────────────────────────────────────────────────────────────

约束 ④（Z 在长波 ≈ 0）→ 提供 2 个方程

  要求：z̄(λ) = 0 对 λ ≥ 650nm 都成立
        即 m31·r̄(λ) + m32·ḡ(λ) + m33·b̄(λ) = 0   ── (eq D)
  
  在长波区，b̄(λ) ≈ 0，所以 (eq D) 退化为：
        m31·r̄(λ) + m32·ḡ(λ) = 0
  
  这给出 m31 : m32 的比例（一个方程）；
  再结合约束 ③ 的 (eq C)，可定出 m31, m32 的绝对大小（第二个方程）。
  
  程序员理解：
  └── 相当于"在长波区把 Z 通道按死为 0"，这是一个比例约束。

─────────────────────────────────────────────────────────────

约束 ①（所有 ≥ 0）→ 提供 2 个不等式约束 + 优化目标

  这个约束本身不直接给等式方程，而是限定"假想原色三角形"必须包住马蹄形。
  
  CIE 选择了一组"恰好相切"的三角形（最紧凑的包含），
  这相当于在剩余自由度里取"最优解"，从而锁定 m11, m12, m13 的具体值。
  
  程序员理解（线性规划视角）：
    minimize:   三角形面积（即不浪费色域）
    subject to: ∀λ, x̄(λ) ≥ 0, ȳ(λ) ≥ 0, z̄(λ) ≥ 0
                + 约束②③④的等式
  → 这是个标准的"带约束最优化问题"，1931 年 CIE 用手工方法解出。
```

##### Step 3：方程汇总 → 解出唯一的 M

```
把所有方程列出来（示意，简化版）：

  ┌─────────────────────────────────────────────┐
  │ 约束②：3 个方程  →  确定 (m21, m22, m23)    │ ← Y 那一行
  │ 约束③：2 个方程  →  约束 X、Z 行的总和       │
  │ 约束④：2 个方程  →  确定 (m31, m32, m33)    │ ← Z 那一行
  │ 约束①：1 个目标  →  确定 (m11, m12, m13)    │ ← X 那一行
  └─────────────────────────────────────────────┘
  
  9 个未知 + 9 个独立条件 → 唯一解！

最终解出来（这就是 1931 年 CIE 公布的官方矩阵）：

  M = (1/0.17) · ┌ 0.4900  0.3100  0.2000 ┐
                 │ 0.1770  0.8124  0.0106 │
                 └ 0.0000  0.0100  0.9900 ┘
                              ↑
                       注意 m31 = 0：这正是约束④的体现！
                       注意 m33 ≈ 0.99：Z 几乎只来自 B 通道
```

##### Step 4：用这个 M 把实验数据"批量转换"成 XYZ 函数

```
拿到 M 之后，再回去对实验数据做一次矩阵乘法即可：

  for λ in [380, 385, ..., 780]:
      [x̄(λ), ȳ(λ), z̄(λ)] = M @ [r̄(λ), ḡ(λ), b̄(λ)]

  → 得到新的三张查找表 x̄/ȳ/z̄，这就是 CIE 1931 XYZ 颜色匹配函数！

伪代码完整版：

  import numpy as np
  
  # 1) 实验得到的"原始 API"（有负值）
  cmf_rgb = load_cie1931_rgb()         # shape: (81, 3)
  V_lambda = load_photopic_curve()     # shape: (81,)
  
  # 2) 解 4 组约束 → 得到 M（CIE 1931 已经替我们解好）
  M = np.array([
      [0.4900, 0.3100, 0.2000],
      [0.1770, 0.8124, 0.0106],
      [0.0000, 0.0100, 0.9900],
  ]) / 0.17
  
  # 3) 一次矩阵乘法 → 得到"工程友好版 API"（全部 ≥ 0）
  cmf_xyz = cmf_rgb @ M.T              # shape: (81, 3)
  
  # 4) 验证：ȳ(λ) 应该等于 V(λ)
  assert np.allclose(cmf_xyz[:, 1], V_lambda, atol=1e-3)
```

        │                   ┌──────┴──────┐                   │

                            │ (3×3 矩阵)│
                            └─────────────┘
                            一次矩阵乘法
                          cmf_xyz = M @ cmf_rgb
```

> **核心要点**：
> - 实验数据（r̄/ḡ/b̄）= "真值锚点"，决定了变换的**起点**
> - 4 组约束 = "工程偏好"，决定了变换的**终点**长什么样
> - 矩阵 M = 连接两端的"桥"，**完全由线性代数推导**，没有任何"经验拍脑袋"
>
> 所以严格地说：
> **XYZ 坐标系 = 实验数据 × 工程约束 × 线性代数**
> 三者缺一不可。

#### 3.3.5 变换矩阵的最终结果

```
CIE 用线性规划方法解出满足上述 4 组约束的矩阵 M：

  ┌ X ┐         ┌ 0.4900  0.3100  0.2000 ┐ ┌ R ┐
  │ Y │ = 1/0.17·│ 0.1770  0.8124  0.0106 │ │ G │
  └ Z ┘         └ 0.0000  0.0100  0.9900 ┘ └ B ┘

  （这里 R、G、B 是 CIE RGB 的原始实验值，
    700.0/546.1/435.8nm 三原色）

程序员视角（如果让你写代码）：

  // 把 CIE RGB 实验数据转成 CIE XYZ
  Vec3 rgb_to_xyz(Vec3 rgb) {
      const float k = 1.0f / 0.17f;
      Mat3 M = {
          {0.4900f, 0.3100f, 0.2000f},
          {0.1770f, 0.8124f, 0.0106f},
          {0.0000f, 0.0100f, 0.9900f},
      };
      return k * (M * rgb);    // 一次矩阵乘法搞定
  }
  
  // 应用到颜色匹配函数（每个波长执行一次）
  for (int lambda = 380; lambda <= 780; lambda += 5) {
      Vec3 rgb_bar = lookup_cie_rgb(lambda);    // 实验查找表
      Vec3 xyz_bar = rgb_to_xyz(rgb_bar);
      cie_xyz_table[lambda] = xyz_bar;          // 新的 API 数据
  }

应用这个矩阵到 r̄、ḡ、b̄ 函数上，得到 x̄、ȳ、z̄：

       响应
        ↑                ȳ(λ)（=亮度）
    1.5 │              ╱╲
        │   x̄(λ)     ╱  ╲
    1.0 │    ╱╲     ╱    ╲
        │   ╱  ╲ ╱      ╲     ╱╲
    0.5 │  ╱   ╳         ╲   ╱  ╲ ← x̄ 在长波有第二峰
        │ ╱  ╱   ╲        ╲ ╱    ╲
    0.0 ─╱─╱───────╲─────────────────→ λ (nm)
       380   450  500  550  600   700
            ↑
          z̄(λ)

  📷 真实的 CIE 1931 XYZ 颜色匹配函数 x̄(λ)、ȳ(λ)、z̄(λ)：
  ├── Wikipedia：https://en.wikipedia.org/wiki/CIE_1931_color_space#Tristimulus_values
  └── Wikimedia 高清图：https://commons.wikimedia.org/wiki/File:CIE_1931_XYZ_Color_Matching_Functions.svg
     （能清楚看到 x̄ 长波处的"第二峰"和三条曲线全部 ≥ 0 的特性）

关键特性（工程师爱不释手的原因）：
├── 三条曲线全部 ≥ 0 ✅       → 代码不用处理负数
├── ȳ(λ) = 亮度响应 ✅        → 亮度独立成一维
├── 色域包含所有可见色 ✅     → 任何颜色都能表达
└── 满足线性性 ✅             → 跟其他空间转换都用矩阵乘法
```

#### 3.3.6 一句话总结：实验 + 理论是如何结合的？

```

### 3.4 XYZ 三刺激值的计算与意义


#### 3.4.1 计算任意光谱 P(λ) 的 XYZ 值

```
  对于任意光源 / 物体反射光，光谱分布为 P(λ)，则其三刺激值为：

       X = k · ∫₃₈₀⁷⁸⁰ P(λ) · x̄(λ) dλ
       Y = k · ∫₃₈₀⁷⁸⁰ P(λ) · ȳ(λ) dλ
       Z = k · ∫₃₈₀⁷⁸⁰ P(λ) · z̄(λ) dλ

  含义（一句话）：把光谱与人眼三种"匹配函数"加权求和。

程序员视角：积分 = 离散加权求和 = 一个 for 循环

  // 给定光谱（每 5nm 一个采样点），计算 XYZ
  Vec3 spectrum_to_xyz(float spectrum[81]) {  // 380~780, step=5
      float X = 0, Y = 0, Z = 0;
      for (int i = 0; i < 81; i++) {
          int lambda = 380 + i * 5;
          X += spectrum[i] * x_bar[lambda];  // 加权累加
          Y += spectrum[i] * y_bar[lambda];
          Z += spectrum[i] * z_bar[lambda];
      }
      const float k = 5.0f;  // Δλ = 5nm
      return {k * X, k * Y, k * Z};
  }

  // 数学公式 ↔ 代码 一一对应：
  //   ∫ P(λ)·x̄(λ) dλ   ≈   Σ P(λᵢ)·x̄(λᵢ)·Δλ   = 上面的 for 循环
  //   ↑                    ↑
  //   连续积分             离散求和（计算机能算）
```

#### 3.4.2 三个分量的物理 / 感知含义
```
  ┌────┬──────────────────────────────────────────────┐
  │ X  │ "类红色"响应（不严格等同于 L 锥）             │
  │    │ 经过数学变换后的红绿混合量                   │
  ├────┼──────────────────────────────────────────────┤
  │ Y  │ ⭐ 亮度（与人眼明视觉光视效率 V(λ) 完全一致） │
  │    │ 所有视频/图像处理中"luminance"的根源         │
  ├────┼──────────────────────────────────────────────┤
  │ Z  │ "类蓝色"响应（接近 S 锥但不严格相等）         │
  │    │ 主要表征短波信息                             │
  └────┴──────────────────────────────────────────────┘

  注意：X 和 Z 不是单纯的"红"和"蓝"！
  └── 它们是数学构造出来的"假想原色"，
      只是恰好与红、蓝有相关性（因为约束 ④ 让 Z 偏短波）
```

#### 3.4.3 XYZ 是所有色彩空间的"中央枢纽"
```
       sRGB ──┐                ┌── Lab
              │                │
   AdobeRGB ──┼──→ CIE XYZ ←──┼── Oklab
              │                │
   BT.2020 ──┘                └── YUV/YCbCr
                  ↑
            设备无关、客观
        所有跨空间转换的"中转站"

  → 任何两个色彩空间之间转换，都可以分两步：
    空间 A →（矩阵 / 公式）→ XYZ →（矩阵 / 公式）→ 空间 B

程序员视角：XYZ 就是色彩空间界的"通用中间表示（IR）"
  ├── LLVM IR：所有语言先编译到 IR，再编译到不同目标平台
  ├── XYZ：所有色彩空间先转到 XYZ，再转到目标空间
  └── 都是"星形拓扑"（hub-and-spoke），N² → N 的复杂度优化
  
  否则：
  ├── 有 10 个色彩空间 → 需要 10×9 = 90 个直接转换函数 ❌
  ├── 用 XYZ 中转 → 只需 10×2 = 20 个函数（A→XYZ + XYZ→A）✅
```

### 3.5 色度图（Chromaticity Diagram）


#### 3.5.1 为什么需要色度图？

本节按 5 步递进讲清楚：**动机 → 思路 → 公式 → 几何 → 疑问 → 总结**。

---

##### 【Step 1】动机：3D 太难用，必须降维

```
  XYZ 是 3 维空间，但作为人/工程师我们经常想问：
    ❓ "这两个颜色是不是同一种颜色？"（不管亮不亮）
    ❓ "这台显示器的色域有多大？"（在纸上画出来比较）
    ❓ "纯红色 700nm 在哪里？"（一个二维坐标即可定位）

  3D 空间的痛点：
  ├── 画不出来（论文/PPT/产品规格书都是 2D 平面）
  ├── 不好比较（两个 3D 立体 gamut 谁大谁小？看不出来）
  └── 维度冗余（"亮度"和"颜色种类"是两件事，不该混在一起）

  → 解决方案：把 3 维 (X, Y, Z) 拆成两部分
       ├── 2D 色度 (x, y)   ← 颜色种类（红？绿？粉？）
       └── 1D 亮度 Y         ← 明暗程度（明？暗？）
```

---

##### 【Step 2】核心思路：颜色种类 = "比例"，与亮度无关

```
  关键洞察：
    "同一种颜色的不同亮度版本"，在 (X, Y, Z) 中是一条从原点出发的射线。

  举个例子（同一种"暖黄色"，三种亮度）：
    昏暗版： (X, Y, Z) = (100, 200,  50)
    正常版： (X, Y, Z) = (200, 400, 100)   ← 全部 ×2
    明亮版： (X, Y, Z) = (300, 600, 150)   ← 全部 ×3

    它们 X:Y:Z 的比例都是 2:4:1 → 同一种颜色！

  所以——
    "颜色种类" = X、Y、Z 三者的比例（方向）
    "亮度"     = 这条射线上点离原点的距离（长度）

  程序员类比：
  ├── vec3 颜色 = vec3 方向 × 标量长度
  ├── 颜色种类 ≈ normalize(颜色)     ← 提取方向
  └── 亮度     ≈ length(颜色)         ← 保留长度
```

---

##### 【Step 3】数学定义：除以 (X+Y+Z) 提取比例

```
  既然要"提取比例"，就用每个分量除以三者之和：

       x = X / (X + Y + Z)
       y = Y / (X + Y + Z)
       z = Z / (X + Y + Z)

  天然性质：
       x + y + z ≡ 1   ← 三者占比之和恒为 1
       → z = 1 − x − y（冗余，所以只需保留 x、y 两个数）

  验证（前面三种亮度的暖黄色）：
       昏暗版：x = 100/350 = 0.286,  y = 200/350 = 0.571
       正常版：x = 200/700 = 0.286,  y = 400/700 = 0.571  ✓
       明亮版：x = 300/1050= 0.286,  y = 600/1050= 0.571  ✓
                                                   ↑
                       不管亮度怎么变，xy 完全相同！
```

---

##### 【Step 4】几何理解：投影到平面 X+Y+Z=1

```
  几何上，"除以 X+Y+Z" 等价于：把 (X,Y,Z) 沿原点射线，
  投影到平面 X+Y+Z=1 上的交点。

         Z
         │
         │     ╱ 平面 X+Y+Z=1
         │    ╱
       1 ●---╱
         │  ╱│ ← 平面与三轴的等距交点构成正三角形
         │ ╱ │
         │╱  │
         ●───●─────── X
        ╱│ 1
       ╱ │
      ╱  │
     Y   │

     ★ 同一种颜色（不同亮度）= 一条从原点射出的射线
     ★ 这条射线与平面 X+Y+Z=1 的交点 = 该颜色的 (x, y) 色度坐标
     ★ 不同亮度的同一种颜色，都映射到平面上的同一点

  程序员类比（GPU 透视除法）：

    GPU 渲染管线：
      [x, y, z, w] →（除以 w）→ [x/w, y/w, z/w]   投影到 w=1 平面
                                                  ↑ 屏幕坐标

    色度坐标：
      [X, Y, Z]   →（除以 X+Y+Z）→ [x, y, 1−x−y]  投影到 X+Y+Z=1 平面
                                                  ↑ 色度坐标

    同样的"齐次坐标除法"思想：
    └── 把"方向"和"长度"分离，丢弃长度，只画方向
```

---

###### 【Step 4.5.1】常见疑问：色度坐标 (x, y) 的坐标系到底在哪？

很多人第一次看到色度图都会困惑：
> "投影点明明在一个**倾斜的 3D 平面 X+Y+Z=1** 上，为什么色度图却是一张平躺的 2D 图？这个 (x, y) 是相对哪个坐标系建立的？"

**短答**：`(x, y)` **不是**在平面 X+Y+Z=1 内部建立的"内蕴 2D 坐标系"，而是**直接借用 3D 笛卡尔空间的 X 轴和 Y 轴的值**——把投影点的 X 分量改名叫 `x`、Y 分量改名叫 `y`，Z 分量丢弃（因为 z = 1 − x − y 可推算）。

```
  ┌─────────────────────────────────────────────────────────┐
  │ 三层坐标系的关系                                         │
  └─────────────────────────────────────────────────────────┘

  第一层：原始 3D 空间 (X, Y, Z)
    坐标系：标准笛卡尔系，X、Y、Z 三轴互相垂直
    例如：暖黄色 = (200, 400, 100)
                          │
                          │  投影到平面 X+Y+Z=1
                          ▼
  第二层：投影后的交点（仍是 3D 点！）
    坐标系：还是同一个 3D 笛卡尔系，只是被约束在斜平面上
    例如：(200, 400, 100) → (0.286, 0.571, 0.143)
                              ↑      ↑      ↑
                              X'     Y'     Z'
    性质：X' + Y' + Z' = 1（恒成立）
                          │
                          │  起别名（不做任何变换！）
                          ▼
  第三层：色度坐标 (x, y)
    定义：x ≡ X',  y ≡ Y',  z ≡ Z'（z 冗余，丢掉）
    例如：(x, y) = (0.286, 0.571)
    ⚠ x 不是斜平面上的"水平方向"，y 也不是"垂直方向"
       它们就是 3D 中投影点的 X 分量、Y 分量本身
```

**关键澄清：色度图上的 x 轴 = 3D 中的 X 轴（不是平面内新建的轴）**

```
  当色度图画在纸上时（CIE 1931 xy 色度图）：

       y ↑
     0.8 │      ★ 520nm
         │     ╱ ╲
     0.6 │   ╱     ╲
         │  ╱       ╲
     0.4 │ ╱         ╲ ★ 700nm
         │╱  ●D65    ╱
     0.2 │ ╲       ╱
         │  ╲___╱
     0.0 │★ 380nm
         └────────────────→ x
         0.0  0.2  0.4  0.6  0.8

  纸上"x 轴向右、y 轴向上"，本质是：
    把 3D 投影点的 X 分量当横轴、Y 分量当纵轴展示。
    并不是在斜平面 X+Y+Z=1 上重新建立了 2D 坐标系。
```

**几何本质：色度图 = 沿 Z 轴俯视斜三角形的"俯视图"**

整个过程其实是"投影的投影"：

```
  Step A：3D 颜色 (X, Y, Z) ──沿原点射线投影到斜平面──→ (X', Y', Z')
  Step B：(X', Y', Z') ──沿 Z 轴正交投影到 X-Y 平面──→ (X', Y') ≡ (x, y)
                                                         ↑
                                                  色度图上画的点

  几何示意：

         Z
         │
       1 ●          ← 斜平面 X+Y+Z=1 与 Z 轴交点
         │\
         │ \
         │  \  ← 倾斜的等边三角形（正三角形）
         │   \
         │    \
         ●─────●─── X
        ╱       1
       ╱
      ●  Y=1
      │
      Y

  从 +Z 方向往下看（沿 Z 轴俯视），三角形的轮廓变成：

         Y
         │
       1 ●
         │ ╲
         │   ╲      ← 直角等腰三角形 x+y ≤ 1
         │     ╲
         │       ╲
         ●─────────●── X
                   1

  色度图上的"马蹄形"可见色域，就在这个直角三角形内部。
  这也是为什么色度图有隐含约束 x + y ≤ 1（因为 z = 1−x−y ≥ 0）。
```

**程序员类比：3D 渲染的正交投影**

```
  3D 渲染管线：
    模型坐标 (X, Y, Z)
         ↓ 投影变换（这里 = 投影到 X+Y+Z=1 平面）
    视图空间 (X', Y', Z')
         ↓ 正交投影（丢弃 Z 分量）
    屏幕坐标 (x, y) = (X', Y')

  色度图的 (x, y) 等价于：
    "把 3D 颜色空间归一化到斜平面后，沿 Z 轴方向看的俯视图坐标"
```

**用暖黄色完整走一遍验证**

```
  原始：(X, Y, Z) = (200, 400, 100)

  Step 1: 投影到 X+Y+Z=1
    s = 200 + 400 + 100 = 700
    投影点 = (200/700, 400/700, 100/700)
           = (0.286, 0.571, 0.143)
    验证：0.286 + 0.571 + 0.143 = 1.000  ✓
    （这是 3D 空间中的一个点，位于斜平面上）

  Step 2: 取前两个分量作色度坐标
    (x, y) = (0.286, 0.571)
    （坐标系 = 3D 空间的 X 轴 + Y 轴，"借用"过来的）

  Step 3: 在色度图纸上绘制
    沿色度图横轴(x) = 0.286，纵轴(y) = 0.571 → 标点
```

**一句话回答**

```
  ┌─────────────────────────────────────────────────────────┐
  │  (x, y) 不是斜平面内部建立的新 2D 坐标系，               │
  │  而是直接借用 3D 笛卡尔空间的 X、Y 轴：                  │
  │                                                          │
  │    x = 投影点的 X 分量    （范围 [0, 1]）                │
  │    y = 投影点的 Y 分量    （范围 [0, 1]）                │
  │    z = 1 − x − y          （冗余，已丢弃）               │
  │                                                          │
  │  色度图 = 倾斜三角形沿 Z 轴方向看的"俯视图"。            │
  └─────────────────────────────────────────────────────────┘
```

---

##### 【Step 5】常见疑问：Y 是亮度，为啥不投影到 Y=1 平面？

直觉："去掉亮度 = 固定 Y = 投到 Y=1 平面"——这是错的。**关键在于"去掉亮度"≠"固定 Y"**。

```
  正解："去掉亮度" = 提取比例 = 除以"总刺激量" (X+Y+Z)，而不是除以 Y。

  Y=1 方案三个致命缺陷：

  ❌ 缺陷 ①：Y=0 时除以零 → 数值崩溃
     纯蓝紫（380nm）：X=0.0014, Y≈0.000004, Z=0.0065
     → X/Y ≈ 350,  Z/Y ≈ 1625    ← 数值爆炸
     → 越深的蓝紫色，坐标越趋向无穷大
     程序员视角：normalize(v) 但 v.length()→0 → NaN

  ❌ 缺陷 ②：丢失大量颜色
     Y=0 的颜色（极暗的纯色）在 Y=1 平面上根本无法表示
     → 部分可见色被"截断"

  ❌ 缺陷 ③：坐标无"占比"语义
     x = X/(X+Y+Z) 满足 x+y+z=1，每个值是"占总量的百分比"，有界 [0,1]
     X/Y 是无界比值，可以是 0.01 也可以是 1000，无直观含义
```

---

##### 【Step 6】完整表示：(x, y, Y) — 色度 + 亮度

```
  色度图丢掉的 Y（亮度）并没有真正消失——把它单独保留即可：

       (X, Y, Z)  ←→  (x, y, Y)
         3D                2D 色度 + 1D 亮度

  正向：XYZ → xyY
       x = X / (X + Y + Z)
       y = Y / (X + Y + Z)
       Y = Y                       ← 亮度直接保留

  反向：xyY → XYZ（无损还原）
       X = (x / y) · Y
       Y = Y
       Z = ((1 − x − y) / y) · Y

  代码实现：
       struct CIE_XYZ { float X, Y, Z; };
       struct CIE_xyY { float x, y, Y; };

       CIE_xyY XYZ_to_xyY(CIE_XYZ c) {
           float s = c.X + c.Y + c.Z;
           return { c.X / s, c.Y / s, c.Y };
       }
       CIE_XYZ xyY_to_XYZ(CIE_xyY c) {
           return { (c.x / c.y) * c.Y,
                     c.Y,
                    ((1 - c.x - c.y) / c.y) * c.Y };
       }
```

---

##### 一句话总结

```
  ┌─────────────────────────────────────────────────────────┐
  │  色度坐标 (x, y) = 把 (X,Y,Z) 沿原点射线投影到           │
  │                    平面 X+Y+Z=1 得到的 2D 坐标            │
  │                                                          │
  │  本质 = 提取 X:Y:Z 的"比例"（颜色种类），丢弃"长度"（亮度）│
  │                                                          │
  │  与 normalize(vec3) 同思想，但用 L1 范数（X+Y+Z）        │
  │  而非 L2 范数（√(X²+Y²+Z²)），保证坐标有"占比"含义        │
  │                                                          │
  │  亮度信息单独用 Y 保留 → 完整表示 (x, y, Y) 可无损还原 XYZ│
  └─────────────────────────────────────────────────────────┘
```

  └── 黑色 = (X=0, Y=0, Z=0)，归一化时 0/0 没定义！

### 3.6 章节回顾：程序员的 5 个 Take-Away

```
  ① CIE XYZ = 一个"标准化的人眼颜色 API"，输入光谱，输出 (X,Y,Z)
  
  ② 这个 API 是怎么实现的？
     实验数据（Wright/Guild）+ 线性变换（CIE 矩阵）= XYZ
  
  ③ 为什么要做 RGB→XYZ 这个变换？
     ├── 消除负值（unsigned 比 signed 友好）
     ├── 让 Y = 亮度（关注点分离）
     ├── 让 (1,1,1) = 白点（标准化）
     └── 让 Z 在长波 ≈ 0（稀疏化）
  
  ④ XYZ 在整个色彩工程中的地位 = LLVM IR 在编译器中的地位
     所有色彩空间 ↔ XYZ ↔ 所有色彩空间，N² → N
  
  ⑤ 色度图 (x, y) = XYZ 的"齐次坐标投影"
     去掉亮度维度，专注"颜色种类"
     → 这就是为什么色域用三角形面积比较

  → 看完这章，下一章讲 RGB 色彩空间时，你会突然发现：
    "原来 sRGB / AdobeRGB / BT.2020 都是在色度图里画三角形啊！"
    "原来 RGB→XYZ 矩阵就是在做基变换啊！"
```

#### 4.0.1 矩阵的"两段式"来源结构

每个色彩空间的矩阵都遵循同一个套路：

```
  ┌────────────────────┐         ┌────────────────────┐
  │  STEP 1：人为约定   │ ──────→ │  STEP 2：数学推导   │
  │  （开会拍板）       │         │  （线性代数）       │
  └────────────────────┘         └────────────────────┘
       │                              │
       │                              │
       ▼                              ▼
  • 原色坐标 (xR,yR)…           • RGB ↔ XYZ 矩阵
  • 白点 (xW,yW)                • RGB ↔ YCbCr 矩阵
  • 亮度参考曲线 ȳ(λ)           • XYZ ↔ Lab 公式
  • 色差通道方向                 • LMS ↔ Oklab 矩阵

  程序员类比：
    STEP 1 = 配置文件（人为指定的常量）
    STEP 2 = 编译过程（按规则自动生成）
    矩阵   = 编译产物（确定且唯一）
```

#### 4.0.3 为什么会有这么多矩阵？

```
  根本原因：STEP 1 的"人为约定"在不同时代/场景下各不相同

  ① 显示技术演进 → 原色变了 → 矩阵变了
     1953: NTSC 荧光粉   → BT.601 系数 (0.299, 0.587, 0.114)
     1990: HD CRT 荧光粉 → BT.709 系数 (0.2126, 0.7152, 0.0722)
     2012: 量子点/激光   → BT.2020 系数 (0.2627, 0.6780, 0.0593)

  ② 应用目的不同 → 通道方向变了 → 矩阵变了
     视频压缩  → YCbCr（亮度+色差，便于色度下采样）
     人类直觉  → HSV/HSL（色相+饱和度+明度）
     感知均匀  → Lab/Oklab（ΔE 距离 ≈ 视觉差异）
     HDR 编码  → ICtCp（高动态范围下的均匀性）

  ③ 算力/带宽约束变了 → 设计取舍变了
     1953 模拟电视：YIQ 让 Q 通道带宽减半
     2010s 流媒体：YCbCr + 4:2:0 色度下采样
     2020s HDR：ICtCp 优化 PQ 曲线下的均匀性
```

### 4.1 为什么选 RGB？

```
为什么显示器用 RGB 而不是其他三原色？

  原因 1：对应人眼三种视锥细胞（L/M/S）
  └── 红绿蓝分别刺激三种视锥细胞，可以混合出大多数颜色

  原因 2：物理实现方便
  └── 红色 LED/荧光粉（~630nm）
      绿色 LED/荧光粉（~530nm）
      蓝色 LED/荧光粉（~460nm）

  原因 3：历史沿革
  └── 彩色摄影（1861年，Maxwell）
      彩色电视（1950s）
      计算机显示器（1970s）
      都采用 RGB 三原色体系

  重要认知：
  ├── 不同设备的 RGB 不一样！
  │   ├── 你的手机 RGB ≠ 你的显示器 RGB ≠ 打印机 RGB
  │   └── 同样的 (255, 0, 0) 在不同设备上显示的颜色不同
  └── 这就是为什么需要"色彩管理"和"色彩空间标准"
```

### 4.2 sRGB（1996年）

```
sRGB（Standard RGB）：
  发布：1996年，HP + Microsoft 联合制定
  目标：为互联网和消费电子建立统一的色彩标准

  sRGB 的定义包含三个部分：

  1. 原色坐标（色度图上的位置）：
     ┌──────┬────────┬────────┐
     │      │   x    │   y    │
     ├──────┼────────┼────────┤
     │  R   │ 0.6400 │ 0.3300 │
     │  G   │ 0.3000 │ 0.6000 │
     │  B   │ 0.1500 │ 0.0600 │
     │ 白点 │ 0.3127 │ 0.3290 │（D65 标准光源）
     └──────┴────────┴────────┘

     ┌─ 这些坐标是怎么来的？─────────────────────────────┐
     │                                                   │
     │  ✗ 不是经验值（有严格的物理测量基础）              │
     │  ✗ 不是矩阵推导（矩阵是从坐标推导出来的，方向相反）│
     │  ✓ 是"物理测量 + 标准委员会选定"的结果            │
     │                                                   │
     │  来源链：                                          │
     │  CIE 1931 颜色匹配实验 → 建立 (x,y) 色度坐标系   │
     │  → 测量 1990s CRT 荧光粉的发光光谱                │
     │  → 计算其在色度图上的坐标                          │
     │  → HP+Microsoft 选取代表性数值并圆整               │
     │  → 写入 IEC 61966-2-1 标准                        │
     │                                                   │
     │  因果关系：先有坐标（物理测量），                   │
     │  再推导出 RGB→XYZ 转换矩阵（数学推导）            │
     └───────────────────────────────────────────────────┘

     ┌─ 来源链中两个关键步骤的技术细节 ────────────────────────────┐
     │                                                              │
     │  上面来源链中有两步可能让程序员困惑：                          │
     │  ① "测量 CRT 荧光粉的发光光谱" —— 怎么测？测出什么？           │
     │  ② "计算其在色度图上的坐标"  —— 用什么公式算？                │
     │  下面分别展开。                                              │
     └──────────────────────────────────────────────────────────────┘

     ────────────────────────────────────────────────────────
     【步骤 ①】测量 CRT 荧光粉的发光光谱
     ────────────────────────────────────────────────────────

     目标：得到红/绿/蓝三个荧光粉各自的"光谱功率分布"
           SPD(λ)，即每个波长 λ 的相对辐射强度

     ┌─ 实验装置（程序员视角）───────────────────────────────────┐
     │                                                            │
     │     ┌─────────────────┐    黑暗室     ┌──────────────────┐│
     │     │  CRT 显示器     │ ──────────→  │ 分光辐射计       ││
     │     │ （只点亮一种    │   (优化信噪   │ (Spectro-        ││
     │     │   荧光粉）      │     比)      │  radiometer)     ││
     │     └─────────────────┘              └──────────────────┘│
     │                                              │             │
     │                                              ▼             │
     │                                       光谱功率分布          │
     │                                       SPD(λ)（数组）       │
     └────────────────────────────────────────────────────────────┘

     操作流程（伪代码视角）：

       # 1) 分别点亮三个荧光粉（不同信号驱动）
       for phosphor in ["red", "green", "blue"]:
           # 让 CRT 只显示该原色（如全屏纯红）
           drive_crt(phosphor, max_intensity=1.0)

           # 2) 用分光辐射计在 380~780nm 范围逐波长扫描
           SPD = {}                       # dict: wavelength → power
           for λ in range(380, 781, 1):   # 1nm 步长（仪器决定）
               SPD[λ] = spectroradiometer.measure(λ)

           save(phosphor, SPD)            # 得到 SPD_R, SPD_G, SPD_B

     ┌─ 那 spectroradiometer.measure(λ) 里的 power 到底怎么测？─┐
     │                                                          │
     │  这是程序员最容易卡住的地方——"光的功率"究竟是个啥？     │
     │  下面分 3 层讲清楚。                                     │
     └──────────────────────────────────────────────────────────┘

     ▶ 第 1 层：power 的物理定义（先搞清是什么）

       power 不是电源插座那种"瓦特"，而是光辐射的能量速率：

         单位：W/(m²·sr·nm)
              │   │   │   └─ 每 1nm 波长间隔
              │   │   └───── 每 1 球面度立体角（光线方向）
              │   └───────── 每 1 平方米发光面
              └───────────── 瓦特（能量/秒）

       一句话翻译：
         "在波长 λ 这个非常窄的小窗口里，从屏幕单位面积、
          沿一个方向、每秒射出的辐射能量有多少。"

       这个量有个专业名：光谱辐射亮度（Spectral Radiance, Lₑ,Ω,λ）
       —— 这就是 SPD[λ] 里存的那个数。

     ▶ 第 2 层：分光辐射计内部到底做了什么（仪器原理）

       一台分光辐射计（spectroradiometer）= 三件套：

       ┌───────────┐   ┌─────────────────┐   ┌──────────────┐
       │ ① 光学   │ → │ ② 分光元件     │ → │ ③ 光电探测  │
       │   入射口 │   │ (光栅/棱镜)    │   │   器(CCD等) │
       └───────────┘   └─────────────────┘   └──────────────┘
            │                  │                    │
        汇聚屏幕         按波长把光"散开"      把光子能量
        发出的光          像三棱镜分白光         转成电流/电压
                                                    │
                                                    ▼
                                              A/D 转换 → 数字

       内部完整流程（程序员视角的伪代码）：

         def measure(λ):
             # ① 入射光通过狭缝，限定方向和面积
             beam = entrance_optics.collect(area, solid_angle)

             # ② 光栅旋转到指定波长 λ（或 CCD 阵列直接选位置 λ）
             beam_λ = diffraction_grating.select(beam, λ,
                                                  bandwidth=1nm)

             # ③ 光电传感器把光子能量 → 电流
             #    每个光子的能量 E = h·c/λ（普朗克公式）
             current = photodetector.photons_to_current(beam_λ)

             # ④ 经积分电路 + A/D 得到数字读数（counts）
             raw_counts = ADC.read(current, integration_time)

             # ⑤ 用工厂标定系数把 counts 换算成绝对辐射量
             #    标定来源：用已知功率的标准灯（如卤钨灯）校准过
             power = raw_counts * calibration_coefficient[λ]
             return power           # 单位 W/(m²·sr·nm)

       关键点：
         • 真正的"测量"动作 = ③ 数光子 → 转电流 → 转数字
         • "已知波长"靠的是 ② 光栅几何（角度 ↔ 波长有精确公式）
         • "已知绝对值"靠的是 ⑤ 出厂前用标准灯做的辐射定标

     ▶ 第 3 层：实际数字长什么样（让概念落地）

       假设测某红荧光粉，仪器返回的 SPD 字典片段：

         SPD = {
           ...
           600: 0.0123,    # W/(m²·sr·nm)
           605: 0.0456,
           610: 0.0892,    # ← 峰值！红原色主要能量集中在这
           615: 0.0501,
           620: 0.0210,
           ...
         }

       工程上常做的简化处理：

         # 绝对值（瓦特）一般无所谓，因为色度只看"形状"
         # 所以工程师常直接归一化为相对 SPD：
         max_val = max(SPD.values())
         SPD_relative = {λ: v / max_val for λ, v in SPD.items()}
         # 现在峰值=1.0，其他点都是 0~1 之间的相对值

       为什么可以归一化？
         回看步骤②的色度计算公式：
            x = X / (X+Y+Z),  y = Y / (X+Y+Z)
         分子分母同乘任何常数 k，(x,y) 不变 → 绝对量级被消掉了
         → 测"相对光谱形状"就够了，色度坐标和亮度无关

       ┌────────────────────────────────────────────────────────┐
       │ power = 分光辐射计在波长 λ 处"数到的光子能量速率"，   │
       │          通过光栅选波长 + 光电探测器转电信号 + 工厂   │
       │          辐射定标得到。对色度计算来说，只关心它的     │
       │          相对大小（光谱形状），不在乎绝对瓦特数。     │
       └────────────────────────────────────────────────────────┘

       三个荧光粉 = 三个这样的向量

     │   SPD(λ)   ──积分──→   (X,Y,Z)   ──归一化──→   (x,y)     │

     Step 2.1：用颜色匹配函数对 SPD 加权积分得到 XYZ

       这是把"物理光谱"翻译成"人眼看到的颜色"的核心公式：

         X = k · ∫ SPD(λ) · x̄(λ) dλ
         Y = k · ∫ SPD(λ) · ȳ(λ) dλ
         Z = k · ∫ SPD(λ) · z̄(λ) dλ

       其中 x̄(λ), ȳ(λ), z̄(λ) 来自第 3 章的 CIE 1931 标准
       观察者数据（实验测出的"标准人眼"），是已知常数表。

       离散版本（实际工程中这样算）：

         X = Σ SPD(λ_i) · x̄(λ_i) · Δλ     # i = 0..400
         Y = Σ SPD(λ_i) · ȳ(λ_i) · Δλ
         Z = Σ SPD(λ_i) · z̄(λ_i) · Δλ

       ┌─ 这个公式到底在做什么？（程序员视角的清晰解释）──────────┐
       │                                                          │
       │  很多人卡在这个公式：一堆积分符号到底什么意思？           │
       │  其实它就是一个超级简单的操作 —— "加权求和"。            │
       └──────────────────────────────────────────────────────────┘

       ▶ 第 1 步：把"积分"翻译成程序员熟悉的"循环求和"

         数学符号：    X = ∫ SPD(λ) · x̄(λ) dλ
         程序员翻译：  X = sum(SPD[i] * x_bar[i] for i in 0..400)

         没了。就是把两个 401 维数组对应位置相乘，再加起来。
         （这在 NumPy 里就是一行 `np.dot(SPD, x_bar)`）

       ▶ 第 2 步：搞清楚两个数组的"角色"

         SPD(λ)：    "光本身发出了什么"（物理世界的信号）
                    例如：在 610nm 这个波长，发出了 0.089 W 能量
                          在 450nm 这个波长，发出了 0.001 W 能量
                    → 是一个客观的、用仪器就能测的物理量

         x̄(λ) 等：  "人眼有多敏感"（生理世界的接收器特性）
                    例如：x̄(610nm) = 1.0  → 人眼对 610nm 的红很敏感
                          x̄(450nm) = 0.3  → 人眼对 450nm 的红没那么敏感
                    → 是 CIE 1931 用 17 个志愿者测出的"标准人眼"

       ▶ 第 3 步：相乘的物理意义 —— "过滤 + 称重"

         在每个波长 λ 上：

           SPD(λ)  ×  x̄(λ)  =  这个波长贡献给"X 通道"的量
           ─────     ─────     ──────────────────────────
           光发出    人眼对该    (类比：信号 × 增益 = 实际感受)
           的能量    波长的反应

         举个例子（程序员能秒懂的版本）：

           想象人眼有 3 个"颜色传感器" X、Y、Z
           每个传感器都有自己的"频率响应曲线"x̄(λ), ȳ(λ), z̄(λ)
           （就像音频领域的均衡器 EQ 曲线！）

           现在有一束光打进来，光谱是 SPD(λ)
           → 每个传感器读数 = ∫ 光谱 × 自己的响应曲线 dλ
           → 这正是上面的积分公式！

       ▶ 第 4 步：为什么是"求和"而不是别的？—— 线性叠加性

         人眼对光的响应是线性的（实验已验证，第 3.2 节）：

           "看 1 份 500nm 光 + 1 份 600nm 光"
           = "看 1 份 500nm 光" + "看 1 份 600nm 光"
                                  ↑ 两次响应直接相加

         所以面对一个连续光谱 SPD(λ)：
           把它切成无数个微小波长段，每段引起的响应叠加起来
           → 数学上就是积分：∫ SPD(λ) · x̄(λ) dλ

       ▶ 第 5 步：一个具象类比（彻底打通直觉）

         把这个公式类比成"考试加权总分"：

           ┌──────────┬──────────┬──────────┬──────────┐
           │  科目    │  你的得分 │ 学校权重 │  贡献分  │
           ├──────────┼──────────┼──────────┼──────────┤
           │ 380nm    │ SPD(380) │ x̄(380) │   相乘   │
           │ 381nm    │ SPD(381) │ x̄(381) │   相乘   │
           │  ...     │   ...    │   ...    │   ...    │
           │ 780nm    │ SPD(780) │ x̄(780) │   相乘   │
           ├──────────┴──────────┴──────────┼──────────┤
           │             总分（X 通道）       │  全加起来 │
           └─────────────────────────────────┴──────────┘

         • 每个波长 = 一个"科目"
         • 你的得分 = 光在该波长发出的能量（物理量）
         • 学校权重 = 人眼对该波长的敏感度（生理量）
         • 总分 = 这盏灯刺激人眼"红通道"的强度 = X

         同理，用 ȳ(λ) 当权重 → 算出 Y（绿/亮度通道）
              用 z̄(λ) 当权重 → 算出 Z（蓝通道）

         三组权重 = 三个"评分标准" → 一份光谱被打成 3 个分数
         这 3 个分数就唯一确定了"这盏灯看起来是什么颜色"。

       ▶ 第 6 步：为什么 3 个数就够描述颜色？

         因为人眼只有 3 种锥细胞（L/M/S）！
         不管光谱多复杂（401 维信号），最终都被压缩到 3 个神经信号。
         这就是色彩科学最核心的事实：

           物理世界：光谱是 ∞ 维的（连续函数）
                            ↓ 人眼是个"3 通道传感器"
           感知世界：只剩 3 个数 (X, Y, Z)

         上面的积分公式 = 模拟人眼这个"401→3 的降维过程"
         → 这就是公式存在的根本原因。

       ▶ 一句话彻底总结

         ┌────────────────────────────────────────────────────────┐
         │  公式 X = ∫ SPD(λ)·x̄(λ) dλ                          │
         │   = 用"标准人眼的红色敏感度曲线 x̄"                    │
         │     去"称量"这束光在每个波长发出的能量，               │
         │     再把所有波长的"称重结果"加起来。                   │
         │                                                        │
         │  程序员一行代码版：                                    │
         │     X, Y, Z = SPD @ CMF   # 矩阵乘法，401→3 降维      │
         └────────────────────────────────────────────────────────┘

     Step 2.2：Python 风格的可运行伪代码

       import numpy as np

       # 输入：荧光粉光谱（401 维）和 CIE 1931 CMF（401×3 表）
       wavelengths = np.arange(380, 781, 1)
       SPD_R   = measure_phosphor("red")        # shape (401,)
       cmf     = load_CIE1931_CMF()             # shape (401, 3)
                                                # 列：x̄, ȳ, z̄
       # ① 加权积分
       X = np.sum(SPD_R * cmf[:, 0])            # ∫ SPD·x̄ dλ
       Y = np.sum(SPD_R * cmf[:, 1])            # ∫ SPD·ȳ dλ
       Z = np.sum(SPD_R * cmf[:, 2])            # ∫ SPD·z̄ dλ

       # ② 归一化得到色度坐标（投影到 X+Y+Z=1 平面）
       x = X / (X + Y + Z)
       y = Y / (X + Y + Z)

       print(f"红荧光粉色度坐标: ({x:.4f}, {y:.4f})")
       # 实际输出 ≈ (0.6400, 0.3300) ← 这就是 sRGB 红原色！

     Step 2.3：直觉理解（一句话总结）

       ┌──────────────────────────────────────────────────────┐
       │  SPD(λ) 描述"灯本身发什么光"（物理量）                │
       │  CMF (x̄,ȳ,z̄) 描述"人眼如何看光"（生理量）          │
       │  两者相乘求和 = "人眼看到这盏灯是什么颜色" → XYZ     │
       │  再做归一化 = 在色度图上找到对应的 (x, y) 点         │
       └──────────────────────────────────────────────────────┘

     ────────────────────────────────────────────────────────
     全流程一图回顾
     ────────────────────────────────────────────────────────

       CRT 荧光粉物理发光
              │
              │ 分光辐射计扫描（步骤①）
              ▼
       SPD_R(λ), SPD_G(λ), SPD_B(λ)        ← 401 维向量 ×3
              │
              │ 与 CIE 1931 CMF 加权积分（步骤②.1）
              ▼
       (X_R,Y_R,Z_R), (X_G,Y_G,Z_G), (X_B,Y_B,Z_B)

              │
              │ HP+Microsoft 圆整 + 标准化
              ▼
       sRGB / BT.709 三原色坐标（写进 IEC 61966-2-1）

     从坐标到矩阵的推导过程（以 sRGB 红色原色为例）：

     ┌─ 输入：物理测量得到的色度坐标 ───────────────────┐
     │  R: (x=0.6400, y=0.3300)                         │
     │  G: (x=0.3000, y=0.6000)                         │
     │  B: (x=0.1500, y=0.0600)                         │
     │  白点 D65: (x=0.3127, y=0.3290)                  │
     └─────────────────────────────────────────────────┘
             ↓
     Step 1: 色度坐标 (x,y) → XYZ 坐标（令 Y=1 归一化）

       公式：X = x/y,  Y = 1,  Z = (1-x-y)/y

       红色原色：X_R = 0.64/0.33 = 1.9394
                 Y_R = 1.0000
                 Z_R = (1-0.64-0.33)/0.33 = 0.0909

       同理算出 G、B 的 XYZ，组成原色矩阵 M：

       M = │ 1.9394  0.5000  3.0000 │  ← 每列 = 一个原色
           │ 1.0000  1.0000  1.0000 │
           │ 0.0909  0.6667 13.1667 │
             ↓
     Step 2: 求缩放系数（让三原色加起来 = 白点）

       白点 XYZ：X_W = 0.3127/0.3290 = 0.9505
                 Y_W = 1.0000
                 Z_W = (1-0.3127-0.3290)/0.3290 = 1.0891

       解方程：M · [S_R, S_G, S_B]ᵀ = [X_W, Y_W, Z_W]ᵀ
       → S_R = 0.6444, S_G = 1.1919, S_B = 0.0950
             ↓
     Step 3: 构建最终矩阵（每列 = 缩放系数 × 原色 XYZ）

       RGB→XYZ = │ S_R·X_R  S_G·X_G  S_B·X_B │
                 │ S_R·Y_R  S_G·Y_G  S_B·Y_B │
                 │ S_R·Z_R  S_G·Z_G  S_B·Z_B │

               = │ 0.4124  0.3576  0.1805 │
                 │ 0.2126  0.7152  0.0722 │  ← 这就是 sRGB→XYZ 矩阵！
                 │ 0.0193  0.1192  0.9505 │
             ↓
     ┌─ 结论 ──────────────────────────────────────────┐
     │  整个过程：纯数学推导，无任何经验参数             │
     │  唯一的物理输入 = 上面那张坐标表（4个点8个数）    │
     │  详细推导见 4.4 节                               │
     └─────────────────────────────────────────────────┘

  2. 白点（White Point）：D65（色温约 6504K）
     └── 模拟正午日光的颜色

  3. Gamma 曲线：约 2.2（实际是分段函数，见第九章）

  sRGB 的局限：
  ├── 色域较小（约覆盖 CIE 色度图的 35%）
  ├── 无法表示高饱和度颜色（如鲜艳的绿色、青色）
  └── 不适合专业摄影/印刷/电影制作

### 4.4 RGB 转换矩阵的推导

> 这是理解所有色彩空间转换的核心！

```
问题：如何在两个 RGB 色彩空间之间转换？
      例如：sRGB → BT.2020

步骤一：确定各色彩空间的原色 XYZ 坐标

  已知原色的色度坐标 (x, y)，求 XYZ：
  
  对于色度坐标 (x, y)，令 Y = 1（归一化亮度）：
  X = x / y
  Y = 1
  Z = (1 - x - y) / y

  例：sRGB 红色原色 (x=0.64, y=0.33)：
  X_R = 0.64/0.33 = 1.9394
  Y_R = 1.0000
  Z_R = (1-0.64-0.33)/0.33 = 0.0909

步骤二：确定白点的 XYZ 坐标

  D65 白点 (x=0.3127, y=0.3290)，令 Y = 1：
  X_W = 0.3127/0.3290 = 0.9505
  Y_W = 1.0000
  Z_W = (1-0.3127-0.3290)/0.3290 = 1.0891

步骤三：求各原色的缩放系数

  设三原色矩阵 M = [R_xyz | G_xyz | B_xyz]（列向量）
  
  白点 = S_R · R_xyz + S_G · G_xyz + S_B · B_xyz
  
  即：[X_W]   [X_R X_G X_B] [S_R]
      [Y_W] = [Y_R Y_G Y_B] [S_G]
      [Z_W]   [Z_R Z_G Z_B] [S_B]
  
  解方程：[S_R, S_G, S_B]ᵀ = M⁻¹ · [X_W, Y_W, Z_W]ᵀ

步骤四：构建 RGB → XYZ 转换矩阵

  M_RGB_to_XYZ = [S_R·R_xyz | S_G·G_xyz | S_B·B_xyz]
  
  即：每列 = 缩放系数 × 原色 XYZ 坐标

步骤五：两个 RGB 空间之间的转换

  RGB_A → XYZ → RGB_B
  
  M_A_to_B = M_B_to_XYZ⁻¹ · M_A_to_XYZ

  ┌─────────────────────────────────────────────────────┐
  │  核心思想：                                          │
  │  每个 RGB 色彩空间都有自己的"坐标系"（原色+白点）    │
  │  XYZ 是"绝对坐标系"（与设备无关）                   │
  │  转换矩阵 = 从一个坐标系变换到另一个坐标系的变换矩阵 │
  └─────────────────────────────────────────────────────┘
```

  │  • 1931：CIE 建立 XYZ 色度坐标系（颜色匹配实验）        │
  │  • 1953：NTSC 首次为电视定义 RGB 原色坐标                │
  │  • 1990：BT.709 定义高清电视原色（与 sRGB 相同）         │
  │  • 1996：HP+Microsoft 发布 sRGB 标准，正式给出此矩阵     │
  │  • 1999：IEC 61966-2-1 国际标准化                        │
  │                                                          │
  │  注意：矩阵中的数值精度取决于原色坐标的精度。            │
  │  不同来源给出的数值可能在小数点后第 5~7 位有微小差异，   │
  │  这是因为中间计算的舍入方式不同，但不影响实际使用。       │
  └──────────────────────────────────────────────────────────┘

  如何理解这个矩阵？
  
  以第一行为例：X = 0.4124·R + 0.3576·G + 0.1804·B
  ├── X 主要由 R 和 G 贡献（红色和绿色对 X 影响大）
  ├── B 的贡献最小（蓝色对 X 影响小）
  └── 这反映了 X 颜色匹配函数 x̄(λ) 的形状特征
  
  以第二行为例：Y = 0.2126·R + 0.7152·G + 0.0722·B
  ├── G 的权重最大（0.7152）！
  ├── R 次之（0.2126），B 最小（0.0722）
  └── 这正是人眼对绿色最敏感的体现（亮度感知）
      → 这也是为什么视频编码中绿色通道最重要！
```

## 五、YCbCr / YUV——视频编码的核心

### 5.1 为什么要从 RGB 转到 YCbCr？

```
三个核心原因：

  原因 1：人眼对亮度更敏感，对色度不敏感
  ├── 人眼有约 1.2 亿个视杆细胞（感知亮度，高分辨率）
  ├── 人眼有约 600 万个视锥细胞（感知颜色，低分辨率）
  └── → 可以对色度信息降采样（4:2:0），节省 50% 数据量！

  原因 2：向后兼容黑白电视
  └── 黑白电视只需要亮度信号 Y
      彩色信号 = Y（亮度）+ Cb/Cr（色差）
      黑白电视只接收 Y，彩色电视接收全部
      → 这是 YUV 最初的设计动机！

  原因 3：压缩效率更高
  └── RGB 三个通道相关性强（同时变化）
      YCbCr 三个通道相关性弱（Y 携带大部分信息）
      → 更容易压缩，DCT 变换后能量更集中

  数据量对比（1080p，4:2:0）：
  ├── RGB：1920 × 1080 × 3 = 6,220,800 字节/帧
  └── YCbCr 4:2:0：1920×1080 + 960×540×2 = 3,110,400 字节/帧
      → 节省 50%！
```

### 5.2 YUV 的历史：从模拟电视说起

```
历史背景：

  1953年：NTSC 彩色电视标准（美国）
  ├── 问题：如何在已有黑白电视基础设施上传输彩色信号？
  ├── 解决：将彩色信号分解为 Y（亮度）+ I、Q（色差）
  └── Y 信号与黑白电视兼容，I/Q 信号携带颜色信息

  1967年：PAL 彩色电视标准（欧洲）
  └── 使用 Y + U、V（色差），即 YUV

  名称混淆说明：
  ├── YUV：模拟电视术语（PAL 系统）
  ├── YIQ：模拟电视术语（NTSC 系统）
  ├── YCbCr：数字视频术语（正式标准名称）
  └── 工程中常混用"YUV"指代 YCbCr，严格来说不准确

  关键区别：
  ├── YUV：模拟信号，U/V 范围不固定
  └── YCbCr：数字信号，Cb/Cr 范围明确定义（0~255 或 16~240）
```

### 5.3 BT.601 转换矩阵推导

```
BT.601（标清数字视频，1982年）：

  ┌─ 历史背景 ──────────────────────────────────────────────┐
  │                                                          │
  │  ITU-R BT.601（原名 CCIR 601）：                         │
  │  • 1982 年由 CCIR（国际无线电咨询委员会）首次发布        │
  │  • 全称：Recommendation ITU-R BT.601                     │
  │  • 目的：为标清数字电视定义统一的编码参数                 │
  │  • 这是第一个数字视频色彩编码国际标准                     │
  │                                                          │
  │  历史渊源：                                              │
  │  • 1953 NTSC → 1967 PAL/SECAM → 1982 BT.601（数字化）   │
  │  • BT.601 的亮度系数直接继承自 NTSC 模拟电视标准         │
  │  • NTSC 的系数来源于 1953 年 CRT 荧光粉的实测光谱        │
  │    在 CIE 1931 亮度函数 ȳ(λ) 下的加权积分               │
  └──────────────────────────────────────────────────────────┘

  设计目标：
  ├── Y 分量 = 人眼感知的亮度
  ├── Cb = 蓝色色差（Blue - Y）
  └── Cr = 红色色差（Red - Y）

  第一步：确定亮度系数

  Y = K_R · R + K_G · G + K_B · B

  系数来源：sRGB → XYZ 矩阵的第二行（亮度行）！
  Y = 0.2126·R + 0.7152·G + 0.0722·B  ← 这是 BT.709 的值

  BT.601 使用的系数（针对标清电视的原色）：
  K_R = 0.299
  K_G = 0.587  （注意：K_R + K_G + K_B = 1）
  K_B = 0.114

  ┌─ 这些系数是怎么来的？────────────────────────────────────┐
  │                                                          │
  │  K_R=0.299, K_G=0.587, K_B=0.114 的来源：               │
  │                                                          │
  │  1. 测量 1953 年 NTSC 标准 CRT 荧光粉的发光光谱         │
  │     • 红色荧光粉峰值 ≈ 610nm                            │
  │     • 绿色荧光粉峰值 ≈ 540nm                            │
  │     • 蓝色荧光粉峰值 ≈ 470nm                            │
  │                                                          │
  │  2. 将各荧光粉光谱与 CIE 1931 亮度函数 ȳ(λ) 做积分     │
  │     得到各原色对亮度的相对贡献                            │
  │                                                          │
  │  3. 归一化使 K_R + K_G + K_B = 1                         │
  │     → 得到 0.299, 0.587, 0.114                          │
  │                                                          │
  │  本质：这些系数 = RGB→XYZ 矩阵的 Y 行（第二行）         │
  │  不同原色 → 不同的 XYZ 矩阵 → 不同的亮度系数            │
  │  所以 BT.601 ≠ BT.709 ≠ BT.2020                         │
  └──────────────────────────────────────────────────────────┘

  为什么 K_G 最大（0.587）？
  └── 因为人眼对绿色最敏感（视锥细胞响应峰值在绿色附近）
      绿色对亮度的贡献最大

  第二步：定义色差分量

  Cb = B - Y = B - (0.299R + 0.587G + 0.114B)
             = -0.299R - 0.587G + 0.886B

  Cr = R - Y = R - (0.299R + 0.587G + 0.114B)
             = 0.701R - 0.587G - 0.114B

  第三步：归一化（使 Cb、Cr 范围对称）

  Cb_norm = Cb / (2 · (1 - K_B)) = Cb / (2 × 0.886) = Cb / 1.772
  Cr_norm = Cr / (2 · (1 - K_R)) = Cr / (2 × 0.701) = Cr / 1.402

  为什么除以 2(1-K_B) 和 2(1-K_R)？
  ├── 当 B=1, R=G=0 时：Cb = 1 - K_B = 0.886（最大值）
  ├── 当 B=0, R=G=1 时：Cb = -K_B = -0.114（最小值）
  └── 归一化后：Cb_norm ∈ [-0.5, +0.5]（对称范围）

  最终 BT.601 转换矩阵（Full Range，浮点）：

  [Y ]   [ 0.299    0.587    0.114  ] [R]
  [Cb] = [-0.16875 -0.33126  0.5    ] [G]
  [Cr]   [ 0.5     -0.41869 -0.08131] [B]

  数字化（8bit，Limited Range，16~235/16~240）：

  Y  = 16  + 65.481·R + 128.553·G + 24.966·B
  Cb = 128 - 37.797·R -  74.203·G + 112.0·B
  Cr = 128 + 112.0·R  -  93.786·G -  18.214·B
```

### 5.4 BT.709 与 BT.2020 转换矩阵

```
BT.709（高清视频，1990/2002年）：

  ┌─ 历史背景 ──────────────────────────────────────────────┐
  │                                                          │
  │  ITU-R BT.709：                                          │
  │  • 1990 年首次发布，2002 年修订（BT.709-5）              │
  │  • 全称：Recommendation ITU-R BT.709                     │
  │  • 目的：为高清数字电视（HDTV）定义色彩参数              │
  │  • sRGB 的原色坐标与 BT.709 完全相同                     │
  │    （1996 年 HP+Microsoft 直接采用了 BT.709 的原色定义） │
  │                                                          │
  │  ITU-R BT.2020：                                         │
  │  • 2012 年发布，面向超高清（4K/8K）和 HDR               │
  │  • 原色选择接近光谱纯色（单色激光），色域极宽            │
  │  • 2020 年的显示技术（量子点/激光）才能接近实现          │
  └──────────────────────────────────────────────────────────┘

  亮度系数（与 BT.601 不同！）：
  K_R = 0.2126
  K_G = 0.7152
  K_B = 0.0722

  为什么系数不同？
  └── BT.709 使用了更宽色域的原色（高清电视的荧光粉不同）
      原色的 XYZ 坐标变了 → 亮度系数也随之变化

  系数来源（与 BT.601 相同的推导方法）：
  ├── 测量 BT.709 原色的发光光谱
  ├── 与 CIE 1931 ȳ(λ) 积分 → 得到各原色亮度贡献
  └── 归一化 → K_R=0.2126, K_G=0.7152, K_B=0.0722

  BT.709 转换矩阵（Full Range）：

  [Y ]   [ 0.2126   0.7152   0.0722 ] [R]
  [Cb] = [-0.11457 -0.38543  0.5    ] [G]
  [Cr]   [ 0.5     -0.45415 -0.04585] [B]

BT.2020（超高清/HDR，2012年）：

  亮度系数：
  K_R = 0.2627
  K_G = 0.6780
  K_B = 0.0593

  注意：BT.2020 的 K_B 更小（0.0593 vs 0.0722）
  └── 因为 BT.2020 的蓝色原色更纯（更接近光谱蓝）
      纯蓝色对亮度贡献更小

  ┌─────────────────────────────────────────────────────┐
  │  重要警告：混用 BT.601 和 BT.709 是视频工程中        │
  │  最常见的颜色错误！                                  │
  │                                                     │
  │  标清视频（DVD）→ 用 BT.601 解码                    │
  │  高清视频（蓝光/流媒体）→ 用 BT.709 解码             │
  │  HDR 视频 → 用 BT.2020 解码                         │
  │                                                     │
  │  如果用错矩阵，颜色会偏色（通常偏绿或偏紫）           │
  └─────────────────────────────────────────────────────┘
```

### 5.5 Full Range vs Limited Range

```
两种数值范围定义：

  Full Range（PC/互联网）：
  ├── Y：0 ~ 255（0 = 纯黑，255 = 纯白）
  ├── Cb/Cr：0 ~ 255（128 = 中性，无色差）
  └── 使用全部 8bit 范围

  Limited Range（广播/视频，也叫 Studio Swing）：
  ├── Y：16 ~ 235（16 = 纯黑，235 = 纯白）
  ├── Cb/Cr：16 ~ 240（128 = 中性）
  └── 保留头尾空间用于同步信号和过冲保护

  为什么 Limited Range 留出边界？
  ├── 历史原因：模拟视频信号需要同步脉冲，占用了边界范围
  ├── 过冲保护：编码过程中可能产生超出范围的值
  └── 广播标准要求：确保信号在传输链路中不被截断

  Full Range vs Limited Range 转换：
  Y_full  = (Y_limited - 16) × 255 / 219
  Cb_full = (Cb_limited - 128) × 255 / 224
  Cr_full = (Cr_limited - 128) × 255 / 224

  常见错误：
  ├── 把 Limited Range 视频当 Full Range 播放 → 画面发灰（对比度低）
  └── 把 Full Range 视频当 Limited Range 播放 → 画面过曝/过暗（截断）
```

### 5.6 色度采样：4:4:4 / 4:2:2 / 4:2:0

```
色度采样格式（利用人眼对色度分辨率不敏感）：

  4:4:4（无采样，完整色彩）：
  ┌─┬─┬─┬─┐  每个像素都有完整的 Y、Cb、Cr
  │Y│Y│Y│Y│  数据量：3 字节/像素
  │C│C│C│C│
  └─┴─┴─┴─┘

  4:2:2（水平方向色度减半）：
  ┌─┬─┬─┬─┐  Y：每像素一个
  │Y│Y│Y│Y│  Cb/Cr：水平方向每 2 像素共享一个
  │C   C  │  数据量：2 字节/像素（节省 33%）
  └─┴─┴─┴─┘  用途：专业视频制作、广播

  4:2:0（水平+垂直方向色度减半）：
  ┌─┬─┬─┬─┐  Y：每像素一个
  │Y│Y│Y│Y│  Cb/Cr：2×2 像素块共享一个
  │Y│Y│Y│Y│  数据量：1.5 字节/像素（节省 50%）
  │C       │  用途：H.264/H.265 视频，蓝光，流媒体
  └─┴─┴─┴─┘

  4:1:1（水平方向色度 1/4）：
  └── 用于 DV 格式（数字摄像机）

  数据量对比（1920×1080）：
  ├── 4:4:4：1920×1080×3 = 6.2 MB/帧
  ├── 4:2:2：1920×1080×2 = 4.1 MB/帧
  └── 4:2:0：1920×1080×1.5 = 3.1 MB/帧
```

## 六、HSV / HSL——面向人类直觉的色彩空间

### 6.1 为什么需要 HSV/HSL？

```
问题：RGB 不符合人类直觉

  "我想要一个更鲜艳的红色" → RGB 怎么调？不知道！
  "我想要一个更暗的蓝色"   → RGB 怎么调？不知道！

  人类描述颜色的自然方式：
  ├── "这是什么颜色？" → 色相（Hue）：红/橙/黄/绿/蓝/紫
  ├── "颜色有多纯？"   → 饱和度（Saturation）：鲜艳/灰暗
  └── "颜色有多亮？"   → 明度（Value/Lightness）：亮/暗

  HSV/HSL 的用途：
  ├── 颜色选择器（Photoshop、Figma 等设计工具）
  ├── 图像处理（按色相范围选取颜色）
  ├── 计算机视觉（颜色分割，对光照变化更鲁棒）
  └── 艺术创作（符合画家的颜色思维）
```

### 6.2 RGB → HSV 转换推导

```
HSV 定义：
  H（Hue，色相）：0° ~ 360°（色轮角度）
  S（Saturation，饱和度）：0 ~ 1（0=灰色，1=纯色）
  V（Value，明度）：0 ~ 1（0=黑色，1=最亮）

  几何直觉：
  └── HSV 是一个倒置的六棱锥（或圆锥）
      顶点 = 黑色（V=0）
      底面中心 = 白色（V=1, S=0）
      底面边缘 = 纯色（V=1, S=1）

RGB → HSV 推导：

  设 R, G, B ∈ [0, 1]
  
  第一步：找最大值和最小值
  M = max(R, G, B)   ← 决定 V
  m = min(R, G, B)   ← 辅助计算
  C = M - m          ← 色度（Chroma），C=0 表示灰色

  第二步：计算 V（明度）
  V = M
  └── 直觉：最亮的通道决定整体亮度

  第三步：计算 S（饱和度）
  S = C / V = (M - m) / M  （当 V ≠ 0 时）
  └── 直觉：最大值和最小值的差越大，颜色越纯

  第四步：计算 H（色相）
  
  当 M = R（红色最大）：H = 60° × ((G - B) / C) mod 360°
  当 M = G（绿色最大）：H = 60° × ((B - R) / C + 2)
  当 M = B（蓝色最大）：H = 60° × ((R - G) / C + 4)
  
  直觉理解：
  ├── 色相 H 表示在色轮上的位置
  ├── 0°/360° = 红色，120° = 绿色，240° = 蓝色
  └── 哪个通道最大，就在对应的 60° 扇区内
      扇区内的位置由另外两个通道的差值决定

  示例：纯红色 (R=1, G=0, B=0)
  M=1, m=0, C=1
  V = 1
  S = 1/1 = 1
  H = 60° × (0-0)/1 = 0°
  → HSV = (0°, 1, 1) ✓
```

### 6.3 HSL 与 HSV 的区别

```
HSL（Hue, Saturation, Lightness）：

  与 HSV 的区别：
  ├── HSV 的 V=1 时，S=1 是纯色，S=0 是白色
  └── HSL 的 L=1 时，永远是白色（无论 S 是多少）
      HSL 的 L=0.5, S=1 才是纯色

  几何直觉：
  ├── HSV：倒置圆锥（顶点=黑，底面=彩色+白）
  └── HSL：双锥体（上顶点=白，下顶点=黑，中间腰部=纯色）

  L（亮度）计算：
  L = (M + m) / 2

  S（饱和度）计算：
  S = C / (1 - |2L - 1|)  （当 L ≠ 0 且 L ≠ 1 时）

  哪个更好？
  ├── HSV：更适合"选取纯色然后调暗"的工作流
  │   （Photoshop 颜色选择器使用 HSB，即 HSV）
  └── HSL：更符合"亮度居中"的直觉
      （CSS color: hsl(120, 100%, 50%) 使用 HSL）

  共同局限：
  └── 感知不均匀！
      HSV/HSL 中相同的 S 变化，人眼感知到的变化量不同
      → 这就是为什么需要 Lab 色彩空间
```

## 七、Lab 色彩空间——感知均匀性

### 7.1 感知均匀性问题

```
什么是感知均匀性（Perceptual Uniformity）？

  理想的色彩空间：
  └── 色彩空间中相同的"距离"= 人眼感知到相同的颜色差异

  CIE XYZ 的问题：
  ├── 在 XYZ 空间中，绿色区域"拥挤"（小变化 = 大感知差异）
  ├── 蓝色区域"稀疏"（大变化 = 小感知差异）
  └── 无法用简单的欧氏距离衡量颜色差异

  实际问题：
  ├── 图像压缩：如何判断两种颜色"看起来一样"？
  ├── 颜色校正：如何量化颜色偏差？
  ├── 设计工具：如何在颜色之间做感知均匀的插值？
  └── 印刷质量控制：产品颜色是否在允许误差范围内？
```

### 7.2 CIE Lab（1976年）

```
CIE L*a*b*（简称 Lab 或 CIELAB）：

  三个分量：
  ├── L*：感知亮度（0=黑，100=白）
  ├── a*：绿色 ↔ 红色轴（负值=绿，正值=红）
  └── b*：蓝色 ↔ 黄色轴（负值=蓝，正值=黄）

  为什么是 a*b* 而不是 Cb/Cr？
  └── a*b* 的设计使得相同的欧氏距离
      对应人眼感知到的相同颜色差异（近似）

  Lab 的优势：
  ├── 感知均匀（近似）
  ├── 设备无关（通过 XYZ 中转）
  ├── 包含所有可见颜色
  └── 适合颜色差异计算（ΔE）
```

### 7.3 XYZ → Lab 转换推导

```
  ┌─ 矩阵/公式的来源与讲究 ────────────────────────────────┐
  │                                                          │
  │  Lab 不是矩阵变换，而是"非线性映射"，原因：              │
  │  ├── 目标：让"数值距离" ≈ "人眼感知差异"（感知均匀）     │
  │  └── 而人眼对亮度/颜色的感知本身就是非线性的             │
  │                                                          │
  │  STEP 1（人为约定）：                                    │
  │  • 参考白点 D65（X_n, Y_n, Z_n）= 标准日光               │
  │  • 立方根 t^(1/3) 作为感知非线性的近似                   │
  │  • 系数 116, 500, 200（缩放使各轴感知尺度匹配）          │
  │                                                          │
  │  STEP 2（数学/实验依据）：                               │
  │  • 立方根 ← 来自 1860 韦伯-费希纳定律的心理物理学实验    │
  │    （感知量 ∝ log(刺激量)，立方根是工程上的近似）        │
  │  • L*=116·f(Y/Yn)−16 ← 让 Y=1 时 L*=100，Y=0 时 L*=0   │
  │  • a*, b* 的系数 500/200 ← 1976 年实验拟合，使 ΔE=1     │
  │    刚好对应人眼"恰可分辨差异"（JND）                     │
  │                                                          │
  │  发布：CIE 1976 L*a*b*（CIELAB）国际标准                 │
  └──────────────────────────────────────────────────────────┘

XYZ → Lab 转换：

  第一步：用参考白点归一化
  f_x = X / X_n
  f_y = Y / Y_n
  f_z = Z / Z_n
  
  其中 (X_n, Y_n, Z_n) 是参考白点的 XYZ 值
  D65 白点：X_n=0.9505, Y_n=1.0000, Z_n=1.0891

  第二步：应用非线性函数 f(t)

  f(t) = t^(1/3)                    （当 t > (6/29)³ ≈ 0.008856）
  f(t) = (1/3)·(29/6)²·t + 4/29    （当 t ≤ 0.008856）

  为什么用立方根？
  └── 人眼对亮度的感知是非线性的（韦伯-费希纳定律）
      亮度增加 1 倍，感知亮度只增加约 26%
      立方根函数近似模拟了这种非线性感知

  第三步：计算 L*, a*, b*

  L* = 116 · f(Y/Y_n) - 16
  a* = 500 · [f(X/X_n) - f(Y/Y_n)]
  b* = 200 · [f(Y/Y_n) - f(Z/Z_n)]

  系数 116、500、200 的来源：
  ├── 116：使 L* 在 [0, 100] 范围内（Y=1 时 L*=100）
  ├── 500：放大 a* 轴，使其与 L* 轴的感知尺度匹配
  └── 200：放大 b* 轴（蓝黄轴比红绿轴感知范围小）

  直觉理解：
  ├── a* = 0, b* = 0 → 无彩色（灰色）
  ├── a* > 0 → 偏红；a* < 0 → 偏绿
  ├── b* > 0 → 偏黄；b* < 0 → 偏蓝
  └── 典型值范围：a*, b* ∈ [-128, +127]
```

### 7.4 色差公式 ΔE

```
ΔE（Delta E）：量化两种颜色的感知差异

  ΔE76（最简单，CIE 1976）：
  ΔE = √[(ΔL*)² + (Δa*)² + (Δb*)²]
  
  └── 就是 Lab 空间中的欧氏距离
      ΔE < 1：人眼无法区分
      ΔE 1~2：专业人员勉强可见
      ΔE 2~10：普通人可见差异
      ΔE > 10：明显不同的颜色

  ΔE94（改进，CIE 1994）：
  └── 对亮度、色相、饱和度分别加权
      更符合人眼对不同颜色属性的敏感度差异

  ΔE2000（最精确，CIE 2000）：
  └── 进一步修正蓝色区域、中性色区域的感知不均匀性
      工业色彩质量控制的标准

  应用场景：
  ├── 印刷质量控制：ΔE < 2 为合格
  ├── 显示器校准：ΔE < 1 为专业级
  ├── 图像压缩质量评估
  └── 颜色匹配（纺织、涂料、汽车喷漆）
```

## 八、其他重要色彩空间

### 8.1 YIQ（NTSC 彩色电视）

```
YIQ（1953年，NTSC 标准）：

  Y：亮度（与 BT.601 Y 相同）
  I（In-phase）：橙色 ↔ 青色轴
  Q（Quadrature）：绿色 ↔ 紫色轴

  转换矩阵：
  [Y]   [0.299   0.587   0.114 ] [R]
  [I] = [0.596  -0.274  -0.322 ] [G]
  [Q]   [0.211  -0.523   0.312 ] [B]

  ┌─ 矩阵的来源与讲究 ──────────────────────────────────────┐
  │                                                          │
  │  STEP 1（人为约定）：                                    │
  │  • Y 行 = NTSC 荧光粉的亮度系数（与 BT.601 同源）        │
  │  • I 轴方向 = 橙-青（人眼最敏感方向，肤色集中区）        │
  │  • Q 轴方向 = 绿-紫（人眼最迟钝方向）                    │
  │  • I/Q 是 YCbCr 的 (Cb, Cr) 旋转 33° 得到的             │
  │                                                          │
  │  STEP 2（数学推导）：                                    │
  │  ┌──┐   ┌ cos33°   sin33° ┐ ┌──┐                       │
  │  │I │ = │                 │ │Cb│                       │
  │  │Q │   │-sin33°   cos33° │ │Cr│                       │
  │  └──┘   └────────────────┘ └──┘                        │
  │  → 把 YCbCr 的色差平面绕 Y 轴旋转 33°，对齐人眼敏感方向 │
  │                                                          │
  │  工程动机（1953 带宽限制）：                             │
  │  • Q 通道（人眼迟钝方向）只用 0.5MHz 带宽                │
  │  • I 通道（人眼敏感方向）用 1.5MHz 带宽                  │
  │  • Y 通道用 4.2MHz 带宽                                  │
  │  → 这是"色度下采样"的鼻祖思想                           │
  └──────────────────────────────────────────────────────────┘

  设计特点：
  ├── I 轴对应人眼最敏感的颜色方向（橙-青）
  ├── Q 轴对应人眼不敏感的颜色方向（绿-紫）
  └── → 可以对 Q 分量用更低带宽传输（节省带宽）

  现代意义：
  └── 已被 YCbCr 取代，但 I/Q 的设计思想
      影响了后来的色彩空间设计
```

### 8.2 Oklab（2020年）

```
Oklab（Björn Ottosson，2020年）：

  背景：CIE Lab 在某些场景下感知均匀性仍不够好
  ├── 蓝色区域的色相旋转问题
  ├── 颜色插值时出现意外的色相偏移
  └── 设计工具中的颜色混合不自然

  Oklab 的改进：
  ├── 基于现代视觉感知数据重新拟合
  ├── 更好的色相均匀性（颜色插值更自然）
  └── 计算简单（两个矩阵变换 + 立方根）

  XYZ → Oklab 转换：
  
  第一步：XYZ → LMS（视锥细胞响应）
  [l]   [0.8189  0.3619  -0.1289] [X]
  [m] = [0.0329  0.9293   0.0361] [Y]
  [s]   [0.0482  0.2643   0.6338] [Z]

  第二步：立方根
  l' = ∛l,  m' = ∛m,  s' = ∛s

  第三步：LMS → Lab
  [L]   [0.2104  0.7936  -0.0040] [l']
  [a] = [1.9780 -2.4285   0.4506] [m']
  [b]   [0.0260  0.7827  -0.8087] [s']

  ┌─ 这两个矩阵的来源与讲究 ────────────────────────────────┐
  │                                                          │
  │  作者：Björn Ottosson（瑞典图形学工程师，2020）          │
  │  方法：用最小二乘法在现代视觉数据集上拟合，不是"开会定" │
  │                                                          │
  │  矩阵 1（XYZ→LMS）的来源：                              │
  │  • 拟合 IPT 色彩空间的 LMS 估计（更接近真实视锥响应）    │
  │  • 不是直接用 CIE 1931 数据，而是综合多组现代实验          │
  │  • 数值精度：用浮点优化器迭代得到的最优解                │
  │                                                          │
  │  矩阵 2（LMS→Lab）的来源：                              │
  │  • 在大量颜色对（人类感知相同 hue 的颜色）上拟合         │
  │  • 目标函数：让"色相直线"在 ab 平面真的是直线（解决 Lab │
  │    的蓝色色相弯曲问题）                                  │
  │  • 拟合细节：见 Ottosson 博客 bottosson.github.io/posts │
  │             /oklab/                                      │
  │                                                          │
  │  关键创新：                                              │
  │  • CIE Lab：基于 1976 年的实验数据                       │
  │  • Oklab： 基于 2020 年的现代数据 + 数值优化             │
  │  → 矩阵不是"标准委员会拍的"，而是"算法跑出来的"           │
  └──────────────────────────────────────────────────────────┘

  应用：
  ├── CSS Color Level 4（oklch() 函数）
  ├── Figma、Framer 等设计工具的颜色插值
  └── 渐变色生成（比 sRGB 渐变更自然）
```

### 8.3 ICtCp（HDR 视频）

```
ICtCp（Dolby，2016年）：

  专为 HDR 视频设计的色彩空间
  
  I：亮度（Intensity）
  Ct：蓝黄色差（Tritan，蓝色盲轴）
  Cp：红绿色差（Protan，红色盲轴）

  完整流程（BT.2020 RGB → ICtCp）：

  Step 1：RGB → LMS（基于 BT.2020 原色拟合）
    [L]    [1688   2146    262 ]   [R]
    [M] = (1/4096) [683    2951    462 ]   [G]
    [S]    [99     309     3688]   [B]

  Step 2：LMS 应用 PQ 传输函数（HDR 非线性）
    L' = PQ(L),  M' = PQ(M),  S' = PQ(S)

  Step 3：LMS' → ICtCp
    [I ]    [2048   2048    0   ]   [L']
    [Ct] = (1/4096) [6610  -13613  7003]   [M']
    [Cp]    [17933 -17390  -543 ]   [S']

  ┌─ 这些矩阵的来源与讲究 ──────────────────────────────────┐
  │                                                          │
  │  发布：ITU-R BT.2100（2016 由 Dolby 提出）               │
  │                                                          │
  │  矩阵 1（RGB→LMS）的来源：                              │
  │  • 输入：BT.2020 三原色 + D65（同 4.4 节套路）           │
  │  • LMS 估计：基于 Hunt-Pointer-Estevez 视锥响应模型      │
  │  • 推导方法：与 4.4 节相同的线性代数推导                 │
  │                                                          │
  │  矩阵 2（LMS'→ICtCp）的来源：                            │
  │  • Ct/Cp 命名 = 对齐两种色盲的混淆轴                     │
  │    - Tritan（蓝色盲）→ Ct 轴（蓝-黄）                   │
  │    - Protan（红色盲）→ Cp 轴（红-绿）                   │
  │  • 目标函数：在 HDR 高亮度下让 ΔE 仍保持均匀             │
  │    （传统 YCbCr 在 HDR 高光区会失真）                    │
  │  • 数值通过实验拟合得出                                  │
  │                                                          │
  │  为什么不直接用 YCbCr？                                  │
  │  • YCbCr 在 SDR（100 cd/m²）下设计，HDR（10000 cd/m²） │
  │    下色差通道感知不均匀                                  │
  │  • ICtCp 在整个 HDR 亮度范围内 ΔE 更均匀                │
  │    → 同样比特数压缩，主观质量更好                        │
  └──────────────────────────────────────────────────────────┘

  特点：
  ├── 基于 BT.2020 色域
  ├── 使用 PQ（ST 2084）或 HLG 传输函数
  ├── 比 YCbCr 更好的感知均匀性（HDR 场景）
  └── 色差分量与人眼色觉缺陷轴对齐（更高效压缩）

  应用：
  ├── Dolby Vision HDR 格式
  ├── HEVC/H.265 HDR 视频编码
  └── 下一代流媒体 HDR 标准
```

## 九、Gamma 校正与线性光

### 9.1 什么是 Gamma？

```
Gamma（γ）：描述输入信号与输出亮度之间的幂次关系

  输出亮度 = 输入信号^γ

  显示器的物理特性（CRT 时代）：
  ├── CRT 显示器的亮度 ∝ 电压^2.2
  ├── 即 γ ≈ 2.2（显示器 Gamma）
  └── 不校正的话，图像会显得过暗

  直觉理解：
  ├── γ = 1：线性（输入=输出）
  ├── γ > 1（如 2.2）：暗部被压缩，亮部被拉伸 → 图像变暗
  └── γ < 1（如 1/2.2 ≈ 0.45）：暗部被拉伸，亮部被压缩 → 图像变亮
```

### 9.2 为什么需要 Gamma 校正？

```
两个原因：

  原因 1：补偿显示器的非线性（历史原因）
  ├── CRT 显示器：亮度 ∝ 电压^2.2
  ├── 如果直接存储线性光值，显示时会变暗
  └── 解决：存储时做 Gamma 编码（γ=1/2.2），显示时自动被 CRT 的 γ=2.2 抵消

  原因 2：更高效地利用有限的比特数（更重要！）
  
  人眼对暗部更敏感（韦伯-费希纳定律）：
  ├── 从 0 到 1 cd/m²：人眼能感知很多层次
  └── 从 100 到 101 cd/m²：人眼几乎感知不到差异
  
  如果线性存储（8bit）：
  ├── 0~127（暗部）：128 个级别，但人眼需要更多
  └── 128~255（亮部）：128 个级别，但人眼不需要这么多
  → 暗部量化噪声明显，亮部比特浪费
  
  Gamma 编码后（8bit）：
  ├── 暗部被"拉伸"→ 更多比特分配给暗部
  └── 亮部被"压缩"→ 更少比特分配给亮部
  → 与人眼感知匹配，量化噪声更均匀！
  
  结论：Gamma 编码 = 感知均匀量化
```

### 9.3 sRGB 的 Gamma 曲线

```
sRGB 的传输函数（不是简单的 γ=2.2！）：

  线性光 → sRGB 编码（Gamma 编码）：
  
  C_srgb = 12.92 · C_linear                    （C_linear ≤ 0.0031308）
  C_srgb = 1.055 · C_linear^(1/2.4) - 0.055   （C_linear > 0.0031308）

  sRGB 解码 → 线性光：
  
  C_linear = C_srgb / 12.92                    （C_srgb ≤ 0.04045）
  C_linear = ((C_srgb + 0.055) / 1.055)^2.4   （C_srgb > 0.04045）

  为什么是分段函数？
  ├── 暗部（C ≤ 0.0031308）：用线性段（避免立方根在 0 附近的斜率无穷大）
  └── 亮部：用幂函数（γ ≈ 2.2，实际指数是 2.4）

  等效 Gamma：
  └── sRGB 的等效 Gamma 约为 2.2（不是精确的 2.2）
      精确值是分段函数，整体效果接近 γ=2.2
```

### 9.4 线性工作流

```
线性工作流（Linear Workflow）：

  问题：在 Gamma 编码的图像上做数学运算会出错！

  错误示例：
  ├── 两张 sRGB 图像叠加：(A + B) / 2
  │   在 sRGB 空间直接平均 → 结果偏暗（错误！）
  └── 正确做法：先解码到线性光，平均后再编码回 sRGB

  线性工作流步骤：
  1. 输入：sRGB 图像 → 解码（去 Gamma）→ 线性光值
  2. 处理：在线性光空间做所有数学运算（混合、滤波、光照计算）
  3. 输出：线性光值 → 编码（加 Gamma）→ sRGB 图像

  需要线性工作流的操作：
  ├── 图像混合/叠加
  ├── 模糊/锐化滤波
  ├── 3D 渲染光照计算
  ├── 颜色空间转换（RGB → YCbCr）
  └── 图像缩放（双线性/双三次插值）

  不需要线性工作流的操作：
  ├── 色相/饱和度调整（在 HSV 空间）
  ├── 对比度曲线调整（直接操作 Gamma 编码值）
  └── 文字/图形叠加（通常在 sRGB 空间）
```

## 十、HDR 与宽色域

### 10.1 SDR vs HDR

```
SDR（Standard Dynamic Range）：
├── 峰值亮度：100 cd/m²（nit）
├── 黑色亮度：0.1 cd/m²
├── 动态范围：约 1000:1（30 EV）
└── 传输函数：Gamma 2.2 / sRGB

HDR（High Dynamic Range）：
├── 峰值亮度：1000 ~ 10000 cd/m²
├── 黑色亮度：0.001 cd/m²
├── 动态范围：约 1,000,000:1（20 EV）
└── 传输函数：PQ（ST 2084）或 HLG

人眼的动态范围：
├── 瞬时：约 10,000:1
└── 适应后：约 1,000,000,000:1（10亿:1）
```

### 10.2 PQ 曲线（ST 2084）

```
PQ（Perceptual Quantizer，感知量化器）：
标准：SMPTE ST 2084（2014年）
设计者：Dolby

设计目标：
└── 在 12bit 精度下，量化噪声低于人眼可见阈值
    覆盖 0.0001 ~ 10000 cd/m² 的亮度范围

PQ 传输函数：
  E = ((c1 + c2·Y^m1) / (1 + c3·Y^m1))^m2

  其中：
  Y = 线性亮度（归一化到 [0, 1]，1 = 10000 cd/m²）
  m1 = 0.1593017578125
  m2 = 78.84375
  c1 = 0.8359375
  c2 = 18.8515625
  c3 = 18.6875

  特点：
  ├── 绝对亮度编码（1 = 10000 cd/m²，固定）
  ├── 12bit 精度下量化步长 ≈ 人眼 JND（刚可察觉差异）
  └── 不向后兼容 SDR（需要色调映射）

  应用：
  ├── HDR10（开放标准，静态元数据）
  ├── Dolby Vision（动态元数据）
  └── HDR10+（Samsung，动态元数据）
```

### 10.3 HLG 曲线

```
HLG（Hybrid Log-Gamma，混合对数 Gamma）：
标准：ITU-R BT.2100（2016年）
设计者：BBC + NHK

设计目标：
└── 向后兼容 SDR（同一信号在 SDR 和 HDR 显示器上都能正常显示）

HLG 传输函数（分段）：
  E = a·ln(12·Y - b) + c    （Y > 1/12，对数段）
  E = √(3·Y)                （Y ≤ 1/12，Gamma 段）

  其中：a=0.17883277, b=0.28466892, c=0.55991073

  特点：
  ├── 相对亮度编码（不固定绝对亮度）
  ├── 低亮度部分（Y ≤ 1/12）= 标准 Gamma 曲线 → SDR 兼容
  ├── 高亮度部分 = 对数曲线 → HDR 扩展
  └── 向后兼容：SDR 显示器直接显示低亮度部分，效果可接受

  应用：
  ├── 广播电视（BBC、NHK、日本 NHK 4K/8K 广播）
  ├── YouTube HDR
  └── 直播 HDR（实时编码，无需元数据）

  PQ vs HLG 选择：
  ├── 点播/电影：PQ（绝对亮度，精确控制）
  └── 直播/广播：HLG（相对亮度，SDR 兼容）
```

### 10.4 色调映射（Tone Mapping）

```
色调映射（Tone Mapping）：
将 HDR 内容映射到 SDR 显示器（或反向）

  问题：HDR 内容亮度范围 0.0001~10000 cd/m²
        SDR 显示器只能显示 0.1~100 cd/m²
        → 直接截断会丢失大量细节

  常用色调映射算法：

  Reinhard（2002年，最经典）：
  L_out = L_in / (1 + L_in)
  └── 简单，但高亮区域细节损失多

  ACES（Academy Color Encoding System）：
  └── 电影工业标准，S 形曲线，保留更多细节
      Unreal Engine、Unity 默认使用

  Filmic（游戏引擎常用）：
  └── 模拟胶片响应曲线，自然的高光过渡

  局部色调映射：
  └── 根据图像局部亮度自适应调整
      效果最好，但计算量大
```

## 十一、转换矩阵的统一理解框架

```
所有色彩空间转换矩阵的统一理解：

  ┌─────────────────────────────────────────────────────────┐
  │              色彩空间转换的本质                          │
  │                                                         │
  │  每个色彩空间 = 一个坐标系                               │
  │  ├── 原点 = 黑色（或某个参考点）                         │
  │  ├── 三个基向量 = 三个原色（或三个感知轴）               │
  │  └── 坐标值 = 在该坐标系中的分量                         │
  │                                                         │
  │  转换矩阵 = 坐标系变换矩阵                               │
  │  └── 将一个坐标系中的坐标，转换到另一个坐标系            │
  └─────────────────────────────────────────────────────────┘

  转换矩阵的来源：

  1. RGB → XYZ 矩阵：
     由原色的色度坐标 + 白点推导（见第四章）
     物理意义：将 RGB 原色坐标系变换到 XYZ 绝对坐标系

  2. XYZ → YCbCr 矩阵：
     Y 行 = XYZ → Y 的亮度系数（来自 ȳ(λ) 函数）
     Cb/Cr 行 = 色差定义（B-Y 和 R-Y 的归一化）
     物理意义：分离亮度和色度信息

  3. XYZ → Lab 变换：
     非线性（立方根）+ 线性变换
     物理意义：将 XYZ 变换到感知均匀的空间

  4. RGB → HSV 变换：
     非线性（max/min 操作）
     物理意义：将笛卡尔坐标变换到柱坐标（色相=角度，饱和度=半径，明度=高度）

  矩阵数值的历史溯源（所有数值都有据可查）：

  ┌──────────────────────────────────────────────────────────────────┐
  │  矩阵/系数          │  数值来源                │  标准文档        │
  ├─────────────────────┼─────────────────────────┼──────────────────┤
  │  sRGB→XYZ 矩阵     │  原色坐标+白点数学推导  │  IEC 61966-2-1   │
  │  BT.601 亮度系数    │  NTSC CRT 荧光粉实测    │  ITU-R BT.601    │
  │  BT.709 亮度系数    │  HDTV 原色实测          │  ITU-R BT.709    │
  │  BT.2020 亮度系数   │  激光原色理论计算       │  ITU-R BT.2020   │
  │  D65 白点坐标       │  CIE 标准光源实测       │  ISO 11664-2     │
  │  CIE XYZ 匹配函数   │  1931 颜色匹配实验      │  ISO 11664-1     │
  └─────────────────────┴─────────────────────────┴──────────────────┘

  关键认知：
  ├── 所有矩阵数值都不是"经验值"或"拍脑袋定的"
  ├── 它们要么来自物理测量（原色光谱、白点光谱）
  ├── 要么来自严格的数学推导（坐标变换、归一化）
  └── 最终由国际标准组织（IEC/ITU/CIE）正式发布

  转换链：
  
  任意 RGB_A → XYZ → 任意 RGB_B
  任意 RGB_A → XYZ → Lab
  任意 RGB_A → XYZ → YCbCr
  
  XYZ 是所有色彩空间的"公共语言"！

  注意非线性步骤：
  ├── Gamma 编码/解码（sRGB、BT.709 等）
  ├── PQ/HLG 传输函数（HDR）
  ├── Lab 的立方根函数
  └── HSV 的 max/min 操作
  
  这些非线性步骤不能用矩阵表示，
  必须在矩阵变换之前或之后单独处理！
```

## 十二、工程实践与代码

### 12.1 Python 实战代码

```python
# 色彩空间转换工具集
# 依赖：pip install numpy matplotlib pillow opencv-python

import numpy as np
import matplotlib.pyplot as plt
import matplotlib.patches as patches
from PIL import Image
import cv2

# ============================================================
# 1. sRGB ↔ 线性光（Gamma 编码/解码）
# ============================================================
def srgb_to_linear(c):
    """sRGB → 线性光（去 Gamma）"""
    c = np.asarray(c, dtype=np.float64)
    return np.where(c <= 0.04045,
                    c / 12.92,
                    ((c + 0.055) / 1.055) ** 2.4)

def linear_to_srgb(c):
    """线性光 → sRGB（加 Gamma）"""
    c = np.asarray(c, dtype=np.float64)
    c = np.clip(c, 0, 1)
    return np.where(c <= 0.0031308,
                    c * 12.92,
                    1.055 * c ** (1/2.4) - 0.055)


# ============================================================
# 2. sRGB ↔ CIE XYZ（D65）
# ============================================================
# sRGB 线性 → XYZ 矩阵
M_sRGB_to_XYZ = np.array([
    [0.4124564, 0.3575761, 0.1804375],
    [0.2126729, 0.7151522, 0.0721750],
    [0.0193339, 0.1191920, 0.9503041]
])

# XYZ → sRGB 线性矩阵（逆矩阵）
M_XYZ_to_sRGB = np.linalg.inv(M_sRGB_to_XYZ)

def srgb_to_xyz(rgb_srgb):
    """sRGB（0~1）→ CIE XYZ"""
    rgb_linear = srgb_to_linear(rgb_srgb)
    return M_sRGB_to_XYZ @ rgb_linear

def xyz_to_srgb(xyz):
    """CIE XYZ → sRGB（0~1）"""
    rgb_linear = M_XYZ_to_sRGB @ xyz
    return linear_to_srgb(np.clip(rgb_linear, 0, 1))


# ============================================================
# 3. CIE XYZ ↔ CIE Lab
# ============================================================
# D65 白点
XYZ_D65 = np.array([0.95047, 1.00000, 1.08883])

def _f_lab(t):
    """Lab 非线性函数"""
    delta = 6/29
    return np.where(t > delta**3,
                    t ** (1/3),
                    t / (3 * delta**2) + 4/29)

def xyz_to_lab(xyz, white=XYZ_D65):
    """CIE XYZ → CIE L*a*b*"""
    xyz_n = xyz / white
    f = _f_lab(xyz_n)
    L = 116 * f[1] - 16
    a = 500 * (f[0] - f[1])
    b = 200 * (f[1] - f[2])
    return np.array([L, a, b])

def lab_to_xyz(lab, white=XYZ_D65):
    """CIE L*a*b* → CIE XYZ"""
    L, a, b = lab
    fy = (L + 16) / 116
    fx = a / 500 + fy
    fz = fy - b / 200
    delta = 6/29
    def f_inv(t):
        return np.where(t > delta, t**3, 3*delta**2*(t - 4/29))
    return white * np.array([f_inv(fx), f_inv(fy), f_inv(fz)])

def delta_e76(lab1, lab2):
    """计算两种颜色的 ΔE76 色差"""
    return np.sqrt(np.sum((np.array(lab1) - np.array(lab2))**2))


# ============================================================
# 4. RGB ↔ YCbCr（BT.709，Full Range）
# ============================================================
# BT.709 亮度系数
KR_709, KB_709 = 0.2126, 0.0722
KG_709 = 1 - KR_709 - KB_709

M_RGB_to_YCbCr_709 = np.array([
    [KR_709,                    KG_709,                    KB_709],
    [-KR_709/(2*(1-KB_709)),   -KG_709/(2*(1-KB_709)),    0.5],
    [0.5,                      -KG_709/(2*(1-KR_709)),   -KB_709/(2*(1-KR_709))]
])

M_YCbCr_to_RGB_709 = np.linalg.inv(M_RGB_to_YCbCr_709)

def rgb_to_ycbcr(rgb, matrix='bt709'):
    """RGB（0~1）→ YCbCr（Y:0~1, Cb/Cr:-0.5~0.5）"""
    if matrix == 'bt709':
        return M_RGB_to_YCbCr_709 @ np.array(rgb)
    elif matrix == 'bt601':
        KR, KB = 0.299, 0.114
        KG = 1 - KR - KB
        M = np.array([
            [KR, KG, KB],
            [-KR/(2*(1-KB)), -KG/(2*(1-KB)), 0.5],
            [0.5, -KG/(2*(1-KR)), -KB/(2*(1-KR))]
        ])
        return M @ np.array(rgb)

def ycbcr_to_rgb(ycbcr, matrix='bt709'):
    """YCbCr → RGB（0~1）"""
    if matrix == 'bt709':
        return np.clip(M_YCbCr_to_RGB_709 @ np.array(ycbcr), 0, 1)


# ============================================================
# 5. RGB ↔ HSV
# ============================================================
def rgb_to_hsv(rgb):
    """RGB（0~1）→ HSV（H:0~360, S:0~1, V:0~1）"""
    r, g, b = rgb
    M = max(r, g, b)
    m = min(r, g, b)
    C = M - m
    
    V = M
    S = C / V if V != 0 else 0
    
    if C == 0:
        H = 0
    elif M == r:
        H = 60 * ((g - b) / C % 6)
    elif M == g:
        H = 60 * ((b - r) / C + 2)
    else:
        H = 60 * ((r - g) / C + 4)
    
    return np.array([H, S, V])

def hsv_to_rgb(hsv):
    """HSV（H:0~360, S:0~1, V:0~1）→ RGB（0~1）"""
    H, S, V = hsv
    C = V * S
    X = C * (1 - abs((H / 60) % 2 - 1))
    m = V - C
    
    if   0   <= H < 60:  r, g, b = C, X, 0
    elif 60  <= H < 120: r, g, b = X, C, 0
    elif 120 <= H < 180: r, g, b = 0, C, X
    elif 180 <= H < 240: r, g, b = 0, X, C
    elif 240 <= H < 300: r, g, b = X, 0, C
    else:                r, g, b = C, 0, X
    
    return np.array([r + m, g + m, b + m])


# ============================================================
# 6. 演示：Gamma 校正的重要性
# ============================================================
def demo_gamma_importance():
    """演示在错误色彩空间做混合的问题"""
    # 黑色和白色的 sRGB 值
    black_srgb = np.array([0.0, 0.0, 0.0])
    white_srgb = np.array([1.0, 1.0, 1.0])
    
    # 错误方式：直接在 sRGB 空间平均
    wrong_mix = (black_srgb + white_srgb) / 2  # = [0.5, 0.5, 0.5]
    
    # 正确方式：在线性光空间平均，再编码回 sRGB
    black_linear = srgb_to_linear(black_srgb)   # = [0, 0, 0]
    white_linear = srgb_to_linear(white_srgb)   # = [1, 1, 1]
    correct_mix_linear = (black_linear + white_linear) / 2  # = [0.5, 0.5, 0.5]
    correct_mix = linear_to_srgb(correct_mix_linear)  # ≈ [0.735, 0.735, 0.735]
    
    print("黑白混合对比：")
    print(f"  错误方式（sRGB 直接平均）：{wrong_mix[0]:.3f} → 亮度 {wrong_mix[0]:.1%}")
    print(f"  正确方式（线性光平均）：   {correct_mix[0]:.3f} → 亮度 {correct_mix[0]:.1%}")
    print(f"  差异：正确方式比错误方式亮 {(correct_mix[0]-wrong_mix[0])/wrong_mix[0]*100:.0f}%")
    
    # 可视化
    fig, axes = plt.subplots(1, 3, figsize=(10, 3))
    for ax, color, title in zip(axes,
                                 [black_srgb, wrong_mix, correct_mix],
                                 ['黑色', '错误混合\n(sRGB直接平均)', '正确混合\n(线性光平均)']):
        ax.add_patch(patches.Rectangle((0,0), 1, 1, color=color))
        ax.set_xlim(0, 1); ax.set_ylim(0, 1)
        ax.set_title(title); ax.axis('off')
    plt.suptitle("Gamma 校正的重要性：黑白混合")
    plt.tight_layout()
    plt.savefig('gamma_demo.png', dpi=150)
    plt.show()


# ============================================================
# 7. 演示：色差计算
# ============================================================
def demo_color_difference():
    """演示 ΔE 色差计算"""
    colors = {
        '纯红 sRGB':    [1.0, 0.0, 0.0],
        '深红 sRGB':    [0.8, 0.0, 0.0],
        '橙红 sRGB':    [1.0, 0.2, 0.0],
        '纯绿 sRGB':    [0.0, 1.0, 0.0],
    }
    
    print("色差 ΔE76 计算（Lab 空间欧氏距离）：")
    labs = {}
    for name, rgb in colors.items():
        xyz = srgb_to_xyz(np.array(rgb))
        lab = xyz_to_lab(xyz)
        labs[name] = lab
        print(f"  {name}: Lab = ({lab[0]:.1f}, {lab[1]:.1f}, {lab[2]:.1f})")
    
    print("\n色差矩阵：")
    names = list(labs.keys())
    for i, n1 in enumerate(names):
        for j, n2 in enumerate(names):
            if i < j:
                de = delta_e76(labs[n1], labs[n2])
                print(f"  {n1} vs {n2}: ΔE = {de:.1f}")


# ============================================================
# 主程序
# ============================================================
if __name__ == "__main__":
    # 1. Gamma 校正演示
    demo_gamma_importance()
    
    # 2. 色差计算演示
    demo_color_difference()
    
    # 3. 颜色转换示例
    red_srgb = np.array([1.0, 0.0, 0.0])
    print(f"\n纯红色 sRGB: {red_srgb}")
    print(f"→ XYZ: {srgb_to_xyz(red_srgb)}")
    print(f"→ Lab: {xyz_to_lab(srgb_to_xyz(red_srgb))}")
    print(f"→ HSV: {rgb_to_hsv(red_srgb)}")
    print(f"→ YCbCr(BT.709): {rgb_to_ycbcr(red_srgb, 'bt709')}")
```

### 12.2 常见陷阱与注意事项

```
工程中最常见的色彩空间错误：

  ❌ 错误 1：混用 BT.601 和 BT.709
  ├── 症状：颜色偏绿或偏紫
  ├── 原因：标清视频用 BT.601，高清视频用 BT.709
  └── 修复：检查视频元数据中的 color_space 字段

  ❌ 错误 2：忽略 Gamma 编码，在 sRGB 空间做线性运算
  ├── 症状：混合/模糊后图像偏暗
  ├── 原因：sRGB 是 Gamma 编码的，不是线性光
  └── 修复：先解码到线性光，运算后再编码回 sRGB

  ❌ 错误 3：混淆 Full Range 和 Limited Range
  ├── 症状：画面发灰（对比度低）或过曝/过暗
  ├── 原因：视频文件元数据中的 color_range 字段未正确读取
  └── 修复：用 ffprobe 检查 color_range 字段

  ❌ 错误 4：在 HDR 内容上用 SDR 的色调映射
  ├── 症状：高亮区域过曝，颜色失真
  └── 修复：使用正确的 PQ/HLG 传输函数解码

  ❌ 错误 5：色彩空间转换时忘记白点适应
  ├── 症状：颜色偏色（偏暖或偏冷）
  ├── 原因：不同色彩空间可能使用不同白点（D65 vs D50）
  └── 修复：使用 Bradford 色适应矩阵进行白点转换

  实用工具命令：

  # 用 ffprobe 查看视频色彩空间信息
  ffprobe -v quiet -print_format json -show_streams video.mp4 | \
    python -c "import json,sys; s=json.load(sys.stdin)['streams'][0]; \
    print('color_space:', s.get('color_space')); \
    print('color_transfer:', s.get('color_transfer')); \
    print('color_primaries:', s.get('color_primaries')); \
    print('color_range:', s.get('color_range'))"

  # 用 ImageMagick 查看图像色彩空间
  identify -verbose image.png | grep -E "Colorspace|Gamma|Type"

  # 用 Python OpenCV 读取并转换
  import cv2
  img_bgr = cv2.imread('image.jpg')          # OpenCV 默认 BGR 顺序！
  img_rgb = cv2.cvtColor(img_bgr, cv2.COLOR_BGR2RGB)
  img_lab = cv2.cvtColor(img_bgr, cv2.COLOR_BGR2Lab)
  img_hsv = cv2.cvtColor(img_bgr, cv2.COLOR_BGR2HSV)
  img_yuv = cv2.cvtColor(img_bgr, cv2.COLOR_BGR2YCrCb)
