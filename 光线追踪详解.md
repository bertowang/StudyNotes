## 0. 光线追踪要解决什么问题？

先看一个最朴素的问题：

> 屏幕上的每一个像素，最终应该显示什么颜色？

在真实世界里，你眼睛看到的每个"点"的颜色，是由**从光源出发、经过反射折射、最终进入你眼睛的那条光线**决定的。

**光线追踪（Ray Tracing）的核心思想**是：

> 与其模拟"光源发出无数条光线"（99.999% 都不会进入眼睛，浪费算力），不如**从眼睛出发反向追踪光线**——每个像素发一条，看它撞到什么、颜色是多少、有没有反射折射……

这就是**逆向光线追踪（Backward Ray Tracing）**，也是几乎所有实用光追算法的起点。

**光追能自然处理这些效果**（这些恰好是光栅化的痛点）：

| 效果 | 光栅化 | 光线追踪 |
|------|--------|----------|
| **硬阴影** | Shadow Map（有走样、漏光）| 一条阴影光线搞定 |
| **精确反射**（镜子里的镜子）| Cubemap 假一层，套多层难 | 递归即可 |
| **折射**（玻璃、水）| 各种 hack | Snell 定律直接算 |
| **透视正确** | 需要额外校正 | 天然正确 |
| **软阴影/全局光照** | 需要 SSAO/GI 等复杂 hack | 采样多条光线自然产生 |

代价是：**慢**。这是过去 40 年光追不能实时的原因，也是 2018 年 NVIDIA RTX 出现后一切开始改变的原因。

## 1. 直观理解：从"光"到"逆向追踪"

**真实世界（正向）**：

```
   ☀️ 光源
   │╲
   │ ╲
   │  ╲     ┌────────────┐
   │   ●───▶│  物体表面    │───▶反射────▶ 👁️ 眼睛
   │        └────────────┘
   │
   └──▶ 大部分光线打到别的地方，跟你无关
```

问题是：光源发出的光线**几乎都不会最终进入你的眼睛**——正向模拟浪费惊人。

**光线追踪（逆向）**：

```
   ☀️ 光源
      ▲
      │ ← ③ 从命中点朝光源发一条阴影光线：能看到光源？→ 亮  被挡？→ 暗
      │
      ●  ← ② 光线撞到物体表面，得到命中点、法线
      ▲
      │
      │  ← ① 从眼睛出发，穿过屏幕像素发一条光线
      │
     👁️ 眼睛
```

三步：

1. **生成光线**：眼睛 → 屏幕像素 → 场景；
2. **求交**：光线撞到最近的哪个物体？在哪里？
3. **着色**：那个点的颜色是多少？（考虑光源方向、反射、折射……）

**递归就发生在第 3 步**：如果那个点是镜面，就再从命中点朝反射方向发一条**次级光线**，重复上面 3 步——这就是"追踪"这个词的含义。

## 2. 数学基础：什么是"光线"

### 2.1 光线的参数方程

光线在数学上是一条**从起点出发、朝某方向前进的射线**，用参数方程表示：

```
    ┌─────────────────────────────┐
    │   P(t) = O + t · D          │
    │                             │
    │   O：起点（Origin）           │
    │   D：方向单位向量（Direction）│
    │   t：距离参数（t > 0）         │
    └─────────────────────────────┘
```

**几何含义**：

```
        O ●────────▶───────▶─────────▶ 无穷远
              t=0       t=1      t=2

    P(0) = O           （光线起点）
    P(1) = O + D       （前进 1 个单位）
    P(t) = O + t·D     （前进 t 个单位）
```

**约定**：`D` 通常取**单位向量**（`|D| = 1`），这样 `t` 就直接等于"从起点走了多远"，很多公式会简化。

### 2.2 从相机产生初始光线

对屏幕上像素 `(i, j)`（`i` 是列，`j` 是行），怎么产生一条光线？

**Step 1**：把像素坐标 `(i, j)` 映射到"标准化设备坐标"（NDC，范围 `[-1, 1]`）：

```
u = (i + 0.5) / W · 2 - 1        (W = 图像宽度)
v = 1 - (j + 0.5) / H · 2        (H = 图像高度，注意 y 反转)
```

**Step 2**：结合视场角（Field of View, `FOV`）和宽高比，得到相机空间中的方向：

```
aspect = W / H
scale  = tan(FOV / 2)          (半视场角的 tan)

x_cam = u · aspect · scale
y_cam = v · scale
z_cam = -1                     (相机看向 -z 方向，这是常规约定)
```

**Step 3**：把方向从相机空间变换到世界空间（乘相机的旋转矩阵 `R_cam`）并归一化：

```
D_world = normalize( R_cam · (x_cam, y_cam, z_cam) )
O_world = 相机位置
```

于是每个像素的光线就是 `P(t) = O_world + t · D_world`。

> 💡 这里涉及到的坐标变换请参考同目录的 [8 个坐标系详解.md](./图形学中的8个坐标系详解.md)。

## 3. 光线-物体求交（Ray-Object Intersection）

有了光线，我们要问：**光线撞到场景中的物体了吗？在哪里？**

对每种几何图元（球、平面、三角形等），我们要能高效解出一个 `t > 0`，使得 `P(t) = O + tD` 恰好在物体表面上。

### 3.1 光线-球体求交（推导）

**球体的隐式方程**：中心 `C`、半径 `r`：

```
|P - C|² = r²
```

**代入光线**：`P = O + tD`：

```
|O + tD - C|² = r²
```

令 `L = O - C`（从球心指向光线起点的向量），展开：

```
|L + tD|² = r²
(L + tD) · (L + tD) = r²
L·L + 2t(L·D) + t²(D·D) = r²
```

因为 `D` 是单位向量 `D·D = 1`，整理成关于 `t` 的**一元二次方程**：

```
    ┌──────────────────────────────────────────┐
    │   t² + 2(L·D)·t + (L·L - r²) = 0         │
    └──────────────────────────────────────────┘
```

对照 `at² + bt + c = 0` 的标准形式：

```
a = 1
b = 2 (L · D)
c = L·L - r²
```

**判别式 Δ = b² - 4ac**：

- **Δ < 0**：无交点（光线擦过或没打到球）；
- **Δ = 0**：切线（一个交点）；
- **Δ > 0**：两个交点，取较小的正根：

```
t₁ = (-b - √Δ) / 2       (较近的交点，光线先碰到)
t₂ = (-b + √Δ) / 2       (较远的交点，光线穿出)
```

**取 t 的规则**：

```
if t₁ > ε:   使用 t₁     (通常在物体外侧)
elif t₂ > ε: 使用 t₂     (光线起点在球内部)
else:        无有效交点
```

`ε` 是一个小正数（如 `1e-4`）用来避免"自相交"（光线打到自己出发的表面上）。

**命中点和法线**：

```
P_hit  = O + t · D
N_hit  = normalize(P_hit - C)      (法线是从球心指向命中点)
```

### 3.2 光线-平面求交

**平面**：过点 `Q`、法线 `N` 的平面方程 `(P - Q) · N = 0`。代入 `P = O + tD`：

```
(O + tD - Q) · N = 0
t · (D · N) = (Q - O) · N
```

```
    ┌─────────────────────────────────────┐
    │   t = (Q - O) · N / (D · N)         │
    └─────────────────────────────────────┘
```

**注意**：

- `D · N = 0`：光线平行于平面，无交点；
- `t < 0`：交点在光线反方向，忽略。

### 3.3 光线-三角形求交（Möller–Trumbore）

三角形是最重要的图元（因为所有网格都是三角形）。经典算法是 **Möller–Trumbore 算法**——直接把光线方程代进三角形的重心坐标表示：

```
P(t) = O + t·D = (1 - u - v)·V₀ + u·V₁ + v·V₂
```

得到 3 个方程 3 个未知数 `(t, u, v)`，用**克莱默法则**解：

```
E₁ = V₁ - V₀
E₂ = V₂ - V₀
P  = D × E₂
det = E₁ · P
if |det| < ε: 平行，无交点

T = O - V₀
u = (T · P) / det          (要求 0 ≤ u)
Q = T × E₁
v = (D · Q) / det          (要求 0 ≤ v，且 u + v ≤ 1)
t = (E₂ · Q) / det         (要求 t > ε)
```

**几何含义**：`(u, v)` 就是重心坐标——正好可以直接用来插值 UV、法线、颜色等（详见 [纹理贴图详解.md](./如何把一张2D图像贴到三角网格立体上-纹理贴图详解.md) 5.1 节）。

## 4. 着色模型（Shading）

求出命中点后，还得知道那点是**什么颜色**。这就是着色。

### 4.1 Phong 光照模型

最经典的经验模型，把颜色分成三部分：

```
    Color =  I_ambient +  I_diffuse +  I_specular
             环境光       漫反射       高光
```

**符号约定**（都是单位向量）：

```
N：命中点的法线
L：从命中点指向光源的方向
V：从命中点指向相机（视点）的方向
R：L 关于 N 的反射方向
```

**三个分量**：

```
I_ambient  = k_a · I_light                            (环境光，常数项)

I_diffuse  = k_d · I_light · max(0, N · L)             (漫反射，Lambert)

I_specular = k_s · I_light · max(0, R · V)^shininess   (高光，Phong)
```

**参数**：

| 参数 | 含义 | 典型值 |
|------|------|--------|
| `k_a, k_d, k_s` | 材质的环境/漫反射/高光系数 | 各 [0, 1] |
| `I_light` | 光源强度（RGB） | 通常 (1, 1, 1) |
| `shininess` | 高光指数（越大高光越集中）| 8 ～ 128 |

**Blinn-Phong 变体**：把 `R·V` 换成 `N·H`（H 是半程向量 `normalize(L+V)`），效果更真实、计算更快，是**现代管线里更常用的写法**。

### 4.2 阴影光线（Shadow Ray）

某点应不应该"被光照亮"？答案很简单：**从命中点朝光源方向发一条光线，看中间有没有东西挡着**。

```
                  ☀️ 光源
                    ▲
                    │ ← 阴影光线
                    │   如果中间有物体挡住 → 该点在阴影里，去掉 diffuse & specular
                    │
                    ●  命中点
```

**伪代码**：

```
def in_shadow(hit_point, light_pos):
    dir = normalize(light_pos - hit_point)
    dist = |light_pos - hit_point|
    shadow_ray = Ray(hit_point + ε·dir, dir)     # ε 偏移防自相交
    for obj in scene:
        t = obj.intersect(shadow_ray)
        if t > 0 and t < dist:
            return True     # 有东西挡住，在阴影里
    return False
```

**这条阴影光线让光追天生就有硬阴影**——完全免费，光栅化里为此要专门做 Shadow Map。

## 5. 递归光线追踪（Whitted-style Ray Tracing）

1980 年 Turner Whitted 的经典论文提出：

> 命中点如果材质是**镜面**，就沿反射方向发一条光线；如果是**透明**，就沿折射方向发一条光线；把追踪回来的颜色按比例混进去。递归执行。

### 5.1 反射方向

给定入射方向 `D`（打到表面的方向）和法线 `N`（都是单位向量）：

```
    R = D - 2 (D · N) N
```

**推导直觉**：

```
             N
             ▲
             │
        D ↘  │  ↗ R
           ↘ │ ↗
             ●─────── 表面

     入射角 = 反射角，且 D、R、N 共面
```

`D · N` 是 `D` 在法线上的投影长度（负数，因为 D 打向表面）；`2(D·N)N` 就是把 D 沿法线"抬起两倍"，从而得到镜像方向 R。

### 5.2 折射方向（Snell 定律）

**Snell 定律**：`n₁ sin θ₁ = n₂ sin θ₂`

给定 `D`（入射方向）、`N`（表面法线，指向入射侧）、`η = n₁/n₂`（相对折射率）：

```
cos θ₁ = -D · N
sin²θ₂ = η² · (1 - cos²θ₁)

if sin²θ₂ > 1:                (全反射，不折射)
    T = 反射方向

else:
    cos θ₂ = √(1 - sin²θ₂)
    T = η·D + (η·cos θ₁ - cos θ₂)·N
```

**典型折射率**：真空/空气 `1.0`、水 `1.33`、玻璃 `1.5`、钻石 `2.42`。

### 5.3 递归伪代码

```python
def trace(ray, depth):
    if depth > MAX_DEPTH:
        return BLACK

    hit = nearest_intersection(ray, scene)
    if hit is None:
        return background_color(ray)

    color = ambient_term(hit)

    # 1) 直接光照 + 阴影
    for light in lights:
        if not in_shadow(hit.point, light.position):
            color += phong_shading(hit, light, ray.direction)

    # 2) 反射（如果材质是镜面）
    if hit.material.k_reflect > 0:
        R = reflect(ray.direction, hit.normal)
        reflect_ray = Ray(hit.point + ε·R, R)
        color += hit.material.k_reflect * trace(reflect_ray, depth + 1)

    # 3) 折射（如果材质透明）
    if hit.material.k_refract > 0:
        T = refract(ray.direction, hit.normal, hit.material.ior)
        refract_ray = Ray(hit.point + ε·T, T)
        color += hit.material.k_refract * trace(refract_ray, depth + 1)

    return color


# 主循环
for j in range(H):
    for i in range(W):
        ray = generate_camera_ray(i, j)
        image[j, i] = trace(ray, depth=0)
```

**这就是完整的 Whitted-style 光追**——加起来不到 30 行伪代码，能画出反射、折射、硬阴影、Phong 高光。

## 6. 加速结构（Acceleration Structures）

### 6.1 为什么需要加速

场景有 100 万个三角形，一张 1920×1080 的图 = 200 万像素，每个像素至少 1 条光线，每条光线要和所有三角形求交：

```
2,000,000 × 1,000,000 = 2 × 10¹² 次求交
```

按现代 CPU 每秒 10⁹ 次简单浮点操作估算，需要**数千秒**。这就是为什么裸算光追一直不能实时——直到我们发明了**加速结构**。

### 6.2 BVH（层次包围盒）

**Bounding Volume Hierarchy** 的核心想法：

> 用一个包围盒（AABB，轴对齐盒）罩住所有物体；再把物体分成两半，各罩一个更小的盒子；递归下去，形成一棵树。

```
                [ 全场景大盒子 ]
                 /            \
          [ 左半盒子 ]      [ 右半盒子 ]
           /       \        /        \
        [子子]    [子子]  [子子]     [子子]
                     ⋯
```

**求交时**：

- 光线先测试**根节点大盒子**——不撞就整棵子树全跳过；
- 撞到就递归进入两个子节点各测一次；
- 直到叶子（放几个三角形），才和三角形做真正的求交。

**复杂度**：从 `O(N)` 降到 `O(log N)`。100 万个三角形从 100 万次求交降到 ~20 次盒子测试 + 少量三角形测试。

**光线-AABB 求交（Slab Method）**：分别对 x/y/z 三个轴计算光线进入/离开该轴切片的 t 区间，取三个区间的交集非空则相交。约 6 次比较 + 6 次除法即可。

### 6.3 KD-Tree 简介

另一种常用结构，思路类似 BVH 但**空间划分**（不是物体划分）：每层沿一个轴切一刀，把空间一分为二。构建慢但求交更快，Embree 等 CPU 光追库常用。

**规则**：静态场景倾向 KD-Tree/BVH（预处理彻底优化），动态场景倾向 BVH（重建/更新更容易）。RTX 硬件用的是 BVH。

## 7. 完整实例：手算一条光线打到一个球

现在用一个**完全可以纸笔手算**的例子把整条链路走完。

### 7.1 场景设置

```
● 相机：位于 O = (0, 0, 0)，看向 -z 方向
● 屏幕：z = -1 平面，居中
● 球体：中心 C = (0, 0, -5)，半径 r = 1
● 光源：位于 L = (2, 2, 0)，白色 I_light = (1, 1, 1)
● 材质：k_a = 0.1，k_d = 0.7，k_s = 0.4，shininess = 32
● 只考虑一个像素：图像正中央
```

场景可视化（俯视图，从 +y 看下去）：

```
        z=0                     z=-5
         │
         │    · · · · · · · · · │ · · · ·
   👁️ ──▶│                      ●   ← 球
         │    · · · · · · · · · │ · · · ·
         │
         O                      C=(0,0,-5)
```

### 7.2 生成初始光线

**中心像素** → NDC 坐标 `(u, v) = (0, 0)`，屏幕平面 z = -1：

```
屏幕点 = (0, 0, -1)
D = normalize((0, 0, -1) - (0, 0, 0)) = (0, 0, -1)
```

**光线**：`P(t) = (0, 0, 0) + t · (0, 0, -1)`

### 7.3 求交（解一元二次方程）

**准备量**：

```
L = O - C = (0,0,0) - (0,0,-5) = (0, 0, 5)
```

**计算系数**：

```
a = D · D = 1                                (D 是单位向量)
b = 2·(L · D) = 2·(0·0 + 0·0 + 5·(-1)) = -10
c = L · L - r² = (0² + 0² + 5²) - 1² = 24
```

**判别式**：

```
Δ = b² - 4ac = (-10)² - 4·1·24 = 100 - 96 = 4
```

**Δ > 0**，有两个交点：

```
t₁ = (-b - √Δ) / 2a = (10 - 2) / 2 = 4       ← 较近，取这个
t₂ = (-b + √Δ) / 2a = (10 + 2) / 2 = 6
```

### 7.4 计算命中点与法线

```
P_hit = O + t₁·D = (0,0,0) + 4·(0,0,-1) = (0, 0, -4)
```

**法线**（球体：从球心指向命中点）：

```
N = normalize(P_hit - C)
  = normalize((0,0,-4) - (0,0,-5))
  = normalize((0, 0, 1))
  = (0, 0, 1)
```

**几何验证**：命中点 `(0,0,-4)` 到球心 `(0,0,-5)` 的距离 = 1 = 半径 ✅

### 7.5 阴影测试

从命中点 `P_hit = (0,0,-4)` 朝光源 `L_pos = (2,2,0)` 发阴影光线：

```
方向 = L_pos - P_hit = (2, 2, 4)
距离 dist = |(2,2,4)| = √(4+4+16) = √24 ≈ 4.899
方向单位向量 L_dir = (2, 2, 4)/√24 ≈ (0.408, 0.408, 0.816)
```

场景里只有一个球，且我们**从球表面出发朝向光源**，加一点 ε 偏移后不会打回自己。可以简单验证阴影光线不会再次和球相交（数学上会得到 `t < 0` 或无解），因此：

```
in_shadow = False → 该点被光源直接照亮
```

### 7.6 Phong 着色

**归一化的关键向量**（P_hit = (0,0,-4)）：

```
N = (0, 0, 1)                                       (已算)
L = normalize(L_pos - P_hit) = (0.408, 0.408, 0.816)
V = normalize(O - P_hit) = normalize((0,0,4)) = (0, 0, 1)
```

**漫反射项**：

```
N · L = 0·0.408 + 0·0.408 + 1·0.816 = 0.816
I_diffuse = k_d · I_light · max(0, N·L)
          = 0.7 · 1 · 0.816
          = 0.571
```

**高光项**（用 Blinn-Phong 半程向量 H 会更简单，这里用经典 Phong）：

反射方向 R（把 L 关于 N 反射）：

```
R = 2(N·L)·N - L
  = 2·0.816·(0,0,1) - (0.408, 0.408, 0.816)
  = (0, 0, 1.632) - (0.408, 0.408, 0.816)
  = (-0.408, -0.408, 0.816)
```

```
R · V = -0.408·0 + -0.408·0 + 0.816·1 = 0.816

I_specular = k_s · I_light · max(0, R·V)^shininess
           = 0.4 · 1 · 0.816^32
```

计算 `0.816^32`：`ln(0.816) ≈ -0.2035`，`32 · (-0.2035) = -6.51`，`e^-6.51 ≈ 0.00149`。

```
I_specular ≈ 0.4 · 0.00149 ≈ 0.000596       (非常小，因为 shininess=32 让高光很集中)
```

**环境光项**：

```
I_ambient = k_a · I_light = 0.1 · 1 = 0.1
```

**最终颜色**：

```
Color = I_ambient + I_diffuse + I_specular
      = 0.1 + 0.571 + 0.000596
      ≈ 0.672
```

### 7.7 结果

对这个中心像素，光追输出灰度值 **≈ 0.672**（若映射到 8 位：`0.672 × 255 ≈ 171`）。

**现象解读**：

- 球正对相机的那一点朝光源"斜 45°"，`N·L = 0.816` 说明它不是最亮点（最亮点应该在球体正对光源方向）；
- 高光项 `0.0006` 几乎为 0，说明中心点不在高光的"闪光锥"里；
- 环境光提供最低 0.1 的保底亮度，这正是 Phong 模型里环境光的作用。

如果要绘制整幅图，只需**对每个像素重复上述过程**——这就是最朴素的光追渲染器的全部逻辑。

## 8. 光线追踪 vs 光栅化

| 维度 | 光栅化 | 光线追踪 |
|------|--------|----------|
| **主循环** | 遍历三角形，投影到屏幕 | 遍历像素，往场景发光线 |
| **时间复杂度** | O(三角形数) | O(像素数 · log(三角形数))（有 BVH）|
| **反射** | 后处理 hack（SSR / Cubemap）| 天生递归 |
| **折射** | 很难 | 天生（Snell）|
| **硬阴影** | Shadow Map（走样） | 阴影光线（精确）|
| **软阴影/GI** | 各种近似（SSAO/GTAO/RTGI hack）| 蒙特卡洛采样自然产生 |
| **实时性** | 极强（GPU 天生擅长） | 直到 2018 RTX 才实时 |
| **代码复杂度** | 极高（管线庞杂） | 相对简单（几百行能写个玩具）|

**现代趋势**：**混合渲染**——光栅化管线里，用硬件光追做反射、阴影、GI 这些光栅化的痛点，性能和质量兼得。UE5 的 Lumen、几乎所有 3A 游戏都在这么做。

---

## 9. 现代扩展：路径追踪与实时光追

### 9.1 路径追踪（Path Tracing）

Whitted-style 光追只处理**理想镜面反射和折射**。真实世界大部分表面是**粗糙的**，反射方向不是唯一的，而是一个"分布"。

**路径追踪**：命中点朝**随机方向**发一条次级光线（服从材质的 BRDF 分布），采样多条路径然后平均——这是**蒙特卡洛积分**在渲染方程上的应用。

**渲染方程**（Kajiya 1986，一切现代真实感渲染的圣经）：

```
L_o(p, ω_o) = L_e(p, ω_o) + ∫Ω  f_r(p, ω_i, ω_o) · L_i(p, ω_i) · (n·ω_i) dω_i
             ↑ 自发光       ↑ 半球积分：来自各方向的入射光被反射成 ω_o 方向
```

- `f_r` = BRDF（双向反射分布函数），描述表面材质；
- 蒙特卡洛法：从半球随机采样若干 `ω_i`，用 `1/N · Σ` 近似积分。

路径追踪能自然产生**软阴影、间接光、色彩溢出、焦散、次表面散射**等所有真实光照现象——代价是**噪声大、需要 1000+ 采样才干净**。因此现代实时路径追踪伴随大量**降噪算法**（NVIDIA OptiX Denoiser、SVGF、ReSTIR 等）。

### 9.2 硬件光追（RTX / DXR / Vulkan RT）

2018 年 NVIDIA Turing 架构（RTX 20 系）第一次在 GPU 上引入**RT Core**——专门做两件事：

1. **BVH 遍历**（硬件加速的树形搜索）；
2. **光线-三角形求交**（硬件电路完成 Möller-Trumbore）。

对应的 API：

- **DXR（DirectX Raytracing）**：微软的 D3D12 扩展；
- **Vulkan Ray Tracing**：Vulkan 1.2+ 扩展；
- **Metal Ray Tracing**：Apple 的方案。

关键 shader 阶段：

```
    Ray Generation Shader   ← 你写的，产生光线
              │
              ▼
    Acceleration Structure  ← 驱动/硬件构建的 BVH
              │
              ▼
    Intersection Shader     ← 自定义几何时用
              │
              ▼
      ┌──────┴──────┐
      ▼             ▼
   Hit Shader   Miss Shader
   （命中）      （没打到）
```

用起来的心智模型接近上面的 Whitted 伪代码，只是要写成 shader 语言（HLSL/GLSL）。

## 10. 光线追踪的常见应用场景

前面把"一条光线怎么打出去、怎么求交、怎么着色、怎么递归"讲透了，那么**光线追踪到底在工业界被用来干什么**？下面把工程上最常见的用法归纳成 9 大类，从"最经典"到"最前沿"依次展开。

> 看似五花八门，但**本质都是一件事**：让像素颜色 = "从相机出发那条光线在场景里遭遇的一切"，只是应用领域不同、对精度/速度的取舍不同。

---

### 10.1 影视级离线渲染

**做什么**：动画电影、真人电影特效里那些以假乱真的画面——皮克斯、迪士尼、工业光魔的镜头，几乎全部用光追渲染。

**为什么用光追**：

- **物理正确的光影** —— 软阴影、次表面散射（皮肤、蜡）、焦散（水下光斑）、体积雾都能自然出现；
- **不惜时间换质量** —— 每帧渲染几分钟到几十小时都可以接受（渲染农场几千台机器并行）；
- **一致性** —— 复杂场景里的间接光、反射不会像光栅化那样"打补丁"。

**代表工具/引擎**：Pixar RenderMan、Arnold、V-Ray、Redshift、Cycles（Blender）。

**典型作品**：《玩具总动员》系列、《阿凡达》、《曼达洛人》、几乎所有漫威/DC 大片。

---

### 10.2 游戏中的实时光追

**做什么**：3A 游戏用 GPU 硬件光追（RTX / DXR / Vulkan RT）实现**局部**光追效果，主渲染仍是光栅化。

**典型子场景**：

| 效果 | 用光追做什么 |
|:--|:--|
| **实时反射** | 打反射光线求命中点颜色，替代 SSR 的漏反射 |
| **软阴影** | 对每个像素向面光源随机采样几条阴影光线 |
| **全局光照** | RTX GI / Lumen 用光追探针采样间接光 |
| **环境光遮蔽 (RTAO)** | 半球随机方向短距离光追，取遮蔽率 |
| **焦散 / 玻璃折射** | 折射光线递归求交 |

**代表游戏/引擎**：《赛博朋克 2077》Overdrive 模式、《控制》、《我的世界 RTX》、UE5 Lumen、Unity HDRP。

**关键取舍**：**每像素只发 1~2 条光线** + **神经网络降噪（DLSS Ray Reconstruction / OptiX Denoiser）**，用"少采样 + AI 补"换实时性能。

---

### 10.3 建筑与工业设计可视化

**做什么**：设计师做完 3D 模型，需要给客户看"这栋楼盖出来长什么样、光照怎么样"。

**为什么用光追**：

- 甲方要看"太阳早上 9 点从东窗照进来是什么样"——需要**物理正确的日照模拟**；
- 材质表现（大理石、玻璃幕墙、金属）光栅化容易露馅；
- 一次渲染几分钟可接受，追求**照片级真实**。

**代表工具**：Enscape、Twinmotion、Lumion、Corona Renderer、V-Ray for SketchUp/Revit。

**典型场景**：房地产售楼处渲染图/漫游动画、建筑投标效果图、室内设计方案演示。

---

### 10.4 产品渲染与广告

**做什么**：手机、汽车、饮料瓶、化妆品的**产品图/广告片**，早已不再实拍——直接建模 + 光追渲染。

**为什么用光追**：

- 精准控制光位/材质，可以任意换配色、换环境；
- 玻璃、液体、金属、皮革的反射折射用光追**开箱即用**；
- 比实拍便宜，还能出 8K 甚至 16K 素材。

**代表工具**：KeyShot、Blender Cycles、V-Ray、Octane。

**典型案例**：苹果发布会所有产品镜头（99% 是渲染）、汽车广告的"车漆流光"镜头、电商平台的商品主图。

---

### 10.5 科学与工程仿真

**做什么**：光线追踪在**非视觉**领域也大量使用，因为它本质上是一种"**沿直线路径的物理传播**求解器"。

| 领域 | 光追用来算什么 |
|:--|:--|
| **雷达/激光雷达仿真** | 电磁波/激光的多径反射，用于自动驾驶 LiDAR 数据合成 |
| **无线通信规划** | 5G/6G 基站信号的多径传播、覆盖率评估（Wireless Ray Tracing）|
| **声学仿真** | 音乐厅混响设计、汽车座舱声学（声线追踪，与光线追踪同构）|
| **热辐射分析** | 卫星表面/发动机内部的辐射换热计算 |
| **地震波正演** | 石油勘探中的射线追踪法 |

**代表工具**：NVIDIA Sionna（无线）、Wireless InSite、ODEON（声学）、Ansys HFSS。

---

### 10.6 CAD / CAM / 光学设计

**做什么**：设计一款相机镜头、望远镜、投影仪、AR 眼镜光机时，工程师要**追踪几百万条光线穿过一堆透镜**，评估像差、畸变、光斑。

**为什么用光追**：

- 折射/反射本来就是几何光学的核心；
- 直接用 Snell 定律递归追踪，非常自然；
- 精度远高于光栅化（后者根本做不了）。

**代表工具**：Zemax OpticStudio、Code V、LightTools、TracePro。

**典型场景**：手机潜望长焦镜头设计、AR/VR 光机设计（Waveguide / Pancake）、车灯 / 大灯配光设计。

---

### 10.7 医学影像与体绘制

**做什么**：把 CT / MRI 的三维体数据（Volume）渲染成医生能直观看的立体图。

**为什么用光追**：

- **体积光追（Volume Ray Casting）** 从相机沿光线在体数据里步进，累计吸收/散射；
- 支持骨骼、器官、血管的半透明分层；
- 比传统 MPR（多平面重建）直观得多。

**代表工具/框架**：NVIDIA IndeX、VTK、3D Slicer、飞利浦/西门子/GE 的高端影像工作站。

**典型场景**：术前规划（比如肝脏血管走向）、放疗剂量可视化、教学演示。

---

### 10.8 VR / AR / 数字孪生

**做什么**：VR 头显、AR 应用、工业数字孪生里，把光追当作"光影质量层"。

**关键子场景**：

- **VR 全景漫游**：一次预渲染 6 面立方体贴图 → 头戴设备实时观看，光追提供电影级画质；
- **AR 融合**：用光追做**虚实反射**——虚拟物体倒映在真实桌面（需要环境光探针 + 光追）；
- **数字孪生工厂**：光追渲染真实光照下的车间/管线，用于培训、操作演示、故障可视化。

**代表框架**：NVIDIA Omniverse（USD + RTX 光追）、Unreal Engine + Pixel Streaming、Varjo。

---

### 10.9 AI 训练数据合成

**做什么**：训练感知模型（自动驾驶、机器人视觉）需要海量带标注的图像，实拍成本高、危险场景（车祸、极端天气）拍不到——用光追**合成数据**。

**为什么用光追**：

- 生成的图像"以假乱真"，能缩小 sim-to-real 差距；
- 天生带**完美标注**：深度图、法线图、语义分割、光流全免费；
- 可以任意变化天气、光照、物体摆放，形成海量 corner case。

## 11. 工程实现要点

1. **偏移量 ε**：所有次级光线（阴影、反射、折射）的起点都要沿方向偏移一个小量（`1e-4` 左右）以避免自相交。
2. **归一化**：光线方向、法线、光源方向……只要参与点积、反射公式，都得是**单位向量**，否则结果全错。
3. **递归深度上限**：反射折射会指数增长，加个 `MAX_DEPTH = 5~10`；深度超限直接返回黑色。
4. **数值稳定性**：解二次方程时，`b > 0` 情况用 `t = -2c/(b+√Δ)` 替代 `-b+√Δ)/(2a)` 避免大数相消。
5. **多线程**：像素之间完全独立，天生尬并行——直接 `parallel_for` 每一行/每一块。
6. **BVH 构建**：SAH（Surface Area Heuristic）是通用最优选择；实时用 LBVH（Morton 码 + 基数排序）能几毫秒重建全场景。
7. **降噪**：路径追踪必配。TAA + 空间滤波（如 SVGF）是入门方案。
8. **RGB 渲染的注意事项**：真正物理正确要在**线性色彩空间**里累加，最后再 gamma 校正到 sRGB 显示。

## 12. 常见疑问 FAQ

### Q1：为什么是"逆向"追踪？

因为从光源正向追踪时，绝大部分光线永远打不到眼睛，浪费算力。**反过来，从眼睛出发，每条光线都对应一个像素，每次计算都有价值**。这个反转是光追高效的根本原因。

### Q2：光追一定比光栅化慢吗？

裸算是的。但**加了 BVH + 硬件加速后，某些效果（真实反射、复杂阴影）光追反而更快**——因为光栅化实现这些效果需要多 pass、后处理，累加起来开销也不小。

### Q3：为什么 shader 里几乎从不实现真正的光追？

传统光栅化管线里 fragment shader 的输入是"这个像素属于哪个三角形"，无法向场景其他部分发起查询。**必须有 acceleration structure 和硬件求交单元支持**，才能在 shader 里发光线——这就是 RTX 引入的能力。DXR/VK-RT 都是把光追作为**独立管线**加进来的。

### Q4：光追和路径追踪有啥区别？

- **光线追踪（Ray Tracing）**：广义术语，泛指所有"发光线求交着色"的算法；狭义常指 Whitted-style；
- **路径追踪（Path Tracing）**：光追的一个具体算法，用蒙特卡洛法解渲染方程，能得到照片级真实感；
- **实时光追（Real-time Ray Tracing）**：光追硬件加速后的实时应用，通常混合光栅化 + 少量光线做特定效果。

### Q5：为什么光追天生做透视正确？

光栅化里，屏幕空间线性插值属性（如 UV）在透视投影下是错的，需要"透视校正插值"（详见 [纹理贴图详解.md](./如何把一张2D图像贴到三角网格立体上-纹理贴图详解.md)）。**光追的插值发生在 3D 空间的重心坐标里**——本来就是 3D 空间的物理量，不需要任何校正。

### Q6：可以只用光追做游戏吗？

理论上可以，实际上现在还太贵。**混合渲染是主流**：光栅化画基础几何，光追做反射、GI、软阴影这些"锦上添花"的效果。UE5 Lumen、Cyberpunk 2077 RTX 都是这个思路。

### Q7：Phong 模型物理正确吗？

不正确，只是**经验模型**（凑出来"看起来像"的效果）。物理正确的模型是**基于微表面理论的 BRDF**（如 Cook-Torrance、GGX），是现代 PBR 渲染的基础。但 Phong 简单直观、教学价值高、性能好，仍在低端场景广泛使用。

### Q8：ε 偏移量选多大？

经验值 `1e-4` ～ `1e-3`（在场景尺度大约 1~100 的情况下）。太小会自相交，太大会导致阴影泄漏。**更好的做法**：沿法线偏移而不是沿光线方向偏移；或用"self-intersection avoidance"专门算法（PBRT 里有讨论）。

## 13. 一句话总结

> **光线追踪 = 一条参数方程 `P = O + tD` + 一堆求交公式（球/平面/三角形）+ 一个着色模型（Phong/PBR）+ 一个递归（反射/折射） + 一棵 BVH（加速）**。
>
> - 想要**镜子里的镜子**？→ 递归多来几层；
> - 想要**玻璃、水**？→ Snell 折射公式；
> - 想要**照片级真实感**？→ 蒙特卡洛路径追踪 + 渲染方程；
> - 想要**实时**？→ 硬件光追（RTX/DXR）+ 降噪。
>
> 无论技术怎么演进，核心就是那条 `P(t) = O + tD` 和"从眼睛发光线撞场景"这两件事。**记住它们，你就掌握了光线追踪的灵魂**。
