## 0. 直觉引入：剥橘子皮的比喻

想象你手里有一个**橘子（3D 立体表面）**，你想知道："如果我把橘子皮上的花纹拍成一张 2D 照片印出来，怎么把这张照片**重新贴回**橘子表面，让花纹和原来对齐？"

```
   现实问题                     图形学问题
   ──────────                  ─────────────
   橘子（立体）       ↔        三角网格 3D 模型
   橘子皮的花纹       ↔        2D 贴图（PNG）
   "把皮剥下铺平"     ↔        UV 展开（Unwrap）
   "把花纹贴回去"     ↔        纹理采样（Texture Sampling）
```

整个"贴图"流程的本质就是**解决两个反向问题**：

1. **离线**（美术阶段）：把 3D 表面"剥皮铺平"到 2D 平面，得到每个顶点的"2D 平面坐标" → 这就是 **UV 坐标**；
2. **运行时**（GPU 渲染阶段）：屏幕上每个像素反过来问："我对应的 3D 表面那一点，在贴图的哪个像素？" → 用 UV 坐标查图。

> **一句话本质**：**给每个 3D 顶点存一对 2D 坐标 `(u, v)`，让 3D 三角形和 2D 贴图区域一一对应起来。**

## 1. 前置概念扫盲

### 1.1 顶点、边、三角形、网格

```
顶点 (Vertex)：
  3D 空间中的一个点，至少包含位置 (x, y, z)
  通常还附带：法线、UV、颜色、切线等属性

边 (Edge)：
  连接两个顶点的线段

三角形 (Triangle / Face)：
  由 3 个顶点定义的平面区域，是 GPU 渲染的最小单元

网格 (Mesh)：
  一堆三角形拼成的整体，用来"近似"任意复杂的曲面
  例：一个 3D 茶壶 = 几千个三角形

数据存储（典型"索引网格"）：
  vertices[] = [V0, V1, V2, V3, V4, ...]
  indices[]  = [0,1,2,  0,2,3,  3,2,4, ...]   ← 每 3 个索引一个三角形
```

**为什么用三角形不用四边形？**
- 三点必共面（三角形天然是平的，不会扭曲）；
- 永远是凸多边形（光栅化算法简单）；
- 任何多边形都可三角化（通用性强）。

### 1.2 像素、纹素、图像数组

| 术语 | 英文 | 含义 |
|------|------|------|
| **像素** | Pixel | 屏幕上的一个色点（Picture Element） |
| **纹素** | Texel | 贴图（纹理）上的一个色点（Texture Element） |
| **图像/纹理** | Texture | 一张二维数组，元素是 RGBA 颜色值 |

```
内存里一张 4×4 的 RGB 贴图：

  texture[y][x] = (R, G, B)     // y 行 x 列
  存储顺序通常是从下到上、从左到右（OpenGL 约定）

         x=0    x=1    x=2    x=3
  y=3 ┌──────┬──────┬──────┬──────┐
      │ T03  │ T13  │ T23  │ T33  │
  y=2 ├──────┼──────┼──────┼──────┤
      │ T02  │ T12  │ T22  │ T32  │
  y=1 ├──────┼──────┼──────┼──────┤
      │ T01  │ T11  │ T21  │ T31  │
  y=0 ├──────┼──────┼──────┼──────┤
      │ T00  │ T10  │ T20  │ T30  │
      └──────┴──────┴──────┴──────┘
```

> 后文我们说"采样贴图"，意思就是"按某种规则从这个二维数组里读出一个颜色"。

### 1.3 涉及的几个坐标系

整个流水线会跨越多个坐标系，先一次性列清楚：

| 编号 | 坐标系 | 范围 | 含义 |
|------|--------|------|------|
| ① | **模型空间** (Model / Object) | 任意 | 顶点在 3D 模型自己的局部坐标系下的位置 |
| ② | **世界空间** (World) | 任意 | 模型放进场景后的全局坐标 |
| ③ | **相机空间** (View / Eye) | 任意 | 以相机为原点 |
| ④ | **裁剪空间** (Clip) | 齐次 `(x,y,z,w)` | 准备做透视除法 |
| ⑤ | **NDC** (Normalized Device Coord.) | `[-1, 1]³` | 透视除法后的归一化坐标 |
| ⑥ | **屏幕空间** (Screen / Pixel) | `[0,W] × [0,H]` | 最终的像素坐标 |
| ⑦ | **UV 空间** (Texture Coord.) | `[0, 1]²` | 贴图上的坐标，**与屏幕无关** |
| ⑧ | **纹素空间** (Texel) | `[0,W_t] × [0,H_t]` | 贴图的整数像素索引 |

> 贴图流程主要用到的是 **⑦ UV 空间** 和 **⑧ 纹素空间**——这两者只差一次"乘以贴图分辨率"。

## 2. 把问题数学化：我们到底要解什么？

我们要在屏幕上画一个 3D 物体。屏幕由几百万个像素组成，每个像素需要一个**最终颜色**。

```
对屏幕上每一个像素 P_screen，要回答：

  ① P_screen 落在哪个 3D 三角形上？     ── 由光栅化解决
  ② 它在那个三角形上的具体位置是？      ── 由重心坐标解决
  ③ 那个位置对应贴图的哪个纹素？        ── 由 UV 插值解决
  ④ 该纹素的颜色是多少？               ── 由纹理采样解决
```

整篇文档剩下的部分就是把这 4 个问题逐一拆开讲清楚。

## 3. UV 坐标：连接 3D 表面与 2D 图像的桥梁

### 3.1 UV 的定义

**UV 坐标**是给每个 3D 顶点附加的一对 2D 数值 `(u, v)`，规定：

```
u, v ∈ [0, 1]

(0, 0)  ──→  贴图的左下角
(1, 0)  ──→  贴图的右下角
(0, 1)  ──→  贴图的左上角
(1, 1)  ──→  贴图的右上角
```

> **为什么叫 U, V？**
> 因为 X, Y, Z 已经被 3D 空间占用了。U 是 X 的右邻字母、V 是 Y 的右邻字母——纯命名习惯，无深意。

> **为什么是 `[0,1]` 而不是直接用像素索引？**
> 1. **分辨率无关**：同一套 UV 适配 512×512 和 4096×4096 的贴图（例如 LOD 切换）；
> 2. **数学统一**：插值、变换都在浮点上做；
> 3. **平铺友好**：超出 `[0,1]` 时按取小数取模，自动平铺。

### 3.2 顶点数据结构

GPU 上一个典型的顶点结构：

```cpp
struct Vertex {
    glm::vec3 position;   // 3D 位置 (x, y, z)
    glm::vec3 normal;     // 法线 (用于光照)
    glm::vec2 uv;         // ★ 纹理坐标 (u, v)
    // 可选：tangent, color, bone weight ...
};
```

**关键认识**：`uv` 和 `position` **同等地位**，都是顶点的固有属性，导出时就写死了。

### 3.3 UV 与图像像素的对应关系

UV 是 `[0,1]` 的归一化坐标，转换为贴图像素索引很简单：

```
设贴图分辨率为 W × H，UV 为 (u, v)，则对应的纹素坐标：

   x_texel = u · W
   y_texel = v · H

  例：贴图 1024×1024，UV = (0.25, 0.75)
  → 纹素坐标 = (256, 768)
```

```
   贴图（W × H）
   v=1 ┌─────────────┐
       │             │
       │   ●(0.25,   │
       │    0.75)    │
       │             │
   v=0 └─────────────┘
       u=0           u=1
```

## 4. UV 展开（Unwrap）：UV 坐标是怎么来的

UV 是**美术在建模软件里手动 / 半自动指定的**。"展 UV"（Unwrap UVs）就是把 3D 模型的表面"剥皮铺平"到 2D 平面的过程：

```
3D 立方体（6 个面）              展开后的 UV 布局（"十字图"）

       ┌─────┐                    ┌─────┐
       │ Top │                    │ Top │
   ┌───┼─────┼───┬───┐        ┌───┼─────┼───┬───┐
   │ L │Front│ R │Back│   →   │ L │Front│ R │Back│
   └───┼─────┼───┴───┘        └───┼─────┼───┴───┘
       │ Bot │                    │ Bot │
       └─────┘                    └─────┘
```

每个 3D 面对应一块 2D UV 区域，美术在这块区域上画对应的图案。常见展 UV 方式：

| 方式 | 适用对象 | 特点 |
|------|----------|------|
| 平面投影 (Planar) | 地面、墙面 | 沿某轴"压扁" |
| 立方体投影 (Box) | 箱子、建筑 | 6 面分别平面投影 |
| 球面 / 柱面投影 | 地球仪、瓶身 | 经纬度展开 |
| LSCM 自动展开 | 复杂角色 | 数学上最小化拉伸 |
| 手工调整 | 高质量资产 | 美术逐面雕琢 |

> **接缝（Seam）**：你不可能把球面无损地铺平（数学上叫 "高斯曲率不为 0 → 不可展曲面"），必须沿某些边"切开"，这些切口就是接缝。展 UV 的艺术之一就是"把接缝藏在不显眼的地方"（比如人物头顶、背后）。

最终输出：模型文件（FBX / glTF / OBJ）里**每个顶点的 UV 都是定值**，运行时直接送 GPU。

### 4.1 手把手完整例子：把"木纹.png"贴到立方体前面

光讲理论容易飘，我们来手工算一遍——**用最简单的例子（立方体的一个面）**，从"3D 顶点位置"一路推到"贴图上的像素索引"，让你看清 UV 的每一个数字是怎么来的。

#### 场景设定

```
【3D 模型】
  一个边长为 1 的立方体，中心在原点。
  取"前面"（Front Face，朝向 +z 方向）来分析：

           y↑
            │
       V₃●─────●V₂        z 指向读者
        ╱│    ╱│
       ╱ │   ╱ │           4 个顶点位置：
   V₀ ●─────●V₁ │             V₀ = (-0.5, -0.5, 0.5)   左下
      │  ●──┼──● ────→ x       V₁ = ( 0.5, -0.5, 0.5)   右下
      │ ╱   │ ╱               V₂ = ( 0.5,  0.5, 0.5)   右上
      │╱    │╱                V₃ = (-0.5,  0.5, 0.5)   左上
      ●─────●
```

```
【贴图】
  一张 512×512 的 wood.png，画满了木纹。
  我们希望它"完整地"贴满整个前面。
```

#### Step 1: 决定 UV 布局（美术的选择）

美术看着这个正方形前面，最自然的做法是：**让整张贴图不缩放、不旋转、正好贴满这个面**。

于是给 4 个顶点分别指定：

```
   V₀ (3D 左下) ──→ UV = (0, 0)   ← 贴图左下角
   V₁ (3D 右下) ──→ UV = (1, 0)   ← 贴图右下角
   V₂ (3D 右上) ──→ UV = (1, 1)   ← 贴图右上角
   V₃ (3D 左上) ──→ UV = (0, 1)   ← 贴图左上角
```

> 💡 **UV 就是这么"人为规定"出来的**——没有算法自动生成，是美术根据"我想让贴图这样铺"这个意图直接指定的。
> 展 UV 软件（Blender、Maya）本质就是提供一个可视化界面，让美术拖动这些点。

#### Step 2: 三角化——4 个顶点变成 2 个三角形

GPU 只吃三角形，所以正方形要拆成 2 个三角形。**沿对角线 V₀→V₂ 切开**，得到：

- **T₁**（下半部分，含 V₁）= `(V₀, V₁, V₂)` —— 逆时针绕序
- **T₂**（上半部分，含 V₃）= `(V₀, V₂, V₃)` —— 逆时针绕序

```
   3D 空间的两个三角形                UV 空间的两个三角形

   V₃●─────●V₂                       (0,1)●─────●(1,1)
     │    ╱│                              │    ╱│
     │ T₂╱ │                              │ T₂╱ │      ← T₂ 在左上（V₃ 那半）
     │  ╱  │        ←→                    │  ╱  │
     │ ╱ T₁│                              │ ╱ T₁│      ← T₁ 在右下（V₁ 那半）
     │╱    │                              │╱    │
   V₀●─────●V₁                       (0,0)●─────●(1,0)

   T₁ = (V₀, V₁, V₂)                 T₁ 的 UV = ((0,0), (1,0), (1,1))
   T₂ = (V₀, V₂, V₃)                 T₂ 的 UV = ((0,0), (1,1), (0,1))
```

> 💡 **对角线方向怎么定的**：两个三角形共享的那条边就是对角线。
> - T₁ 用了 V₀ 和 V₂，T₂ 也用了 V₀ 和 V₂ → 共享边 **V₀↔V₂**（左下↔右上）→ 对角线是 **`╱` 方向**。
>
>
> **绕序（Winding Order）说明**：`(V₀, V₁, V₂)` 在屏幕上是**逆时针**方向。OpenGL 默认把逆时针视为"正面"，用于背面剔除。写索引时顺序不能乱。

#### Step 3: 顶点数据（GPU 眼中的样子）

写成 C++ 顶点数组：

```cpp
Vertex frontFace[] = {
    // position               uv
    {{-0.5f, -0.5f, 0.5f}, {0.0f, 0.0f}},  // V₀
    {{ 0.5f, -0.5f, 0.5f}, {1.0f, 0.0f}},  // V₁
    {{ 0.5f,  0.5f, 0.5f}, {1.0f, 1.0f}},  // V₂
    {{-0.5f,  0.5f, 0.5f}, {0.0f, 1.0f}},  // V₃
};
uint32_t indices[] = { 0,1,2,   0,2,3 };   // T₁ 和 T₂
```

**重点观察**：这里 UV 是**手动写死的常量**，不是算出来的——它就是"美术意图"的直接体现。

#### Step 4: 手工验证 UV 是否合理

假设 GPU 光栅化后要画三角形 T₁ 内部一个点 P，它的 3D 位置正好是三角形 T₁ 的**重心**：

```
P₃D = (V₀ + V₁ + V₂) / 3
    = ((-0.5+0.5+0.5)/3, (-0.5-0.5+0.5)/3, 0.5)
    = (0.1667, -0.1667, 0.5)
```

由重心坐标定义（第 5 章会详推），重心的 (α, β, γ) = (1/3, 1/3, 1/3)。

按第 5.3 节的插值公式，P 的 UV 是：

```
uv_P = (1/3)·(0,0) + (1/3)·(1,0) + (1/3)·(1,1)
     = (2/3, 1/3)
     ≈ (0.6667, 0.3333)
```

#### Step 5: UV → 贴图像素索引

贴图分辨率 512×512，按第 3.3 节公式：

```
x_texel = u × W = 0.6667 × 512 ≈ 341
y_texel = v × H = 0.3333 × 512 ≈ 171

⇒ 该像素最终读取 wood.png[171][341] 的颜色
```

```
   wood.png (512×512)
   ┌─────────────────────┐
   │                     │
   │                     │
   │           ● (341,171)   ← 屏幕上那个像素
   │           采样这里        对应到贴图这个位置
   │                     │
   │                     │
   └─────────────────────┘
```

#### Step 6: 全流程回顾——所有数字是怎么串起来的

| 阶段 | 数据 | 谁产生 |
|------|------|--------|
| ① 建模 | `V₀..V₃` 的 3D 位置 | 美术在 Blender 里建 |
| ② 展 UV | `V₀..V₃` 的 `(u,v)` 值 | 美术手动指定（或软件辅助）|
| ③ 三角化 | 索引 `[0,1,2, 0,2,3]` | 建模软件自动生成 |
| ④ 导出 | `.gltf` 文件 | 建模软件保存 |
| ⑤ 加载 | 上传到 GPU 的 VBO | 引擎运行时 |
| ⑥ 光栅化 | 每像素的 `(α,β,γ)` | GPU 硬件 |
| ⑦ UV 插值 | 每像素的 `(u,v)` | GPU 硬件（透视正确） |
| ⑧ 采样 | 从 wood.png 读颜色 | GPU 纹理单元 |

**核心结论**：
- **UV 不是"算出来"的，是"设计出来"的**——美术意图 → 顶点属性 → GPU 数据；
- **一旦 4 个顶点的 UV 被指定，三角形内部所有像素的 UV 都由重心坐标插值自动完成**；
- **UV 与贴图分辨率解耦**：把 wood.png 换成 4096×4096 的高清版，代码一行都不用改，贴图自动"高清化"。

#### 变体：如果我想让贴图"重复 3×3 次"呢？

只需把 UV 从 `[0,1]` 扩到 `[0,3]`：

```cpp
Vertex frontFace[] = {
    {{-0.5f, -0.5f, 0.5f}, {0.0f, 0.0f}},
    {{ 0.5f, -0.5f, 0.5f}, {3.0f, 0.0f}},   // ← 变了
    {{ 0.5f,  0.5f, 0.5f}, {3.0f, 3.0f}},   // ← 变了
    {{-0.5f,  0.5f, 0.5f}, {0.0f, 3.0f}},   // ← 变了
};
```

配合 `GL_REPEAT` 的 wrap 模式（第 9 章详解），GPU 会把 `u=1.7` 当成 `u=0.7` 来采样——效果就是同一张木纹在这个面上平铺了 9 次。**没有改贴图、没有改几何，只改了 4 个数字**。

> 这个例子是"手工指定 UV"的最简版本。真实项目中：
> - 简单几何（平面、方块、球体）——引擎/程序化生成 UV 就够了；
> - 复杂角色（人脸、盔甲）——必须美术在 Blender / Maya / Substance Painter 里手工展 + 调整。
> 但**不管多复杂，UV 的本质就是这里演示的：给每个顶点一对 (u,v) 数值，交由 GPU 光栅化时按重心坐标插值。**

## 5. 重心坐标与三角形内插值

### 5.1 重心坐标的定义与几何含义

**问题**：屏幕上某个像素 P 落在了三角形 V₀V₁V₂ 内部，我怎么知道它的 UV 是多少？只有 3 个顶点的 UV 已知。

**答案**：用重心坐标插值。

---

#### 为什么可以这样？——推导

我们只知道 3 个顶点上的属性值（UV、颜色、深度……都可以），要给三角形内部任意一点 P 一个"合理"的属性值。先问一句：**"合理的插值"到底该满足什么？**

##### 第 1 步：合理的插值必须满足的 3 条性质

设 `f(V₀), f(V₁), f(V₂)` 是 3 个顶点上已知的属性（比如 UV），`f(P)` 是我们想推出的、P 点上的属性。任何一个"讲道理"的插值方案 `f(P)`，都必须满足下面 3 条：

**性质 ①（顶点还原）**：如果 P 恰好落在 V₀，那 `f(P)` 必须等于 `f(V₀)`；落在 V₁、V₂ 同理。
——不然连"顶点自己的值都对不上"，就算不上插值了。

**性质 ②（连续、无跳变）**：P 从三角形一端平滑移动到另一端时，`f(P)` 也必须平滑变化，不能有台阶或突变。
——不然贴图上会看到破洞和撕裂。

**性质 ③（线性 / 无偏）**：不能凭空多出"来源不明"的成分——`f(P)` 必须完全由 `f(V₀), f(V₁), f(V₂)` 三者组合出来，且**权重之和 = 1**（不放大也不缩小）。
——数学上这叫"仿射组合"，是"平均"这个概念的推广。

用式子写就是：我们要找一组**只跟位置 P 有关、跟具体属性无关**的权重 `(α, β, γ)`，使得对**任何**属性 f 都有：

```
   f(P) = α · f(V₀) + β · f(V₁) + γ · f(V₂)
```

并且这组权重要满足：

```
   α + β + γ = 1              ← 性质 ③（权重和为 1）
   P = V₀ 时 (α,β,γ) = (1,0,0) ← 性质 ①
   P = V₁ 时 (α,β,γ) = (0,1,0)
   P = V₂ 时 (α,β,γ) = (0,0,1)
   (α,β,γ) 随 P 连续变化       ← 性质 ②
```

##### 第 2 步：这样的权重一定存在，而且唯一——就是"重心坐标"

现在把上面对 f 的要求，**同时应用到 x 坐标和 y 坐标本身**（因为 P 的坐标也可以看作定义在三角形上的一个"属性"，顶点处的值就是顶点自己）：

```
   P.x = α · V₀.x + β · V₁.x + γ · V₂.x
   P.y = α · V₀.y + β · V₁.y + γ · V₂.y
   1   = α       + β       + γ
```

这是 **3 个未知数 (α, β, γ)、3 个方程**的线性方程组。

只要三角形不退化（三点不共线，也就是有正的面积），这个 3×3 系数矩阵就是可逆的——由**克莱默法则**：

```
              α = △PV₁V₂ / △V₀V₁V₂
   解得：      β = △PV₂V₀ / △V₀V₁V₂
              γ = △PV₀V₁ / △V₀V₁V₂
```

（三角形面积用叉积就能算，具体推导见 §5.2。）

**这组** `(α, β, γ)` **就是我们要找的权重，它就叫"重心坐标"**。

它天然满足前面 3 条性质：

| 要求 | 重心坐标为什么满足 |
| --- | --- |
| ① 顶点还原 | P=V₀ 时 △PV₁V₂ = △V₀V₁V₂，α=1；另两个子三角形退化为线段，面积 0 → β=γ=0 ✅ |
| ② 连续变化 | P 平移 → 3 个子三角形面积连续变化 → (α,β,γ) 连续 ✅ |
| ③ 权重和为 1 | 3 个子三角形恰好把大三角形铺满：△PV₁V₂ + △PV₂V₀ + △PV₀V₁ = △V₀V₁V₂ ✅ |

**唯一性**：由 3×3 方程组可逆保证，同一个 P 不会解出两组不同的权重。

##### 第 3 步：几何 / 物理直观——为什么"面积比"就是权重

想象在 V₀、V₁、V₂ 三个顶点各挂一个质量分别为 α、β、γ 的重物（α+β+γ=1）。由物理里的**杠杆平衡 / 质心公式**：

```
   质心位置 = α · V₀ + β · V₁ + γ · V₂
```

如果我们希望这个质心正好落在 P 点，那么 α、β、γ 的取值就唯一确定了——**这就是"重心坐标"这个名字的来源**（Bary- 在希腊语里就是"重"）。

而"重物越重，质心离它越近"这条常识，翻译成三角形语言就是：

```
   α 越大 ⇒ P 越靠近 V₀（P 对面的子三角形 △PV₁V₂ 越大）
```

——正好和面积比公式一致。

##### 一句话总结推导

> "**任意一个三角形内部的点 P，都能被 3 个顶点唯一地、线性地、连续地表达出来**"
>
> ——这个"唯一的线性表达系数"就是重心坐标 (α, β, γ)；
> ——所以任何一个属性（UV、颜色、深度）都能用**同一组** (α, β, γ) 从 3 个顶点插值到 P。

理解了这个之后，下面的定义就是水到渠成的形式化：

---

**重心坐标**（Barycentric Coordinates）的定义：对三角形内任意一点 P，存在唯一一组 `(α, β, γ)` 使得：

```
   P = α·V₀ + β·V₁ + γ·V₂
   且  α + β + γ = 1
   且  α, β, γ ≥ 0  （P 在三角形内或边上）
```

**几何含义**（一定要记住）：

```
     V₀
      ●
     / \
    /   \
   /  P  \         α = △PV₁V₂ / △V₀V₁V₂   （P 对面的子三角形面积比）
  /   ●   \        β = △PV₂V₀ / △V₀V₁V₂
 /    α    \       γ = △PV₀V₁ / △V₀V₁V₂
●───────────●
V₁    γ      V₂

α 越大 ⇒ P 越靠近 V₀
β 越大 ⇒ P 越靠近 V₁
γ 越大 ⇒ P 越靠近 V₂

特殊点：
  V₀ = (1, 0, 0)        三角形顶点
  V₁ = (0, 1, 0)
  V₂ = (0, 0, 1)
  重心 = (1/3, 1/3, 1/3)  三条中线的交点
```

### 5.2 求解重心坐标（推导）

给定 P、V₀、V₁、V₂（都是 2D 点），求 (α, β, γ)。

**方法一：面积比法**（直观，适合理解）

```
  α = Area(P, V₁, V₂) / Area(V₀, V₁, V₂)
  β = Area(V₀, P, V₂) / Area(V₀, V₁, V₂)
  γ = Area(V₀, V₁, P) / Area(V₀, V₁, V₂)
```

其中 2D 三角形的有向面积可用叉积算：

```
  Area(A, B, C) = 0.5 · |(B - A) × (C - A)|
                = 0.5 · |(B.x - A.x)(C.y - A.y) - (B.y - A.y)(C.x - A.x)|
```

**方法二：解线性方程组**（GPU 实际用法）

由 `P = αV₀ + βV₁ + γV₂` 和 `α+β+γ=1`，代入 `α = 1-β-γ`：

```
   P = (1-β-γ)V₀ + βV₁ + γV₂
     = V₀ + β(V₁-V₀) + γ(V₂-V₀)

⇒  P - V₀ = β·e₁ + γ·e₂      其中 e₁ = V₁-V₀, e₂ = V₂-V₀
```

**把上面那个向量等式按分量拆开**（一个向量方程 = 两个标量方程）：

设：

- `e₃ = P - V₀`（P 相对 V₀ 的位移向量）
- `e₁ = V₁ - V₀`、`e₂ = V₂ - V₀`（三角形的两条邻边）

那么 `e₃ = β·e₁ + γ·e₂` 按 x、y 两个分量分别写就是：

```
  ┌ β · e₁.x  +  γ · e₂.x  =  e₃.x     …… (方程 ①：x 分量)
  │
  └ β · e₁.y  +  γ · e₂.y  =  e₃.y     …… (方程 ②：y 分量)
```

写成矩阵形式更直观：

```
  ┌ e₁.x   e₂.x ┐   ┌ β ┐     ┌ e₃.x ┐
  │             │ · │   │  =  │      │
  └ e₁.y   e₂.y ┘   └ γ ┘     └ e₃.y ┘
        A（2×2）        x         b
```

**未知数**只有 `β` 和 `γ` 两个，方程也正好两个（x 一个、y 一个），可以唯一解出（只要三角形不退化，即 `det(A) ≠ 0`）。

**用克莱默法则解**：

先算系数矩阵行列式（几何意义就是 `2·三角形有向面积`）：

```
  d = det(A) = e₁.x · e₂.y  -  e₂.x · e₁.y
```

然后把 A 的第 1 列换成 `b` 求 β，把第 2 列换成 `b` 求 γ：

```
             ┌ e₃.x   e₂.x ┐
  β = det │             │ / d  =  (e₃.x · e₂.y  -  e₂.x · e₃.y) / d
             └ e₃.y   e₂.y ┘

             ┌ e₁.x   e₃.x ┐
  γ = det │             │ / d  =  (e₁.x · e₃.y  -  e₃.x · e₁.y) / d
             └ e₁.y   e₃.y ┘

  α = 1 - β - γ
```

这三行**正是**下面 GLSL 伪代码中 `d / beta / gamma / alpha` 四行的一一对应实现。

GLSL 伪代码：

```glsl
vec3 barycentric(vec2 P, vec2 V0, vec2 V1, vec2 V2) {
    vec2 e1 = V1 - V0;
    vec2 e2 = V2 - V0;
    vec2 e3 = P  - V0;
    float d  = e1.x * e2.y - e2.x * e1.y;     // = 2·总面积
    float beta  = (e3.x * e2.y - e2.x * e3.y) / d;
    float gamma = (e1.x * e3.y - e3.x * e1.y) / d;
    float alpha = 1.0 - beta - gamma;
    return vec3(alpha, beta, gamma);
}
```

> 实际 GPU 中重心坐标由硬件**光栅化器**直接生成，开发者通常不用手写。

### 5.3 任意顶点属性的插值公式

一旦得到 `(α, β, γ)`，**任何附在顶点上的属性**都可以这样插值：

```
属性插值通式：
   Attr(P) = α · Attr(V₀) + β · Attr(V₁) + γ · Attr(V₂)

例：
  位置：     P    = α·P₀  + β·P₁  + γ·P₂
  UV：       uv_P = α·uv₀ + β·uv₁ + γ·uv₂      ← ★ 我们想要的
  颜色：     c_P  = α·c₀  + β·c₁  + γ·c₂
  法线：     n_P  = α·n₀  + β·n₁  + γ·n₂   （需重新归一化）
```

> **⚠️ 但这有个隐藏陷阱**：直接对屏幕空间做线性插值，在透视投影下会得到**错误的 UV**。下一节专门解决。

---

## 6. 透视正确插值：为什么不能"直接线性插值 UV"

### 6.1 反例：直接线性插值会扭曲

想象一条铁轨向远处延伸——离相机近的地方枕木间距大，远的地方间距小（透视收缩）。

```
若把贴图（一张棋盘格）贴到地面上，直接对屏幕空间线性插值 UV，
会得到这种结果：

   错误（线性插值）              正确（透视正确插值）
   ──────────────                ──────────────────
   ┌───────┐                     ┌───────┐
   │ □ □ □ │   近                │ □ □ □ │   近
   │ □ □ □ │                     │ □ □ □ │
   │ □ □ □ │                     │ ▢ ▢ ▢ │
   │ □ □ □ │   远                │  ▫ ▫ ▫│   远
   └───────┘                     └───────┘
   远处格子大小不变（错）         远处格子收缩（正确）
```

**原因**：透视投影是非线性的（涉及除法 `1/z`）。屏幕上等距的两点，对应到 3D 世界的距离不等距——所以**屏幕空间的"中点"≠ 3D 空间的中点**。

### 6.2 推导：正确公式是什么

**关键事实 1**：在透视投影前，`1/z`（深度的倒数）在屏幕空间是**线性**的。
**关键事实 2**：任何属性 `attr/z` 在屏幕空间也是线性的。

> 这两个事实可以通过对透视投影矩阵展开 + 透视除法推导得到，本文聚焦应用，推导细节略（结论是图形学的标准结果）。

利用上述事实，**正确的属性插值**应这样做：

```
设 V₀, V₁, V₂ 在相机空间的 z 值为 z₀, z₁, z₂，
屏幕空间得到的重心坐标为 (α, β, γ)。

Step 1: 对 1/z 线性插值（这一步是合法的）
        1/z_P = α·(1/z₀) + β·(1/z₁) + γ·(1/z₂)

Step 2: 对 attr/z 线性插值（也合法）
        attr_P/z_P = α·(attr₀/z₀) + β·(attr₁/z₁) + γ·(attr₂/z₂)

Step 3: 还原
        attr_P = (attr_P/z_P) / (1/z_P)
               = [α·(attr₀/z₀) + β·(attr₁/z₁) + γ·(attr₂/z₂)] / [α·(1/z₀) + β·(1/z₁) + γ·(1/z₂)]
```

写成更常用的形式（用裁剪空间的 `w` 替代 `z`，因为 `w_clip = z_view`）：

```
                α·attr₀/w₀ + β·attr₁/w₁ + γ·attr₂/w₂
   attr_P  =  ─────────────────────────────────────
                α/w₀      + β/w₁       + γ/w₂
```

> **GLSL 中如何获得这种"透视正确"行为？**
> 默认的 `varying` / `in/out` 变量已经是**透视正确插值**——GPU 硬件帮你做了。
> 只有显式声明 `noperspective` 修饰符时才是普通线性插值。
>
> ```glsl
> in vec2 vUV;             // 默认：透视正确插值 ✓
> noperspective in vec2 vUV2;  // 普通线性插值（很少用）
> ```

## 7. 纹理采样：从 UV 到颜色

光栅化阶段为每个像素插值出了 `(u, v)`。下一步：**把 UV 翻译成颜色**。

### 7.1 UV → 像素坐标

```
设贴图分辨率 W × H：
   x = u · W
   y = v · H

注意 x, y 通常是浮点数（不是整数），例如 (256.7, 511.3)
   ⇒ 落在 4 个纹素之间，怎么办？
```

这就需要**过滤模式**（Filter Mode）。

### 7.2 最近邻采样（Nearest）

最简单：**四舍五入到最近的整数纹素**。

```glsl
ivec2 ij = ivec2(floor(vec2(x, y) + 0.5));
color = texelFetch(tex, ij, 0);
```

```
特点：
  ✓ 速度最快（1 次访存）
  ✓ 像素艺术风格保留锐利块状
  ✗ 放大时锯齿严重
  ✗ 移动镜头时会"跳动"
```

### 7.3 双线性采样（Bilinear，带推导）

**思想**：取最近的 4 个纹素，按距离加权混合。

设浮点纹素坐标 `(x, y)` 落在 4 个整数纹素 `T00, T10, T01, T11` 围成的方块内：

```
     T01 ●─────────● T11
         │    ▴    │
         │    │ 1-fy
         │ ●(x, y) │     fx = x - floor(x)
         │    │ fy │     fy = y - floor(y)
         │    ▾    │
     T00 ●─────────● T10
         ←── 1-fx──→← fx →
```

**Step 1**：先沿 x 方向插值两次

```
   A = (1-fx) · T00 + fx · T10      （下边水平插值）
   B = (1-fx) · T01 + fx · T11      （上边水平插值）
```

**Step 2**：再沿 y 方向插值一次

```
   color = (1-fy) · A + fy · B
```

**展开成最终公式**：

```
   color = (1-fx)(1-fy)·T00
         +    fx (1-fy)·T10
         + (1-fx)   fy ·T01
         +    fx    fy ·T11
```

```
特点：
  ✓ 平滑（无块状）
  ✓ GPU 硬件加速，单次采样代价就 1 次
  ✗ 放大太多会发糊
```

> 现代 GPU 默认开启双线性，调用 `texture(tex, uv)` 自动完成。

## 8. Mipmap：远处贴图为什么不闪烁

### 8.1 走样问题

考虑一个极端场景：一张 1024×1024 的棋盘格贴图，贴在远处的地面上。

```
此时屏幕上的 1 个像素，可能覆盖了贴图的 100×100 个纹素。
如果只采 1 个纹素，结果几乎是随机的——
镜头稍微一动，采到的纹素从 (500, 500) 跳到 (517, 503)，
颜色完全变了 ⇒ 屏幕闪烁！
```

这就是**走样（Aliasing）**——本质是采样率低于信号频率（违反奈奎斯特定理）。

### 8.2 Mipmap 的构造与选层

**核心思想**：预先把贴图缩小成一系列层级，远处用小的、近处用大的。

```
Level 0: 1024×1024  （原图）
Level 1:  512× 512  （下采样）
Level 2:  256× 256
Level 3:  128× 128
...
Level 10:   1×   1

每层是上一层的 2×2 平均
内存增加：1 + 1/4 + 1/16 + ... ≈ 4/3 倍（即多了 33%）
```

```
   Lv0 ┌─────────────────┐
       │  原图 1024²     │
       └─────────────────┘
   Lv1 ┌────────┐
       │  512²  │
       └────────┘
   Lv2 ┌────┐
       │256²│
       └────┘
   ...
```

**怎么选层级？**

```
GPU 会算"屏幕上 1 像素跨越了贴图多少纹素"——
通过偏导数 ddx/ddy（相邻像素的 UV 差异）：

   ρ = max( length(∂uv/∂x · W), length(∂uv/∂y · H) )
   level = log₂(ρ)

ρ ≈ 1   ⇒  level=0    （1 像素 = 1 纹素，用原图）
ρ ≈ 4   ⇒  level=2    （1 像素 = 4 纹素，用 256²）
ρ ≈ 16  ⇒  level=4    （1 像素 = 16 纹素，用 64²）
```

### 8.3 三线性与各向异性过滤

**三线性过滤（Trilinear）**：在双线性基础上，再在两个相邻层级之间插值。

```
   level 算出来 = 2.7
   ⇒ 在 Level 2 和 Level 3 各做一次双线性采样
   ⇒ 然后按 0.7 比例混合：
       color = 0.3 · sample_lv2 + 0.7 · sample_lv3
```

代价：8 次纹素读取 vs 双线性的 4 次。

**各向异性过滤（Anisotropic）**：当贴图被斜着看时（如地面延伸到远方），双线性/三线性还是会糊。

```
问题：地面斜视角下，1 个屏幕像素覆盖的纹素区域是"长条形"的。
      但 mipmap 假设各方向一致，会按"最长边"选层 ⇒ 选了过小的层 ⇒ 模糊。

解决：沿"长条方向"采样多次（2x / 4x / 8x / 16x AF）。
      游戏画面"AF×16" 选项就是它。
```

## 9. 边界处理：UV 越界了怎么办

UV 不一定都在 `[0, 1]`——美术可以故意用 `(0, 0) → (3, 3)` 表示"重复 9 次"。GPU 通过**Wrap Mode**（包裹模式）处理边界：

| 模式 | 英文 | 行为 | 效果（u 方向举例） |
|------|------|------|------|
| 重复 | `REPEAT` | 取小数部分 | `1.3 → 0.3`（无限平铺，最常用于地面、墙砖） |
| 镜像重复 | `MIRRORED_REPEAT` | 偶数次正常，奇数次翻转 | `1.3 → 0.7`（消除接缝感） |
| 边缘钳制 | `CLAMP_TO_EDGE` | 超出就用边缘像素 | `1.3 → 1.0`（防止边缘渗色） |
| 边框色 | `CLAMP_TO_BORDER` | 超出返回固定颜色 | `1.3 → border_color`（特殊用途） |

```glsl
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_REPEAT);  // S = u
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_REPEAT);  // T = v
```

## 10. 端到端示例：把"木纹.png"贴到立方体上

### 10.1 数据准备

**立方体顶点**（部分，仅展示前两个面）：

```cpp
Vertex cube[] = {
    // Front 面（z = +0.5）
    //   position             normal             uv
    {{-0.5,-0.5, 0.5}, { 0, 0, 1}, {0.0, 0.0}},  // 左下
    {{ 0.5,-0.5, 0.5}, { 0, 0, 1}, {1.0, 0.0}},  // 右下
    {{ 0.5, 0.5, 0.5}, { 0, 0, 1}, {1.0, 1.0}},  // 右上
    {{-0.5, 0.5, 0.5}, { 0, 0, 1}, {0.0, 1.0}},  // 左上

    // Back 面（z = -0.5）
    {{ 0.5,-0.5,-0.5}, { 0, 0,-1}, {0.0, 0.0}},
    // ...（共 24 个顶点，每面 4 个，6 面）
};

uint32_t indices[] = {
    0,1,2,  0,2,3,    // Front
    4,5,6,  4,6,7,    // Back
    // ...（共 36 个索引，12 个三角形）
};
```

**加载贴图**：

```cpp
// 1. 用 stb_image 把 PNG 解码成像素数组
int W, H, channels;
uint8_t* pixels = stbi_load("wood.png", &W, &H, &channels, 4);

// 2. 上传到 GPU
GLuint tex;
glGenTextures(1, &tex);
glBindTexture(GL_TEXTURE_2D, tex);
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, W, H, 0,
             GL_RGBA, GL_UNSIGNED_BYTE, pixels);

// 3. 生成 mipmap 链
glGenerateMipmap(GL_TEXTURE_2D);

// 4. 配置过滤 + wrap
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR_MIPMAP_LINEAR);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_REPEAT);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_REPEAT);
```

### 10.2 GPU 流水线全过程

每帧渲染时：

```
CPU 端：
  ① 计算 MVP = P · V · M
  ② 绑定贴图到 unit 0：glActiveTexture(GL_TEXTURE0); glBindTexture(...)
  ③ 设置 uniform：uMVP, uTex(=0)
  ④ 提交 draw call：glDrawElements(GL_TRIANGLES, 36, ...)

GPU 端（每个三角形并行）：
  ⑤ 顶点着色器（×24 个顶点）
       gl_Position = uMVP * vec4(v.position, 1.0);
       vUV = v.uv;                           ← 透传
  ⑥ 图元装配 + 裁剪剔除
  ⑦ 光栅化（×百万级像素）
       对每个像素，由硬件计算重心坐标 (α,β,γ)
       插值 vUV = α·uv₀ + β·uv₁ + γ·uv₂      ← 透视正确插值
  ⑧ 片元着色器（×百万级像素并行）
       FragColor = texture(uTex, vUV);       ← Mipmap 选层 + 三线性 + AF
  ⑨ 深度测试 + 写 framebuffer

最终：
  ⑩ 显示器扫描 framebuffer ⇒ 你看到一个贴着木纹的立方体 ✓
```

### 10.3 完整 GLSL 着色器代码

**顶点着色器**：

```glsl
#version 450

layout(location = 0) in vec3 aPosition;
layout(location = 1) in vec3 aNormal;
layout(location = 2) in vec2 aUV;

uniform mat4 uMVP;     // CPU 预乘好的 MVP
uniform mat4 uModel;   // 模型矩阵（用于法线变换）

out vec2 vUV;
out vec3 vWorldPos;
out vec3 vWorldNormal;

void main() {
    gl_Position = uMVP * vec4(aPosition, 1.0);
    vUV = aUV;                                  // 透传（GPU 自动透视正确插值）
    vWorldPos = (uModel * vec4(aPosition, 1.0)).xyz;
    vWorldNormal = mat3(transpose(inverse(uModel))) * aNormal;
}
```

**片元着色器**：

```glsl
#version 450

in vec2 vUV;
in vec3 vWorldPos;
in vec3 vWorldNormal;

uniform sampler2D uTex;      // 木纹贴图
uniform vec3 uLightDir;      // 平行光方向

out vec4 FragColor;

void main() {
    // 1. 采样贴图（自动 mipmap + 双线性 + 各向异性）
    vec3 albedo = texture(uTex, vUV).rgb;

    // 2. 简单 Lambert 光照
    vec3 N = normalize(vWorldNormal);
    vec3 L = normalize(-uLightDir);
    float diff = max(dot(N, L), 0.0);

    // 3. 输出最终颜色
    FragColor = vec4(albedo * (diff + 0.2), 1.0);
}
```

> 短短两个着色器 + 几行 CPU 代码，就完成了完整的"图像贴立体"流程。

## 11. 进阶：PBR 多贴图协作

现代游戏一个材质往往用 **4~7 张贴图**，**全部共享同一套 UV**，分别承担不同物理意义：

| 贴图 | 含义 | 通道用途 |
|------|------|----------|
| **Albedo / BaseColor** | 物体本色（去掉光影） | RGB |
| **Normal Map** | 切线空间下的法线偏移 | RGB → 法线 (x,y,z) |
| **Roughness** | 表面粗糙度 0~1 | 单通道 |
| **Metallic** | 金属度 0~1 | 单通道 |
| **AO (Ambient Occlusion)** | 环境光遮蔽 | 单通道 |
| **Emissive** | 自发光颜色 | RGB |
| **Height / Displacement** | 高度图（视差/位移） | 单通道 |

> 优化技巧：把 Roughness / Metallic / AO 三个单通道合到一张 RGB 贴图（俗称 "ORM" / "MRA"），节省采样次数。

**典型 PBR 片元着色器骨架**：

```glsl
vec3 albedo    = texture(uAlbedo,    vUV).rgb;
vec3 nMap      = texture(uNormal,    vUV).rgb * 2.0 - 1.0;
float rough    = texture(uRoughness, vUV).r;
float metallic = texture(uMetallic,  vUV).r;
float ao       = texture(uAO,        vUV).r;

// 把切线空间法线变换到世界空间（TBN 矩阵）
vec3 N = normalize(TBN * nMap);

// PBR 光照计算（Cook-Torrance + GGX + Fresnel）...
```

**为什么法线贴图能"假装凹凸"？**
法线只决定光照计算时表面"朝向"——把每个像素的法线替换成贴图里的，光照立刻表现出凹凸感，但**几何依然是平的**（侧视看会穿帮，所以才有视差贴图、位移贴图等改进）。

## 12. 常见问题答疑（FAQ）

**Q1：UV 一定要在 `[0, 1]` 吗？**
不必。`(2.5, -0.3)` 都合法，根据 wrap 模式处理（重复/镜像/钳制）。

**Q2：两个 3D 点重合，UV 一定相同吗？**
**不一定！** 在接缝处（如立方体的棱），同一个 3D 位置往往**复制成多个顶点**，每个顶点对应不同 UV——这就是为什么立方体只有 8 个角，但 GPU 端要存 24 个顶点。

**Q3：贴图为什么有时候上下颠倒？**
OpenGL 默认 (0,0) 在左下，DirectX 默认 (0,0) 在左上。导入图片时常常需要 `stbi_set_flip_vertically_on_load(true)` 翻转一次。

**Q4：贴图为什么要求边长是 2 的幂（POT）？**
历史原因：旧 GPU 计算 mipmap 和过滤时要求 `W = 2^k`。现代 GPU（GL ≥ 3.0、所有 D3D11+）已没有这个限制，但 2 的幂仍然有更好的内存对齐和缓存命中率。

**Q5：UV 重叠会怎样？**
两个 3D 三角形映射到贴图同一区域——它们会"共享"同一块图案。常用于左右对称模型（左右脸用同一块脸贴图，节省一半空间）。

**Q6：采样到的 UV 出现"螺旋纹"或"撕裂"是为什么？**
通常是因为**接缝处 UV 不连续 + 双线性采样跨接缝**——采样器跨过了接缝边界，把不该混在一起的两块图案混在了一起。解决：让接缝两侧贴图边缘颜色相近、留 padding（"UV 岛屿之间隔几像素的安全带"）。

**Q7：能不能完全跳过 UV，让 GPU 自动算？**
有"三平面投影（Triplanar Mapping）"等技巧，按法线方向选择投影方向、用世界空间坐标当 UV——适合地形、岩石等无需精确 UV 的场景。但损失了美术控制力，且采样代价 3 倍。

**Q8：GPU 怎么知道相邻像素的 UV 差异（用于选 mipmap 层）？**
GPU 不是孤立处理一个像素，而是以 **2×2 像素块（Quad）** 为最小单位调度——块内 4 个像素的 UV 一比较，差分就出来了。这也是为什么 `discard` 掉的像素仍占 quad 资源。

```
   离线（美术）                       运行时（GPU）
   ─────────────                      ──────────────
   ① 建模 + 展 UV                    ⑥ 加载到显存 (VBO + Texture)
       └── 给每顶点指定 (u,v)              │
   ② 绘制贴图 (PNG)                       ▼
   ③ 导出 .glTF                       ⑦ Draw Call 提交
                                            │
                                            ▼
                              ┌───────────────────────────────────┐
                              │ VS：MVP 变换 + 透传 UV            │
                              │  ↓                                │
                              │ 光栅化：重心坐标 + 透视正确插值     │
                              │  ↓                                │
                              │ 每像素得到精确的 UV               │
                              │  ↓                                │
                              │ FS：texture(tex, uv)              │
                              │   └── UV→纹素坐标                 │
                              │   └── Mipmap 选层（ddx/ddy）      │
                              │   └── 三线性 + 各向异性过滤        │
                              │  ↓                                │
                              │ 光照 + 深度测试 + 写 framebuffer  │
                              └───────────────────────────────────┘
                                            │
                                            ▼
                                       ⑧ 屏幕显示 ✓
```

**贴图的本质再总结一次**：
1. **数据上**：给每个 3D 顶点附加一对 `(u, v)`，把 3D 三角形和 2D 贴图区域一一映射；
2. **几何上**：用**重心坐标**把"3 个顶点的 UV"插值到三角形内每个像素（透视正确）；
3. **采样上**：用**双线性 + Mipmap + 各向异性**，从贴图二维数组里读出抗走样的颜色；
4. **工程上**：UV 在离线由美术指定，运行时全程在 GPU 流水线里高度并行完成。

至此，你应该对"一张 PNG 怎么变成 3D 模型表面那块木纹"的全过程有了从直觉、数学、到代码的完整理解。🎉

## 附录 A：光栅化（Rasterization）名词详解

> 正文里我们多次用到"光栅化"这个词（例如 §5、§6、§10 的流水线图），但一直没停下来讲它是什么、名字从哪来、为什么这么叫。这一节把它彻底讲清楚。

### A.1 一句话定义

> **光栅化（Rasterization）**：把用**顶点**描述的**几何图元**（三角形、线段、点），转换成屏幕上一个个**离散像素**的过程。

用程序员熟悉的话说：

- **输入**：3 个顶点（含屏幕坐标 + UV + 颜色 + 法线 + …）；
- **输出**：一堆"我这三角形覆盖了哪些像素"，以及每个像素上插值好的属性（UV、深度、颜色…）；
- **执行者**：GPU 里的**光栅化器（Rasterizer）**——一块专用的固定功能硬件单元。

一句话再抽象一层：**把连续的几何 → 变成离散的像素网格**。

### A.2 词源与来历："raster" 到底是什么意思

"光栅"这个中文名初次读会觉得别扭，其实是从英文 **raster** 直译过来的。

#### A.2.1 词源追溯

- **拉丁语 `rastrum`** = "耙子（raking tool，农具）"，形状是一排平行齿；
- **德语 `Raster`** = "格栅 / 网格"（保留了拉丁词根"平行线"的含义）；
- **英语 `raster`** 从德语借入，指"平行扫描线的图案"。

一句话：`raster` 的原始意象就是**"一排一排平行的线条 / 格子"**。

#### A.2.2 从 CRT 电视到显示器：raster 是怎么进图形学的

CRT（阴极射线管）电视的成像原理正好是这样的：

```
   电子束
     │
     ▼
   ┌──────────────────────────┐
   │ ── ── ── ── ── ── ── ─→ │  ← 第 1 行：从左扫到右
   │←── ── ── ── ── ── ── ── │  ← 回到最左，往下一行
   │ ── ── ── ── ── ── ── ─→ │  ← 第 2 行
   │      ...  逐行扫描  ...   │
   │ ── ── ── ── ── ── ── ─→ │  ← 第 N 行
   └──────────────────────────┘
   一屏 = 一整套水平平行的扫描线 = raster
```

- 电子束**逐行**扫过荧光屏，像"耙子"一样一行一行地耙 → 这套扫描图案就叫 **raster scan（光栅扫描）**；
- 由此产生的图像叫 **raster image（光栅图像）**——本质就是"由一行行像素组成的位图"。

对应到今天：

| 名词 | 中文 | 含义 |
|------|------|------|
| Raster image | 光栅图像 / 位图 | 由像素网格构成的图像（PNG、JPG、BMP…） |
| Vector image | 矢量图 | 由数学公式描述的图像（SVG、字体轮廓…） |
| **Rasterization** | **光栅化** | **矢量 → 光栅（把几何变成像素网格）** |

**"光栅化"三个字拆开看**：
- **光**：来自显示器发光的历史印象（电子束 → 荧光）；
- **栅**：格栅、网格，也就是"像素网格"；
- **化**：转化 / 变成。

合起来：**"把东西变成像素网格"**——这就是这个中文名的字面含义。

> 💡 如果当年翻译时叫"像素化"或"网格化"，含义其实一样，只是"光栅化"是从德/英直译，历史沿用下来了。**在图形学语境里，"光栅化"就等于"变成像素"**，不用想复杂。

### A.3 为什么图形学要"光栅化"：矢量世界 → 像素世界

3D 图形学里，我们描述物体的方式是**矢量的**：

- 顶点：`(x, y, z)` 是**连续**的浮点数；
- 三角形：由 3 个顶点定义，理论上是**连续**的一片平面；
- 相机变换、投影 → 得到的仍然是**连续**的屏幕坐标。

但显示器是**离散**的：

- 一块 1920×1080 的屏幕就是 1920×1080 个"格子"（像素），每个格子只能显示一种颜色；
- 屏幕**不认识**"三角形"，只认识"这个位置的这个像素涂什么颜色"。

**中间的鸿沟**由光栅化来跨越：

```
  连续的几何世界             离散的像素世界
  ─────────────              ─────────────
   ▲                              ┌──┬──┬──┬──┐
   │  三角形（一大片平面）           ├──┼──┼──┼──┤
   │       ╱╲                      ├──┼██┼██┼──┤
   │      ╱  ╲       光栅化         ├──┼██┼██┼██┤
   │     ╱    ╲    ─────────►      ├──┼██┼██┼██┤
   │    ╱______╲                    └──┴──┴──┴──┘
   │                              （被三角形覆盖的像素被点亮）
   └──────────────►
```

**光栅化的核心工作**就两件事：

1. **覆盖测试（Coverage）**：屏幕上哪些像素被这个三角形盖住了？
2. **属性插值（Interpolation）**：对每个被覆盖的像素，用重心坐标把 3 个顶点的属性（UV、深度、颜色…）插值到该像素上。

正文 §5 讲的**重心坐标**、§6 讲的**透视正确插值**，全部就是"光栅化的第 2 步"要用到的数学。

### A.4 GPU 里的光栅化到底做了什么

放大看，GPU 的光栅化阶段大致这样运作：

```
输入（一个三角形，来自 VS 输出）：
  V₀ = (x₀, y₀, z₀, 1/w₀,   uv₀,   color₀, ...)
  V₁ = (x₁, y₁, z₁, 1/w₁,   uv₁,   color₁, ...)
  V₂ = (x₂, y₂, z₂, 1/w₂,   uv₂,   color₂, ...)
                                 │
                                 ▼
        ┌──────────────────────────────────────┐
        │ ① 计算包围盒（bounding box）         │  只在这个矩形里循环，避免全屏扫
        │ ② 逐像素做覆盖测试（in / out）       │  用叉积或 edge function
        │ ③ 若覆盖 → 计算重心坐标 (α, β, γ)    │  §5.2 的公式
        │ ④ 用 (α, β, γ) 做透视正确属性插值    │  §6.2 的公式
        │ ⑤ 生成 fragment（"候选像素"）        │  一个 fragment ≈ 一个待处理像素
        └──────────────────────────────────────┘
                                 │
                                 ▼
输出：一大堆 fragment，每个带有插值好的 (uv, z, color, ...)
       → 送给片段着色器（FS）
```

几个关键术语顺便讲清楚：

| 术语 | 中文 | 含义 |
|------|------|------|
| Fragment | 片元 / 片段 | 光栅化输出的"候选像素"。一个 fragment 未必成为最终像素（可能被深度测试丢弃）。 |
| Rasterizer | 光栅化器 | GPU 里执行光栅化的固定功能硬件（不可编程）。 |
| Fragment Shader | 片段着色器 | 光栅化器把 fragment 交给它来算最终颜色。 |
| Edge function | 边函数 | 判断"点在三角形内 / 外"的 O(1) 数学工具，是覆盖测试的主流实现。 |

> 💡 一个常见的困惑：**片段（fragment）和像素（pixel）是一回事吗？**
> 不完全一样。**fragment 是"候选像素"**，光栅化产生 fragment，再经过深度测试、模板测试、混合等阶段，**幸存下来**的 fragment 才写进 framebuffer 变成"最终像素"。可以粗略理解成："fragment 是像素的胚胎"。

### A.5 一段最小可读伪代码

把 A.4 的流程写成程序员能一眼看懂的伪代码：

```cpp
// 光栅化一个三角形（简化版，忽略透视除法、抗锯齿等细节）
void rasterizeTriangle(Vertex v0, Vertex v1, Vertex v2, FrameBuffer& fb) {
    // ① 计算屏幕空间的包围盒（只扫这个矩形，不扫全屏）
    int xMin = floor(min({v0.x, v1.x, v2.x}));
    int xMax = ceil (max({v0.x, v1.x, v2.x}));
    int yMin = floor(min({v0.y, v1.y, v2.y}));
    int yMax = ceil (max({v0.y, v1.y, v2.y}));

    // ② 遍历包围盒里的每个像素
    for (int y = yMin; y <= yMax; ++y) {
        for (int x = xMin; x <= xMax; ++x) {
            // ③ 计算像素中心 (x+0.5, y+0.5) 的重心坐标 (α, β, γ)
            //    ——公式见正文 §5.2
            auto [alpha, beta, gamma] = barycentric(x + 0.5f, y + 0.5f, v0, v1, v2);

            // ④ 覆盖测试：三个重心坐标都非负 ⇒ 点在三角形内
            if (alpha < 0 || beta < 0 || gamma < 0) continue;

            // ⑤ 属性插值（透视正确公式见正文 §6.2）
            float u = perspectiveCorrect(alpha, beta, gamma,
                                         v0.u, v1.u, v2.u,
                                         v0.w, v1.w, v2.w);
            float v = perspectiveCorrect(alpha, beta, gamma,
                                         v0.v, v1.v, v2.v,
                                         v0.w, v1.w, v2.w);
            float z = alpha*v0.z + beta*v1.z + gamma*v2.z;

            // ⑥ 生成 fragment，交给片段着色器 + 深度测试
            Fragment frag { x, y, u, v, z, ... };
            fragmentShaderAndDepthTest(frag, fb);
        }
    }
}
```

一眼可以对应上：

- 第 ① 步 = "只在有可能被三角形覆盖的矩形范围里循环"，节省算力；
- 第 ③④ 步 = 正文 §5 的**重心坐标 + 覆盖判定**；
- 第 ⑤ 步 = 正文 §6 的**透视正确插值**；
- 第 ⑥ 步 = 把结果打包成 fragment 送去片段着色器（正文 §7 的纹理采样就发生在这里）。

**光栅化 = 双重 for 循环 + 重心坐标 + 插值**，就是这么简单直白。真实 GPU 只是把这个循环用海量并行硬件跑得非常快。

### A.6 光栅化 vs 光线追踪：对比理解

图形学里另一个大名鼎鼎的名词是**光线追踪（Ray Tracing）**。放在一起对比，光栅化的含义会更清晰：

| 维度 | 光栅化（Rasterization） | 光线追踪（Ray Tracing） |
|------|------------------------|------------------------|
| **核心问题** | "**这个三角形**盖住了哪些像素？" | "**这个像素**看到了哪个三角形？" |
| **循环方向** | 遍历三角形 → 找像素 | 遍历像素 → 找三角形 |
| **优势** | 极快，硬件流水线成熟，实时游戏首选 | 天然支持反射/折射/阴影，画质好 |
| **代价** | 反射、折射、软阴影都要靠 hack | 计算量大，实时化难 |
| **典型应用** | 所有 3D 游戏、CAD、VR | 电影 CG、离线渲染、部分现代游戏（RTX） |

**记忆口诀**：
- **光栅化 = 从物体出发**，"三角形往屏幕上盖章"；
- **光线追踪 = 从眼睛出发**，"每个像素射出一根探测射线"。

本文全篇讨论的"贴图"流程是**基于光栅化的传统 GPU 流水线**——这是目前 99% 实时渲染的做法。

### A.7 程序员常见误解澄清

**误解 1：光栅化 = 抗锯齿？**
> ❌ 不是。光栅化是"变成像素"这个动作本身；抗锯齿（AA）是**改善光栅化产生的锯齿边缘**的一系列后续技术（MSAA、FXAA、TAA…）。

**误解 2：光栅化 = 画三角形边线？**
> ❌ 不是。光栅化默认是**填充三角形内部**（每个内部像素都生成 fragment）。只画边线是另一种模式，叫 line rasterization / wireframe。

**误解 3：CPU 也能做光栅化吗？**
> ✅ 完全可以。早期没有 GPU 时，游戏（如《雷神之锤》软件模式）全是 CPU 光栅化。今天的 CPU 软光栅（如 Mesa 的 llvmpipe）也能跑，只是慢得多。**光栅化是算法，不是硬件**——只是现代 GPU 把它硬件化了。

**误解 4：光栅化和着色器（Shader）是一回事？**
> ❌ 不是，正好相反：**光栅化器是不可编程的固定功能硬件**，它夹在**顶点着色器（VS）**和**片段着色器（FS）**中间——VS 输出的三角形交给光栅化器，光栅化器产出 fragment 再交给 FS。三者是流水线上的不同阶段。

```
   VS（可编程）  →  光栅化器（固定功能）  →  FS（可编程）
   处理顶点          产生 fragment           计算像素颜色
```

**误解 5：光栅化只处理三角形？**
> 大体上是。现代 GPU 的光栅化针对三角形做了极致优化，**线段和点**也能被光栅化，但都是"退化"用法。多边形（四边形以上）会先被拆成三角形再交给光栅化器——这就是正文 §4 里"正方形要拆成 2 个三角形"的原因。

---

**一句话总结附录 A**：**光栅化 = "把矢量几何变成像素网格" 的算法/硬件阶段**，名字来自 CRT 时代的"光栅扫描"，本质就是"逐像素做覆盖测试 + 属性插值"，正文里所有 UV、颜色、深度的"每像素值"都是它插出来的。
