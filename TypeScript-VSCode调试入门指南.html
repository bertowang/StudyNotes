工程一：prj_1 — Node.js 命令行输出

功能说明

TypeScript 源码 → 编译为 JavaScript → Node.js 执行 → 输出字符串到 VSCode 调试控制台。

完整执行流程

src/index.ts          ← 你编写的 TypeScript 源码
      │
      │  tsc 编译（根据 tsconfig.json）
      ▼
dist/index.js         ← 编译生成的 JavaScript 文件
      │
      │  Node.js 执行
      ▼
console.log(...)      ← 输出字符串
      │
      ▼
VSCode 调试控制台      ← 你看到结果的地方

源码文件

src/utils/helper.ts — 工具函数模块：

export function greet(name: string): string {
  return `Hello, ${name}!`;
}

export const PI: number = 3.14159;

src/index.ts — 主入口文件：

// 导入其他模块（注意：编译后需要 .js 扩展名）
import { greet } from './utils/helper.js';

// 定义接口（TypeScript 特有的类型系统）
interface User {
  name: string;
  age: number;
}

// 使用类型
const user: User = {
  name: '张三',
  age: 25
};

// 调用函数，console.log 输出到终端
console.log(greet(user.name));

// 类定义
class Calculator {
  add(a: number, b: number): number {
    return a + b;
  }
}

const calc = new Calculator();
console.log(calc.add(5, 3));

工程二：prj_2 — 浏览器网页输出

功能说明

TypeScript 源码 → 编译为 JavaScript → http-server 托管 → 浏览器加载 → 网页显示输出内容。

⚡ 与 prj_1 的核心区别：谁来"运行" JS？

这是理解两个工程最关键的问题。

prj_1：Node.js 直接运行 JS

dist/index.js
      │
      │  Node.js 进程直接执行（就像运行一个程序）
      ▼
console.log() 输出到终端
      │
      ▼
程序执行完毕，进程退出
  • Node.js 是一个 JS 运行时，可以直接在操作系统上执行 .js 文件，就像 Python 执行 .py 文件一样。
  • 程序运行一次就结束，没有持续监听。
  • 输出目标是终端（标准输出 stdout）。

prj_2：http-server 提供文件，浏览器运行 JS

dist/index.js
dist/index.html
      │
      │  http-server 进程持续运行，监听 8080 端口
      │  （http-server 本身是 Node.js 写的，但它的职责只是"传文件"）
      ▼
浏览器发起 HTTP 请求：GET http://localhost:8080/index.html
      │
      ▼
http-server 把 index.html 文件内容原样返回给浏览器
      │
      ▼
浏览器解析 HTML，遇到 <script src="./index.js">
      │
      ▼
浏览器再次发起请求：GET http://localhost:8080/index.js
      │
      ▼
http-server 把 index.js 文件内容原样返回给浏览器
      │
      ▼
浏览器内置的 JS 引擎（V8）执行 index.js
      │
      ▼
操作 DOM，网页显示内容
  • http-server 本质上只是一个文件传输工具，它把 dist/ 目录下的文件通过 HTTP 协议"送"给浏览器，自己并不执行 JS。
  • 真正执行 JS 的是浏览器（Chrome 内置的 V8 引擎），不是 Node.js。
  • http-server 持续运行，每次浏览器请求都响应，不会自动退出。
  • 输出目标是浏览器页面（DOM）。

关键对比表

对比项 prj_1 prj_2
谁执行 JS Node.js 直接执行 浏览器（V8 引擎）执行
Node.js 的角色 JS 运行时，直接运行程序 运行 http-server 工具（间接参与）
http-server 的角色 不涉及 文件传输工具，把 JS/HTML 文件送给浏览器
JS 可用的 API Node.js API（fs、path、http 等） 浏览器 API（document、window、fetch 等）
输出方式 console.log() → 终端 DOM 操作 → 网页
进程生命周期 运行完毕自动退出 http-server 持续监听，不退出

一句话总结：

  • prj_1：Node.js 是演员，直接上台表演（执行 JS）。
  • prj_2：http-server 是快递员（送文件），浏览器才是演员（执行 JS）。 http-server 虽然底层也是 Node.js 写的，但在 prj_2 中它只负责传文件，不负责运行你的业务代码。

完整执行流程

src/index.ts          ← 你编写的 TypeScript 源码
      │
      │  tsc 编译（根据 tsconfig.json）
      ▼
dist/index.js         ← 编译生成的 JavaScript 文件
      │
dist/index.html       ← HTML 页面（手动创建，引用 index.js）
      │
      │  http-server 托管 dist/ 目录
      ▼
http://localhost:8080  ← 本地 HTTP 服务
      │
      │  Chrome 访问
      ▼
浏览器加载 index.html
      │
      │  <script type="module" src="./index.js"> 加载 JS
      ▼
JS 执行：调用 TS 函数，操作 DOM
      │
      ▼
网页显示输出内容        ← 你在 Chrome 中看到的结果

关键说明：为什么需要 http-server？

浏览器的安全策略（CORS）禁止通过 file:// 协议加载 ES6 模块（<script type="module">）。 必须通过 http:// 协议访问，所以需要 http-server 提供本地 HTTP 服务。

源码文件

src/utils/helper.ts — 工具函数模块：

// 问候函数：接收名字，返回问候语
export function greet(name: string): string {
    return `你好，${name}！欢迎使用 TypeScript！`;
}

// 计算两数之和
export function add(a: number, b: number): number {
    return a + b;
}

// 获取当前时间字符串
export function getCurrentTime(): string {
    const now = new Date();
    return now.toLocaleString("zh-CN");
}

src/index.ts — 主入口文件：

import { greet, add, getCurrentTime } from "./utils/helper.js";

// 将内容渲染到页面上
function renderToPage(): void {
    const app = document.getElementById("app");
    if (!app) return;

    // 调用 TypeScript 函数，获取输出结果
    const greeting = greet("World");
    const sum = add(3, 5);
    const time = getCurrentTime();

    // 通过 DOM 操作将结果写入 HTML 页面
    app.innerHTML = `
        <h1>🎉 TypeScript → Chrome 输出 Demo</h1>
        <div class="card">
            <h2>📢 问候语</h2>
            <p>${greeting}</p>
        </div>
        <div class="card">
            <h2>🔢 计算结果</h2>
            <p>3 + 5 = <strong>${sum}</strong></p>
        </div>
        <div class="card">
            <h2>🕐 当前时间</h2>
            <p>${time}</p>
        </div>
    `;
}

// 页面加载完成后执行
window.addEventListener("DOMContentLoaded", renderToPage);

dist/index.html — 浏览器入口页面（放在 dist 目录）：

<!DOCTYPE html>
<html lang="zh-CN">
<head>
    <meta charset="UTF-8">
    <title>TypeScript Demo - prj_2</title>
</head>
<body>
    <div id="app">
        <p>加载中...</p>
    </div>
    <!-- type="module" 支持 ES6 模块导入 -->
    <script type="module" src="./index.js"></script>
</body>
</html>

VSCode 配置文件详解

.vscode/launch.json — 调试启动配置

作用：定义按下 F5 后的调试行为。

{
    "version": "0.2.0",
    "configurations": [
        {
            // ── prj_2：Chrome 调试配置 ──
            "type": "chrome",                          // 调试器类型：Chrome 浏览器
            "request": "launch",                       // 启动模式
            "name": "运行 TypeScript (Chrome) - prj_2",// 显示在下拉框中的名称
            "preLaunchTask": "启动 prj_2",             // F5 前先执行的任务（编译+启动服务器）
            "url": "http://localhost:8080",            // Chrome 打开的地址
            "webRoot": "${workspaceFolder}/prj_2/dist",// 网页根目录（用于 sourceMap 映射）
            "sourceMaps": true,                        // 开启 sourceMap（支持在 .ts 文件打断点）
            "sourceMapPathOverrides": {
                "../src/*": "${workspaceFolder}/prj_2/src/*"
            }
        },
        {
            // ── prj_1：Node.js 调试配置 ──
            "type": "node",                            // 调试器类型：Node.js
            "request": "launch",
            "name": "运行 TypeScript (Node.js) - prj_1",
            "preLaunchTask": "tsc: build - prj_1/tsconfig.json", // F5 前先编译
            "program": "${workspaceFolder}/prj_1/dist/index.js", // 运行的 JS 文件
            "outFiles": ["${workspaceFolder}/prj_1/dist/**/*.js"],
            "sourceMaps": true,
            "skipFiles": ["<node_internals>/**"]
        }
    ]
}

重要规则：

  • configurations 数组中第一个配置是 F5 的默认配置
  • 切换工程时，必须在调试面板顶部下拉框手动选择，VSCode 会记住上次选择

.vscode/tasks.json — 构建任务配置

作用：定义可执行的构建任务，被 launch.json 的 preLaunchTask 引用。

{
    "version": "2.0.0",
    "tasks": [
        {
            // ── prj_1 编译任务 ──
            "type": "shell",
            "command": "npx tsc -p tsconfig.json",     // 执行的命令
            "options": {
                "cwd": "${workspaceFolder}/prj_1"      // 在 prj_1 目录下执行
            },
            "label": "tsc: build - prj_1/tsconfig.json", // 任务名（供 launch.json 引用）
            "problemMatcher": ["$tsc"]                 // 识别 tsc 的错误输出
        },
        {
            // ── prj_2 编译任务 ──
            "type": "shell",
            "command": "npx tsc -p tsconfig.json",
            "options": {
                "cwd": "${workspaceFolder}/prj_2"
            },
            "label": "tsc: build - prj_2/tsconfig.json",
            "problemMatcher": ["$tsc"]
        },
        {
            // ── prj_2 启动 http-server 任务 ──
            "type": "shell",
            "command": "npx http-server ./dist -p 8080 -c-1",  // -c-1 禁用缓存
            "options": {
                "cwd": "${workspaceFolder}/prj_2"
            },
            "label": "http-server: prj_2",
            "isBackground": true,                      // 后台持续运行
            "problemMatcher": {
                "pattern": { "regexp": "^$" },
                "background": {
                    "activeOnStart": true,
                    "beginsPattern": "Starting up http-server",
                    "endsPattern": "Hit CTRL-C to stop the server"  // 检测到此输出则认为服务器就绪
                }
            }
        },
        {
            // ── prj_2 复合任务：先编译，再启动服务器 ──
            "label": "启动 prj_2",
            "dependsOrder": "sequence",                // 按顺序执行
            "dependsOn": [
                "tsc: build - prj_2/tsconfig.json",    // 第一步：编译
                "http-server: prj_2"                   // 第二步：启动服务器
            ],
            "group": {
                "kind": "build",
                "isDefault": true                      // 设为默认构建任务
            }
        }
    ]
}

重要规则：

  • PowerShell 不支持 && 语法，多步骤操作必须用 dependsOn 组合任务，或用 options.cwd 指定工作目录
  • isBackground: true 用于持续运行的任务（如服务器），需配合 problemMatcher.background 告知 VSCode 何时"就绪"

常见问题

Q1：PowerShell 报错 && 不是有效语句分隔符

原因：PowerShell 不支持 && 语法（这是 bash/cmd 的语法）。

解决：在 tasks.json 中使用 options.cwd 指定工作目录，而不是用 cd xxx && command：

// ❌ 错误写法
"command": "cd prj_1 && npx tsc"

// ✅ 正确写法
"command": "npx tsc -p tsconfig.json",
"options": { "cwd": "${workspaceFolder}/prj_1" }

Q2：prj_2 页面显示"加载中..."不更新

原因：通过 file:// 协议直接打开 HTML 文件时，浏览器 CORS 策略阻止 ES6 模块加载。

解决：必须通过 http:// 协议访问，即使用 http-server 托管后再访问 http://localhost:8080。

Q3：F5 总是运行错误的工程

原因：VSCode 记住了上次选择的调试配置。

解决：按 Ctrl+Shift+D 打开调试面板，在顶部下拉框手动切换到目标工程的配置，再按 F5。

Q4：http-server 端口被占用

原因：上次调试结束后 http-server 进程没有关闭。

解决：在终端执行以下命令释放端口：

npx kill-port 8080

Q5：import 路径报错，提示需要 .js 扩展名

原因：tsconfig.json 中 module 设置为 nodenext 时，ESM 规范要求显式写出文件扩展名。

解决：导入时写 .js 扩展名（即使源文件是 .ts，编译后是 .js）：

// ❌ 错误
import { greet } from './utils/helper';

// ✅ 正确
import { greet } from './utils/helper.js';

附录

A. .vscode 目录的作用

.vscode 目录是 VSCode 专属的工作区配置目录，由 VSCode 自动识别和读取，与项目代码本身无关。

typescript/
└── .vscode/           ← VSCode 读取此目录下的所有配置文件
    ├── launch.json    ← 调试配置
    └── tasks.json     ← 任务配置

谁来使用它？

使用者 说明
VSCode 编辑器 打开工作区时自动加载，控制调试、任务、代码风格等行为
团队协作 可以提交到 Git，让团队成员共享同一套调试配置
CI/CD 流程 不会被 Node.js、tsc、浏览器等读取，仅 VSCode 使用

⚠️ .vscode 目录对 Node.js 运行时、TypeScript 编译器、浏览器完全透明，它们不会读取这里的任何内容。

tsconfig.json — TypeScript 编译器配置

官方文档：https://www.typescriptlang.org/tsconfig

官方字段速查：https://aka.ms/tsconfig

{
  "compilerOptions": {
    "rootDir": "./src",
    // 源码根目录。tsc 从此目录开始扫描 .ts 文件。
    // 同时影响 outDir 中的目录结构：src/utils/helper.ts → dist/utils/helper.js

    "outDir": "./dist",
    // 编译输出目录。所有 .ts 编译生成的 .js 文件都放在这里。
    // dist 目录通常加入 .gitignore，不提交到版本控制。

    "target": "es6",
    // 编译目标版本。tsc 将 TypeScript 代码转换为此版本的 JavaScript。
    // 常用值：
    //   "es5"     → 兼容旧浏览器（IE11）
    //   "es6"     → 现代浏览器（Chrome/Firefox/Safari）← prj_2 使用
    //   "esnext"  → 最新 JS 特性 ← prj_1 使用

    "module": "es6",
    // 模块系统格式。编译后的 JS 使用哪种模块语法。
    // 常用值：
    //   "commonjs"  → Node.js 传统模块（require/module.exports）
    //   "es6"       → ES Module（import/export）← 浏览器原生支持，prj_2 使用
    //   "nodenext"  → Node.js ESM 模式 ← prj_1 使用

    "moduleResolution": "bundler",
    // 模块解析策略。tsc 如何查找 import 的模块文件。
    // 常用值：
    //   "node"     → 模拟 Node.js require() 的查找规则
    //   "bundler"  → 模拟 Webpack/Vite 等打包工具的查找规则（允许省略扩展名）
    //   "nodenext" → Node.js ESM 严格模式（必须写 .js 扩展名）

    "sourceMap": true,
    // 生成 Source Map 文件（.js.map）。
    // Source Map 记录了编译后 JS 代码与原始 TS 代码的行号映射关系，
    // 是 VSCode 断点调试能定位到 .ts 文件的关键。

    "strict": true,
    // 开启所有严格类型检查选项（相当于同时开启以下多个选项）：

    //   strictFunctionTypes    → 严格检查函数参数类型
    //   noImplicitAny          → 禁止隐式 any 类型

    "skipLibCheck": true

package-lock.json — 依赖版本锁定文件

官方文档：https://docs.npmjs.com/cli/v10/configuring-npm/package-lock-json

{
  "name": "prj_2",
  "version": "1.0.0",
  "lockfileVersion": 3,
  // lock 文件格式版本（npm v7+ 使用版本 3）

  "packages": {
    "": { ... },
    // 根项目自身的信息

    "node_modules/typescript": {
      "version": "6.0.3",
      // 精确锁定的版本号（不再是 ^6.0.3，而是确切的 6.0.3）

      // 下载地址（npm 仓库 URL）

      // 文件完整性校验哈希（SHA-512），确保下载的文件未被篡改
    }
  }
}

为什么需要 package-lock.json？

package.json 中写的是版本范围（如 ^6.0.3），不同时间 npm install 可能安装不同的小版本。package-lock.json 精确记录了实际安装的每个包的确切版本，确保团队所有成员、CI 服务器安装的依赖完全一致。

✅ 建议：package-lock.json 应提交到 Git 版本控制。

1. ${workspaceFolder} 变量

VSCode 内置变量，代表当前工作区根目录的绝对路径。

2. Source Map（源码映射）工作原理

编译过程：
  src/index.ts  ──tsc──►  dist/index.js
                           dist/index.js.map   ← 记录行号映射关系

调试过程：
  浏览器/Node.js 执行 dist/index.js
        │
        │ 遇到断点或错误
        ▼
  读取 dist/index.js.map
        │
        │ 反向映射
        ▼
  VSCode 定位到 src/index.ts 的对应行

涉及的配置项：

  • tsconfig.json → "sourceMap": true：让 tsc 生成 .map 文件
  • launch.json → "sourceMaps": true：让调试器使用 .map 文件
  • launch.json → "sourceMapPathOverrides"：修正路径映射偏差

3. npx 与 npm 的区别

命令 作用
npm install 下载依赖包到 node_modules
npm run build 执行 package.json 中 scripts.build 定义的命令
npx tsc 直接运行 node_modules/.bin/tsc，无需全局安装 TypeScript
npx http-server 直接运行本地安装的 http-server

npx 的优势：每个项目可以使用自己版本的工具，不同项目之间互不干扰。
